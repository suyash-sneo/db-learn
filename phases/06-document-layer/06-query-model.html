6.6 — The query model & evaluator

Goal: Define what a query is in your engine — a filter that's a predicate over documents — design a small filter representation, and write the evaluator that decides whether a document matches, independent of how the candidates were found.

Why this matters to your DB

A query database is only as useful as the questions it can answer. MongoDB-style queries are filter documents: "{ field: value } matches documents where field equals the specified value," and operators like $gt, $lt, $in express ranges and membership. [src: MongoDB — query-filter syntax (equality and operators)] This lesson turns that idea into a concrete data structure and a matching function — the heart of find.

A query is a predicate

Strip away syntax and a filter is a boolean function over a document: matches(doc) -> bool. "Find users in London with age over 30" is doc.city == "London" AND doc.age > 30. Everything else — indexes, scans, planning — is just how you avoid calling this function on every document. The predicate itself is the ground truth of correctness: a query plan is correct iff it returns exactly the documents the predicate accepts.

A small filter representation

Model the filter as a tree (an enum in Rust): leaf comparisons combined by boolean connectives.

rust — the filter shape (you implement the evaluator in the build) 
// leaf: compare one field against a constant
// enum Cmp { Eq, Ne, Gt, Ge, Lt, Le, In }
// enum Filter {
//   Compare { field: String, op: Cmp, value: Value },
//   And(Vec<Filter>),
//   Or(Vec<Filter>),
//   Not(Box<Filter>),
//   All,            // matches every document (empty filter)
// }

This mirrors MongoDB's model: a leaf is {field: value} or {field: {$op: value}}; an implicit AND joins multiple fields; $and/$or combine sub-filters. [src: MongoDB — filters; equality and $gt/$lt/$in operators] An empty filter (All) returns the whole collection.

The evaluator

Evaluation is a recursive walk of the filter tree against a single document:

  • Compare: resolve field in the document (supporting dot notation for nested fields and array indices, as MongoDB does — e.g. "name.last", "tags.0" [src: MongoDB — dot notation for embedded fields and array elements]); if the field is missing, the comparison is false (unless the op specifically tests for absence). Then apply op using your defined value ordering (6.5).
  • And: all children match. Or: any child matches. Not: the child doesn't match. All: always true.

Crucially, the evaluator takes a document, not a collection — it doesn't know or care whether that document came from a full scan or an index lookup. This separation (predicate vs. candidate-generation) is what lets the planner (6.7) swap strategies without changing correctness.

The simplest correct query: full scan + filter

Before any optimization, find(filter) can always be implemented as: iterate every document in the collection (via the primary index or a heap scan), run matches(doc), and collect the ones that return true. This is O(n) but always correct for any filter — it's your baseline and your test oracle. Every clever plan in 6.7 must return the same set as this.

⚠️ Define comparison semantics before you optimize

The trap in a schemaless store is comparing across types and missing fields. Pin down: How do you order a number vs a string? What does age > 30 mean for a document whose age is the string "old" or is absent? MongoDB has a defined BSON type ordering for exactly this. Choose your rules, encode them in one comparison function, and make both the evaluator and the index use it — otherwise an indexed query and a scanned query can disagree, which is the worst kind of bug.

✅ Concept check

1. Fundamentally, what is a query in this model?

Show answer
A predicate — a boolean function matches(doc) — built from field comparisons combined by AND/OR/NOT.

2. Why does the evaluator take a single document rather than the collection?

Show answer
So matching is independent of how candidates were produced (scan or index); the planner can change strategy without affecting correctness.

3. What is the always-correct baseline implementation of find?

Show answer
Full scan + filter: evaluate matches on every document and collect the true ones. O(n) but correct for any filter; it's the oracle for optimized plans.

4. Why must comparison semantics be defined in one place?

Show answer
In a schemaless store you compare across types and handle missing fields; if the evaluator and the index use different rules, an indexed query and a scanned query can return different results.
