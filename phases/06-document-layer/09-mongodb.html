6.9 — MongoDB data model & CRUD

Goal: Use MongoDB — the canonical document database — as the reference for the external behavior your engine emulates: its document model, the _id contract, size limits, dot notation, and the CRUD + filter surface, with the precise facts sourced.
📂 Where to read

The MongoDB Manual is the authority for the model your engine imitates. [src: Manual — Documents; Manual — CRUD]

The data model, precisely

  • Documents are BSON. "MongoDB stores data records as BSON documents … a binary representation of JSON documents that contains more data types than JSON." Documents are field-and-value pairs; a value can be another document, an array, or an array of documents. [src: Manual — Documents]
  • Field order is preserved (BSON documents are ordered, unlike a JS object), and field names must be unique within a document; names cannot contain the null character. [src: Manual — field names & order]
  • _id: required, unique, the primary key; "always the first field"; auto-generated as an ObjectId if omitted; immutable; may be any BSON type "except arrays, regex, or undefined"; MongoDB creates a unique index on it. [src: Manual — the _id field]
  • Size limit: "The maximum BSON document size is 16 mebibytes (16 MB)" — to bound per-document RAM and bandwidth; larger blobs use GridFS. [src: Manual — document size limit]
  • Dot notation reaches into structure: "name.last" for an embedded field, "contribs.2" for the third (zero-based) array element. [src: Manual — dot notation]

Query filters

Reads, updates, and deletes share one filter syntax:

  • Equality: { field: value } matches documents where field equals value.
  • Operators: { field: { $gt: v } }, with $gt (greater than), $lt (less than), $in (matches any value in an array), and many more. [src: Manual — query-filter syntax]

This is exactly the filter tree of 6.6: leaves are {field: value} or {field: {$op: value}}, multiple fields imply AND, and $and/$or combine sub-filters.

⚠️ MongoDB is the spec to imitate, not a property you inherit

Citing the MongoDB Manual establishes the behavior your engine targets — the document model, the _id contract, the filter semantics. It does not mean your engine has MongoDB's performance, sharding, replication, or isolation. You are matching an interface, not importing an implementation. Keep your claims to what you actually built (e.g. integer _id not ObjectId; single-node not replicated).
