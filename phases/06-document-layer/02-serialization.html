6.2 — Serialization: BSON vs MessagePack vs JSON

Goal: Understand how a document becomes bytes and compare JSON, MessagePack, and BSON without assuming one format wins every workload—then choose a deliberately limited codec for the build.

Why this matters to your DB

A document lives in memory as a tree of fields and values; on disk it's a byte string in a record (Phase 4). The serialization format is the bidirectional rule that converts between the two. It's not a cosmetic choice: it decides how compact your data is, how fast you can read a single field, whether you can tell a string from binary, and whether you can navigate a document without parsing all of it.

What "self-describing" means

This project has no external field schema, so its document bytes need enough tags and framing to reconstruct each value's type and boundaries. MessagePack uses a one-byte format code (sometimes followed by length/type data); BSON places a one-byte type code before each named element. [src: pinned MessagePack specification spec.md] [src: BSON spec — each element is type_code · name · value]

A typed store can instead keep types in schema metadata and encode fields positionally. Isar v3's object builder is an example: it writes schema-known properties at calculated offsets and puts variable-size data in a dynamic region. [src: pinned Isar v3.1.0+1 object_builder.rs] Self-description is required by this schemaless format choice, not by all databases.

JSON — the baseline (text)

  • Human-readable and ubiquitous. Its values are object, array, number, string, boolean, and null; it has no native byte-string type. [src: RFC 8259 §§3–5]
  • Binary needs an application convention. Base64 is common; it emits four encoded characters for each complete three-byte input group, plus padding for a short final group—not a fixed 33% for every tiny value. [src: RFC 4648 §4]
  • Numbers require a compatibility contract. JSON's grammar permits decimal numbers, but interoperable exact integer range depends on the producer and consumer; RFC 8259 notes exact agreement for integers in the IEEE-754 binary64 range [-(2**53)+1, (2**53)-1]. [src: RFC 8259 §6]
  • No container length prefix. A reader generally has to parse structural syntax to find a nested value's end. That can be acceptable for a learning build and debugging, but is a trade-off rather than a universal disqualification.

MessagePack — compact, self-describing binary

MessagePack encodes a JSON-like object model plus binary and extension values using a one-byte format code followed, where needed, by lengths and payload. [src: pinned MessagePack specification] Its first-byte families pack small values into a single byte:

First byteFormat
0x00–0x7fpositive fixint (small ints in one byte)
0x80–0x8ffixmap (maps up to 15 entries)
0x90–0x9ffixarray (arrays up to 15 elements)
0xa0–0xbffixstr (strings up to 31 bytes)
0xc0, 0xc2, 0xc3nil, false, true
0xc4–0xdbvariable-length bin / float / int / str / array / map
0xe0–0xffnegative fixint

[src: pinned MessagePack specification — format table]

Unlike JSON it distinguishes binary (bin) from text (str) and "encodes type information in the binary format itself," with serializers told they "SHOULD use the format which represents the data in the smallest number of bytes." [src: pinned MessagePack specification — bin/str and serialization rules] The cost: it's compact but not random-access — to reach the 5th field you generally walk the preceding ones (their length prefixes let you skip quickly, but it's still sequential).

BSON — the database-shaped binary format

BSON ("Binary JSON") is what MongoDB stores. A BSON document is int32 · e_list · 0x00: a 4-byte little-endian total length, then a list of elements, then a terminating zero byte. [src: BSON spec — document = int32 length, element list, trailing 0x00] Each element is a one-byte type code, the field name as a C-string, then the value. Key type codes:

CodeType
0x0164-bit binary floating point (double)
0x02UTF-8 string
0x03embedded document
0x04array
0x05binary data
0x07ObjectId
0x08boolean
0x09UTC datetime (int64 ms since Unix epoch)
0x0Anull
0x1032-bit integer
0x1264-bit integer

[src: BSON spec — element type codes]

A BSON string is encoded as an int32 giving "the number of bytes … plus one for the trailing null byte," then the UTF-8 bytes, then 0x00. [src: BSON spec — string framing] BSON length-prefixes documents and strings; BSON arrays use the same embedded-document representation with numeric field names. A decoder can skip many values after reading their type-specific framing, but BSON has no field-offset table: finding a field by name still requires walking preceding elements. Length prefixes improve skipping and validation; they do not create true random field access.

Format choice depends on the supported value types, compatibility requirements, update/read patterns, and measured data. For this build, use UTF-8 JSON through serde_json::Value so bytes are inspectable while the lesson focuses on storage, indexes, and queries. That deliberately omits a native binary value and BSON-specific types. A later format change needs a file-format version and migration plan; it is not a transparent codec swap.

🦀 Rust just-in-time

serde separates the data model from the format: the same #[derive(Serialize, Deserialize)] type can be encoded as JSON (serde_json) or MessagePack (rmp_serde) by swapping the codec. A dynamic document can be modeled as serde_json::Value (an enum of null/bool/number/string/array/object). See the serde guide and the Rust Book ch.6 (enums).

✅ Concept check

1. Why does this engine use a self-describing format?

Show answer
This build has no external field schema, so the bytes must carry enough names, types, and boundaries to reconstruct the document (or refer to an equivalent dictionary). MessagePack and BSON demonstrate binary forms of that approach; JSON is the simpler format chosen for the exercise.

2. Name two concrete weaknesses of JSON as an on-disk database format.

Show answer
No binary type (blobs become bloated base64 or another convention) and no container-length prefix, so structure must be parsed to locate an arbitrary nested value. Number interoperability also requires a documented numeric range.

3. What does BSON's length-prefixing of documents/arrays/strings buy you, and at what cost?

Show answer
Traversability — you can skip a whole sub-document/string once its start is known, and validate boundaries. It adds framing bytes, but does not provide a field-offset table or O(1) lookup by field name.

4. Which BSON byte starts a UTF-8 string element, and how is the string framed?

Show answer
Type code 0x02; the string is an int32 (byte count including the trailing null) then the UTF-8 bytes then 0x00.
