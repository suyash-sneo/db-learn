6.2 — Serialization: BSON vs MessagePack vs JSON

Goal: Understand how a document becomes bytes, why a self-describing binary format beats text for a database, and the concrete trade-offs between JSON, MessagePack, and BSON — so you can pick one and know what you gave up.

Why this matters to your DB

A document lives in memory as a tree of fields and values; on disk it's a byte string in a record (Phase 4). The serialization format is the bidirectional rule that converts between the two. It's not a cosmetic choice: it decides how compact your data is, how fast you can read a single field, whether you can tell a string from binary, and whether you can navigate a document without parsing all of it.

What "self-describing" means and why a DB needs it

Because your store is schemaless (6.1), the bytes themselves must say what each value is — there's no external schema to consult. A self-describing format tags every value with its type inline. MessagePack is described exactly this way: it "uses a type-prefixed byte format where the first byte determines the data type … This makes it self-describing — deserializers can determine type information from the binary stream itself." [src: MessagePack spec — format overview] BSON does the same with a one-byte type code before each element. [src: BSON spec — each element is type_code · name · value]

JSON — the baseline (text)

  • Human-readable, ubiquitous, and self-describing in a loose sense (you can see the structure).
  • But: numbers are decimal text (slow to parse, lossy for large integers); there's no binary type (blobs must be base64-encoded, inflating size ~33%); no length prefixes, so to skip a value you must parse it; every key is repeated as text.
  • Verdict: great as an interchange/debug format, poor as an on-disk database format. Many engines accept JSON at the edge and store a binary form internally.

MessagePack — compact, self-describing binary

MessagePack is "an object serialization specification like JSON" that encodes the same data model in far fewer bytes by tagging each value with a one-byte format code. [src: MessagePack spec] Its first-byte families pack small values into a single byte:

First byteFormat
0x00–0x7fpositive fixint (small ints in one byte)
0x80–0x8ffixmap (maps up to 15 entries)
0x90–0x9ffixarray (arrays up to 15 elements)
0xa0–0xbffixstr (strings up to 31 bytes)
0xc0, 0xc2, 0xc3nil, false, true
0xc4–0xdbvariable-length bin / float / int / str / array / map
0xe0–0xffnegative fixint

[src: MessagePack spec — first-byte format families]

Unlike JSON it distinguishes binary (bin) from text (str) and "encodes type information in the binary format itself," with serializers told they "SHOULD use the format which represents the data in the smallest number of bytes." [src: MessagePack spec — bin vs str, smallest-format rule] The cost: it's compact but not random-access — to reach the 5th field you generally walk the preceding ones (their length prefixes let you skip quickly, but it's still sequential).

BSON — the database-shaped binary format

BSON ("Binary JSON") is what MongoDB stores. A BSON document is int32 · e_list · 0x00: a 4-byte little-endian total length, then a list of elements, then a terminating zero byte. [src: BSON spec — document = int32 length, element list, trailing 0x00] Each element is a one-byte type code, the field name as a C-string, then the value. Key type codes:

CodeType
0x0164-bit binary floating point (double)
0x02UTF-8 string
0x03embedded document
0x04array
0x05binary data
0x07ObjectId
0x08boolean
0x09UTC datetime (int64 ms since Unix epoch)
0x0Anull
0x1032-bit integer
0x1264-bit integer

[src: BSON spec — element type codes]

A BSON string is encoded as an int32 giving "the number of bytes … plus one for the trailing null byte," then the UTF-8 bytes, then 0x00. [src: BSON spec — string framing] The defining trait: BSON length-prefixes documents, arrays, and strings. That makes it traversable — you can skip an entire embedded document or string by reading its length and jumping, without parsing the contents. That's exactly what a database wants for fast field access and partial reads. The price is size: BSON is often larger than MessagePack (and sometimes larger than the equivalent JSON for small values) because of those length prefixes and richer type tags.

For a document database, a self-describing binary format is the right call; BSON's traversability is the feature that justifies its extra bytes. For your build the learning is in the storage/index/query layers, not in hand-rolling a serializer, so you'll lean on Rust's serde (Phase 0.9) with a self-describing format and treat the encoded bytes as the document payload. Picking MessagePack (via a serde codec) keeps documents compact; picking JSON keeps them debuggable. Either is fine for the build as long as it round-trips losslessly — just record which one and why.

🦀 Rust just-in-time

serde separates the data model from the format: the same #[derive(Serialize, Deserialize)] type can be encoded as JSON (serde_json) or MessagePack (rmp_serde) by swapping the codec. A dynamic document can be modeled as serde_json::Value (an enum of null/bool/number/string/array/object). See the serde guide and the Rust Book ch.6 (enums).

✅ Concept check

1. Why must a schemaless store use a self-describing format?

Show answer
There's no external schema, so the bytes must carry each value's type inline; both MessagePack and BSON tag every value with a type code.

2. Name two concrete weaknesses of JSON as an on-disk database format.

Show answer
No binary type (blobs become bloated base64) and no length prefixes (you must parse a value to skip it); also numbers are decimal text and keys repeat as text.

3. What does BSON's length-prefixing of documents/arrays/strings buy you, and at what cost?

Show answer
Traversability — you can skip a whole sub-document/string by reading its length and jumping, enabling fast field access — at the cost of extra bytes (BSON is often larger than MessagePack).

4. Which BSON byte starts a UTF-8 string element, and how is the string framed?

Show answer
Type code 0x02; the string is an int32 (byte count including the trailing null) then the UTF-8 bytes then 0x00.
