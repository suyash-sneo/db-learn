6.3 — Storing documents

Goal: Assemble a collection from machinery you already built — serialize a document to bytes (6.2), store those bytes as a record (Phase 4), and key them by _id through a primary index (Phase 3) — and understand the two-step lookup that results.

Why this matters to your DB

This is the lesson where the whole stack clicks together. A collection isn't a new storage engine; it's a thin composition: the record store gives you "put these bytes somewhere, get a stable handle back," the B+tree gives you "map a key to a value in sorted order," and the serializer turns documents into bytes. Storing a document is just chaining those three.

The two structures behind a collection

  • The heap (record store, Phase 4): holds the serialized document bytes. Inserting returns a RecordId (page + slot) — a physical handle to where the bytes live. It knows nothing about _id or fields.
  • The primary index (B+tree, Phase 3): maps the logical key _id → the physical RecordId. A balanced B+tree lookup visits logarithmically many tree levels; fetching the record is the second step. Range scans also benefit from ordered keys.

This separation — logical key in an index, physical bytes in a heap — is the classic indirection: the document can move (compaction relocates its bytes, an update may rewrite it elsewhere) and only the primary index entry changes; references by _id stay valid.

Insert, step by step

  1. If the document has no _id, assign one (e.g. an auto-incrementing counter persisted in collection metadata. Updating that counter and inserting the document must be one transaction. Isar v3 is a reference for the policy: it assigns an auto-increment id when requested and does not reuse deleted ids. [src: pinned Isar v3.1.0+1 docs/docs/schema.md]
  2. Serialize the document to bytes (6.2).
  3. Store the bytes in the record store → get a RecordId (Phase 4).
  4. Insert _id → RecordId into the primary index (Phase 3). If _id already exists, that's a duplicate-key error (the primary index is unique).
  5. Do all of this inside one transaction so a crash can't leave the heap and index disagreeing (Phase 5) — see the warning below.

An explicit id must also advance the future auto-id counter when necessary. Use checked arithmetic: if no larger i64 id can be assigned, report exhaustion rather than wrapping into an old key.

Read, step by step

  1. Look up _id in the primary index → RecordId (or "not found").
  2. Read the bytes at that RecordId from the record store.
  3. Deserialize back into a document.
⚠️ The heap and the index must stay consistent — that's a transaction

An insert touches two structures (heap + index); an update or delete may touch three (heap + primary index + secondary indexes, 6.5). If a crash lands between them you get an orphaned record (bytes with no index entry) or a dangling pointer (index entry to freed bytes). This is exactly why Phase 5 exists: wrap the whole multi-structure mutation in one transaction so it commits atomically or not at all. The document layer must never write the heap and index in separate, independently-durable steps.

Where secondary indexes fit (preview)

The primary index answers "by _id." Querying by any other field (6.5) needs another mapping: field value → the _id(s) of documents that have it. Note it stores _id, not RecordId — so when a document's bytes relocate, only the primary index entry updates and the secondary indexes (which point at the stable _id) don't have to change. That's a deliberate consequence of the indirection above.

Updating a document's size

Documents are mutable and updates change their size. Two strategies, both built on Phase 4:

  • Update in place if the new bytes fit the existing cell (cheapest).
  • Relocate if they don't: write the new bytes (possibly a new page / overflow chain), repoint the primary index entry to the new RecordId, and free the old cell. The _id is unchanged, so secondary indexes and external references survive.
✅ Concept check

1. What are the two structures that make up a collection, and what does each store?

Show answer
A primary index (B+tree) mapping _id → RecordId, and a record-store heap mapping RecordId → the serialized document bytes.

2. Why store _id (not RecordId) in secondary indexes?

Show answer
_id is stable; the document's physical location can change (relocation/compaction). Pointing secondary indexes at _id means only the primary index updates when bytes move.

3. Why must an insert run inside a transaction?

Show answer
It mutates two structures (heap + primary index); a crash between them leaves an orphan record or a dangling index entry. A transaction makes the whole change atomic.

4. What happens to references when an update grows a document beyond its cell?

Show answer
The bytes relocate to a new RecordId and the primary index entry is repointed; the _id is unchanged, so secondary indexes and external references by _id stay valid.
