6.8 — Source dive: Isar

Goal: Study Isar — an embedded, schema-annotated database for Flutter/Dart whose core is written in Rust and exposed over an FFI — because it is the closest existing project to what you're building, and extract the design decisions worth borrowing (while being scrupulous about which version you're describing).

⚠️ Version discipline (this is a hard rule for Isar)

Isar has materially different generations: the documented stable v3 and a later v4 rewrite. They differ in storage engine and internals. The published docs at isar.dev describe v3; the homepage itself does not print a version number ⚠️VERIFY the exact version of any page/commit you cite. Do not blend v3, v4, or the underlying storage engine into one description. Treat the docs-confirmed data model facts below as solid; treat any internal architecture claim as ⚠️VERIFY until you read it in the repo at a known commit.

Design decisions worth borrowing

  • Compile-time schema from annotations + codegen. Isar uses a generator to turn an annotated Dart class into (de)serialization and query boilerplate, so queries are statically typed. Your engine is schemaless at the storage layer, but you can still offer a typed convenience layer at the edge.
  • Rust core behind an FFI, Dart as the consumer. This is precisely your Phase 1 / Phase 8 plan: the heavy logic in Rust, exposed over a C ABI, called from Dart. Isar is proof the shape works in production.
  • Indexes as first-class, declared per field, with explicit unique/composite/ multi-entry variants — the same taxonomy as 6.5.
  • Integer primary keys with auto-increment and no id reuse — a clean, B+tree-friendly choice you can copy verbatim.

How Isar differs from your engine (so you don't over-copy)

  • Isar's public model is typed (a class per collection); yours is schemaless (arbitrary documents). Don't import Isar's codegen assumption into your storage layer.
  • Isar targets Dart/Flutter specifically; your C ABI (Phase 8) aims to be consumable beyond Dart.
  • Isar's exact durability/concurrency guarantees come from its implementation and storage engine — your engine only has the guarantees you built (Phase 5), not whatever Isar provides. Don't claim inherited properties.
