6.8 — Source dive: Isar v3.1.0+1

Goal: Read one pinned Isar generation without blending it with v4: identify its public typed-document model, MDBX-backed storage, schema-driven object bytes, secondary-index representation, and Rust/C/Dart boundary—then separate reusable ideas from implementation details your engine does not inherit.
📂 Exact source under discussion

This lesson describes tag 3.1.0+1, commit 6643d064abf22606b6c6a741ea873e4781115ef4. Every internal link is pinned to that commit. The repository's current README warns that Isar v4 is not production-ready and directs production users to stable v3; that current status is separate from the pinned v3 implementation below. [src: Isar v3.1.0+1 source; current Isar README, checked June 21, 2026]

Public model: typed collections, ids, indexes, transactions

Isar v3 is not schemaless in the same way as this curriculum's document store. A collection maps to one generated Dart object type. Its schema names properties and their types; an Id property uniquely identifies each object. Id is a Dart int alias recognized by the generator. Auto-increment ids are available and deleted ids are not reused until the database is cleared. [src: pinned v3 docs/docs/schema.md; type.dart]

Its documented indexes include unique, composite, and multi-entry forms. Its transaction guide says read and write access uses transactions and that errors roll transactions back. [src: pinned v3 docs/docs/indexes.md; docs/docs/transactions.md]

Storage engine: Isar v3 wraps libmdbx

The v3 Rust core depends on a local crate named mdbx-sys. That crate's build script obtains Isar's libmdbx fork and builds/binds the C library. The core's environment, transaction, cursor, and database wrappers call MDBX through those generated FFI bindings. Therefore, v3's page management, B+tree, transactions, locking, and crash behavior are largely properties of MDBX—not a from-scratch Isar pager. [src: pinned v3 isar_core/Cargo.toml; mdbx_sys/build.rs; isar_core/src/mdbx/]

Do not attribute MDBX internals to Isar's own storage design

It is correct to say “Isar v3 uses MDBX through a Rust wrapper.” It is not correct to inspect an MDBX meta-page, copy-on-write rule, or lock and then present it as a from-scratch Isar algorithm. Cite the layer that actually implements the behavior.

Object bytes: schema-driven, not BSON or MessagePack

Because v3 knows each collection schema, object bytes do not repeat field names and type tags like BSON. ObjectBuilder computes a fixed “static properties” region from property offsets. Fixed-size values are written at those offsets. Variable-size values store three-byte little-endian offsets into a dynamic region; the dynamic value begins with a three-byte length. The object begins with a two-byte little-endian static-region size. [src: pinned v3 object_builder.rs; matching reads in isar_object.rs]

This is the counterexample to “all document databases need self-describing object bytes.” Isar v3 stores document-like objects, but schema metadata supplies property names, types, and offsets. Your schemaless engine cannot use the same positional layout unless it also stores a per-document or shared schema dictionary.

Secondary indexes: a separate MDBX database

For each collection index, v3 opens a named MDBX database. Unique indexes use one value per key. Non-unique indexes enable MDBX duplicate sorting (MDBX_DUPSORT), so one encoded index key can have multiple sorted id values. Index maintenance writes the object's id as the index value; deletion locates the key/id pair. Multi-entry indexes generate multiple keys for one object. [src: pinned v3 schema_manager.rs::open_index_db; mdbx/db.rs::Db::open; index/mod.rs::create_for_object]

Your build's BTreeMap<Key, Vec<id>> is conceptually closer to a posting list. Isar v3's non-unique representation is closer to “one index key, duplicate-sorted id values.” Both represent the same logical relation, but their update and scan mechanics differ.

The boundary: exported C functions plus Dart integration

The isar_core_ffi crate exports C-ABI functions. For example, isar_instance_create receives C strings, a schema-JSON string, configuration values, and an out-pointer for an IsarInstance; successful creation stores an Arc-backed opaque pointer. isar_txn_begin creates an opaque transaction and isar_txn_finish consumes it to commit or abort. Async operations use Dart native ports to post completion codes. [src: pinned v3 instance.rs; txn.rs; dart.rs]

The pinned code predates Rust edition 2024 and uses the older #[no_mangle] spelling. Learn the ownership pattern, not the obsolete syntax: this curriculum's new exports use #[unsafe(no_mangle)] and must document every pointer's lifetime and deallocator.

What to borrow—and what not to borrow

  • Borrow: explicit collection metadata, generated typed APIs above a storage core, ids as stable logical references, transactional index maintenance, and opaque FFI handles.
  • Do not borrow blindly: v3's positional object layout assumes a known schema; this project's arbitrary documents do not. Its MDBX guarantees also do not transfer to your WAL.

✅ Concept check

1. Which storage engine implements Isar v3's low-level B+tree and transactions?

Show answer
libmdbx, reached through the local mdbx-sys crate and Rust wrappers in isar_core/src/mdbx.

2. Why can v3 omit field names and per-value type tags from each object?

Show answer
Its generated collection schema already defines properties, types, and offsets. Fixed values are positional and variable values are reached through offsets into a dynamic region.

3. How does a non-unique v3 index represent several objects with one key?

Show answer
Its separate MDBX index database enables duplicate-sorted values; the encoded field key can have multiple object-id values.

4. Why must the v3 FFI source not be copied verbatim into new Rust 2024 code?

Show answer
It uses older export syntax and its raw pointer contracts must be re-audited. Rust 2024 requires #[unsafe(no_mangle)] for this curriculum, with explicit ownership, lifetime, panic, and deallocation rules.
