6.1 — From records to documents

Goal: Define what a "document" is in your engine, how it differs from a relational row, why schemaless changes what the storage layer must do, and how documents group into collections keyed by a primary _id.

Why this matters to your DB

You're building a MongoDB-like, schemaless, embedded database. Until now the engine stored opaque variable-length byte records (Phase 4). A "document" is the meaning your engine assigns to those bytes: a nested map of fields to values, encoded in this phase as self-describing JSON. Deciding precisely what a document is — and what is and isn't enforced about its shape — determines everything above it: serialization, indexing, and queries.

Row vs document

In the conventional relational model, a row is interpreted through a table schema: declared columns and their constraints live in catalog metadata rather than being repeated in every row. Particular engines vary in how strictly they enforce column types, but the shared schema still supplies the row's field names and positions.

A document is logically a tree of field/value pairs. In MongoDB, BSON stores those pairs with field names and value types in each document, and values can include nested documents and arrays. Consequently, documents in one collection can have different shapes. A typed document store can represent the same logical tree with help from external schema metadata, so “document” does not by itself require one physical encoding. [src: MongoDB Manual — Documents]

What "schemaless" really means

"Schemaless" does not mean "no structure." It means this project does not require one fixed field layout for every document in a collection. A document still has a precise logical tree and encoded byte representation. Other document databases may offer optional schema validation; the no-validator rule below is a deliberate choice for this engine.

  • Field names travel with this project's encoded document. A separate per-type schema could support positional encoding—as Isar v3 does—but arbitrary documents need their keys or an equivalent dictionary in the encoded representation.
  • Documents are variable-length and irregular. Exactly the case Phase 4's slotted pages and overflow chains were built for. A document is a variable-length record.
  • No collection-wide declared field types. This project therefore chooses a self-describing value encoding. That is a design consequence here, not a law of every document database: a typed document store can use a schema-driven binary layout.
  • The engine can't assume any field exists except the primary key. Indexing and queries must tolerate missing fields.

The one field the engine does require: _id

This engine needs a stable, unique handle for each document so it can be addressed, updated, and referenced by indexes. That handle is the primary key. MongoDB requires a unique _id primary key in a standard collection, places that field first, creates a unique _id index, and has the driver generate an identifier when an inserted document omits one. [src: MongoDB Manual — Documents: the _id field]

Your engine adopts the same contract: every document has an _id; if the caller doesn't supply one, the engine assigns it; the _id uniquely identifies the document within its collection and is the key of the primary index (Phase 3 B+tree) that maps _id → the document's storage location (a Phase 4 RecordId).

⚠️ Design choice, not a universal law

MongoDB allows an _id of most BSON types and its drivers commonly generate an ObjectId when the field is absent. Your build will simplify to an integer _id (like Isar's Id, an alias for int [src: pinned Isar v3.1.0+1 docs/docs/schema.md]), because integer keys are trivial to order in a B+tree and auto-increment. State this assumption in your code; it's a simplification, not a requirement of the document model.

Collections

Documents are grouped into collections. MongoDB uses collections as namespaces for documents whose shapes may differ. Isar v3 uses a stricter typed model: a collection corresponds to one Dart object type. [src: MongoDB — databases and collections; pinned Isar v3.1.0+1 docs/docs/schema.md] In your engine a collection is the pairing of (a) a record store holding the document bytes and (b) a primary index mapping _id → RecordId, plus any secondary indexes (6.5). Each collection is independent: its own key space, its own indexes.

The shape of the layer you're about to build

  • This build serializes a document to self-describing JSON bytes → 6.2.
  • Those bytes are stored as a record; _id → location lives in the primary index → 6.3.
  • Insert/find/update/delete operate through that mapping → 6.4.
  • Indexing other fields needs a second mapping value → _id → 6.5.
  • A query is a filter evaluated against documents → 6.6, and choosing index-vs-scan is planning → 6.7.
✅ Concept check

1. What is the key structural difference between a relational row and a document?

Show answer
A row stores values positionally against shared schema metadata; this build's JSON document carries field names and value syntax inline, so documents in one collection can differ in shape. Other document stores may use a declared schema and positional encoding.

2. Does "schemaless" mean documents have no structure? What does it actually mean?

Show answer
No — each document has a precise byte layout. Here it means the collection has no mandatory fixed field set or field-type declaration; documents can differ in shape. Optional validation or a typed convenience layer could still be added above storage.

3. Which single field does the engine require, and what role does it play?

Show answer
The _id primary key: unique within the collection, auto-assigned if omitted, and the key of the primary index that maps to the document's storage location.

4. In your engine, what two things make up a collection?

Show answer
A record store holding the document bytes, and a primary index mapping _id → RecordId (plus optional secondary indexes).
