6.1 — From records to documents

Goal: Define what a "document" is in your engine, how it differs from a relational row, why schemaless changes what the storage layer must do, and how documents group into collections keyed by a primary _id.

Why this matters to your DB

You're building a MongoDB-like, schemaless, embedded database. Until now the engine stored opaque variable-length byte records (Phase 4). A "document" is the meaning your engine assigns to those bytes: a self-describing, nested map of fields to values. Deciding precisely what a document is — and what is and isn't enforced about its shape — determines everything above it: serialization, indexing, queries.

Row vs document

A relational row conforms to a table schema: a fixed, declared list of typed columns. The schema lives in the catalog, separate from the data; every row in the table shares it, so a row can be stored as just its values in column order.

A document is a self-describing tree of field/value pairs. Each document carries its own field names and types inline, so two documents in the same collection can have entirely different shapes. MongoDB's model is exactly this: "MongoDB stores data records as BSON documents … composed of field-and-value pairs" where "the value of any field can be any BSON data type, including other documents, arrays, and arrays of documents." [src: MongoDB Manual — Documents]

What "schemaless" really means

"Schemaless" does not mean "no structure" — every document is a precise byte string with a defined layout. It means the structure is per-document and not enforced by the engine. Consequences for the storage layer:

  • Field names travel with the data. You can't store values positionally; each document repeats its keys. (This is why binary document formats and field-name reuse matter — Phase 6.2.)
  • Documents are variable-length and irregular. Exactly the case Phase 4's slotted pages and overflow chains were built for. A document is a variable-length record.
  • No declared types to validate against. Type information is encoded inside each value, so the serializer must be self-describing (it records each value's type), not schema-driven.
  • The engine can't assume any field exists except the primary key. Indexing and queries must tolerate missing fields.

The one field the engine does require: _id

Even a schemaless store needs a stable, unique handle for each document so it can be addressed, updated, and referenced by indexes. That handle is the primary key. Following MongoDB: "each document stored in a standard collection requires a unique _id field that acts as a primary key," it "is always the first field," and "if an inserted document omits the _id field, the MongoDB driver automatically generates" one; MongoDB "creates a unique index on the _id field." [src: MongoDB Manual — Documents: the _id field]

Your engine adopts the same contract: every document has an _id; if the caller doesn't supply one, the engine assigns it; the _id uniquely identifies the document within its collection and is the key of the primary index (Phase 3 B+tree) that maps _id → the document's storage location (a Phase 4 RecordId).

⚠️ Design choice, not a universal law

MongoDB allows an _id of almost any BSON type (it auto-generates a 12-byte ObjectId when absent). Your build will simplify to an integer _id (like Isar's Id, an alias for int [src: Isar — the Id field is "an alias for int"]), because integer keys are trivial to order in a B+tree and auto-increment. State this assumption in your code; it's a simplification, not a requirement of the document model.

Collections

Documents are grouped into collections — "like a database table … can only contain a single type of Dart object" in Isar's words, or a namespace of related documents in MongoDB. [src: Isar — "A collection is like a database table"] In your engine a collection is the pairing of (a) a record store holding the document bytes and (b) a primary index mapping _id → RecordId, plus any secondary indexes (6.5). Each collection is independent: its own key space, its own indexes.

The shape of the layer you're about to build

  • A document is serialized to bytes by a self-describing format → 6.2.
  • Those bytes are stored as a record; _id → location lives in the primary index → 6.3.
  • Insert/find/update/delete operate through that mapping → 6.4.
  • Indexing other fields needs a second mapping value → _id → 6.5.
  • A query is a filter evaluated against documents → 6.6, and choosing index-vs-scan is planning → 6.7.
✅ Concept check

1. What is the key structural difference between a relational row and a document?

Show answer
A row stores values positionally against an external shared schema; a document is self-describing — it carries its own field names and types inline, so documents in one collection can differ in shape.

2. Does "schemaless" mean documents have no structure? What does it actually mean?

Show answer
No — each document has a precise byte layout. It means structure is per-document and not enforced by the engine; types are encoded inside each value rather than declared once.

3. Which single field does the engine require, and what role does it play?

Show answer
The _id primary key: unique within the collection, auto-assigned if omitted, and the key of the primary index that maps to the document's storage location.

4. In your engine, what two things make up a collection?

Show answer
A record store holding the document bytes, and a primary index mapping _id → RecordId (plus optional secondary indexes).
