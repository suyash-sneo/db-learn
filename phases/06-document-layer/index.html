Phase 6

The document layer

Phases 2–5 built a durable byte store: pages, an ordered index, variable-length records, and crash-safe transactions. Everything so far has been content-agnostic — it stores opaque bytes. Phase 6 gives those bytes meaning. You'll serialize schemaless documents, store them in collections keyed by _id, run CRUD through the engine, add secondary indexes on fields, and evaluate filter queries — with a first taste of query planning (use an index or scan?). This is the layer that turns "a storage engine" into "a document database."

Why this comes after durability

A document is just a variable-length record (Phase 4) whose bytes happen to be a serialized object, written inside a transaction (Phase 5) and looked up through an index (Phase 3). The document layer is not a new storage mechanism — it's a meaning layer assembled from the primitives you already built. Doing it last means each new concept (serialization, secondary indexes, query evaluation) rests on machinery you fully understand.

What you'll be able to do after Phase 6

Compare document encodings, serialize the build's JSON subset to bytes and back, store it in a collection keyed by a primary _id index, perform insert/find/update/delete, build a secondary index that maps a field's values to documents, evaluate a filter query against documents, and choose between an index lookup and a full scan. The build keeps indexes derived/in-memory until the Phase 3 and Phase 5 layers are integrated behind one transactional API.
