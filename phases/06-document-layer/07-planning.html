6.7 — Query planning basics

Goal: Understand that planning is choosing how to produce candidate documents for the evaluator — most importantly, when to use a secondary index versus a full scan — and the simple cost intuition (selectivity) that drives that choice.

Why this matters to your DB

6.6 gave you a correct-but-slow baseline: scan everything, filter. The whole point of building secondary indexes (6.5) was to avoid touching documents that can't match. The query planner is the component that decides, for a given filter, which access path to use. A good plan can be thousands of times faster than a scan; a bad plan can be slower than a scan. And — the non-negotiable — every plan must return the same documents the predicate accepts.

Plan = access path + residual filter

A plan has two parts:

  • Access path: how to generate candidate documents. Either a full scan (every document) or an index scan (use a secondary index to fetch only documents in a key range / equal to a value).
  • Residual filter: the part of the predicate the access path didn't already enforce, applied by the evaluator (6.6) to each candidate.

Example: filter city == "London" AND age > 30 with an index on city. Plan: index-scan city = "London" to get candidate _ids; for each, fetch the document and apply the residual age > 30. The index did the selective part; the evaluator finishes the rest. Correctness is preserved because the residual is still the full predicate minus what the access path guaranteed.

Which access path? Selectivity

Selectivity is the fraction of documents a condition keeps. A condition matching 5 of 1,000,000 documents is highly selective (great for an index); one matching 800,000 is not (an index would do a huge number of random primary-index lookups — often slower than a straight scan). The core heuristic:

  • If a filter condition is on an indexed field and is selective, use that index and apply the rest as a residual.
  • If no useful index exists, or the most selective condition still matches most of the collection, full scan.
  • With several usable indexes, prefer the one expected to be most selective (fewest candidates).

Real planners estimate selectivity from statistics (histograms, distinct-value counts) and compare estimated costs. That cost-based machinery is large; for your engine a simple rule-based planner is enough and is honestly labeled as such.

A workable rule-based planner for your engine

  1. Flatten the top-level AND into a list of conditions.
  2. Find conditions whose field has an index and whose op is index-usable (Eq, range, In).
  3. If any exist, pick one (e.g. an equality over a range; if you have stats, the most selective), use its index to produce candidate _ids, and apply the remaining conditions as a residual filter.
  4. Otherwise, full scan with the whole filter as residual.

OR across different fields is harder (you may need to union index results or fall back to a scan); it's fine for a first engine to scan on top-level OR and optimize only AND-of-conditions — just say so.

⚠️ A plan that changes results is a bug, not an optimization

The single rule that must never break: the planned query returns exactly the same set as the full-scan baseline (6.6). Two classic ways to violate it: (1) using an index that omits documents missing the field when the query should include them; (2) forgetting the residual filter so you return everything in the key range, including documents the rest of the predicate rejects. Test every plan against the scan oracle on the same data.

Honest scope

This is "planning basics." You are not building a cost-based optimizer with join ordering, statistics, and plan caching. You're building the one decision that matters most for a single-collection document store: index this condition, or scan — with everything else handled as a residual filter, verified against the scan.

✅ Concept check

1. What two parts make up a query plan?

Show answer
An access path (full scan or index scan that generates candidates) and a residual filter (the rest of the predicate applied by the evaluator to each candidate).

2. What is selectivity, and how does it drive the choice of access path?

Show answer
The fraction of documents a condition keeps. Highly selective + indexed → use the index; low selectivity or no index → full scan, since a non-selective index does many lookups for little gain.

3. Why is a residual filter necessary after an index scan?

Show answer
The index only enforced one condition; the remaining conditions of the predicate must still be checked on each candidate to keep the result correct.

4. What invariant must every plan satisfy?

Show answer
It returns exactly the same document set as the full-scan baseline for the same filter and data — same correctness, different cost.
