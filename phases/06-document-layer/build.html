Build: a document store with CRUD, a secondary index & a filter planner

Goal: Compose your record store (Phase 4) into a collection of schemaless documents keyed by _id, support insert/get/replace/delete, build a secondary index on a field, evaluate filter queries, and choose between an index lookup and a full scan — verified against a scan oracle.
⚠️ AI off

The document layer — id assignment, index maintenance, the query evaluator, and the planner — is core database logic. No worked implementation here: only contracts, stubs, failing tests, and graduated hints. Cite the BSON/MessagePack specs and the MongoDB Manual for behavior; write the code yourself.

Deliverable

A Rust 2024 module docstore built on your Phase 4 recstore: a DocStore holding documents (modeled as serde_json::Value objects), each with an integer _id. It supports insert, get, replace, delete, create_index, and find(&Filter) — with find using a secondary index when one applies and a full scan otherwise, always returning the same set as the scan baseline.

Fixed design decisions

  • Document model: serde_json::Value that is a JSON object at the top level. (You may swap the on-disk codec to MessagePack via rmp_serde; JSON keeps the build debuggable. Record your choice.)
  • _id: an i64 stored in the document under the key "_id". If absent on insert, assign next_id (a monotonically increasing counter; never reuse a deleted id). If present, use it and reject a duplicate.
  • Heap: serialized document bytes live in RecStore; insert returns a RecordId.
  • Primary index: _id → RecordId, kept as an in-memory BTreeMap<i64, RecordId>. Simplification: rebuild it by scanning the heap on open; persisting it as the Phase 3 B+tree is a labeled stretch goal. The heap (in the pager file) is the source of truth.
  • Secondary index: field → (key → list of _id) as BTreeMap<Key, Vec<i64>>, also rebuilt on open. Keys are ordered by one total order (the Key enum below).
  • Indexable key types: only Null, Bool, integer Number, and String are indexable; floats, arrays, and objects yield no index entry (document this). This sidesteps float ordering and keeps one comparison rule for both the index and the evaluator.
  • Atomicity: in this build the in-memory indexes plus the heap write happen together per operation; in a real engine all of it is one Phase 5 transaction. Note the gap.

Ordered milestones

  1. M1 — Collection & primary index. create/open; insert (assign/validate _id, serialize, store, index); get by id; open rebuilds the primary index from a heap scan.
  2. M2 — Replace & delete. Maintain the heap + primary index (and later secondary indexes) on update/delete; relocation repoints the primary index; delete frees the record.
  3. M3 — The evaluator. key_of, dot-notation field access, and matches(&Filter, &Value) -> bool over Eq/Ne/Gt/Ge/Lt/Le/In + And/Or/Not/All.
  4. M4 — Full-scan find. find = iterate all docs, keep those that match. This is your oracle.
  5. M5 — Secondary index + planner. create_index(field) (one-time scan + incremental maintenance in insert/replace/delete); find uses an index for a selective indexed condition and applies the rest as a residual — returning the same set as M4.

Acceptance criteria

  • All five Rust tests below pass.
  • insert assigns an _id when absent, honors an explicit one, and rejects a duplicate; get round-trips the document.
  • find with a full scan correctly evaluates equality, range, and boolean-combined filters.
  • After create_index, find on the indexed field returns exactly the same document set as the full-scan oracle (asserted by comparing the two).
  • replace and delete keep every index consistent: a query on a changed field no longer returns the stale match, and a deleted document disappears from id, scan, and index results.
  • Manual/external: wrapping each operation in one Phase 5 transaction (crash atomicity) and persisting the indexes as B+trees are labeled stretch goals, not covered by these tests.

// One total order across the indexable value types (design decision above).
// Derived Ord ranks by variant order first, then by contained value.
#[derive(Clone, Debug, PartialEq, Eq, PartialOrd, Ord)]
pub enum Key { Null, Bool(bool), Int(i64), Str(String) }

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum Cmp { Eq, Ne, Gt, Ge, Lt, Le, In }

#[derive(Clone, Debug)]
pub enum Filter {
    Compare { field: String, op: Cmp, value: Value },
    And(Vec<Filter>),
    Or(Vec<Filter>),
    Not(Box<Filter>),
    All,
}

#[derive(Debug)]
pub enum DocError {
    Rec(RecError),
    Serde(String),
    Duplicate(i64),
    NotFound,
    BadDocument(&'static str),
}
impl From<RecError> for DocError {
    fn from(e: RecError) -> Self { DocError::Rec(e) }

#[cfg(test)]
mod tests {
    use super::*;
    use serde_json::json;
    use pager::MemBlock;

    fn fresh() -> DocStore<MemBlock> {
        DocStore::create(MemBlock { data: Vec::new() }).unwrap()
    }
    // scan oracle: brute-force the filter over every document
    fn oracle(s: &mut DocStore<MemBlock>, f: &Filter) -> Vec<i64> {
        let ids: Vec<i64> = s.primary.keys().copied().collect();
        let mut out = Vec::new();
        for id in ids {
            let d = s.get(id).unwrap().unwrap();
            if DocStore::<MemBlock>::matches(f, &d) { out.push(id); }
        }
        out.sort();
        out
    }
    fn ids_of(rows: Vec<(i64, Value)>) -> Vec<i64> {
        let mut v: Vec<i64> = rows.into_iter().map(|(id, _)| id).collect();
        v.sort(); v
    }

    #[test]
    fn insert_assigns_id_and_roundtrips() {
        let mut s = fresh();
        let id = s.insert(json!({"city": "London", "age": 31})).unwrap();
        let got = s.get(id).unwrap().unwrap();
        assert_eq!(got["city"], json!("London"));
        assert_eq!(got["_id"], json!(id));
    }

    #[test]
    fn explicit_id_honored_and_dup_rejected() {
        let mut s = fresh();
        assert_eq!(s.insert(json!({"_id": 100, "x": 1})).unwrap(), 100);
        assert!(matches!(s.insert(json!({"_id": 100, "x": 2})), Err(DocError::Duplicate(100))));
    }

    #[test]
    fn full_scan_equality_and_range() {
        let mut s = fresh();
        s.insert(json!({"city": "London", "age": 31})).unwrap();
        s.insert(json!({"city": "London", "age": 25})).unwrap();
        s.insert(json!({"city": "Paris",  "age": 40})).unwrap();
        let f = Filter::And(vec![
            Filter::Compare { field: "city".into(), op: Cmp::Eq, value: json!("London") },
            Filter::Compare { field: "age".into(),  op: Cmp::Gt, value: json!(30) },
        ]);
        assert_eq!(ids_of(s.find(&f).unwrap()), oracle(&mut s, &f));
        assert_eq!(ids_of(s.find(&f).unwrap()).len(), 1);
    }

    #[test]
    fn index_returns_same_as_scan() {
        let mut s = fresh();
        for i in 0..50 {
            let city = if i % 3 == 0 { "London" } else { "Paris" };
            s.insert(json!({"city": city, "age": i})).unwrap();
        }
        s.create_index("city").unwrap();
        let f = Filter::Compare { field: "city".into(), op: Cmp::Eq, value: json!("London") };
        // index-served result must equal the scan oracle
        assert_eq!(ids_of(s.find(&f).unwrap()), oracle(&mut s, &f));
    }

    #[test]
    fn replace_and_delete_maintain_indexes() {
        let mut s = fresh();
        let a = s.insert(json!({"city": "London"})).unwrap();
        let _b = s.insert(json!({"city": "London"})).unwrap();
        s.create_index("city").unwrap();
        let f = Filter::Compare { field: "city".into(), op: Cmp::Eq, value: json!("London") };
        assert_eq!(ids_of(s.find(&f).unwrap()).len(), 2);
        // move a out of London, delete the other
        s.replace(a, json!({"_id": a, "city": "Paris"})).unwrap();
        assert_eq!(ids_of(s.find(&f).unwrap()), oracle(&mut s, &f));
        assert_eq!(ids_of(s.find(&f).unwrap()).len(), 1);
        assert!(s.delete(a).unwrap());
        assert!(s.get(a).unwrap().is_none());
    }
}

The tests read s.primary directly in the oracle; keep that field crate-visible to the test module (same file). The oracle re-evaluates the filter by brute force, so every index-served find is checked against the ground-truth predicate on the same data.

Stuck? — graduated hints

M1: open() can't find next_id or the index after reopening
Nudge: The heap is the source of truth; the indexes are derived.
Technique: on open, iterate every record in the RecStore, deserialize each, read its _id, and fill primary; set next_id = max(_id) + 1. You'll need a way to enumerate all records — expose an iterator over (RecordId, bytes) from your Phase 4 crate.
M3: comparing a number and a string panics or gives nonsense
Nudge: One total order, defined once.
Technique: convert both sides through key_of to the Key enum and compare Keys; the derived Ord ranks by variant (Null<Bool<Int<Str) then value. If either side isn't an indexable key (float/array/object), define the comparison result explicitly (e.g. only Eq on deep-equal Values, everything else false) and use the same rule in the index.
M5: indexed find returns fewer/more rows than the scan
Nudge: A plan = access path + residual filter; both halves must be right.
Technique: use the index only to produce candidate ids for one condition; then still run matches(full_filter, doc) on each candidate (the residual). For equality, candidates = secondary[field][key_of(value)]; for a range, scan the BTreeMap sub-range. Never return index hits without re-checking the whole predicate.
M2: after replace, a stale value still matches
Nudge: You must remove the old entry before adding the new one.
Technique: read the existing document first, compute its old key for each indexed field, remove that id from secondary[field][old_key], then write the new doc and insert the new key. Do the same removal in delete. (In a real engine this whole sequence is one Phase 5 transaction.)
