6.4 — CRUD on the engine

Goal: Define the create/read/update/delete API of your document store, what each operation must touch to keep heap and indexes consistent, and the document-level atomicity guarantee you can honestly make.

Why this matters to your DB

CRUD is the surface the rest of the world uses. Getting its contract right — what's atomic, what happens to indexes, what an update means for a missing field — is what makes the engine usable and correct. MongoDB's CRUD surface is the model: create with insertOne/insertMany, read with find, update with updateOne/updateMany/replaceOne, delete with deleteOne/deleteMany. [src: MongoDB Manual — CRUD operations]

Create (insert)

The 6.3 sequence: assign _id if absent → serialize → store bytes → insert _id → RecordId into the primary index → insert into every secondary index (6.5) → commit. MongoDB insert operations can create an absent collection, and the driver supplies an identifier when an inserted document omits _id. [src: MongoDB — Create operations; _id auto-generation]

Read (find / get)

  • By _id: the two-step lookup from 6.3 — primary index then heap.
  • By filter: a query (6.6) that yields a set of matching documents, served either by scanning the heap or by consulting a secondary index (6.7 decides which). MongoDB read operations use query filters to select returned documents; a simple { field: value } filter expresses equality. [src: MongoDB — Read operations & query-filter syntax]

Update

Two flavors, mirroring MongoDB:

  • Replace (replaceOne): swap the whole document for a new one. _id is immutable in MongoDB. For this project's replace(id, doc), inject id if the replacement omits _id, and reject a different supplied value; then re-serialize, relocate if necessary, and maintain secondary indexes. [src: MongoDB replaceOne() — replacement cannot change _id]
  • Modify (updateOne with operators like $set): apply a partial change to specific fields. MongoDB uses a filter to identify targets and update operators such as $set to describe modifications. [src: MongoDB — Update operations] Mechanically: read → apply the change in memory → write back like a replace.
⚠️ An update must fix every index that points at it

If you change an indexed field's value, the old secondary-index entry is now wrong. An update is: remove the old index entries for the fields that changed, write the new document bytes (in place or relocated), repoint the primary index if it moved, then insert the new index entries — all in one transaction. Forgetting to maintain a secondary index on update is the classic source of "the query misses a document that's actually there."

Delete

Reverse of insert: look up _id → read the document (to know its indexed field values) → remove every secondary-index entry → remove the primary-index entry → free the heap record (and its overflow chain, Phase 4) → commit. deleteOne removes the first match; deleteMany removes all matches of a filter. [src: MongoDB — Delete operations]

The atomicity guarantee you can make

After the document layer is actually routed through one Phase 5 transaction, a single insert/update/delete can be crash-atomic across the heap and every index. That is the storage guarantee built so far. MongoDB separately documents single-document write atomicity, including behavior under concurrent updates; reproducing that wider concurrency guarantee also depends on Phase 7. [src: MongoDB — atomicity and transactions]

⚠️ Document-level ≠ multi-document transactional

Atomic per operation is not the same as a multi-document transaction spanning several inserts/updates. A deleteMany touching 100 documents being "atomic at the document level" means each document's removal is atomic, not that all 100 commit together — unless you explicitly open one transaction around the whole batch. Be precise about which you offer; don't imply serializable multi-document isolation. (Concurrency & isolation are Phase 7.)

✅ Concept check

1. Which structures does an insert touch, and how is consistency guaranteed?

Show answer
The heap (document bytes), the primary index (_id → RecordId), and every secondary index — all inside one transaction so the change is crash-atomic; Phase 7 is still needed for concurrency semantics.

2. What's the difference between a replace and an operator update ($set)?

Show answer
Replace swaps the whole document for a new one; an operator update modifies specific fields. Mechanically the operator update is read → apply change → write back like a replace.

3. When an update changes an indexed field, what must happen to the secondary index?

Show answer
Remove the old entry for that field's old value and insert a new entry for the new value, in the same transaction; otherwise queries on that field go wrong.

4. What does "atomic at the document level" guarantee, and what does it not?

Show answer
Each single write fully happens or not at all across heap + indexes when routed through one transaction. It does not define concurrent-update behavior or make a multi-document batch one unit unless those contracts are explicitly added.
