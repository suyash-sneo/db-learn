6.5 — Secondary indexes on fields

Goal: Build the mapping that makes "find documents where field = x" fast — a secondary index from a field's values to the _ids that have them — and understand keys, duplicates, composite and multi-entry indexes, and the maintenance cost.

Why this matters to your DB

The primary index only answers "by _id." Every other query — "users in London," "orders over $100" — would otherwise scan every document. A secondary index is the same B+tree machinery (Phase 3) pointed at a different key. An equality lookup costs the tree search plus work proportional to the number of matching ids and their primary-record fetches; an index is not automatically faster when most documents match.

The core mapping: field value → _id

A secondary index on field f is a B+tree whose key is the value of f and whose payload is the _id of a matching document (recall from 6.3: store _id, not RecordId, so document relocation doesn't disturb the secondary index). To answer f = x: look up x in this tree → get the _id(s) → fetch each document via the primary index. Because the tree is ordered, it also answers ranges (f > x, x ≤ f < y) by scanning a contiguous run of keys — the same reason Phase 3 chose an ordered B+tree over a hash.

Duplicates: an index key usually isn't unique

Many documents can share a city, so the key "London" maps to several _ids. Two standard encodings, both common in real engines:

  • Posting list: the key's payload is a list/set of _ids.
  • Composite key: make the B+tree key the pair (field_value, _id) so every entry is unique; all documents with a given value form a contiguous key range you scan. SQLite ordinary secondary-index records use the indexed columns followed by the table rowid for rowid tables. [src: SQLite file format §2.4 — index keys followed by rowid]

A unique index is the special case where you forbid a second entry for the same value (how the primary _id index itself behaves; Isar exposes this as a unique @Index). [src: pinned Isar v3.1.0+1 docs/docs/indexes.md]

Composite and multi-entry indexes

  • Composite (multi-field) index on (a, b): the key is the tuple, so the tree is ordered by a then b. It accelerates queries on a alone (a prefix) and on a AND b, but not on b alone — the leftmost-prefix rule. Isar v3 documents that a prefix of a composite index can be used. [src: pinned Isar v3.1.0+1 docs/docs/queries.md]
  • Multi-entry index on a list/array field: index each element separately, so a document with tags: ["math","poetry"] produces two index entries both pointing at its _id. This is how "array contains x" stays fast. Isar calls these multi-entry indexes. [src: pinned Isar v3.1.0+1 docs/docs/indexes.md]

The cost: every write maintains every index

Indexes are not free. Each one you add is another structure that every insert, update (of the indexed field), and delete must keep in sync — within the same transaction as the document write (6.4). The trade-off is the universal database bargain: faster reads, slower writes, more space. Index the fields you actually query; don't index everything.

⚠️ Missing fields and type ordering

Schemaless documents may lack the indexed field. This project chooses a sparse policy: missing fields produce no entry, while an explicit null does. That differs from MongoDB's default non-sparse indexes, which include missing fields as null-like entries; MongoDB sparse indexes omit them. [src: MongoDB — sparse versus non-sparse indexes] Also define a total byte ordering for index keys, but do not confuse sort order with query semantics: this project permits range comparisons only when both operands have the same supported type.

Building an index on existing data

Creating an index on a non-empty collection is a one-time scan: read every document, extract the field, insert (value → _id) for each. From then on, incremental maintenance on each write keeps it current. (Doing this safely while other writers run is a Phase 7 concern.)

✅ Concept check

1. What does a secondary index map, and why to _id rather than RecordId?

Show answer
Field value → the _id(s) of matching documents. Using _id means relocating a document's bytes doesn't require touching the secondary index—only the primary index repoints. Output cost still grows with the number of matching ids.

2. Give two ways to handle a non-unique index key.

Show answer
Store a posting list of _ids per key, or make the B+tree key a composite (value, _id) so every entry is unique and a value's documents form a contiguous range.

3. Why can a composite index on (a,b) serve a query on a but not on b alone?

Show answer
The tree is ordered by a then b; a (and a+b) is a contiguous prefix range, but b alone is scattered across the whole tree (leftmost-prefix rule).

4. What's the cost side of adding an index?

Show answer
Every insert/relevant-update/delete must maintain it (in the same transaction), and it consumes space — faster reads at the cost of slower writes and more storage.
