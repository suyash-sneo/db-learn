4.6 — Source dive: SQLite records, cells & freelist

Goal: Confirm the slotted-page, overflow, and freelist concepts against SQLite's spec and code, and collect the precise field layouts you'll mirror in the build.
📂 Where to read

For Phase 4 the most authoritative reading is the file-format specification (it is the contract the code implements), backed by src/btree.c for the page-manipulation routines. Code links are pinned to SQLite 3.53.2, commit ccc132c5be20ab5c755c97a08d06b1b592fef330. [src: file format §1.5, §1.6, §2; pinned src/btree.c]

⚠️ Source dive discipline

The names above were checked at the pinned revision, but their internal contracts are still more complex than this curriculum's format. The file-format specification is the authority for SQLite bytes. The build's record-page and freelist layouts are explicitly different project choices.

✅ Concept check

1. For Phase 4 layouts, which SQLite document is the authority — the code or the spec?

Show answer
The file-format spec; the code implements it. Cite §1.5/§1.6/§2 for byte layouts.

2. Which spec offsets hold the freelist head and free-page count?

Show answer
Offset 32 (first freelist trunk page) and offset 36 (total freelist pages).

3. What does the first 4 bytes of an overflow page hold?

Show answer
The page number of the next overflow page, or 0 if it's the last in the chain.

4. Which routine corresponds to your 4.4 compaction?

Show answer
defragmentPage—it packs live cells to reclaim fragmented free space.
