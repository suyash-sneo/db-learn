4.1 — Fixed- vs variable-length records

Goal: Understand why a record's layout is a design choice, what fixed-length records buy and cost, and why a schemaless document store is inherently variable-length — which forces the slotted-page design that follows.

Why this matters to your DB

A "record" is the unit of data inside a page (a row, a document, an index entry). How you lay it out determines whether you can address it cheaply, update it in place, and reclaim its space. Your engine is schemaless and stores documents of wildly differing sizes, so this lesson establishes why you can't take the easy fixed-length road.

Fixed-length records: simple but rigid

If every record is exactly R bytes, a page is an array of slots and record i lives at offset header + i × R — the same O(1) addressing trick as pages within a file. Deletion is trivial (mark the slot free; any record fits any slot). The cost:

  • Wasted space. Every record must be padded to the largest possible size. A schema with one rare long field bloats every row.
  • No variable data. Strings, blobs, and documents have no natural fixed size; forcing one means truncation or huge padding.
  • Schema rigidity. Changing R means rewriting every record.

Fixed-length layouts still appear inside databases — for fixed-width columns, for the slot array itself, for page headers — but rarely for whole user records.

Variable-length records: flexible but needs bookkeeping

Allow each record to be its own size and you stop wasting space, but you lose O(1) addressing: you no longer know where record i is without help, because records before it have different sizes. You also can't reuse a freed hole for a record of a different size without thought. The solution to both problems is the slotted page (4.2): an indirection array that maps a stable slot index to the current byte offset of a variable-length record.

What "a record" is in your schemaless engine

Your engine stores documents serialized to bytes (BSON/MessagePack, Phase 6). A document is a variable-length blob: {} is a few bytes; a document with arrays and nested objects can be kilobytes. There is no fixed R. Therefore:

  • Records are variable-length → you need slotted pages (4.2).
  • Some records exceed a page → you need overflow pages (4.3).
  • Updates change a record's size → you need in-page compaction and possibly relocation (4.4).
  • Deletes free whole pages → you need a freelist (4.5).

This lesson is the "why"; the next four are the "how."

SQLite's record format, for reference

SQLite serializes each row as a "record" with a header of varint "serial types" followed by the column values, and stores that record as the payload of a b-tree cell. [src: SQLite file format §2 "Record Format"] The key idea you'll reuse: a record is a self-describing, variable-length byte string, and the page layer treats it as opaque payload. ⚠️VERIFY the exact serial-type encoding against §2 if you cite specific type codes.

✅ Concept check

1. What's the addressing advantage of fixed-length records, and its main cost?

Show answer
O(1) offset = header + i×R, but every record is padded to the maximum size, wasting space and forbidding truly variable data.

2. Why do variable-length records lose direct index addressing?

Show answer
Records have different sizes, so you can't compute record i's position; you need a stored offset (the slot array).

3. Why is a schemaless document store inherently variable-length?

Show answer
Documents differ in shape and size with no common fixed width; serializing them yields variable-length blobs.

4. Name the four mechanisms variable-length records force you to build.

Show answer
Slotted pages, overflow pages, in-page compaction/relocation, and a freelist.
