4.5 — The freelist

Goal: Reclaim whole pages that are no longer used (freed tree nodes, emptied overflow chains) by tracking them in a persistent freelist, so allocation reuses freed pages before growing the file.

Why this matters to your DB

Within a page, compaction (4.4) reclaims space. But when an entire page becomes empty — a merged B+tree node, a deleted record's overflow chain — that whole page must go somewhere reusable, or the file only ever grows. The freelist is the engine's "recycle bin" for pages and is the difference between a database that stays compact and one that bloats endlessly.

The allocation contract

Page allocation now has two sources, checked in order:

  1. Reuse: if the freelist is non-empty, pop a page number off it and return that page (no file growth).
  2. Grow: otherwise, extend the file by one page (the Phase 2 allocate_page) and return the new page number.

Freeing a page pushes its number onto the freelist. Net effect: the file grows only when there are genuinely no reusable pages.

How SQLite structures its freelist (your reference)

SQLite's freelist is a linked list of trunk pages, each holding an array of free leaf page numbers. The database header stores the page number of the first freelist trunk page at offset 32 and the total count of freelist pages at offset 36. [src: SQLite file format §1.3 header (offsets 32 & 36) and §1.5 "The Freelist"] Each trunk page is an array of 4-byte big-endian integers: the first is the page number of the next trunk page (0 if last), the second is the count of leaf page numbers that follow, then those leaf page numbers. [src: SQLite file format §1.5, freelist trunk page layout] This two-level structure (trunks pointing to many leaves) means freeing or allocating a page usually touches only one trunk page, and SQLite need not even read the leaf pages' contents.

A simpler freelist for your build

You don't need SQLite's exact two-level structure to start. Two valid simpler designs:

  • Singly-linked free pages: the header stores the first free page number; each free page's first 4 bytes hold the next free page number (0 = end). Allocation pops the head; freeing pushes onto the head. One page read/write per operation, like a stack. This is the easiest correct design.
  • Trunk-style (SQLite-like): trunk pages batching many free page numbers, fewer reads under heavy churn. More code; adopt it if you measure the simple list as a bottleneck.

Either way, the freelist head lives in the file header so it survives reopen, and updates to it must be made durable together with the operation that freed/allocated (Phase 5).

⚠️ The freelist is on the crash-safety critical path

If a page is removed from the freelist and used, but a crash loses the freelist update, the same page could be handed out twice on the next run — silent corruption. Freelist mutations are part of the transaction that caused them and must be journaled/WAL-logged like any other page change (Phase 5). Treat the freelist not as a side cache but as durable database state. Also: never put a page on the freelist while it's still referenced — double-check the structure that pointed at it was updated first.

Optional: shrinking the file

A freelist lets you reuse pages but the file still doesn't shrink. Reclaiming disk back to the OS (SQLite's VACUUM / auto-vacuum) requires moving live pages from the end of the file into freed slots, then truncating. This is an optional advanced feature; a freelist alone keeps the file from growing unboundedly, which is the essential property. ⚠️VERIFY SQLite's auto-vacuum mechanics before describing them in detail.

✅ Concept check

1. What two sources does page allocation check, and in what order?

Show answer
First the freelist (reuse a freed page); only if empty, grow the file by one page.

2. Why store the freelist head in the file header?

Show answer
So the recycle bin persists across reopen; otherwise freed pages would be forgotten and leak on every restart.

3. What corruption can a lost freelist update cause?

Show answer
A page could be allocated twice (handed out again after a crash), overwriting live data — silent corruption.

4. Does a freelist shrink the file? What does?

Show answer
No — it only enables reuse. Shrinking requires relocating live pages and truncating (vacuum), an optional feature.
