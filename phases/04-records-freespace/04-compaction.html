4.4 — Deletion, fragmentation & compaction

Goal: Understand how deletes and in-place updates fragment a page's free space, the difference between contiguous free space and total free space, and how compaction reclaims dead space so a page can keep accepting inserts.

Why this matters to your DB

Slotted-page deletion (4.2) leaves "dead" bytes behind. Over time a page can have plenty of total free space yet still reject an insert because none of it is contiguous. Compaction is the step that turns scattered holes back into one usable gap. Without it, pages bloat and inserts fail prematurely.

How fragmentation happens

Recall the layout: a free gap in the middle, plus possibly dead cells scattered in the cell-content region after deletes. Two kinds of free space arise:

  • Contiguous free space — the central gap between the slot array and the start of live cell content. Only this can satisfy a new insert directly.
  • Fragmented free space — dead bytes left by deleted or shrunk cells, interspersed among live cells. It counts toward total free space but can't host an insert until reclaimed.

SQLite tracks both: its page header records the start of the cell-content area and a count of "fragmented free bytes," and maintains a linked list of larger "freeblocks" within the page. [src: SQLite file format §1.6 — freeblocks & fragmented free bytes]

Compaction: the algorithm

Compaction rebuilds the page so all live cells are packed and free space is contiguous:

  1. Walk the slot array (logical order) and collect each live cell's bytes.
  2. Rewrite them packed against the back of the page (or front of content area), in any consistent order — physical order doesn't matter, only that slots are updated to the new offsets.
  3. Update each slot's offset to the cell's new location.
  4. Reset cell_content_start and the fragmented-bytes counter; the gap is now one block.

This is an O(page) memory shuffle done in a scratch buffer, then written back as one page. It's cheap (one page, RAM-speed) and only triggered when an insert can't otherwise fit.

Updates: the size-change problem

An in-place update is only easy when the new value is the same size or smaller. If a value grows:

  • If it still fits the page (possibly after compaction), delete the old cell and insert the new one — its slot index can stay the same so external references (e.g. the key→slot mapping) survive.
  • If it no longer fits the page, the record must relocate (in a B+tree, this may trigger a split; in a record store, move it to another page and update the index pointer).

This is why a stable logical slot index (not a raw byte offset) is valuable: you can move a cell's bytes around freely as long as the slot keeps pointing at it.

⚠️ Compaction must be all-or-nothing within the page

Build the compacted page in a scratch buffer and write it back in one shot, rather than mutating the live page in place mid-shuffle. A crash partway through an in-place compaction could corrupt the page. Writing a fully-formed page is also what the durability layer (Phase 5) expects — it ships whole pages, not partial edits. (Per-page atomicity against torn writes is itself a Phase 5 concern.)

✅ Concept check

1. Why can a page reject an insert despite having lots of free space?

Show answer
The free space is fragmented into small dead holes; only contiguous free space can host a new cell until compaction merges the holes.

2. What does compaction actually do?

Show answer
Packs all live cells together and updates their slot offsets, turning scattered free bytes into one contiguous gap.

3. Why does physical cell order not matter after compaction?

Show answer
Logical order lives in the slot array; cells can sit anywhere as long as slots point to their new offsets.

4. Why prefer a stable slot index over a raw byte offset for external references?

Show answer
Cells move during compaction/updates; a stable slot index keeps references valid while the underlying bytes relocate.
