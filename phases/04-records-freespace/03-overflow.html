4.3 — Cell formats & overflow pages

Goal: Define what's inside a cell, and handle records too large to fit in a page by spilling the tail to a chain of overflow pages — removing the bounded-size restriction from Phase 3.

Why this matters to your DB

A document can be larger than your page. Without overflow handling, your engine simply can't store it — a fatal limitation for a document database. Overflow pages are how every page-based engine stores values larger than one page while keeping fixed-size pages. The maximum is still bounded by the format's length field, page-number space, available storage, and implementation limits.

What's in a cell

A cell is the unit a slot points at. Its contents depend on the page type (3.3), but the recurring fields are:

  • A length (or it's derivable from the slot) — how many payload bytes.
  • The key (for ordered pages) — used by binary search.
  • The payload — the value/record bytes, possibly only a prefix if the rest overflows.
  • An optional overflow pointer — the page number of the first overflow page, when the payload didn't fit inline.
  • For internal B+tree cells, a child page number instead of a payload.

The overflow decision

When a record is too large to fit inline (it would consume too much of the page, hurting fanout), the engine stores an inline prefix plus a pointer, and writes the remainder into a linked list of overflow pages:

  1. Keep up to some threshold of payload inline in the cell.
  2. Allocate an overflow page; write the next page_size − overflow_header bytes of payload into it; its header identifies the page type and stores a next-page pointer (0 = end).
  3. Repeat, chaining pages, until all payload is written.
  4. Store the first overflow page number in the cell.

Reading reverses this: read the inline prefix, then follow the chain, concatenating each page's payload until the next-pointer is 0.

How SQLite does it (your reference)

SQLite stores large cell payloads partly inline and spills the rest to a linked list of overflow pages; each overflow page begins with a 4-byte big-endian pointer to the next overflow page (zero on the last), followed by payload bytes. [src: SQLite file format §1.6 "Cell payload overflow pages"] SQLite computes local payload using formulas based on usable page size and cell type; those exact formulas are in §1.6 and belong to SQLite's format, not this project's simpler threshold. [src: SQLite file format §1.6, local payload formulas and overflow pages]

⚠️ Overflow pages must be freed too

When a record is deleted or shrunk, its overflow chain must be returned to the freelist (4.5), or you leak pages. Conversely, allocating overflow pages draws from the freelist first. Overflow and the freelist are two halves of the same space-management story.

Keeping it tractable in your build

The build fixes a self-describing cell header: flags:u8 | total_len:u32 | inline_len:u16 | first_overflow:u32, followed by inline_len bytes. The 11-byte header uses big-endian integers. A set overflow flag requires a nonzero first page; a clear flag requires zero. The u32 total_len bounds one record to at most u32::MAX bytes. Keep the complete cell—including its 4-byte slot—at or below one quarter of usable page bytes; spill the remainder. A project overflow page has an 8-byte header: type:u8 | reserved:[u8;3] | next:u32, then payload. This differs from SQLite's 4-byte-only overflow header and makes page scanning/type validation simpler.

✅ Concept check

1. Why store a value partly inline instead of always fully overflowing?

Show answer
A small inline prefix avoids an extra page read for short-ish values and keeps the cell self-locating, while still bounding inline size to protect fanout.

2. How is a multi-page value reassembled on read?

Show answer
Read the inline prefix, then follow the overflow-page chain, concatenating each page's payload until the next-pointer is 0.

3. What must happen to overflow pages when a record is deleted?

Show answer
The whole chain must be returned to the freelist, or those pages leak.

4. Why bound how much payload stays inline?

Show answer
Large inline cells reduce how many cells fit per page, lowering fanout and deepening the tree.
