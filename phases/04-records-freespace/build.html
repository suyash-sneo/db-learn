Build: a slotted record store with overflow & a freelist

Goal: Store variable-length records of any size in slotted pages on your pager, spill big ones to overflow chains, compact pages to reclaim holes, and recycle whole pages through a freelist.
⚠️ AI off

Slotted pages, compaction, overflow, and the freelist are core storage components — no worked implementation here, only contracts, stubs, failing tests, and graduated hints. Cite the SQLite file-format spec for layouts; write the code yourself.

Deliverable

A Rust 2024 module recstore on the Phase 2 pager: insert/get/delete of arbitrary-length byte records, returning a stable RecordId (page number + slot). Pages are slotted; oversized records use overflow chains; freed pages return to a persistent freelist.

Fixed design decisions

  • Slot = 2-byte big-endian offset into the page; the slot array is at the front (after the page header), cells grow from the back.
  • Page header (this phase): 1-byte type, 2-byte cell count, 2-byte cell-content-start, 2-byte fragmented-bytes counter.
  • Inline threshold: a record larger than ¼ usable page space overflows. Inline portion = a 4-byte first-overflow-page pointer plus whatever prefix you choose (document the choice).
  • Overflow page: first 4 bytes = next overflow page (0 = last), rest = payload.
  • Freelist: singly-linked (header holds first free page; each free page's first 4 bytes = next). Head stored in the file header.
  • RecordId is stable across compaction (slot index does not change when cells move).

Ordered milestones

  1. M1 — Slotted page. Insert/read/delete a cell within one page; maintain the header counters and the gap invariant.
  2. M2 — Compaction. defragment packs live cells; an insert that doesn't fit triggers it before failing.
  3. M3 — Freelist. alloc_page pops the freelist or grows; free_page pushes; head persisted in the header.
  4. M4 — Overflow. Write/read records larger than the inline threshold across an overflow chain; freeing a record frees its chain.
  5. M5 — Record store API. insert(bytes) -> RecordId, get(id) -> Vec<u8>, delete(id), gluing M1–M4 together.

Acceptance criteria

  • All Rust tests below pass.
  • Insert/get round-trips records of varied sizes, including ones that overflow multiple pages.
  • After many delete+insert cycles, a page still accepts inserts thanks to compaction (no premature "page full" while total free space suffices).
  • Freeing records returns their pages (including overflow pages) to the freelist; subsequent allocations reuse them instead of growing the file (assert file/page count doesn't grow when reuse is possible).
  • A RecordId obtained before a compaction still resolves to the same record after it.

#[cfg(test)]
mod tests {
    use super::*;
    use pager::{Pager, MemBlock};

    fn fresh() -> RecStore<MemBlock> {
        let p = Pager::create(MemBlock { data: Vec::new() }, 4096, 64).unwrap();
        RecStore { pager: p }
    }

    #[test]
    fn small_records_roundtrip() {
        let mut s = fresh();
        let a = s.insert(b"hello").unwrap();
        let b = s.insert(b"").unwrap(); // empty record
        assert_eq!(s.get(a).unwrap(), b"hello");
        assert!(s.get(b).unwrap().is_empty());
    }

    #[test]
    fn large_record_uses_overflow() {
        let mut s = fresh();
        let big: Vec<u8> = (0..20_000).map(|i| (i % 251) as u8).collect();
        let id = s.insert(&big).unwrap();
        assert_eq!(s.get(id).unwrap(), big);
    }

    #[test]
    fn compaction_lets_page_keep_accepting() {
        let mut s = fresh();
        // fill a page with medium records, delete most, insert more:
        // without compaction this would falsely report "full".
        let mut ids = Vec::new();
        for i in 0..50 { ids.push(s.insert(&vec![i as u8; 60]).unwrap()); }
        for id in ids.iter().step_by(2) { s.delete(*id).unwrap(); }
        for i in 0..20 { s.insert(&vec![200 + i as u8; 60]).unwrap(); }
    }

    #[test]
    fn freed_pages_are_reused() {
        let mut s = fresh();
        let big: Vec<u8> = vec![7u8; 30_000]; // several overflow pages
        let id = s.insert(&big).unwrap();
        let before = s.pager.page_count;
        s.delete(id).unwrap();
        let _ = s.insert(&big).unwrap();
        // reuse means page_count should not have grown the second time
        assert_eq!(s.pager.page_count, before);
    }
}

Stuck? — graduated hints

M1: "page full" even though I deleted a lot
Nudge: Distinguish contiguous free from total free (4.4).
Technique: when the central gap is too small but total free (gap + fragmented) suffices, call defragment and retry once before reporting full.
M3: a page seems to get allocated twice
Nudge: Order of operations on the freelist head matters.
Technique: on alloc, read head → set head = that page's next → return the page; on free, write current head into the page → set head = page. Never reuse a page before updating the head. (Crash-safety of these updates is Phase 5.)
M4: my reassembled big record is wrong length
Nudge: Each overflow page carries payload after its 4-byte next-pointer.
Technique: track the total length in the cell; read exactly that many bytes across the chain, not "until the page ends," so the final partial page isn't over-read.
M5: RecordId breaks after compaction
Nudge: A RecordId is (page, slot index), not a byte offset.
Technique: compaction may move cell bytes but must keep each live cell's slot index unchanged; only offsets inside slots change.
