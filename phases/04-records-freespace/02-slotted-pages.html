4.2 — Slotted pages

Goal: Master the slotted-page layout in full: a header, a slot (cell-pointer) array growing from the front, cell content growing from the back, and the operations to insert, read, and delete a variable-length cell.

Why this matters to your DB

The slotted page is the single most reused on-disk structure in your engine: B+tree nodes are slotted pages, and so are the record pages you build here. If you understand it deeply, both the index and the record store fall out of one design.

The layout

A slotted page has three regions:

  1. Header (fixed, at the front): page type, number of cells, the offset where free space begins (end of the slot array), and the offset where cell content begins (the back edge of the used cells). Optionally a pointer to a freed-cell list within the page.
  2. Slot array (grows toward higher offsets, just after the header): one small entry per cell, typically a 2-byte offset (and sometimes a length) pointing to where the cell lives. The slot array is kept in the page type's logical order: sorted by key for the Phase 3 B+tree, but stable insertion/identity order for record pages.
  3. Cell content (grows toward lower offsets, from the end of the page): the actual variable-length record bytes, placed wherever they fit. Cells need not be in key order physically.

Free space is the gap in the middle between the end of the slot array and the start of cell content. A cell fits if the gap is at least cell_size + slot_size.

The operations

Insert a cell

  1. Check the free gap is ≥ cell_size + slot_size. If not, try compaction (4.4); if it still doesn't fit, the page is full → split (B+tree) or move on (record store).
  2. Place the cell bytes at cell_content_start − cell_size; update cell_content_start.
  3. Insert a slot at the correct logical position (shifting later slots over by one slot width); write the new cell's offset into it; bump the cell count.

Read a cell

Look up the slot by logical index → get the offset (and length) → slice the cell bytes. One indirection, O(1).

Delete a cell

Deletion depends on what the slot number means:

  • Ordered B+tree page: the external identity is the key, not the slot number. Removing a slot and shifting later slots is acceptable because search re-derives positions.
  • Record page with RecordId(page, slot): shifting slots would retarget every later identifier. Mark the slot as a tombstone instead and leave its index in place. The build uses a 4-byte slot—offset:u16, length:u16—where offset == 0 means deleted. Compaction changes live offsets, not slot indices.

In either case, the old cell bytes become dead space until compaction or an in-page freeblock policy reclaims them.

⚠️ Two offsets, two directions — don't cross them

The invariant is slot_array_end ≤ cell_content_start (the gap is non-negative). Every insert must check this before writing, accounting for both the new cell bytes and the new slot. A bug that lets the slot array grow into cell content silently overwrites records. Assert the invariant after every mutation in debug builds.

How SQLite names these parts

SQLite's b-tree page is exactly this design: the header records the number of cells, the start of the cell-content area, the first freeblock, and the count of fragmented free bytes; a cell-pointer array follows the header; cell content grows from the end of the page. [src: SQLite file format §1.6 — "cell pointer array", "cell content area", "freeblocks", "fragmented free bytes"] Reusing this vocabulary makes the source dive (4.6) readable.

🦀 Rust just-in-time
  • Read/write the 2-byte slot offsets with u16::from_be_bytes/to_be_bytes over page[a..a+2] slices (same discipline as the header in 2.2).
  • Shifting the slot array is copy_within on the page byte slice — be careful with overlapping ranges; copy_within handles overlap correctly, manual loops may not.
  • Bounds-check every slice into the page; an out-of-range index is corruption, not a panic you want in production — return a Corrupt error.
✅ Concept check

1. Why do slots grow from the front and cells from the back?

Show answer
So both can grow into a shared middle gap; free space is simply the distance between them, easy to compute and check.

2. Why must a stable record page tombstone a slot instead of removing it?

Show answer
Removing it would shift later indices and make existing RecordId(page, slot) values point at different records. A tombstone preserves every other slot number.

3. What invariant must every insert preserve?

Show answer
slot_array_end ≤ cell_content_start — the free gap must stay non-negative after adding both the cell and its slot.

4. Why does the slot array stay in logical order while cells don't?

Show answer
Logical order (e.g. by key) lets you binary-search slots; physical cell placement is just "wherever it fits," decoupled from order.
