8.5 — Pinned cbindgen + Dart binding

Goal: Pin cbindgen and ffigen, generate a C header and Dart binding from one ABI, inspect both generated artifacts, and prove the chain with a C compile/link smoke test plus a Dart ownership round-trip.
📂 Where to read

This lesson pins cbindgen 0.29.4, ffigen 20.1.1, and package:ffi 2.2.0, the versions shown by their upstream registries on June 21, 2026. [src: cbindgen release 0.29.4; pub.dev — ffigen 20.1.1; pub.dev — ffi 2.2.0]

Why this matters to your DB

You designed an ABI (8.1–8.4); now two artifacts make it usable: a C header declaring the functions (so C/Dart know the signatures) and a Dart binding that loads the library and looks up the symbols. Generating the header from the Rust source keeps the two in sync — but only if you configure and check the generator carefully, because it transcribes declarations without verifying any of the safety contracts you wrote.

cbindgen: generate the header

cbindgen reads a Rust crate's public C-facing declarations and emits a C/C++ header. Version 0.29.4 recognizes Rust 2024's #[unsafe(no_mangle)]; it follows function signatures into referenced types and emits only a forward declaration when a type has no guaranteed public layout, which is exactly what opaque Db/Cursor handles need. [src: cbindgen 0.29.4 — writing a C API; 0.29.x changes]

Three configuration facts matter:

  • The default output is C++. You must ask for C explicitly: "Default output is C++; use --lang c flag or language = "C" in cbindgen.toml for C output." [src: cbindgen 0.29.4 quick start]
  • Map Rust lengths intentionally. Set usize_is_size_t = true so Rust usize parameters become C size_t rather than uintptr_t. [src: cbindgen 0.29.4 configuration]
  • It can run via CLI, a build.rs script, or a cbindgen.toml. Pin the cbindgen version so the generated header is reproducible.
# cbindgen.toml
language = "C"
usize_is_size_t = true
include_guard = "MINIDB_H"
autogen_warning = "/* Generated by cbindgen 0.29.4. Do not edit. */"

# Install and invoke the pinned tool:
# cargo install --locked cbindgen --version 0.29.4
# cbindgen --config cbindgen.toml --crate minidb_capi --output minidb.h
⚠️ cbindgen transcribes; it does not verify

cbindgen does not prove ownership, pointer validity, threading, or panic containment. It also does not understand Rust module namespacing, so same-named types in different modules can be ambiguous. [src: cbindgen 0.29.4 limitations] So treat the generated header as a draft to inspect, not gospel: confirm the opaque handles are forward-declared, the buffer pointers/lengths match your contracts, and the error codes are present. Compile the header as both C11 and C++ only if you promise C++ compatibility; for this curriculum, C11 is the required surface. Then link a tiny C program to the built library and execute an ABI-version call.

The Dart binding: match the generated C header

When handwritten, each binding has a native signature and a Dart-callable signature. Fixed-width C integers use matching Dart FFI markers, and C size_t uses ffi.Size, not an assumed 32- or 64-bit integer. [src: Dart 3.12.2 Size]

import 'dart:ffi' as ffi;

// Direct loading is useful for a desktop/dev smoke test. Packaging differs in 8.7.
final dylib = ffi.DynamicLibrary.open('libminidb.so');

final class Db extends ffi.Opaque {}

// C: int32_t minidb_open(const uint8_t*, size_t, Db**);
typedef _OpenC = ffi.Int32 Function(
  ffi.Pointer<ffi.Uint8>, ffi.Size, ffi.Pointer<ffi.Pointer<Db>>);
typedef _OpenDart = int Function(
  ffi.Pointer<ffi.Uint8>, int, ffi.Pointer<ffi.Pointer<Db>>);
final _open = dylib.lookupFunction<_OpenC, _OpenDart>('minidb_open');

The binding restates each signature twice: once with native types (Int32, Size, Pointer) and once with Dart value types (int, Pointer). NativeType subclasses are signature/layout markers rather than ordinary Dart values. [src: Dart 3.12.2 NativeType]

Memory on the Dart side

  • Allocate native input and out-parameter storage with calloc or malloc from package:ffi. Release every Dart allocation in finally, including failure paths.
  • For engine-allocated output buffers, copy the bytes into Dart memory, then call your minidb_free_buf — never Dart's free on engine memory (8.1: matching deallocator). Put the engine free in a nested finally so decoding or copying errors cannot leak it.
  • Use Uint8List.fromList(ptr.asTypedList(len)) to make a Dart-owned copy before freeing the engine buffer. Pointer.asTypedList is a view over native memory, not an owning copy. [src: Dart FFI asTypedList]

Generate the Dart side too

Just as cbindgen generates the C header from Rust, ffigen can generate the Dart bindings from the C header — turning minidb.h into the typedef/lookup boilerplate above. In ffigen 20.1.1, the upstream getting-started guide recommends a Dart generator script with FfiGenerator; run it again whenever the header or generator configuration changes. [src: ffigen 20.1.1 getting started]

// tool/ffigen.dart — configuration shape, not a complete binding.
import 'package:ffigen/ffigen.dart';

void main() {
  FfiGenerator(
    output: Output(dartFile: Uri.file('lib/src/minidb.g.dart')),
    headers: Headers(entryPoints: [Uri.file('native/minidb.h')]),
  ).generate();
}

The Flutter 3.44 package_ffi template currently contains ffigen.yaml and generates @Native external functions resolved against a build-hook code asset. That is a packaging integration choice, not a different ABI; 8.7 explains it. [src: Flutter 3.44 — generating bindings]

✅ Concept check

1. What must you configure for cbindgen, and why?

Show answer
Set the output language to C (--lang c / language = "C"), because cbindgen defaults to C++ and a C++ header used as C breaks subtly. Also pin the version for reproducibility.

2. cbindgen produced a header with no errors — what have you proven, and not proven?

Show answer
You've proven the declarations were transcribed. You have not proven ownership, pointer, threading, or panic safety — those are your contracts. Inspect the header and compile a C smoke test to prove it links and runs.

3. Why does each Dart binding restate a function's types twice?

Show answer
Once in native marker types (Int32, Pointer) for the C signature and once in Dart types (int) for the Dart-callable form; the native markers aren't instantiable Dart values.

4. Who frees an engine-allocated output buffer on the Dart side?

Show answer
The engine's own minidb_free_buf, not Dart's free — the allocating library must provide the matching deallocator, and native memory is outside the Dart GC.
🦀 Rust just-in-time
  • crate-type = ["cdylib", "rlib"] lets the same crate produce a C-loadable dynamic library and a Rust library used by in-process tests. [src: Rust Reference — linkage]
  • Keep exported declarations in one shallow module with unique names. cbindgen does not understand Rust module namespaces the way rustc does.
  • A generated header reflects declarations, not behavior. Treat it like another build artifact: pin the generator, inspect diffs, and compile a consumer.
