8.5 — cbindgen header + Dart binding

Goal: Generate a C header from your Rust ABI with cbindgen — correctly, in C (not C++) — verify it, then bind those functions from Dart with dart:ffi, so the engine is actually callable from the consumer side.
📂 Where to read

cbindgen's behavior is documented in its repo docs; the Dart side in dart:ffi. [src: cbindgen docs; dart.dev — C interop]

Why this matters to your DB

You designed an ABI (8.1–8.4); now two artifacts make it usable: a C header declaring the functions (so C/Dart know the signatures) and a Dart binding that loads the library and looks up the symbols. Generating the header from the Rust source keeps the two in sync — but only if you configure and check the generator carefully, because it transcribes declarations without verifying any of the safety contracts you wrote.

cbindgen: generate the header

cbindgen "generates C/C++11 headers from Rust libraries that expose a public C API," scanning for #[no_mangle] pub extern fn, exported statics, and pub const items. [src: cbindgen docs] Two configuration facts are critical:

  • The default output is C++. You must ask for C explicitly: "Default output is C++; use --lang c flag or language = "C" in cbindgen.toml for C output." [src: cbindgen docs; AGENTS.md] A C++ header consumed as C is a subtle source of breakage.
  • It can run via CLI, a build.rs script, or a cbindgen.toml. Pin the cbindgen version so the generated header is reproducible.
# cbindgen.toml
language = "C"
# run it (pinned version), then read the output before trusting it:
# cargo install cbindgen --version X.Y.Z
# cbindgen --config cbindgen.toml --crate minidb --output minidb.h --lang c
⚠️ cbindgen transcribes; it does not verify

"cbindgen transcribes declarations; it does not prove ownership, pointer, threading, or panic safety." [src: AGENTS.md — Rust/FFI baseline] It also "does not understand Rust's module system or namespacing," so identically named types in different modules can clash. [src: cbindgen docs] So treat the generated header as a draft to inspect, not gospel: confirm the opaque handles are forward-declared, the buffer pointers/lengths match your contracts, and the error codes are present. Then compile a tiny C smoke test against the header and the built library to prove they actually link and run. AGENTS requires exactly this: "inspect generated output, and compile a C smoke test."

The Dart binding

dart:ffi lets Dart "call native C APIs, and … read, write, allocate, and deallocate native memory." You load the compiled library and look up each function:

import 'dart:ffi' as ffi;
import 'package:ffi/ffi.dart'; // malloc/free helpers

// Library file name differs per platform (8.7).
final dylib = ffi.DynamicLibrary.open('libminidb.so');

// C: int32_t minidb_open(const char* path, Db** out);
typedef _OpenC = ffi.Int32 Function(ffi.Pointer<ffi.Char>, ffi.Pointer<ffi.Pointer<ffi.Void>>);
typedef _OpenDart = int Function(ffi.Pointer<ffi.Char>, ffi.Pointer<ffi.Pointer<ffi.Void>>);
final _open = dylib.lookupFunction<_OpenC, _OpenDart>('minidb_open');

The binding restates each signature twice: once with native types (Int32, Pointer) and once with Dart types (int), because dart:ffi marker types like Int32 "serve as markers in type signatures" and aren't Dart values. [src: dart.dev — C interop]

Memory on the Dart side

  • Allocate native input buffers with malloc from package:ffi, copy your bytes in, pass the pointer + length, then free them after the call returns (inputs are borrowed only for the call, per 8.2).
  • For engine-allocated output buffers, copy the bytes into Dart memory, then call your minidb_free_buf — never Dart's free on engine memory (8.1: matching deallocator). Native memory is outside the Dart GC; nothing frees it for you.

Generating the Dart side too

Just as cbindgen generates the C header from Rust, ffigen can generate the Dart bindings from the C header — turning minidb.h into the typedef/lookup boilerplate above. It is the symmetric tool and worth using once the header is stable; like cbindgen, inspect its output and don't assume it encodes your ownership rules. ⚠️VERIFY current ffigen configuration/options against pub.dev/packages/ffigen for the version you use.

✅ Concept check

1. What must you configure for cbindgen, and why?

Show answer
Set the output language to C (--lang c / language = "C"), because cbindgen defaults to C++ and a C++ header used as C breaks subtly. Also pin the version for reproducibility.

2. cbindgen produced a header with no errors — what have you proven, and not proven?

Show answer
You've proven the declarations were transcribed. You have not proven ownership, pointer, threading, or panic safety — those are your contracts. Inspect the header and compile a C smoke test to prove it links and runs.

3. Why does each Dart binding restate a function's types twice?

Show answer
Once in native marker types (Int32, Pointer) for the C signature and once in Dart types (int) for the Dart-callable form; the native markers aren't instantiable Dart values.

4. Who frees an engine-allocated output buffer on the Dart side?

Show answer
The engine's own minidb_free_buf, not Dart's free — the allocating library must provide the matching deallocator, and native memory is outside the Dart GC.
