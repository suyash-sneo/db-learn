8.2 — The full API with byte buffers

Goal: Nail the pointer-plus-length contract used to move documents and queries across the boundary in both directions — including the exact rules for empty buffers, null pointers, and who frees engine-allocated output — so the surface is safe to call from C and Dart.

Why this matters to your DB

Documents are variable-length serialized bytes (Phase 6). You can't pass a Rust Vec<u8> across the ABI, and inventing a C struct per document type would freeze the schema into the ABI. The answer is uniform: everything variable is a (pointer, length) pair of bytes. Inputs are buffers the caller owns; outputs are buffers the engine allocates and the caller later returns for freeing. Get this contract exactly right and the rest of the API is just functions over byte buffers.

Input buffers: borrowed, caller-owned

An input is a *const u8 plus a usize length. The engine reads it during the call and must not retain the pointer afterward (the caller may free it the instant the call returns). Converting it to a slice has a precise precondition: slice::from_raw_parts "requires a non-null aligned pointer even for length zero." [src: AGENTS.md — "slice::from_raw_parts requires a non-null aligned pointer even for length zero"]

// Map a borrowed C buffer to &[u8], honoring the (NULL, 0) special case.
unsafe fn as_slice<'a>(ptr: *const u8, len: usize) -> Option<&'a [u8]> {
    match (ptr.is_null(), len) {
        (true, 0)  => Some(&[]),                 // empty input allowed as (NULL, 0)
        (true, _)  => None,                      // (NULL, nonzero) is a caller bug → reject
        (false, _) => Some(unsafe { std::slice::from_raw_parts(ptr, len) }),
    }
}

This is the AGENTS rule made concrete: "If the ABI permits (NULL, 0), branch first and map it to &[]; reject (NULL, nonzero)." [src: AGENTS.md — Rust/FFI baseline]

Output buffers: engine-allocated, caller-returned

For results (a found document, a query's serialized rows), the engine allocates. The curriculum's fixed convention: back the output with a Box<[u8]>, hand out a (pointer, length) pair, and reclaim "the unchanged pair" later. "Represent empty output as (NULL, 0)." [src: AGENTS.md — "prefer pointer + length backed by Box<[u8]> … reclaim the unchanged pair … Represent empty output as (NULL, 0)"]

// Hand ownership of bytes to the caller as (ptr, len).
fn export_bytes(bytes: Vec<u8>, out_ptr: *mut *mut u8, out_len: *mut usize) {
    if bytes.is_empty() { /* write NULL, 0 */ return; }
    let boxed: Box<[u8]> = bytes.into_boxed_slice();
    let len = boxed.len();
    let raw = Box::into_raw(boxed) as *mut u8; // leak it to the caller
    // write raw, len through the out-params
}

#[unsafe(no_mangle)]
pub extern "C" fn minidb_free_buf(ptr: *mut u8, len: usize) {
    if ptr.is_null() { return; }
    // reconstruct the SAME pair and drop it
    unsafe { let _ = Box::from_raw(std::ptr::slice_from_raw_parts_mut(ptr, len)); }
}

The pair must come back unchanged: the same pointer and the same length you handed out. Reconstructing a Box<[u8]> from a different length is undefined behavior. This is why the length is part of the free call, not re-derived.

A representative surface

FunctionShape

minidb_find(*mut Db, *const u8 filter, usize len, *mut *mut u8 out, *mut usize out_len) -> i32
minidb_free_buf(*mut u8, usize) -> void

Documents and filters are serialized bytes (your Phase 6 codec). Adding a query operator or document field changes the bytes, not the ABI — exactly the stability win from 8.1.

⚠️ A null check is not a validity check

Checking ptr.is_null() catches exactly one mistake. It cannot detect a dangling, misaligned, undersized, read-only, or incorrectly aliased pointer: "Checking for null cannot detect dangling, misaligned, undersized, read-only, or incorrectly aliased memory." [src: AGENTS.md — Rust/FFI baseline] So the safety of every buffer rests on the documented contract (length is correct, memory is valid for that length, not aliased during the call), which the caller must uphold — your job is to state it precisely next to each function and reject the one case you can detect.

✅ Concept check

1. Why are documents passed as byte buffers instead of C structs?

Show answer
Documents are variable-length and schemaless; a (ptr,len) byte pair keeps the schema out of the ABI, so changing document shape changes the bytes, not the binary interface.

2. How must (NULL, 0) and (NULL, nonzero) be handled for an input?

Show answer
(NULL,0) maps to an empty slice &[] (because from_raw_parts needs non-null even at len 0); (NULL,nonzero) is a caller bug and must be rejected.

3. Why must an output buffer be freed with the exact same (ptr, len) it was issued with?

Show answer
The free reconstructs a Box<[u8]> from that pointer and length; a different length (or a foreign free) is undefined behavior.

4. What does a successful null check fail to guarantee?

Show answer
Everything else: the pointer could be dangling, misaligned, too short, read-only, or aliased. Validity rests on the documented contract, not on the null check.
🦀 Rust just-in-time
  • Vec::into_boxed_slice drops spare capacity so the (ptr,len) round-trips exactly; pairing it with Box::into_raw/Box::from_raw is the curriculum's standard output-buffer idiom.
  • slice_from_raw_parts_mut(ptr, len) builds the *mut [u8] fat pointer Box::from_raw needs to reclaim the allocation.
  • Mark FFI bodies that dereference raw pointers unsafe and document the contract above each — see Rust Book ch. 20.
