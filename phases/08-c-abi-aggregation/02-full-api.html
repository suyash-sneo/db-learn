8.2 — Full lifecycle + byte-buffer CRUD

Goal: Specify open, close, insert, find, update, and delete with one consistent pointer-plus-length contract — including empty/null rules, output publication, deallocation, and every raw-pointer precondition the caller must uphold.

Why this matters to your DB

Documents, filters, and paths are variable-length data. Passing Rust Vec, String, or slices directly would expose Rust-specific layouts or lifetimes, while defining a C struct per document would contradict the schemaless model. This API therefore uses one rule: all variable data is bytes plus a length. The path is UTF-8 bytes; documents, filters, and result sets use the Phase 6 serialization format. Inputs are borrowed for one call. Outputs are allocated by the engine and returned through out-parameters.

Input buffers: borrowed, caller-owned

An input is a *const u8 plus a usize length. The engine reads it during the call and must not retain the pointer afterward (the caller may free it the instant the call returns). slice::from_raw_parts requires a non-null, aligned pointer even for a zero-length slice; the bytes must lie within one allocated object, the computed range must not wrap, and its total size must not exceed isize::MAX. The memory must remain readable and unmutated for the returned slice's lifetime. [src: slice::from_raw_parts safety contract]

// Map a borrowed C buffer to &[u8], honoring the (NULL, 0) special case.
unsafe fn as_slice<'a>(ptr: *const u8, len: usize) -> Option<&'a [u8]> {
    match (ptr.is_null(), len) {
        (true, 0)  => Some(&[]),                 // empty input allowed as (NULL, 0)
        (true, _)  => None,                      // (NULL, nonzero) is a caller bug → reject
        (false, _) => Some(unsafe { std::slice::from_raw_parts(ptr, len) }),
    }
}

The ABI deliberately accepts (NULL, 0) as empty, so it must branch before calling from_raw_parts. It rejects (NULL, nonzero) as MINIDB_ERR_NULL_ARG. A non-null pointer with an incorrect length is still a caller contract violation; Rust cannot discover the true allocation size from the address.

⚠️ Do not let the helper manufacture a long-lived borrow

The illustrative helper's lifetime is inferred by its caller, so it belongs inside the FFI function and its slice must not escape the call. A safer implementation shape is to validate the pair, create the slice in a small unsafe block, parse or copy it immediately, and retain only owned Rust data. The exported function itself should be unsafe extern "C" because its correctness depends on foreign memory satisfying this contract.

The path is bytes too

minidb_open(path_ptr, path_len, out_db) uses the same borrowed-buffer rule, then requires valid UTF-8 for v1. This avoids a second NUL-terminated-string convention and makes embedded NUL bytes an ordinary validation error rather than a terminator. The limitation is explicit: v1 cannot represent an operating-system path that is not valid UTF-8. If that becomes a requirement, add a platform-specific or encoded-path entry point instead of silently changing this function's meaning.

Output buffers: engine-allocated, caller-returned

For results, the engine owns the allocation and transfers it to the caller. This curriculum backs a nonempty output with Box<[u8]>, returns its data pointer and exact length, and later reconstructs that same boxed slice in minidb_free_buf. Box::from_raw is safe only for a pointer/layout that came from the matching Box allocation, and using it twice can double-free. [src: Box::from_raw safety] Empty output is normalized to (NULL, 0).

// Hand ownership of bytes to the caller as (ptr, len).
fn export_bytes(bytes: Vec<u8>, out_ptr: *mut *mut u8, out_len: *mut usize) {
    if bytes.is_empty() { /* write NULL, 0 */ return; }
    let boxed: Box<[u8]> = bytes.into_boxed_slice();
    let len = boxed.len();
    let raw = Box::into_raw(boxed) as *mut u8; // leak it to the caller
    // write raw, len through the out-params
}

#[unsafe(no_mangle)]
pub unsafe extern "C" fn minidb_free_buf(ptr: *mut u8, len: usize) -> i32 {
    if ptr.is_null() {
        return if len == 0 { MINIDB_OK } else { MINIDB_ERR_NULL_ARG };
    }
    // reconstruct the SAME pair and drop it
    unsafe { let _ = Box::from_raw(std::ptr::slice_from_raw_parts_mut(ptr, len)); }
    MINIDB_OK
}

The pair must come back unchanged: the same pointer and same length, exactly once. minidb_free_buf(NULL, 0) is a no-op; (NULL, nonzero) is rejected. A foreign, shifted, already-freed, or wrong-length non-null pair violates the safety contract and cannot be made trustworthy by a null check.

Out-parameters are a tiny commit protocol

A result-producing call first validates that each out-pointer is non-null and writable. It then writes safe defaults — null handle, null buffer, zero length, or zero ID — before parsing input or touching the database. Only after all work succeeds does it publish the real result. On every error, the caller therefore sees the documented defaults and has nothing to free. Out-pointers must also not alias memory the function reads or other out-pointers in a way that violates Rust's access rules.

The complete v1 CRUD surface

FunctionContract
minidb_open(path, path_len, out_db)UTF-8 path bytes in; new opaque handle out.
minidb_close(db)Release one handle exactly once; null is a documented no-op.
minidb_insert(db, doc, doc_len, out_id)Serialized document in; assigned positive ID out.
minidb_find(db, filter, filter_len, out, out_len)Serialized filter in; serialized result array out. No matches is a successful empty array, not NOT_FOUND.
minidb_update(db, id, doc, doc_len)Replace the document at ID; missing ID returns NOT_FOUND.
minidb_delete(db, id)Delete ID; missing ID returns NOT_FOUND.
minidb_free_buf(ptr, len)Return one exact engine-issued output pair.

Documents and filters are serialized bytes (your Phase 6 codec). Adding a query operator or document field changes the payload format, not these function signatures. That does not make payload changes automatically compatible: the serialization/query format needs its own versioning rules.

⚠️ A null check is not a validity check

Checking ptr.is_null() catches exactly one mistake. It cannot detect a dangling, misaligned, undersized, read-only, or incorrectly aliased pointer. The safety of every buffer rests on the documented contract (length is correct, memory is valid for that length, not aliased during the call), which the caller must uphold — your job is to state it precisely next to each function and reject the one case you can detect.

✅ Concept check

1. Why are paths, documents, and filters passed as byte buffers instead of C structs or strings?

Show answer
Documents are variable-length and schemaless; a (ptr,len) pair keeps their shape out of the ABI and avoids a separate NUL-terminated convention. The path contract then explicitly requires UTF-8.

2. How must (NULL, 0) and (NULL, nonzero) be handled for an input?

Show answer
(NULL,0) maps to an empty slice &[] (because from_raw_parts needs non-null even at len 0); (NULL,nonzero) is a caller bug and must be rejected.

3. Why must an output buffer be freed with the exact same (ptr, len) it was issued with?

Show answer
The free reconstructs a Box<[u8]> from that pointer and length; a different length (or a foreign free) is undefined behavior.

4. Why initialize output parameters before doing real work?

Show answer
So every error leaves the caller with harmless, documented defaults rather than stale pointers, lengths, handles, or IDs that might be mistaken for successful output.

5. What does a successful null check fail to guarantee?

Show answer
Everything else: the pointer could be dangling, misaligned, too short, read-only, or aliased. Validity rests on the documented contract, not on the null check.
