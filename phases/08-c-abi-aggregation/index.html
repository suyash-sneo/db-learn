Phase 8

Real C ABI + aggregation

Phase 1 proved one byte buffer could cross from Rust to Dart. This phase turns that spike into a real, stable public interface: an opaque-handle C ABI for open/CRUD/query/cursor, with disciplined ownership, error reporting, and versioning; a generated C header and a Dart binding that consumes it; a carefully scoped optional aggregation pipeline layered on the Phase 6 query engine; and Flutter 3.44 build-hook packaging that ships the native library for iOS and Android. This is where the engine stops being only a Rust crate and becomes a database another runtime can call.

Why the C ABI comes last

A public ABI is a promise: once code depends on your symbols, types, and ownership rules, you cannot casually change them. You only want to make that promise once the thing behind it is real — a durable, indexed, queryable document store (Phases 2–7). Designing the boundary now means freezing a surface over machinery you actually trust, and aggregation is added here because it is a feature built on the query engine rather than a new storage layer.

What you'll be able to do after Phase 8

Design an ABI-stable C surface using opaque handles and byte buffers; specify and uphold ownership, lifetime, and panic contracts at the boundary; iterate query results across the FFI with a cursor; report errors and version the ABI without breaking callers; generate a C header with cbindgen and call it from Dart; specify a small optional aggregation pipeline on top of your evaluator; and package the native library with Flutter's current package_ffi build-hook flow.
