8.6 — Aggregation (v2): the pipeline

Goal: Understand the aggregation pipeline as a sequence of stages that transform a stream of documents — filter, group, sort, reshape — and see how it layers on the Phase 6 query engine rather than replacing it, using MongoDB's pipeline as the reference model.
📂 Where to read

MongoDB 8.3's aggregation manual is the reference model. This lesson defines a much smaller engine subset and does not claim wire-format or operator parity. [src: MongoDB 8.3 — aggregation pipeline]

Why this matters to your DB

Phase 6 gave you find: a filter that selects documents. Real applications also need to summarize — count per category, average per group, top-N by a field. The aggregation pipeline is the general mechanism for that: a list of stages, each transforming the stream of documents flowing through it. Crucially, it is built on your existing evaluator, not beside it — the first stage is usually just the filter you already have.

A pipeline is a sequence of stages

A pipeline is an ordered list. Stage 1 consumes the collection or access-path output; each later stage consumes the previous stage's documents. A stage may preserve cardinality, discard documents, emit one output for many inputs, or expand one input into several outputs. [src: MongoDB 8.3 — stage behavior]

The common stages

StagePurpose
$matchFilter documents by a condition (your Phase 6 filter).
$groupGroup documents and compute aggregate values (sum, avg, count, …).
$sortOrder documents ascending/descending.
$projectReshape: include/exclude/compute fields.
$limit / $skipCap the count / skip a prefix.
$countCount the documents reaching the stage.
$unwindDeconstruct an array field into one document per element.

[src: MongoDB 8.3 — stage reference]

MongoDB permits most stages to appear repeatedly, with documented exceptions including $out, $merge, and $geoNear. Its normal aggregation pipeline is read-only unless it contains $merge or $out. [src: MongoDB 8.3 — additional stage details] This curriculum's subset has no write stage, so every aggregation is read-only.

Work one pipeline by hand

Suppose the input is:

{"city":"Oslo",   "amount":4}
{"city":"London", "amount":7}
{"city":"Oslo",   "amount":9}
{"city":"London", "amount":2}

The pipeline “match amount ≥ 4 → group by city and sum amount → sort total descending” produces:

{"_id":"Oslo",   "total":13}
{"_id":"London", "total":7}

The filter removes London's amount-2 document; grouping creates one accumulator state per distinct city; sorting orders the two new group documents. The original documents are not modified.

How it layers on Phase 6

Model a pipeline as a chain of iterators over documents — exactly the cursor idea from 8.3 with a transformation per stage:

  • $match is your existing evaluator (6.6) used as a streaming filter.
  • $sort and the simple hash-based $group taught here are blocking: they consume all input before final output. MongoDB also documents $group as blocking and potentially memory-intensive. [src: MongoDB 8.3 — $group performance]
  • $count consumes the complete input and emits one count document at end; for empty input MongoDB emits no document, not a document containing zero. [src: MongoDB 8.3 — $count behavior] Your subset must state whether it copies that empty-input rule or deliberately returns zero.
  • $limit, $skip, $project, $unwind are streaming: they transform documents one at a time.

So aggregation reuses the planner/evaluator and adds stage operators on top. It is a feature built on the query engine, which is why it belongs in this phase rather than Phase 6.

⚠️ Filter early only when it means the same thing

A leading $match can use a source-collection index and reduce work for later stages. MongoDB recommends putting a source-document match at the beginning when possible. [src: MongoDB 8.3 — document filters and indexes] But stage order is semantics, not decoration: a match after $group filters aggregate output (similar to SQL HAVING), and a match on a field computed by $project cannot simply move before that computation. Reorder only when you can prove equivalence.

Define the subset before implementing it

A stage name is not a complete semantic contract. For this engine, write down:

  • which field paths exist and what a missing field does;
  • which value types may be compared, sorted, grouped, or summed;
  • whether numeric accumulation can overflow and which error it returns;
  • the total ordering used by $sort, plus a deterministic tie-breaker;
  • the exact output document for $group and the empty-input behavior of $count;
  • memory/size limits for blocking stages and the error returned when a limit is exceeded.

Without these rules, two implementations can both “support $group” and still return different results.

Honest scope for your engine

You are not building MongoDB's full operator set or its query optimizer. A credible optional v2 implements a handful of stages — $match, $project, $sort, $limit, and a simple $group with count/sum — composed as a stage chain, with $match delegating to your evaluator. Say so explicitly, and list which stages and accumulators you support, so a reader isn't misled into expecting parity with MongoDB.

✅ Concept check

1. What is an aggregation pipeline, in one sentence?

Show answer
An ordered sequence of stages, each transforming the stream of documents and passing its output to the next stage.

2. Why is "a stage need not output one document per input" important?

Show answer
Stages change cardinality: $match drops documents, $group collapses many into few, $unwind expands one into many. The stream's length is not fixed.

3. Which stages are blocking vs. streaming, and why does it matter?

Show answer
Blocking stages ($sort, the simple $group and $count) must see all input before final output; streaming stages ($match, $project, $limit, $unwind) act per document. Blocking stages bound memory/latency, so feed them fewer documents.

4. When may you move $match to the front?

Show answer
Only when it filters source-document fields and moving it is semantically equivalent. A match on grouped or computed output must stay after the stage that creates those values.

5. What must a custom $group specification define beyond its name?

Show answer
At least group-key/missing-field behavior, supported accumulator types, overflow handling, output shape, memory limits, and deterministic ordering where ordering is promised.
🦀 Rust just-in-time
  • Represent the parsed pipeline as an enum such as Stage::Match, Stage::Sort, and Stage::Limit; parsing bytes and executing stages are separate responsibilities.
  • Streaming stages can wrap an iterator; blocking stages collect owned state. That ownership difference should be visible in the design and memory-limit tests.
  • Return a typed error for unsupported stages, invalid operands, overflow, and memory limits rather than using panic!. Review Rust Book iterators.
