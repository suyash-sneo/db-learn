8.6 — Aggregation (v2): the pipeline

Goal: Understand the aggregation pipeline as a sequence of stages that transform a stream of documents — filter, group, sort, reshape — and see how it layers on the Phase 6 query engine rather than replacing it, using MongoDB's pipeline as the reference model.
📂 Where to read

MongoDB's aggregation pipeline is the reference. Quotes below are from its manual. [src: Manual — Aggregation Pipeline]

Why this matters to your DB

Phase 6 gave you find: a filter that selects documents. Real applications also need to summarize — count per category, average per group, top-N by a field. The aggregation pipeline is the general mechanism for that: a list of stages, each transforming the stream of documents flowing through it. Crucially, it is built on your existing evaluator, not beside it — the first stage is usually just the filter you already have.

A pipeline is a sequence of stages

An aggregation pipeline is "one or more stages that process documents." "Each stage performs an operation on the input documents" and "the documents that a stage outputs are then passed to the next stage." A key property: "A stage does not need to output one document for every input document. Some stages produce new documents or filter documents out." [src: Manual — Aggregation Pipeline]

The common stages

StagePurpose
$matchFilter documents by a condition (your Phase 6 filter).
$groupGroup documents and compute aggregate values (sum, avg, count, …).
$sortOrder documents ascending/descending.
$projectReshape: include/exclude/compute fields.
$limit / $skipCap the count / skip a prefix.
$countCount the documents reaching the stage.
$unwindDeconstruct an array field into one document per element.

[src: Manual — common stages]

Two precise rules worth copying: a stage "can appear multiple times in a pipeline" (with a few documented exceptions like $out/$merge), and an aggregate() pipeline "do[es] not modify documents in a collection" unless it contains a $merge or $out stage — i.e. aggregation is read-only by default. [src: Manual — Aggregation Pipeline]

How it layers on Phase 6

Model a pipeline as a chain of iterators over documents — exactly the cursor idea from 8.3 with a transformation per stage:

  • $match is your existing evaluator (6.6) used as a streaming filter.
  • $sort, $group, $count are blocking stages: they must consume their entire input before producing output (you can't sort a stream without seeing all of it).
  • $limit, $skip, $project, $unwind are streaming: they transform documents one at a time.

So aggregation reuses the planner/evaluator and adds stage operators on top. It is a feature built on the query engine, which is why it belongs in this phase rather than Phase 6.

⚠️ Put $match first, and reuse the index path

A pipeline's cost is dominated by how many documents flow through the expensive (blocking) stages. Filtering early — $match as the first stage — means $group/$sort see far fewer documents, and a leading $match can use a secondary index via your Phase 6 planner (6.7) instead of a full scan. A $match buried after a $group has lost that opportunity. This is the aggregation version of "selective access path first." Treat ordering as a correctness-preserving optimization: reordering must not change results (the same rule as 6.7).

Honest scope for your engine

You are not building MongoDB's full operator set or its query optimizer. A credible v2 implements a handful of stages — $match, $project, $sort, $limit, and a simple $group with count/sum — composed as a stage chain, with $match delegating to your evaluator. Say so explicitly, and list which stages and accumulators you support, so a reader isn't misled into expecting parity with MongoDB.

✅ Concept check

1. What is an aggregation pipeline, in one sentence?

Show answer
An ordered sequence of stages, each transforming the stream of documents and passing its output to the next stage.

2. Why is "a stage need not output one document per input" important?

Show answer
Stages change cardinality: $match drops documents, $group collapses many into few, $unwind expands one into many. The stream's length is not fixed.

3. Which stages are blocking vs. streaming, and why does it matter?

Show answer
Blocking stages ($sort, $group, $count) must see all input before emitting; streaming stages ($match, $project, $limit, $unwind) act per document. Blocking stages bound memory/latency, so feed them fewer documents.

4. Why put $match first?

Show answer
It shrinks the stream before expensive stages and lets the planner use a secondary index for the filter; a late $match loses both benefits. Reordering must not change the result.
