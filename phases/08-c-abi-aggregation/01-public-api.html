8.1 — Designing a stable public C API

Goal: Design a small C ABI whose handles, types, errors, ownership, and concurrency rules are explicit enough for C and Dart callers to use correctly — and whose internals can evolve without changing the binary contract.

Why this matters to your DB

A C ABI is a practical interoperability target because many runtimes can call C-compatible functions. It is not one universal ABI: extern "C" selects the ABI used by the dominant C compiler for the current Rust target. It says how a call crosses the machine boundary; it does not prove that a Rust type, pointer, lifetime, or failure mode is safe for foreign code. [src: Rust Reference — ABI] The public contract must therefore define every handle, buffer, result code, allocator, lifetime, and threading rule.

📂 SQLite's useful shape, not its whole API

SQLite's core C interface is built around opaque lifecycle objects: sqlite3_open() creates a database connection, sqlite3_prepare() creates a statement, sqlite3_step() advances it, and matching close/finalize calls destroy them. [src: SQLite C interface §3–4] Your API can copy that shape while remaining much smaller: database and cursor handles, operations that return status codes, and explicit destroy functions.

In the generated SQLite 3.53.0 sqlite3.h, line 275 forward-declares typedef struct sqlite3 sqlite3;; the same header declares runtime version, close, free, open, and error-message functions at lines 191, 356, 3322, 3990, and 4246 respectively. [src: SQLite 3.53.0 sqlite3.h in the official amalgamation archive] These are source-navigation anchors, not an instruction to copy SQLite's full surface.

Rule 1: opaque handles, never exposed layout

The database is a complex Rust struct. Do not expose its fields or layout across the boundary — callers get an opaque pointer (a forward-declared, incomplete type) they pass back to every call. This keeps the struct's internals free to change without breaking the ABI, and keeps Rust ownership on the Rust side. The Rustonomicon describes the same C pattern as struct Foo; plus functions that accept struct Foo *. [src: Rustonomicon — representing opaque structs]

// Rust side: fields are private and are not part of the C API.
pub struct Db { inner: DbInner } // no repr(C): layout stays private

#[unsafe(no_mangle)]
pub unsafe extern "C" fn minidb_open(
    path: *const u8, path_len: usize,
    out: *mut *mut Db
) -> i32 { /* boundary logic */ }
// A checked cbindgen header should contain an incomplete Db declaration.

Functions whose safety depends on raw-pointer validity should be pub unsafe extern "C" fn. C and Dart do not display Rust's unsafe, but it keeps the Rust-facing API honest: safe Rust must not be able to trigger undefined behavior merely by calling a supposedly safe function with an arbitrary raw pointer. #[unsafe(no_mangle)] gives the item an exported symbol name without Rust name mangling in Rust 2024; a globally distinctive minidb_ prefix reduces collisions. [src: Rust 2024 Edition Guide — unsafe attributes]

⚠️ A name attribute is not an export test

Linking and stripping still vary by target. After producing the final library, inspect its exported symbol table and compile a C consumer. The Flutter native-linking guide likewise warns that link-time optimization can discard symbols unless the native build retains them. [src: Flutter 3.44 — native linking]

Rule 2: only ABI-safe types cross the line

Keep Rust-owned containers and default-layout Rust structs out of public signatures. The useful vocabulary is deliberately small:

  • Fixed-width integers whose C header uses matching int32_t/uint64_t types. Use an integer with documented accepted values instead of exposing Rust bool.
  • usize only where the generated header is intentionally checked as C size_t; Dart binds that as ffi.Size, whose width is ABI-specific. [src: Dart 3.12.2 Size]
  • Raw pointers (*const T, *mut T) with documented contracts.
  • #[repr(C)] structs only when you intentionally freeze their field layout and every field is itself suitable for C. Rust's default representation gives no layout guarantee. [src: Rust Reference — C representation]
  • Opaque handles (rule 1) and pointer+length pairs for variable data (8.2).

String, Vec, slices/references with foreign lifetimes, traits, generics, and ordinary Rust structs stay behind the boundary.

Rule 3: errors are return codes, not panics or exceptions

Make every fallible operation return an i32 status code and write successful results through out-parameters. For this project, 0 means success, 1 is the cursor's normal end-of-results status, and negative values are stable error categories. Initialize every out-parameter to a harmless value before work begins, then publish the real result only after the operation succeeds. This prevents callers from mistaking stale pointers or IDs for valid output.

An unwinding Rust panic that reaches a non-unwinding extern "C" boundary causes a safe process abort; catch_unwind can catch only unwinding panics. Lesson 8.4 defines the exact boundary policy and its limits. [src: Rustonomicon — unwinding at ABI boundaries]

Rule 4: explicit ownership at the boundary

State, at the point each pointer is introduced: who allocates it, who may read or write it, its valid length, how long it remains valid, which operation releases it, and whether another thread may use it. A handle from minidb_open must be returned to minidb_close; a buffer the engine allocates must be freed by minidb_free_buf, not by the caller's free(). SQLite makes the same allocator-family rule explicit for memory returned by its allocation APIs. [src: SQLite memory-allocation routines]

⚠️ Null checks do not make lifecycle misuse safe

close(NULL) may be defined as a harmless no-op, but double-close, use-after-close, freeing a foreign pointer, or returning a buffer with the wrong length still violate the contract and can cause undefined behavior. Prefer API designs that make misuse harder, then test every detectable case; do not claim that an FFI can validate arbitrary foreign pointers.

Rule 5: design for a stable, narrow surface

  • Narrow: a handful of functions (open, close, CRUD, query, cursor, free, error) rather than mirroring every internal method. Each exported symbol is a promise.
  • Stable: preserve shipped names, signatures, layouts, ownership rules, and numeric meanings. Adding a symbol does not break an already-built old caller, but a new caller must still check the loaded ABI version before assuming that symbol exists (8.4).
  • Self-describing data, not C structs: pass documents and queries as serialized bytes (8.2), so adding a field doesn't change the ABI.

The handle and concurrency caveat from Phase 1

If boundary code turns a raw handle into &mut Db, two simultaneous calls on that handle can create aliased mutable references before the lock manager gets a chance to help. The honest v1 contract is therefore: one handle is single-caller and non-reentrant; the Dart wrapper confines it to one database isolate. Supporting concurrent calls later requires interior synchronization and a new, explicitly documented handle contract.

✅ Concept check

1. Why expose the database as an opaque handle rather than a #[repr(C)] struct?

Show answer
An opaque handle hides the layout, so internals can change without breaking the ABI and Rust keeps ownership. A repr(C) struct would freeze the layout and leak internals into every caller.

2. Does extern "C" make a Rust type safe to pass across the boundary?

Show answer
No. It only selects the C calling convention. ABI safety of each type is a separate property; String/Vec/generics/native structs must stay out of public signatures.

3. Why must every allocating call have a matching free in the same library?

Show answer
There is no stable cross-library deallocation contract. The library that allocated must provide and be used for the matching release operation.

4. What should v1's ABI say about concurrent calls on one handle?

Show answer
That it is not safe for concurrent/reentrant use; callers must serialize access or confine the handle to one isolate. Any concurrency must be a deliberate, documented engine design.
🦀 Rust just-in-time
  • unsafe fn means callers must uphold a documented safety contract; it does not make every operation in the function body automatically correct.
  • Use #[repr(C)] only for layouts intentionally exposed to C. Leave opaque handle internals in Rust's private/default representation.
  • Box::into_raw/Box::from_raw transfer and reclaim ownership; they do not protect against double-close or fabricated pointers. Review Rust Book ch. 20.
