8.1 — Designing a stable public C API

Goal: Learn the design rules that make a C ABI usable from many languages and stable over time — opaque handles, a narrow surface of ABI-safe types, return-code errors, and explicit ownership — so the boundary you freeze in this phase is one you can live with.

Why this matters to your DB

The C ABI is the only application binary interface that virtually every language can call, which is why it's the lingua franca for shipping a library. But "C ABI" is a calling convention, not a safety net: extern "C" "selects the target's C calling convention; it does not make arbitrary Rust types ABI-safe." Everything about correctness — what's a valid pointer, who frees memory, what happens on panic — is a contract you design and document. A good design makes the boundary small, boring, and hard to misuse; a bad one leaks Rust types and undefined behavior into every caller.

Rule 1: opaque handles, never exposed layout

The database is a complex Rust struct. Do not expose its fields or layout across the boundary — callers get an opaque pointer (a forward-declared, incomplete type) they pass back to every call. This keeps the struct's internals free to change without breaking the ABI, and keeps Rust ownership on the Rust side.

// Rust side: a handle is just a pointer the caller can't look inside.
pub struct Db { /* private: pager, btree, docstore, lockmgr ... */ }

#[unsafe(no_mangle)]
pub extern "C" fn minidb_open(path: *const c_char, out: *mut *mut Db) -> i32 { /* ... */ }
// C side sees: typedef struct Db Db;  (incomplete type — opaque)

Exported symbols use #[unsafe(no_mangle)] and a globally unique prefix (minidb_) so they don't collide with other libraries in the same process. [src: AGENTS.md — "Exported unmangled symbols use #[unsafe(no_mangle)], with globally unique prefixed names"]

Rule 2: only ABI-safe types cross the line

Keep Rust types that have no stable C representation out of public signatures. AGENTS is explicit: keep "String, Vec, references with unprovable borrow requirements, generics, and native-layout structs out of public C signatures." [src: AGENTS.md — Rust/FFI baseline] The vocabulary that is safe:

  • Fixed-width integers (i32, u64, …), bool via an integer.
  • Raw pointers (*const T, *mut T) with documented contracts.
  • #[repr(C)] structs whose fields are themselves ABI-safe (use sparingly).
  • Opaque handles (rule 1) and pointer+length pairs for variable data (8.2).

Rule 3: errors are return codes, not panics or exceptions

C has no exceptions, and letting a Rust panic cross a non-unwinding extern "C" boundary aborts the process. The convention: every fallible function returns an i32 status code (0 = OK, negative = a specific error), and produces results through out-parameters. Errors are data, not control flow. Panic handling is detailed in 8.4; the design rule here is simply: no failure mode should be a panic that escapes.

Rule 4: explicit ownership at the boundary

The single most common FFI bug is unclear ownership of memory. State, at the point each pointer is introduced: who allocates, who frees, the valid length, the lifetime, and the threading rules. AGENTS: "The API that allocates memory must provide the matching deallocation API." A handle from minidb_open must be returned to minidb_close; a buffer the engine allocates must be freed by a minidb_free, not by the caller's free(). [src: AGENTS.md — "The API that allocates memory must provide the matching deallocation API"]

⚠️ "Same allocator" is not a stable contract

Do not assume Rust and C/Dart share a heap allocator and that either side can free the other's memory. The correct claim is that "sharing an underlying allocator is not a stable deallocation contract." Always pair every allocating call with the matching free from the same library. [src: AGENTS.md — Rust/FFI baseline]

Rule 5: design for a stable, narrow surface

  • Narrow: a handful of functions (open, close, CRUD, query, cursor, free, error) rather than mirroring every internal method. Each exported symbol is a promise.
  • Stable: additive change is safe (new functions); changing a signature, a struct layout, or an error-code meaning breaks callers. Version the ABI deliberately (8.4).
  • Self-describing data, not C structs: pass documents and queries as serialized bytes (8.2), so adding a field doesn't change the ABI.

The handle and concurrency caveat from Phase 1

Phase 1 warned that boundary code creating &mut Db makes the handle non-reentrant. Now that Phase 7 exists, the ABI must say which calls may run concurrently. The simplest honest answer for v1: the handle is not safe for concurrent calls; callers must serialize access (or confine it to one isolate). If you later allow concurrent reads, that is a deliberate synchronization design inside the engine, documented in the ABI — not an accident of two threads sharing a pointer. [src: AGENTS.md — "A live opaque handle … is not concurrently or reentrantly usable when boundary code creates &mut Db"]

✅ Concept check

1. Why expose the database as an opaque handle rather than a #[repr(C)] struct?

Show answer
An opaque handle hides the layout, so internals can change without breaking the ABI and Rust keeps ownership. A repr(C) struct would freeze the layout and leak internals into every caller.

2. Does extern "C" make a Rust type safe to pass across the boundary?

Show answer
No. It only selects the C calling convention. ABI safety of each type is a separate property; String/Vec/generics/native structs must stay out of public signatures.

3. Why must every allocating call have a matching free in the same library?

Show answer
Rust and C/Dart do not have a stable shared allocator contract; freeing across allocators is undefined. The library that allocated must provide and be used for the free.

4. What should v1's ABI say about concurrent calls on one handle?

Show answer
That it is not safe for concurrent/reentrant use; callers must serialize access or confine the handle to one isolate. Any concurrency must be a deliberate, documented engine design.
