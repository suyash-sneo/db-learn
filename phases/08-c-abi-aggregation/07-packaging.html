8.7 — Flutter build-hook packaging

Goal: Package the Rust library with Flutter 3.44's recommended package_ffi build-hook flow, understand when the legacy plugin flow is still needed, and verify architecture, loading, signing, exported symbols, and Android 16 KB compatibility on real artifacts.
📂 Where to read

This lesson is pinned to the Flutter documentation for Flutter 3.44.0, last updated June 8, 2026. Since Flutter 3.38, the recommended new-project flow is flutter create --template=package_ffi with Dart build hooks. [src: Flutter 3.44 — bind to native code]

Why this matters to your DB

A correct Dart typedef is useless if the app bundle lacks a library for the device architecture, the code asset points at the wrong artifact, the linker removed the symbols, or Apple cannot validate the embedded code. Packaging is part of the ABI delivery system, not a final copy step.

The current vehicle: package_ffi + a build hook

# Flutter 3.44.0
flutter create --template=package_ffi minidb_flutter

The generated package contains a Dart API under lib/, native source/header files under src/, hook/build.dart, ffigen configuration, and pubspec.yaml. The Flutter/Dart tool invokes the build hook, bundles the resulting code asset, and the generated @Native functions resolve against that asset at runtime; application code does not hard-code Android/iOS library names. [src: Flutter 3.44 — package contents; code-asset resolution]

Adapt the hook for Rust

The stock template demonstrates C compilation with package:native_toolchain_c. A Rust package replaces or extends that hook so it:

  1. reads the hook's target OS, architecture, and build mode;
  2. selects the matching installed Rust target and platform linker/SDK;
  3. invokes Cargo for the cdylib (or downloads a trusted prebuilt artifact);
  4. places the produced library in the hook's shared output directory; and
  5. registers a CodeAsset with the correct asset name and link mode.

Dart build hooks are scripts in hook/; they can compile or download native assets, including Rust libraries, and are invoked automatically by the Dart SDK. [src: Dart 3.12.2 — build hooks] The hook does not install Rust targets, the Android NDK, or Apple SDKs for you. Fail with a clear message when a required toolchain is absent, and make the selected target visible in build logs.

Know the legacy path, but do not teach it as the default

The older plugin_ffi template uses OS-specific Gradle/CMake and CocoaPods files. Flutter 3.44 labels it legacy, but still documents it for cases that need the Flutter Plugin API, static linking on Apple platforms, or a Google Play services runtime on Android. [src: Flutter 3.44 — legacy FFI plugin]

DynamicLibrary.executable() exposes symbols from the running executable; process() exposes global process symbols; open(path) loads a library file. [src: Dart 3.12.2 DynamicLibrary]

Platform artifacts still differ

  • Android: dynamic native code is packaged as ELF .so files for the supported device architectures. The app bundle/install process selects a compatible artifact; a missing device ABI prevents successful installation or loading.
  • iOS: the chosen code asset may be linked/bundled according to the hook and Xcode build. Dynamically embedded frameworks must be signed; statically linked symbols use process/executable resolution only in the legacy path.
  • Do not infer packaging from the filename alone: the code asset's link mode, architecture, deployment target, and bundle placement must agree with the final app.
⚠️ Verify the final packaged binary
  • Code signing (iOS): an embedded dynamic framework must be correctly placed and signed by the Xcode packaging flow. Flutter's legacy guide instructs “Embed & Sign” for a compiled dynamic framework. [src: Flutter 3.44 — iOS dynamic library]
  • 16 KB pages (Android): since November 1, 2025, new apps and updates submitted to Google Play that target Android 15 or later must support 16 KB page sizes on 64-bit devices. Apps with native code must rebuild compatible libraries and test the packaged ELF alignment. [src: Android Developers — 16 KB page sizes, updated May 28, 2026]
  • Symbol survival: aggressive stripping/LTO can remove your exported symbols if the linker considers them unused. Inspect symbols in the packaged artifact, not only Cargo's intermediate output.
  • Architecture and deployment target: inspect every slice/artifact and test at least one real release-mode build for each supported platform family.

Verify the whole chain, not just the build

The release check is: build hook receives target → Cargo emits the matching library → hook registers the code asset → Flutter bundles it → inspect the APK/AAB or iOS app/framework → install/launch → generated @Native binding resolves → ABI version matches → open/insert/find/free/close round-trips. Run an Android 16 KB emulator/device test and an iOS release-mode device test when those are claimed as supported. A host-side unit test proves none of this.

✅ Concept check

1. What is Flutter 3.44's recommended starting template for a new direct-FFI package?

Show answer
package_ffi, whose hook/build.dart builds/registers a code asset and whose generated @Native functions resolve against it.

2. What must a Rust-aware build hook do beyond the stock C example?

Show answer
Map the hook target to a Rust target/toolchain, invoke Cargo or select a trusted prebuilt, place the artifact in the shared output, and register the correct CodeAsset/link mode.

3. When is the legacy plugin_ffi path still relevant?

Show answer
When you need the Flutter Plugin API, static Apple linking, a Google Play services runtime, or another OS-specific integration not handled by the normal code-asset flow.

4. What is the current Android 16 KB requirement?

Show answer
Since November 1, 2025, new Play submissions and updates targeting Android 15+ must support 16 KB page sizes on 64-bit devices; native libraries must be rebuilt/aligned and tested accordingly.
🦀 Rust just-in-time
  • cdylib asks rustc for a dynamic system library intended for foreign consumers; the final filename and linker behavior remain target-specific. [src: Rust Reference — linkage]
  • rustup target add installs Rust standard-library support for a target, but a mobile build still needs the platform linker, SDK/NDK, deployment settings, and signing tools.
  • The Dart hook is the orchestrator. Keep target selection and Cargo arguments explicit and logged; do not hide cross-compilation assumptions in a shell command that only works on one machine.
