8.7 — Packaging for iOS/Android

Goal: Understand how the compiled Rust library actually ships inside a Flutter app on both mobile platforms — the FFI plugin structure, the per-platform native build, and the library-loading and signing details that differ by target — so the binding from 8.5 has something to load at runtime.
📂 Where to read

Flutter documents native packaging under "developing packages & plugins" (FFI plugins) and the platform interop guides. Packaging mechanics evolve across Flutter versions, so confirm specifics against the docs for your version. [src: docs.flutter.dev — developing packages]

Why this matters to your DB

An FFI binding is useless if the operating system can't find and load the native library. Each platform has its own build system, library format, loading rule, and signing requirement. Flutter's FFI plugin packaging exists to automate this — but you still need to understand what it produces, because cross-compiling a Rust library for ARM phones and bundling it correctly is where "works on my laptop" meets reality.

FFI plugins: the packaging vehicle

An FFI package is "a specialized Dart package that enables calling native code using dart:ffi," created with a Flutter FFI plugin/package template. The defining difference from a regular plugin: FFI plugins "do not use method channels" — they call native code directly through dart:ffi. [src: docs.flutter.dev] The package bundles native code compiled per platform; a typical layout has the Dart API in lib/, the native source in src/, and a build hook that compiles it. ⚠️VERIFY the exact template name, directory layout, and build-hook mechanism against the Flutter version you target — these have changed across releases (e.g. the native build-hook/native-assets approach is newer).

For a Rust engine, the native build cross-compiles the crate for each target triple (e.g. aarch64-linux-android, aarch64-apple-ios) and produces the platform's library form. The plugin's build hook wires that into Gradle/CocoaPods so a normal flutter build picks it up.

Loading differs by platform

From 8.5, the Dart side opens the library — but how depends on the target:

  • Android: a separate .so is shipped in the APK/AAB; DynamicLibrary.open('libminidb.so') loads it by name.
  • iOS: symbols are commonly statically linked into the app binary; you then use DynamicLibrary.process() (or .executable()) to look up symbols already in the process rather than opening a separate file. ⚠️VERIFY the exact open/process choice for your linkage and Flutter version.
  • Library naming differs everywhere: lib*.so (Linux/Android), *.dylib/framework (Apple), *.dll (Windows). Don't hard-code one name for all targets — branch on the platform.

This is exactly AGENTS's warning that platform assumptions must be stated: "C ABIs, symbol display, dynamic-library names, linking, fsync behavior, and Flutter packaging differ by target." [src: AGENTS.md — Rust/FFI baseline]

⚠️ Signing, stripping, and ABI coverage will bite you
  • Code signing (iOS): all native code in an app must be signed; an unsigned or improperly embedded library fails at install/launch, not at build. Let the CocoaPods/Xcode flow embed and sign it.
  • 16 KB page sizes / ABI coverage (Android): you must ship a .so for every ABI you support, built with the right toolchain/NDK; a missing ABI means the app crashes on those devices only. ⚠️VERIFY current Android ABI and page-size requirements for your target API level.
  • Symbol survival: aggressive stripping/LTO can remove your exported symbols if the linker thinks they're unused. Keep them exported (the #[unsafe(no_mangle)] + prefix from 8.1) and verify they're present in the shipped binary.
  • Size: a Rust static can be large; strip debug info for release and measure the added app size.

Verify the whole chain, not just the build

"It compiled" is not "it loads and runs on a device." The end-to-end check: cross-compile for the device's ABI → bundle via the plugin → install on a real device (or device-accurate simulator) → DynamicLibrary opens → a symbol lookup succeeds → a real minidb_open/insert/find round-trips. Each arrow is a place it can silently fail, and on mobile the failure often appears only at runtime on hardware.

✅ Concept check

1. How does an FFI plugin differ from a regular Flutter plugin?

Show answer
An FFI plugin calls native code directly via dart:ffi and bundles per-platform compiled native code; it does not use method channels.

2. Which build systems compile the native code on Android and iOS?

Show answer
Android uses CMake integrated with Gradle; iOS uses CocoaPods. The Rust crate is cross-compiled per target triple and wired in via the plugin's build hook.

3. Why might library loading use DynamicLibrary.open on Android but DynamicLibrary.process on iOS?

Show answer
Android ships a separate .so opened by name; iOS commonly links the library statically into the app binary, so symbols are looked up in the already-loaded process rather than a separate file.

4. Name two packaging failures that appear only at runtime on a device.

Show answer
Any two of: missing/unsigned native library (iOS signing), a missing ABI .so (Android), or stripped-away exported symbols — none of which necessarily fail the build.
