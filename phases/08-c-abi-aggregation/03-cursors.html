8.3 — Cursors: delivery, execution & lifetime

Goal: Return many query results through an opaque cursor with next, while distinguishing FFI delivery from query execution and specifying ownership, end-of-stream, database lifetime, mutation, and cleanup without leaving use-after-free traps.

Why this matters to your DB

A query can match thousands of documents. One API can serialize the whole result into one buffer; another can return a cursor that yields one document or batch per call. The cursor keeps each FFI output allocation small and lets the caller stop early. It does not automatically make the query engine lazy: Phase 6's teaching signature returns a Vec, so a cursor wrapped around that vector still materializes all matches inside Rust. Bounded end-to-end memory requires an incremental scan/index executor as well as a cursor-shaped ABI.

The cursor as an opaque handle

A cursor is another opaque handle (8.1): the caller opens it from a query, calls next in a loop, and closes it. Each next yields one document as an engine-allocated (ptr,len) buffer (8.2) or signals "no more."

// Opaque cursor handle; internals (the iterator state) stay in Rust.
pub struct Cursor { /* private: owned execution state + shared engine owner */ }

#[unsafe(no_mangle)]
pub unsafe extern "C" fn minidb_find_cursor(
    db: *mut Db, filter: *const u8, len: usize,
    out: *mut *mut Cursor) -> i32 { /* ... */ }

// Returns: 0 = produced a document (out_*), 1 = end of results, <0 = error.
#[unsafe(no_mangle)]
pub unsafe extern "C" fn minidb_cursor_next(
    cur: *mut Cursor,
    out: *mut *mut u8, out_len: *mut usize) -> i32 { /* ... */ }

#[unsafe(no_mangle)]
pub unsafe extern "C" fn minidb_cursor_close(cur: *mut Cursor) -> i32 { /* release once */ }

Note the three-valued return: a separate "end of results" code (1) is cleaner than overloading null or an error. The per-document buffer from next follows the 8.2 output rule — the caller frees each one with minidb_free_buf. On both end and error, next leaves (out, out_len) as (NULL, 0). This curriculum does not use a cursor-owned reusable output buffer because that would give the bytes a different lifetime from every other engine output. A separate minidb_cursor_last_error copies diagnostic detail from the cursor handle.

Do not encode the parent lifetime as a raw borrow

A cursor that merely stores a borrowed pointer to Db becomes dangling when the caller closes the database first. Writing "close cursors first" in documentation does not make that API robust. Choose one of these ownership designs:

  • Shared ownership: the database handle and each cursor own a reference-counted engine/session object. Closing the database handle releases only that handle; the underlying engine remains alive until the last cursor closes.
  • Busy close: the database tracks child cursors and minidb_close returns BUSY while any remain. SQLite's ordinary close similarly returns BUSY if unfinalized statements or unfinished backups still exist. [src: SQLite close contract]

The Phase 8 build chooses shared ownership. A copied, double-closed, or completely fabricated cursor pointer is still invalid foreign input, but closing the original Db before a valid cursor no longer creates a use-after-free.

  • Each cursor handle is single-threaded and non-reentrant: do not call next on the same cursor from two threads (8.1's concurrency rule applies per handle).
  • After MINIDB_END, later next calls keep returning END with null/zero output until the cursor is closed.
  • cursor_close(NULL) is a no-op; every non-null cursor must be closed exactly once.
⚠️ Define what mutation means to an open cursor

A materialized v1 cursor freezes its result documents when the cursor is created; later writes do not change that owned result list. A future lazy cursor needs a real policy: hold a Phase 7 read transaction/snapshot, or detect a generation change and return a documented invalidation error. "Whatever the underlying iterator happens to do" is not a policy.

Delivery shape versus execution shape

DesignWhat it boundsTrade-off
One serialized result bufferNothing until the whole result is readySimplest lifecycle; one potentially large allocation.
Cursor over a materialized VecEach FFI output and Dart copyCan stop Dart delivery early, but Rust already holds all matches.
Cursor over a lazy executorFFI output and query execution, except blocking operatorsNeeds page/index scan state plus snapshot/invalidation rules.

Batching is a separate knob: one next call can return several documents to reduce FFI calls while keeping a maximum batch size. It does not change whether the underlying query was materialized. Start with one document per call; add batching only with explicit byte/count limits and ownership tests.

Mapping to Phase 6

For v1, the planner (6.7) and evaluator (6.6) produce the same result set as find, and the cursor owns that materialized set plus its current position. A later lazy version can replace the private execution state without changing the C signatures. That is the value of the opaque handle: execution strategy remains an implementation detail.

✅ Concept check

1. What does a three-valued return from cursor_next encode, and why is it better than overloading null?

Show answer
0 = a document was produced, 1 = end of results, <0 = error. Distinct codes separate "no more data" from "something failed," which a null pointer alone can't express unambiguously.

2. Why should a cursor own shared engine state instead of a raw borrow of Db?

Show answer
So closing the original database handle does not leave a valid cursor pointing at freed memory. Shared ownership keeps the engine alive until the last cursor closes.

3. What is cursor invalidation and how can you handle it?

Show answer
Writing through the Db while a cursor is open can change what a lazy iterator would observe. A materialized cursor freezes its owned results; a lazy cursor needs a snapshot or explicit invalidation error.

4. Does a cursor automatically bound query-engine memory?

Show answer
No. A cursor over a materialized Vec bounds each FFI result allocation but still holds every match in Rust. End-to-end bounded memory requires a lazy executor.

5. When is one big result buffer preferable to a cursor?

Show answer
When result sets are small: it gives a trivial ABI with one allocation and no lifetime/invalidation contracts. Cursors win when results are large or you want to stop early.
🦀 Rust just-in-time
  • A materialized cursor can own a Vec<Vec<u8>> plus an index or an owning iterator such as IntoIter; it does not need a self-referential borrow into Db.
  • Arc<T> provides shared ownership, not automatic thread safety. The handle may still be single-caller unless T and all access paths are deliberately synchronized. [src: std::sync::Arc]
  • Drop runs when the Rust cursor owner is reclaimed, but the C caller must still invoke the matching close exactly once. See Rust Book ch. 13.
