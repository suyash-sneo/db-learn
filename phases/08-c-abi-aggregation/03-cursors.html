8.3 — Cursors / iteration across FFI

Goal: Return a query's many results across the boundary without materializing everything at once, using an opaque cursor handle with a next call — and get its ownership, lifetime, and invalidation rules right.

Why this matters to your DB

A query can match thousands of documents. Two ways to return them across the FFI: serialize the whole result set into one giant buffer, or hand back a cursor the caller pulls one document at a time. The single buffer is simple but forces the entire result into memory at once — bad on a phone. A cursor streams results, bounding memory, but introduces a stateful object whose lifetime must be managed across the boundary. This is the same iterator concept you know from Rust, made ABI-safe.

The cursor as an opaque handle

A cursor is another opaque handle (8.1): the caller opens it from a query, calls next in a loop, and closes it. Each next yields one document as an engine-allocated (ptr,len) buffer (8.2) or signals "no more."

// Opaque cursor handle; internals (the iterator state) stay in Rust.
pub struct Cursor { /* private: result iterator over the docstore */ }

#[unsafe(no_mangle)]
pub extern "C" fn minidb_find_cursor(
    db: *mut Db, filter: *const u8, len: usize,
    out: *mut *mut Cursor) -> i32 { /* ... */ }

// Returns: 0 = produced a document (out_*), 1 = end of results, <0 = error.
#[unsafe(no_mangle)]
pub extern "C" fn minidb_cursor_next(
    cur: *mut Cursor,
    out: *mut *mut u8, out_len: *mut usize) -> i32 { /* ... */ }

#[unsafe(no_mangle)]
pub extern "C" fn minidb_cursor_close(cur: *mut Cursor) { /* drop it */ }

Note the three-valued return: a separate "end of results" code (1) is cleaner than overloading null or an error. The per-document buffer from next follows the 8.2 output rule — the caller frees each one with minidb_free_buf (or the cursor owns a reusable buffer; pick one and document it).

Lifetime: the cursor borrows the database

A cursor iterates over engine state, so it is only valid while the Db it came from is open. In Rust this would be a borrow the compiler enforces; across the FFI the borrow checker is gone, so the lifetime relationship becomes a documented contract:

  • A cursor must be closed before its Db is closed. Closing the Db first leaves a dangling cursor — undefined behavior if used.
  • Each cursor handle is single-threaded and non-reentrant: do not call next on the same cursor from two threads (8.1's concurrency rule applies per handle).
⚠️ Mutation can invalidate a cursor

If the caller inserts, updates, or deletes through the same Db while a cursor is open, the cursor's view may become inconsistent — the classic "iterator invalidation" problem, now across a boundary the compiler can't police. Decide and document a policy: (a) the cursor reads a stable snapshot taken at open (the snapshot-isolation idea from 7.3), so later writes don't affect it; or (b) writing while a cursor is open is forbidden / returns an error. Option (a) is friendlier; option (b) is simpler. Whichever you choose, state it in the ABI — silence here is a latent crash.

Why not just return one big buffer?

ApproachProsCons
One serialized result buffertrivial ABI; one alloc, one free; no lifetime jugglingwhole result set in memory at once; large allocations on a constrained device
Cursor + nextbounded memory; can stop early; streamsstateful handle; lifetime & invalidation contracts; more calls across the boundary

A pragmatic middle ground is batching: next returns N documents per call to amortize boundary-crossing cost while still bounding memory. For a first implementation, a simple one-document cursor is clear and correct; batching is an optimization to add later with its own tests.

Mapping to Phase 6

The cursor wraps your Phase 6 query path: the planner (6.7) picks an access path, the evaluator (6.6) filters candidates, and the cursor yields surviving documents one at a time. The cursor is purely a delivery mechanism over machinery you already built — it adds no query semantics, only iteration and ABI plumbing.

✅ Concept check

1. What does a three-valued return from cursor_next encode, and why is it better than overloading null?

Show answer
0 = a document was produced, 1 = end of results, <0 = error. Distinct codes separate "no more data" from "something failed," which a null pointer alone can't express unambiguously.

2. Why is the cursor's lifetime relative to the Db a documented contract rather than a compiler-enforced borrow?

Show answer
Across the FFI there is no borrow checker, so the rule "close the cursor before the Db" must be stated and upheld by the caller; violating it dangles the cursor.

3. What is cursor invalidation and how can you handle it?

Show answer
Writing through the Db while a cursor is open can make its view inconsistent. Handle it by reading from a stable snapshot taken at open, or by forbidding writes while a cursor is open — and documenting the choice.

4. When is one big result buffer preferable to a cursor?

Show answer
When result sets are small: it gives a trivial ABI with one allocation and no lifetime/invalidation contracts. Cursors win when results are large or you want to stop early.
