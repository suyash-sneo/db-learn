8.4 — Error reporting, versioning, ABI stability

Goal: Give every ordinary failure a stable code and retrievable detail, contain unwinding panics as far as Rust permits, poison state that may be inconsistent, and version the binary contract so callers can reject incompatible libraries.

Why this matters to your DB

The previous lessons defined the happy path. This one defines ordinary errors, internal panics, and incompatible change. They are different: an invalid document is expected data, a panic is an internal failure that may invalidate the handle, and an ABI mismatch means the caller and library disagree before useful work begins.

Error reporting: stable codes + copied detail

Every fallible function returns an i32: 0 for success, negative values for specific error categories (not-found, bad-document, I/O, busy, poisoned, …); cursor end is the positive non-error value 1. Once shipped, a numeric value keeps its meaning. Callers must still treat unknown negative values as failure so a later library can add categories without old wrappers accidentally treating them as success.

// Stable numeric categories — never renumber these once shipped.
pub const MINIDB_OK: i32 = 0;
pub const MINIDB_END: i32 = 1;
pub const MINIDB_ERR_NOT_FOUND: i32 = -1;
pub const MINIDB_ERR_BAD_DOC: i32 = -2;
pub const MINIDB_ERR_IO: i32 = -3;
pub const MINIDB_ERR_POISONED: i32 = -5;
pub const MINIDB_ERR_PANIC: i32 = -99;

// Copy this handle's latest detail into a normal engine-owned output buffer.
#[unsafe(no_mangle)]
pub unsafe extern "C" fn minidb_last_error(
    db: *mut Db,
    out: *mut *mut u8,
    out_len: *mut usize
) -> i32 { /* boundary logic */ }

The detail belongs to the database handle, not a process-global or OS-thread-local slot, and the accessor returns a copied engine allocation using the normal minidb_free_buf contract. That avoids a borrowed string whose lifetime changes on the next call and avoids requiring two Dart FFI calls to run on the same native thread. The string is diagnostic, may change between versions, and must not be parsed for program logic. A matching minidb_cursor_last_error(cursor, ...) keeps cursor diagnostics available even after the original database handle closes. An open failure has no usable handle, so v1 guarantees only its stable numeric code; a later additive API can return richer open diagnostics.

SQLite is a useful comparison: its error code and message accessors are associated with a database connection, and the docs warn that simultaneous use can make results unpredictable. [src: SQLite error-code/message contract]

Panics must not cross the boundary

The exact Rust behavior is narrower than “catch every crash”:

  • If an unwind reaches a non-unwinding extern "C" boundary, Rust safely aborts the process. [src: Rustonomicon — panic stopped at ABI boundary]
  • extern "C-unwind" is the explicit variant for cross-language unwinding; this database API does not expose it. [src: Rust Reference — unwind ABI variants]
  • catch_unwind catches only unwinding Rust panics. It does not catch an aborting panic, invalid-pointer undefined behavior, segmentation faults, or arbitrary foreign exceptions. A configured panic hook runs before the panic is caught. [src: std::panic::catch_unwind]

The boundary pattern is therefore conditional: make the catch_unwind wrapper the outermost Rust operation in every nontrivial export, compile the tested profiles with unwinding enabled, and map a caught panic to MINIDB_ERR_PANIC. This contains that class of panic; it is not a general process-crash sandbox.

#[unsafe(no_mangle)]
pub unsafe extern "C" fn minidb_insert(db: *mut Db, doc: *const u8, len: usize, out_id: *mut i64) -> i32 {
    let r = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| -> Result<(), i32> {
        // ... validate pointers, do transactional work ...
        Ok(())
    }));
    match r {
        Ok(Ok(())) => MINIDB_OK,
        Ok(Err(code)) => code,
        Err(_panic) => MINIDB_ERR_PANIC, // caught — do NOT let it unwind into C
    }
}
⚠️ Catching a panic ≠ a still-valid handle

AssertUnwindSafe only overrides the compiler's unwind-safety check; it does not prove the database survived logically intact. A panic mid-mutation may leave in-memory state inconsistent even if Phase 5 prevents an uncommitted root from becoming durable. The build therefore poisons the handle after any caught panic: later operations return MINIDB_ERR_POISONED, while last_error and close remain available. Reopen the file to recover from its last committed state.

ABI versioning: change without breaking

Once callers link against your symbols and header, the binary interface is a contract. Classify changes:

ChangeCompatible?
Add a new functionDoes not break an old binary; new callers must verify the loaded ABI before looking it up.
Add a new negative error codeCompatible only because callers are required to treat every unknown negative value as failure.
Change a function's signature/parametersNo — breaks every caller compiled against the old header.
Change a #[repr(C)] struct's layoutNo — silent memory misinterpretation.
Reuse/renumber an existing error codeNo — changes the meaning of a shipped value.

Practical discipline:

  • Expose minidb_abi_version() -> u32. Check it immediately after loading and before looking up optional/newer symbols.
  • Keep the ABI version separate from the package/library release version. A release may fix internals without changing the ABI; an incompatible ABI change must change the ABI major and the library's public versioning policy.
  • Prefer additive evolution; pass extensible data as serialized bytes (8.2) so new fields don't touch the ABI.
  • Keep the exported symbol set minimal and prefixed (8.1) — fewer promises to keep.

Symbol & layout stability

The ABI includes exported symbol names, calling conventions, parameter/return types, integer widths, and the memory layout of any exposed #[repr(C)] type. It also has behavioral contracts that compilers cannot check: pointer ownership, valid lengths, error meanings, threading, and lifetimes. Opaque handles keep internal layouts outside that promise; generated headers and symbol inspection check part of it, while C/Dart smoke tests check that consumers agree.

✅ Concept check

1. What happens if an unwinding Rust panic reaches a default extern "C" boundary?

Show answer
Rust safely aborts the process. catch_unwind can contain an unwinding panic before it reaches the boundary, but cannot catch aborting panics or arbitrary crashes.

2. After catching a panic, why can't you just return and keep using the handle?

Show answer
The handle may be logically inconsistent even if uncommitted disk state was not published. Poison it, reject normal operations, permit diagnostics/close, and reopen the file from committed state.

3. Which of these is an ABI-compatible change: adding a function, or changing a struct's layout?

Show answer
Adding a function is additive and compatible. Changing a repr(C) struct's layout breaks callers via silent memory misinterpretation.

4. Why keep error codes stable but allow the detail strings to change?

Show answer
Codes are the machine-readable contract callers branch on, so renumbering changes meaning. Strings are human-facing detail, not part of the programmatic contract.
