8.4 — Error reporting, versioning, ABI stability

Goal: Make failures, panics, and change all safe at the boundary: report errors as codes plus optional detail, stop Rust panics from crossing into C as undefined behavior, and version the ABI so you can evolve the library without breaking existing callers.

Why this matters to your DB

The previous lessons defined the happy path. This one defines what happens when things go wrong or change — which is where FFIs actually break. A panic that escapes can abort the host app; an error with no detail is undebuggable; a careless signature change silently corrupts every caller compiled against the old header. Each needs an explicit, conservative policy.

Error reporting: codes + retrievable detail

Every fallible function returns an i32: 0 for success, negative values for specific error categories (a small stable enum: not-found, bad-document, io, busy, …). Codes are stable and language-neutral. For human-readable detail, the common pattern is a last-error accessor the caller can query after a failure:

// Stable numeric categories — never renumber these once shipped.
pub const MINIDB_OK: i32 = 0;
pub const MINIDB_ERR_NOT_FOUND: i32 = -1;
pub const MINIDB_ERR_BAD_DOC: i32 = -2;
pub const MINIDB_ERR_IO: i32 = -3;
pub const MINIDB_ERR_PANIC: i32 = -99;

// Detail for the most recent failure on this thread; borrowed, valid until the next call.
#[unsafe(no_mangle)]
pub extern "C" fn minidb_last_error() -> *const c_char { /* thread-local CString ptr */ }

Keep the detail string thread-local so two threads don't clobber each other's last error, and document its lifetime (valid until the next call on that thread). The numeric codes — not the strings — are the stable contract; never reuse or renumber a shipped code.

Panics must not cross the boundary

This is the highest-stakes rule. The precise behaviors (all from the curriculum's FFI baseline):

  • "If an unwinding Rust panic reaches non-unwinding extern "C", Rust aborts the process." So an unguarded panic in your ABI function kills the host app.
  • "extern "C-unwind" permits unwinding" — a different ABI you opt into deliberately, not the default.
  • "A foreign exception entering a non-unwind Rust ABI is undefined behavior."
  • "catch_unwind catches only unwinding Rust panics, not panic = "abort" or arbitrary crashes."

[src: AGENTS.md — Rust/FFI panic baseline]

The practical pattern: wrap each extern "C" body in catch_unwind, convert a caught panic into MINIDB_ERR_PANIC, and return normally — so an internal bug becomes an error code instead of a process abort.

#[unsafe(no_mangle)]
pub extern "C" fn minidb_insert(db: *mut Db, doc: *const u8, len: usize, out_id: *mut i64) -> i32 {
    let r = std::panic::catch_unwind(|| {
        // ... real work; may return Err(code) ...
    });
    match r {
        Ok(Ok(())) => MINIDB_OK,
        Ok(Err(code)) => code,
        Err(_panic) => MINIDB_ERR_PANIC, // caught — do NOT let it unwind into C
    }
}
⚠️ Catching a panic ≠ a still-valid handle

"Catching a panic does not prove a database handle remains logically valid." A panic mid-mutation may leave the engine half-updated. After a caught panic, don't pretend nothing happened: "Publish state only after successful work, use transactional mutation, or poison the handle after a caught panic." [src: AGENTS.md — Rust/FFI baseline] The cleanest fit with your Phase 5/7 work is transactional mutation: a caught panic means the transaction did not commit, so committed state is untouched. If you can't guarantee that, mark the handle poisoned and reject further calls.

ABI versioning: change without breaking

Once callers link against your symbols and header, the binary interface is a contract. Classify changes:

ChangeCompatible?
Add a new functionYes — additive; old callers keep working.
Add a new error codeMostly — only if callers treat "any negative" as failure, which you should document.
Change a function's signature/parametersNo — breaks every caller compiled against the old header.
Change a #[repr(C)] struct's layoutNo — silent memory misinterpretation.
Reuse/renumber an existing error codeNo — changes the meaning of a shipped value.

Practical discipline:

  • Expose a version: minidb_abi_version() -> u32 and/or minidb_version_string(), so a caller can check what it loaded at runtime.
  • Follow semantic versioning for the library: bump major on any incompatible ABI change.
  • Prefer additive evolution; pass extensible data as serialized bytes (8.2) so new fields don't touch the ABI.
  • Keep the exported symbol set minimal and prefixed (8.1) — fewer promises to keep.

Symbol & layout stability

The ABI is the exact set of symbol names, their signatures, and the memory layout of any #[repr(C)] type that crosses it. #[unsafe(no_mangle)] fixes the names; the prefix keeps them unique; opaque handles (8.1) keep layouts private so most of the engine can change freely. The only layouts you must keep stable are the few repr(C) types you deliberately expose — keep that set as small as possible.

✅ Concept check

1. What happens if an unwinding Rust panic reaches a default extern "C" boundary?

Show answer
Rust aborts the process. You must catch panics at the boundary (catch_unwind) and convert them to an error code instead.

2. After catching a panic, why can't you just return and keep using the handle?

Show answer
The handle may be logically inconsistent (half-updated). Use transactional mutation so an uncommitted change is discarded, publish state only on success, or poison the handle and reject further calls.

3. Which of these is an ABI-compatible change: adding a function, or changing a struct's layout?

Show answer
Adding a function is additive and compatible. Changing a repr(C) struct's layout breaks callers via silent memory misinterpretation.

4. Why keep error codes stable but allow the detail strings to change?

Show answer
Codes are the machine-readable contract callers branch on, so renumbering changes meaning. Strings are human-facing detail, not part of the programmatic contract.
