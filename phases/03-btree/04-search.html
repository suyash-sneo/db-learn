3.4 — Search & traversal

Goal: Implement point lookup and range scan over the B+tree: descend from the root through internal nodes using binary search on separator keys, land on a leaf, then for ranges follow the leaf links.

Why this matters to your DB

Search is the operation every read goes through and the skeleton that insert and delete reuse (both first find where a key belongs). Getting the separator-key comparison convention exactly right is the difference between a correct index and one that silently returns wrong rows.

Point lookup: descend, choosing a child at each internal node

Starting at the root:

  1. Read the node. If it's a leaf, binary-search its sorted entries for the key; found ⇒ return its value, not found ⇒ key absent.
  2. If it's internal, binary-search the separator keys to pick the correct child pointer, load that child page, and repeat.

Each step reads exactly one page, so a lookup reads height pages — the shallow tree pays off here.

The separator convention (pin it down once)

An internal node with separators [k₀, k₁, …, kₙ₋₁] has children [c₀, c₁, …, cₙ]. A common convention: child cᵢ holds keys < kᵢ, and child cₙ (the rightmost) holds keys ≥ kₙ₋₁. To route a search key K: find the first separator kᵢ with K < kᵢ and descend into cᵢ; if none, descend into the rightmost child.

⚠️ Off-by-one here corrupts every query

Whether a separator means "<" or "≤", and whether equal keys go left or right, must be one fixed rule used identically in search, insert, and split. If split copies up a separator using one convention and search routes with another, lookups will miss keys that exist. Write the convention as a comment at the top of the node module and never deviate.

Range scan: descend once, then walk the leaf links

For "all keys in [lo, hi]": point-search for lo to find the starting leaf and position, then emit entries in order, following each leaf's next-leaf pointer to the sibling, stopping when a key exceeds hi or the chain ends. This is why B+tree leaves are linked (3.2) — the scan never returns to the root.

Binary search within a node

Within a node, entries are sorted, so use binary search, not a linear scan — it turns per-node cost from O(entries) to O(log entries). Rust's slices give you binary_search_by which returns Ok(index) on a match or Err(insertion_point) otherwise — the Err case is exactly the child index / insert position you want. ✅ Concept check

1. How many pages does a point lookup read, and why?

Show answer
One per level (the height), because each descent step reads exactly one node page.

2. Why must the separator convention be identical in search and split?

Show answer
If they disagree on "<" vs "≤" or where equal keys go, search can route past keys that split placed elsewhere, missing existing keys.

3. Why don't range scans return to the root for each result?

Show answer
Leaves are linked in key order, so the scan follows next-leaf pointers after a single initial descent.

4. What does binary_search_by's Err value give you?

Show answer
The insertion point — the index where the key would go — which is the child index to descend into or the slot to insert at.
