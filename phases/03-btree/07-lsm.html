3.7 — B-tree vs LSM, revisited

Goal: Understand the LSM-tree alternative, the read/write/space trade-offs between B+trees and LSMs, and articulate why this project deliberately chose a B+tree.

Why this matters to your DB

B+trees and LSM trees are two major families of ordered storage structures. Neither dominates every workload. You should be able to identify where each family moves its costs, then state which constraints—not slogans—drive this curriculum's choice.

The LSM idea in one paragraph

In the classic LSM design, incoming updates are buffered in a memory-resident component and made recoverable by a log. Full memory components are written as sorted on-disk runs. Reads may consult multiple components, while background or scheduled merges reorganize runs and eventually discard overwritten values and deletion markers (“tombstones”). Concrete LSM engines differ in levels, tiering, Bloom filters, and compaction scheduling. [src: O'Neil et al., “The Log-Structured Merge-Tree” (1996)]

The trade-offs (the "amplification" lens)

B+treeLSM tree
WritesModify page-sized nodes; splits and logging can rewrite more than the changed keyBuffer and flush sorted runs; compaction may rewrite the same data multiple times
Point readsFollow one root-to-leaf path, subject to cachingMay probe memory plus multiple runs; indexes/Bloom filters reduce unnecessary probes
Range scansWalk ordered leaves in this project's layoutMerge ordered results from the relevant runs
SpacePartially filled pages and old versions/logs can amplify spaceObsolete versions and tombstones remain until compaction
MaintenanceMay still need checkpointing, vacuum, or page reclamationCompaction policy is central and consumes I/O/CPU

The RUM conjecture frames access methods in terms of read, update, and memory overhead: reducing two generally increases the third. It is a design lens, not a benchmark result proving one family always wins. [src: Athanassoulis et al., “Designing Access Methods: The RUM Conjecture” (2016)] Write amplification in particular is not inherently lower for every LSM: compaction strategy, update distribution, run sizing, and workload determine how often bytes are rewritten.

Who uses which

  • B+tree examples: SQLite and LMDB are the source teachers used in this phase.
  • LSM examples: RocksDB and LevelDB document memtables, sorted-table files, and compaction as core structures.
⚠️ Isar is not the LSM example here

Isar v3.1.0+1 pins its native core to mdbx-sys, the libmdbx binding. The current Isar main branch includes native libmdbx and SQLite backend dependencies. Those checked revisions do not support describing Isar as an LSM engine. [src: Isar v3 commit packages/isar_core/Cargo.toml; current-main commit Cargo.toml]

Why this project chose a B+tree

  1. Required behavior. The project needs ordered point/range access, which a page-oriented B+tree exposes directly.
  2. Scope control. A minimal B+tree plus one WAL keeps the exercise focused on pages, splits, logging, and recovery. A useful LSM would add run files, manifests, compaction, and tombstone lifecycle.
  3. Single-file goal. The selected design maps tree pages and WAL frames into the single-database-file architecture being taught. LSMs can be embedded, but their common multi-run organization would require a different file/metadata design.
  4. It's the better teacher. The B+tree forces you to confront pages, splitting, and balance directly — the foundational ideas. An LSM could be a future v2.
✅ Concept check

1. What does an LSM do instead of updating data in place?

Show answer
Buffers writes in a memtable + log, flushes immutable sorted SSTables, and periodically compacts them, using tombstones for deletes.

2. Why can LSM point reads be slower than B+tree point reads?

Show answer
A key may live in the memtable or any of several SSTables, so a read may probe multiple files (Bloom filters help but don't eliminate this).

3. Which structure is better for range scans and why?

Show answer
The B+tree — linked sorted leaves give a contiguous ordered walk, whereas an LSM must merge across sorted files.

4. Give one concrete reason this project picked a B+tree.

Show answer
Any of: read/range-heavy mobile workload, direct ordered access, narrower implementation scope, alignment with the single-file design, or its value for learning page splits and balance.
