3.7 — B-tree vs LSM, revisited

Goal: Understand the LSM-tree alternative, the read/write/space trade-offs between B+trees and LSMs, and articulate why this project deliberately chose a B+tree.

Why this matters to your DB

"B-tree or LSM?" is the central storage-engine decision. You've committed to a B+tree, but you should be able to defend that choice and recognize when the other would be better — both because it deepens understanding and because reviewers will ask.

The LSM idea in one paragraph

A Log-Structured Merge tree never updates data in place. Writes go to an in-memory sorted structure (the "memtable") plus a sequential write-ahead log. When the memtable fills, it is flushed to disk as an immutable sorted file (an "SSTable"). Over time many SSTables accumulate and are periodically merged ("compaction") into fewer, larger sorted files, discarding overwritten and deleted entries. Deletes are recorded as "tombstone" markers that suppress older values until compaction removes them.

The trade-offs (the "amplification" lens)

B+treeLSM tree
WritesUpdate in place; may do random page writes; can re-write a page for one change (write amplification)Sequential appends; generally lower write amplification, higher raw throughput
Reads (point)One root-to-leaf descentMay check memtable + several SSTables (mitigated by Bloom filters)
Range scansExcellent: follow linked leavesMust merge across multiple sorted files
SpaceCan have partially-empty pages (internal fragmentation)Temporary space amplification from un-compacted overwrites/tombstones
Background workMostly noneContinuous compaction (CPU/IO cost, tuning)

The classic framing is the RUM conjecture / read-write-space amplification trade-off: you can optimize for read, write, or space, but improving one tends to cost another. LSMs trade read and space cost for write throughput; B+trees trade write amplification for simpler, faster reads and scans. ⚠️VERIFY attribute the RUM conjecture precisely (Athanassoulis et al., 2016) before citing it as a hard result.

Who uses which

  • B+tree engines: SQLite, classic relational engines, LMDB, Berkeley DB. Strong reads and range scans; the default for embedded read-heavy workloads.
  • LSM engines: RocksDB/LevelDB, Cassandra, and notably Isar's storage in some versions. Strong write throughput; popular for write-heavy and large-dataset stores. ⚠️VERIFY Isar's exact storage engine and version before stating it relies on a specific LSM/B-tree design — Isar's internals differ across major versions (see 6.8).

Why this project chose a B+tree

  1. Workload fit. An embedded, on-device database for a mobile app is read-heavy and relies on range/ordered queries — the B+tree's strengths.
  2. No background threads needed. Compaction implies ongoing background work and tuning; a B+tree needs none, which suits a single-file embedded engine and a from-scratch learning project.
  3. Simpler crash story. One in-place file with a WAL (Phase 5) is conceptually tighter than coordinating many immutable files plus a log plus compaction.
  4. It's the better teacher. The B+tree forces you to confront pages, splitting, and balance directly — the foundational ideas. An LSM could be a future v2.
✅ Concept check

1. What does an LSM do instead of updating data in place?

Show answer
Buffers writes in a memtable + log, flushes immutable sorted SSTables, and periodically compacts them, using tombstones for deletes.

2. Why can LSM point reads be slower than B+tree point reads?

Show answer
A key may live in the memtable or any of several SSTables, so a read may probe multiple files (Bloom filters help but don't eliminate this).

3. Which structure is better for range scans and why?

Show answer
The B+tree — linked sorted leaves give a contiguous ordered walk, whereas an LSM must merge across sorted files.

4. Give one concrete reason this project picked a B+tree.

Show answer
Any of: read/range-heavy mobile workload, no background compaction, simpler single-file crash story, better learning value.
