3.2 — B-tree vs B+tree; internal vs leaf nodes

Goal: Learn the precise difference between a B-tree and a B+tree, why disk-oriented systems commonly use the B+ variant, and the distinct jobs of internal and leaf nodes.

Why this matters to your DB

"B-tree" gets used loosely, but the variant you build changes the layout, the scan behavior, and the code. You're building a B+tree, and you need to know exactly what that implies so your node formats and traversal are correct.

The core distinction

B-treeB+tree
Where values liveIn every node (internal and leaf)Only in leaf nodes
Internal nodes holdKeys + values + child pointersKeys + child pointers only (no values)
Keys in leavesEach key appears once, anywhereAll keys appear in leaves; internal keys are copies used only as routers
Range scansUse in-order tree navigation unless another link structure is addedCommonly use ordered leaf links

The B+tree's defining move is to keep record payloads in leaves and use internal nodes as routing pages. Linking leaves is a common implementation choice, and it is part of this curriculum's format because it makes range scans direct; do not treat that link as a universal requirement of every structure called a B+tree.

Why databases prefer B+trees

  • Higher fanout. Because internal nodes store only keys and child pointers (no payloads), each internal page holds more keys, so the tree is even shallower. The payloads that would bloat internal nodes live only in leaves.
  • Cheap range scans. Leaves form a linked list in sorted order. A range query descends once to the start leaf, then walks the leaf links — no repeated root-to-leaf traversals. This is the killer feature for the query layer.
  • Uniform leaf access. Every full-record read ends at a leaf, so the access cost is predictable (always full height), which simplifies reasoning and caching.

Internal vs leaf: two node types, one page each

You'll implement two on-page formats:

  • Internal node: an ordered list of separator keys and one more child pointer than keys. For separators k₀…kₙ₋₁, this project uses: c₀: K<k₀; middle child cᵢ: kᵢ₋₁≤K<kᵢ; and cₙ: K≥kₙ₋₁.
  • Leaf node: an ordered list of (key, value) entries plus a pointer to the next leaf (the sibling link for range scans).

A one-byte type tag at the start of the page distinguishes them — exactly how SQLite encodes its four page types (interior/leaf × table/index) with the type byte at offset 0 of the page header (values 2, 5, 10, 13). [src: SQLite file format §1.6, "B-tree Pages" — page-type byte]

SQLite's table b-tree is a special case worth knowing

SQLite's table b-trees are B+trees keyed by a 64-bit integer rowid, with the row data stored only in the leaves; its index b-trees store the indexed keys. [src: SQLite file format §1.6] Your schemaless engine will key its primary store by a document id, which is structurally the same idea: an ordered B+tree from id → document bytes, with data in the leaves.

✅ Concept check

1. In a B+tree, where do values live, and what do internal nodes contain?

Show answer
Values live only in leaves; internal nodes hold separator keys and child pointers only (routing).

2. Why does a B+tree achieve higher fanout than a B-tree?

Show answer
Internal nodes carry no payloads, so more keys+pointers fit per page, making the tree shallower.

3. What makes range scans cheap in a B+tree?

Show answer
Leaves are linked in key order, so after descending to the start you just follow sibling links instead of re-traversing from the root.

4. For an internal node with n separator keys, how many children?

Show answer
n + 1 children.
