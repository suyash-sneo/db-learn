Build: a persistent B+tree on the pager

Goal: Implement an ordered map from byte-string keys to byte-string values, stored as a B+tree whose nodes are pages from your Phase 2 pager: get, insert (with split), delete (with merge/borrow), and an ordered range scan.
⚠️ AI off

The B+tree is the core index — per the project rules there is no worked implementation here, only contracts, stubs, failing tests, and graduated hints. Use the lessons, the SQLite file-format spec, and the compiler. Splitting and balancing are exactly the skills being tested; do not have an AI write them.

Deliverable

A Rust 2024 module btree built on the Phase 2 pager crate (as a path or workspace dependency). It exposes an ordered key/value map persisted in a single file.

Fixed design decisions

  • Keys and values are &[u8]. Order is lexicographic byte order ([u8]'s natural Ord).
  • Two node types (leaf, internal) with a 1-byte type tag, stored as slotted pages in a pager page.
  • Bound key+value sizes so any single entry fits in a page with room to split (e.g. an entry may be at most ¼ of usable page space). Oversized entries return a typed error; overflow pages are deferred to Phase 4. Document this limit.
  • The root page number is stored in the file header (extend your Phase 2 header with a 4-byte "root page" field, or keep it in a dedicated metadata page).
  • Separator convention (write it as a comment and never deviate): child cᵢ holds keys < kᵢ; rightmost child holds keys ≥ the last separator.

Ordered milestones

  1. M1 — Node codec. Encode/decode a leaf and an internal node to/from a page's bytes (slotted layout). Round-trip tested in isolation.
  2. M2 — Search. get(key): descend internal nodes, binary-search the leaf. Build the root-to-leaf path stack you'll reuse.
  3. M3 — Insert, no split. Insert into a leaf with room; keep entries sorted.
  4. M4 — Leaf split. Split a full leaf, fix sibling links, copy a separator up; handle the parent-has-room case.
  5. M5 — Internal split + new root. Propagate splits upward (move the median up); grow a new root when the old root splits.
  6. M6 — Delete + rebalance. Remove from a leaf; borrow from a sibling or merge; propagate; shrink the root when it has one child.
  7. M7 — Range scan. Iterate [lo, hi] in order via the leaf links.

Acceptance criteria

  • All Rust tests below pass.
  • Inserting many keys in random order, then reading them all back, returns every value (split correctness).
  • A range scan returns keys in sorted order and exactly the keys in range.
  • After deleting half the keys (random order), the remaining keys are all still found and the deleted ones are absent (merge/borrow correctness).
  • The tree survives close+reopen: persist, drop, reopen from the file, and all data is intact (assuming a sync before drop).
  • An over-limit entry returns the typed EntryTooLarge error (tested).

#[cfg(test)]
mod tests {
    use super::*;
    use pager::MemBlock;

    fn fresh() -> BTree<MemBlock> {
        let pager = Pager::create(MemBlock { data: Vec::new() }, 4096, 64).unwrap();
        BTree::create(pager).unwrap()
    }

    #[test]
    fn insert_get_roundtrip_forces_splits() {
        let mut t = fresh();
        let n = 5000u32;
        // pseudo-random order via a simple LCG so a split cascade happens
        let mut x = 1u32;
        for _ in 0..n {
            x = x.wrapping_mul(1664525).wrapping_add(1013904223);
            let k = x.to_be_bytes();
            t.insert(&k, &k).unwrap();
        }
        x = 1;
        for _ in 0..n {
            x = x.wrapping_mul(1664525).wrapping_add(1013904223);
            let k = x.to_be_bytes();
            assert_eq!(t.get(&k).unwrap().as_deref(), Some(&k[..]));
        }
    }

    #[test]
    fn range_is_sorted_and_bounded() {
        let mut t = fresh();
        for i in 0u32..1000 { let k = i.to_be_bytes(); t.insert(&k, &k).unwrap(); }
        let out = t.range(&100u32.to_be_bytes(), &200u32.to_be_bytes()).unwrap();
        assert_eq!(out.len(), 101); // 100..=200 inclusive
        assert!(out.windows(2).all(|w| w[0].0 < w[1].0)); // sorted, strictly increasing keys
    }

    #[test]
    fn delete_then_search_forces_merges() {
        let mut t = fresh();
        for i in 0u32..2000 { let k = i.to_be_bytes(); t.insert(&k, &k).unwrap(); }
        for i in (0u32..2000).step_by(2) { assert!(t.delete(&i.to_be_bytes()).unwrap()); }
        for i in 0u32..2000 {
            let got = t.get(&i.to_be_bytes()).unwrap().is_some();
            assert_eq!(got, i % 2 == 1); // odds remain, evens gone
        }
    }

    #[test]
    fn rejects_oversized_entry() {
        let mut t = fresh();
        let big = vec![0u8; 4096];
        assert!(matches!(t.insert(b"k", &big), Err(BTreeError::EntryTooLarge { .. })));
    }
}

Stuck? — graduated hints

M1: how do I store variable-length entries in a page?
Nudge: Re-read 3.3 — sorted slot array at the front, cells at the back.
Bigger: Each slot is a 2-byte offset to a cell; the header tracks count and the start of free space.
Technique: keep slots sorted by key; to insert, shift slots (cheap) and place the cell wherever it fits (Phase 4 adds compaction).
M4/M5: my keys go missing after splits
Nudge: Leaf split copies the separator up; internal split moves the median up (3.5).
Bigger: Verify your separator convention is identical in get and in split.
Technique: after each split, assert the parent's separators still partition the children correctly before descending in a test.
M5: borrow-checker fights me passing pages up the path
Nudge: You can't hold mutable borrows of two pages from the same pager at once.
Technique: work one page at a time — read a node into an owned buffer, mutate, write it back via the pager, then move to the parent using the path stack from M2. Avoid holding two live page borrows simultaneously.
M6: deletes corrupt the parent separator
Nudge: After borrow/merge the separator must describe the new boundary (3.6).
Technique: re-derive the separator from the children after redistribution rather than patching it incrementally.
