3.8 — Source dive: SQLite's B-tree

Goal: Map the concepts from 3.2–3.6 onto SQLite's real B-tree implementation so you can read it yourself, and see how a production engine structures cells, cursors, and balancing.
📂 Where to read

The B-tree layer is src/btree.c (with btree.h and btreeInt.h for internals) in the public-domain SQLite tree. The on-disk layout it produces is specified precisely in the file-format document §1.6. [src: sqlite.org/src · src/btree.c; file format §1.6 "B-tree Pages"]

How to verify any claim below: open the file, search the symbol, read the comment. The §1.6 spec is the authority for byte layout; btreeInt.h is the authority for the data structures.

The on-disk facts (these are spec-confirmed)

  • Four page types, distinguished by the type byte at offset 0 of the page header: interior index (2), interior table (5), leaf index (10), leaf table (13). [src: file format §1.6]
  • The b-tree page header is 8 bytes for leaves and 12 bytes for interior pages (the extra 4 bytes are the right-most child pointer). [src: file format §1.6 header table]
  • Cells are stored in a slotted arrangement: a cell-pointer array near the front of the page and cell content growing from the end — exactly the layout from 3.3. [src: file format §1.6 "cell pointer array" / "cell content area"]
  • Cells too large for a page spill to overflow pages. [src: file format §1.6]

SQLite's rebalancing is famously consolidated: instead of separate "split" and "merge" routines, a balance routine redistributes cells across a node and its siblings, which can both split (add a page) and merge (remove one). Reading how one routine subsumes both operations from 3.5 and 3.6 is the highlight of this dive. ⚠️VERIFY the exact routine names and their division of labor against the current source.

What's different from your build (and fine to skip)

  • Table vs index b-trees. SQLite specializes: table b-trees are keyed by integer rowid with data in leaves; index b-trees store keys. Your schemaless engine can use a single id-keyed B+tree, conceptually the table b-tree.
  • The cursor abstraction. SQLite exposes navigation through cursors that the VDBE (its bytecode VM) drives. You don't need a bytecode VM; a direct API is fine.
  • Decades of edge cases. Auto-vacuum, incremental vacuum, integer-key optimizations — extract the core, leave these.

✅ Concept check

1. How does SQLite mark a page's type and tell leaf from interior?

Show answer
A type byte at offset 0 (2/5/10/13); leaf headers are 8 bytes, interior 12 (extra right-most child pointer).

2. How does SQLite's balance relate to your split and merge?

Show answer
One redistribution routine covers both: it can add a page (split) or remove one (merge) while rebalancing cells across siblings.

3. Where is the authoritative byte layout, vs the data structures?

Show answer
The file-format §1.6 spec for on-disk bytes; btreeInt.h for in-memory structs like MemPage/BtCursor.

4. Which SQLite-specific features can your build safely omit?

Show answer
The bytecode-VM cursor coupling, separate table/index b-tree specialization, and auto/incremental vacuum.
