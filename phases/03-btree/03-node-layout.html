3.3 — Node layout inside a page

Goal: Decide the exact byte layout of an internal node and a leaf node within one page, including a node header, the entries, and how variable-length keys/values are placed — then estimate fanout.

Why this matters to your DB

A "node" is an abstraction; on disk it's just bytes in a page. The layout you choose determines how many keys fit (fanout, hence tree height), how splits and deletes work, and how scans read entries. This is where the abstract tree meets the concrete page from Phase 2.

The node header

Every node page begins with a small fixed header. A reasonable minimal set:

  • Type byte — internal vs leaf (3.2).
  • Entry count — how many keys/cells are stored.
  • Free-space pointer(s) — where unused space begins (needed for variable-length entries; detailed in Phase 4's slotted-page lesson).
  • Right/next pointer — for a leaf, the page number of the next leaf (the sibling link); for an internal node, the rightmost child pointer.

SQLite's b-tree page header is 8 bytes for leaf pages and 12 bytes for interior pages (the extra 4 bytes hold the right-most child pointer), with fields for the page type, the first freeblock, the cell count, the cell-content start, and fragmented free bytes. [src: SQLite file format §1.6, "B-tree Page" header table (8-byte leaf / 12-byte interior header)] Your header can be simpler, but the same ingredients appear.

Fixed-length vs variable-length entries

If keys and values are all the same size, layout is trivial: an array of fixed-size slots, binary search by index. Real keys (strings, document ids) and values (documents) are variable-length, which is why databases use a slotted page: a growing array of small offset "pointers" at the front of the page, and the actual variable-length cells growing from the back. The slot array stays sorted by key even though the cells are placed wherever they fit.

This is important enough that Phase 4 is devoted to it. For now, know that a B+tree node is almost always a slotted page, and the "entries" are slots pointing at cells.

Internal vs leaf cell contents

  • Internal cell: a separator key + a child page number. (Plus the header's rightmost-child pointer for the "≥ last key" case.)
  • Leaf cell: a key + the value (or, if the value is large, a pointer to overflow pages — Phase 4). Plus the leaf's next-leaf link in the header.

Estimating fanout (why this drives height)

Fanout = how many children an internal node can hold = how many (separator key + child pointer) cells fit in the usable page space. Roughly:

approximate fanout 
usable = page_size − node_header − slot_overhead_per_entry × entries
fanout ≈ usable_for_cells / (avg_key_size + child_pointer_size)

With a 4096-byte page, 4-byte child pointers, and small keys, an internal node can route to on the order of a few hundred children, giving a tree only ~3–4 levels deep for millions of keys. The precise number depends on your exact header and key sizes — ⚠️VERIFY by computing it for your chosen layout rather than trusting a round figure.

⚠️ A key+value must fit; otherwise you need overflow

If a single entry is larger than the page can hold (a big document, a long key), it cannot live inline. Databases handle this with overflow pages: store a prefix plus a pointer to a chain of pages holding the rest. SQLite does exactly this for large records. [src: SQLite file format §1.6 — overflow pages for cells that exceed the page] You'll implement overflow in Phase 4; for the Phase 3 build you can bound key/value sizes so every entry fits inline, and note the limitation.

✅ Concept check

1. Why is a B+tree node almost always a slotted page?

Show answer
Keys/values are variable-length; a sorted slot array plus back-growing cells keeps entries logically sorted without moving the cell bytes.

2. What does an internal cell contain vs a leaf cell?

Show answer
Internal: separator key + child page number. Leaf: key + value (or overflow pointer), with the next-leaf link in the leaf header.

3. Why does fanout determine tree height?

Show answer
Height ≈ log base fanout of N; more cells per internal node → larger base → fewer levels → fewer page reads.

4. What happens when one entry can't fit in a page?

Show answer
It spills to overflow pages: an inline prefix/pointer plus a chain of pages holding the remainder.
