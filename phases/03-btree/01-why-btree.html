3.1 — Why an ordered, disk-oriented index

Goal: Understand why disk databases need a structure that is both ordered and shallow, and why the cost model on disk is "number of pages touched," not "number of comparisons."

Why this matters to your DB

Without an index, finding a record means scanning every page — O(pages). With the right index, it's O(height of a very shallow tree). The choice of index structure determines whether your engine is usable on real data. And because disk is slow, the metric that matters is how many pages a lookup reads, not how many CPU comparisons it does.

The disk cost model

In RAM, the cost of a search is roughly the number of comparisons. On disk, one access to a page that isn't cached can be many thousands of times slower than a comparison. So the design goal flips: minimize the number of distinct pages a lookup must read. A structure with one key per node forces one page read per level; a structure with hundreds of keys per node collapses many levels into one page.

Why "ordered" matters too

A hash index gives O(1) point lookups but stores keys in no useful order. Many real queries are not point lookups:

  • Range: "all orders between two dates."
  • Ordering: "list users sorted by name."
  • Prefix: "names starting with 'Sa'."
  • Min/max: "the earliest event."

An ordered index answers all of these by walking a contiguous run of the structure. A hash index cannot, because order is destroyed by hashing. Your engine will need ordered scans for its query layer (Phase 6), so the primary index must be ordered.

The two properties an index for your DB must have

  1. Shallow on disk. Height grows logarithmically with a very large base (keys per page), so even billions of keys are a few levels deep — few page reads per lookup.
  2. Ordered. Supports range, sort, prefix, and min/max, not just equality.

The B+tree has both. The next lessons build it; the rest of this one is the "why" that justifies the work.

A note on logarithms with a big base

Tree height is roughly logb(N) where b is the branching factor (keys per node) and N is the number of keys. With b = 2 (a binary tree) and a billion keys, height ≈ 30. With b in the hundreds (a page full of keys), height drops to about 3–4. Each level is potentially one page read, so the high branching factor is the whole game. The exact b depends on key size and page size — you'll compute it in 3.3. ⚠️VERIFY any specific fanout number against your chosen page and key sizes.

✅ Concept check

1. Why is "pages touched" the right cost metric on disk, not "comparisons"?

Show answer
A non-cached page read can be thousands of times slower than a comparison, so the dominant cost is the number of distinct page reads.

2. Why does a high branching factor make the tree shallow?

Show answer
Height ≈ log base (keys-per-node) of N; a large base shrinks the exponent, so many keys fit in few levels.

3. Give two queries an ordered index supports that a hash index cannot.

Show answer
Any two of: range scans, sorted output, prefix matches, min/max — all rely on key order, which hashing destroys.

4. Why is a plain in-memory BST a poor on-disk index?

Show answer
One key per node makes it deep, so each level is another page read; it wastes the page as the I/O unit.
