3.1 — Why an ordered, disk-oriented index

Goal: Understand why disk databases need a structure that is both ordered and shallow, and why the cost model on disk is "number of pages touched," not "number of comparisons."

Why this matters to your DB

Without an index, finding a record may require scanning every data page—O(pages). With a B+tree, point lookup follows one root-to-leaf path. For data not already cached, page transfers usually dominate the handful of CPU comparisons inside each node, so the first cost to minimize is the number of distinct pages visited.

The disk cost model

In RAM, comparison count and cache behavior matter. In a disk-oriented index, a cache miss may also require a storage read, so the design packs many routing keys into each page: minimize page visits while keeping the work inside a page cheap. A structure with one key per page wastes most of the transfer; a wide node performs many comparisons after one page has arrived.

Why "ordered" matters too

A hash index can provide expected O(1) point lookup, but hash order is not key order. Many queries are not point lookups:

  • Range: "all orders between two dates."
  • Ordering: "list users sorted by name."
  • Prefix: "names starting with 'Sa'."
  • Min/max: "the earliest event."

An ordered index answers all of these by walking a contiguous run of the structure. A hash index cannot, because order is destroyed by hashing. Your engine will need ordered scans for its query layer (Phase 6), so the primary index must be ordered.

The two properties an index for your DB must have

  1. Shallow on disk. Height grows logarithmically with the branching factor (children per internal page), so a large fanout means few page visits.
  2. Ordered. Supports range, sort, prefix, and min/max, not just equality.

The B+tree has both. The next lessons build it; the rest of this one is the "why" that justifies the work.

A note on logarithms with a big base

Tree height is roughly logb(N) where b is the branching factor and N is the number of reachable entries. As an illustration, 230 is about one billion, while 2564 is over four billion. That does not predict your exact tree height: leaf capacity, root occupancy, variable key sizes, and fill policy all matter. It shows why a page-sized fanout changes the scale. In 3.3 you calculate one explicit layout instead of trusting a round fanout claim.

✅ Concept check

1. Why is "pages touched" the right cost metric on disk, not "comparisons"?

Show answer
A cache miss may require a storage transfer, while many comparisons can run after that page is already in memory. Page visits are therefore the first-order disk-oriented cost, though CPU/cache costs still exist.

2. Why does a high branching factor make the tree shallow?

Show answer
Height ≈ log base (keys-per-node) of N; a large base shrinks the exponent, so many keys fit in few levels.

3. Give two queries an ordered index supports that a hash index cannot.

Show answer
Any two of: range scans, sorted output, prefix matches, min/max — all rely on key order, which hashing destroys.

4. Why is a plain in-memory BST a poor on-disk index?

Show answer
One key per node makes it deep, so each level is another page read; it wastes the page as the I/O unit.
