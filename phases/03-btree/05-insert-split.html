3.5 — Insert & node splits

Goal: Insert a key/value into the correct leaf, and when a node overflows, split it and push a separator up — recursively, possibly growing the tree by a new root — so the tree stays balanced.

Why this matters to your DB

Insertion is where the B+tree earns its "balanced" property. Done right, every leaf stays at the same depth and the tree never degenerates. Splitting is the single trickiest algorithm in the whole index; understanding it precisely is the core of this phase.

The happy path: leaf has room

Descend (3.4) to the leaf where the key belongs. If the leaf has free space, insert the (key, value) entry in sorted position and you're done. Most inserts are this simple.

The split: leaf is full

If the target leaf has no room for the new entry:

  1. Conceptually add the new entry to the full leaf's entries, giving an over-full set.
  2. Allocate a new leaf page (via the pager).
  3. Split the entries roughly in half: lower half stays in the original leaf, upper half moves to the new leaf.
  4. Fix the sibling links: the new leaf's next-pointer becomes the old leaf's old next, and the old leaf now points to the new leaf.
  5. Copy the first key of the new (right) leaf up into the parent as a separator, pointing at the new leaf. (For leaves, the separator is a copy — the key still exists in the leaf, because all data lives in leaves.)

Internal splits and the recursive cascade

Pushing a separator into the parent may overflow the parent. An internal node splits similarly, but with one crucial difference: when an internal node splits, the middle separator is moved up (not copied), because internal separators are pure routers — the moved key is represented below it by the child pointers, not duplicated as data. The left half keeps the separators below the median; the right half gets those above it; the median rises to the grandparent.

This can cascade all the way to the root. If the root itself splits, you allocate a brand-new root whose single separator is the median and whose two children are the two halves. This is the only way a B+tree grows in height, and it grows from the top, which is why all leaves stay at the same depth — the balance property.

⚠️ Leaf split copies up; internal split moves up

This asymmetry is the most common B+tree bug. Leaf separator = a copy of the right leaf's first key (the key still lives in the leaf). Internal separator = the median key removed from the node and promoted (it must not remain, or you'd route to it as if it were a real key with a child on both sides). Mixing these up either loses keys or creates phantom routes.

Top-down vs bottom-up splitting

Two implementation strategies:

  • Bottom-up (insert then split): descend to the leaf, insert, and if it overflows, split and propagate upward, repeating at each ancestor. Needs the path back to the root (a stack of visited pages).
  • Top-down (pre-emptive split): on the way down, proactively split any full node you pass through, guaranteeing the parent always has room for a promoted key. Avoids the back-up pass but may split nodes unnecessarily.

This curriculum's build uses bottom-up splitting and records the root-to-leaf path. Do not infer that a production source uses the same strategy merely because its result is a balanced tree.

Choosing the split point

Splitting by entry count is adequate only when entries are similarly sized. With variable-length cells, choose a boundary near half of the encoded bytes, while preserving key order and ensuring both resulting pages encode successfully. A separate occupancy policy is needed if deletion later promises a minimum fill ratio. The Phase 3 build bounds entry size so one entry cannot consume most of a page.

✅ Concept check

1. When a leaf splits, is the separator pushed up a copy or a move? Why?

Show answer
A copy — all data lives in leaves, so the key must remain in the leaf while a copy routes to it from the parent.

2. When an internal node splits, is the median a copy or a move? Why?

Show answer
A move — internal separators are pure routers; leaving it behind would create a duplicate route with no real key.

3. How does a B+tree increase in height, and why does that keep it balanced?

Show answer
Only by the root splitting into a new root above it; growth at the top keeps every leaf at equal depth.

4. Why does bottom-up splitting need the root-to-leaf path?

Show answer
To insert promoted separators into each parent in turn as splits propagate upward.
