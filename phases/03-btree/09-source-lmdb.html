3.9 — Source dive: LMDB / MDBX copy-on-write B+tree

Goal: See a completely different B+tree durability strategy — copy-on-write with no write-ahead log — and understand precisely what COW does and does not give you for free.
📂 Where to read

This lesson verifies LMDB itself at commit 22a8571f4506f853a101506465893486cdea3776. Its core implementation is libraries/liblmdb/mdb.c. libmdbx is related and Isar v3 uses it, but this page does not merge the two codebases' exact protocols. [src: pinned LMDB mdb.c; LMDB API documentation]

The big idea: copy an older-snapshot page before changing it

LMDB uses copy-on-write/shadow-paging behavior for pages from an older snapshot. In mdb_page_touch, if the cursor's page is not mutable in the current transaction, LMDB allocates a replacement page, records the old page for later reclamation, and updates either the parent pointer or the database root to the replacement. It then copies the original page bytes into the writable page. [src: pinned LMDB mdb_page_touch] Across a tree update, pages on the modified path are touched as needed:

  1. Obtain a transaction-writable copy of the page being modified.
  2. Applies the change to the copy.
  3. Update the parent pointer; when the changed page is the root, update the transaction's root descriptor.
  4. Commit the new database descriptors through one of the metadata pages after the data-page write/synchronization steps required by the selected durability flags.

A page already writable in the same transaction need not be copied again. Copy-on-write is therefore not “rewrite the entire tree”; it preserves older reachable pages while publishing new roots for changed paths.

LMDB's two metadata pages

mdb.c defines NUM_METAS as 2 and states that pages 0–1 are metadata pages; transaction N writes metadata page N % 2. Each MDB_meta contains the free-space and main-database descriptors, the last page number, and the transaction id that committed the metadata page. [src: pinned LMDB NUM_METAS and MDB_meta] On open, LMDB reads the metadata pages and selects the newer acceptable transaction id. Alternating pages means one commit does not overwrite both metadata copies.

What the complete protocol can buy you

  • Recovery without replaying a WAL. With the default ordered data/meta commit path and its documented storage assumptions, an open chooses metadata for a committed snapshot rather than replaying logical changes. Relaxed LMDB flags weaken synchronization behavior and must be analyzed separately.
  • Snapshot reads. A read transaction identifies a committed version while a single writer creates new pages. Older pages cannot be reused until they are no longer needed by readers, so reader tracking and free-page reclamation are essential parts of the design. [src: LMDB Getting Started — transactions and concurrency; pinned mdb.c free-space database]
⚠️ COW does not give you these "for free"

This is the lesson's whole point, and a common over-claim. Copy-on-write provides the mechanism, but you still must implement the protocol around it:

  • Root publication protocol. The new roots, transaction id, and last-page boundary must be encoded and ordered so recovery can choose a usable metadata page.
  • Durability policy. In the normal commit path, _mdb_txn_commit flushes dirty pages, performs environment synchronization unless disabled by flags, and then calls mdb_env_write_meta. [src: pinned LMDB _mdb_txn_commit] COW does not remove synchronization or storage assumptions.
  • Space reclamation. Old pages from no-longer-referenced versions must be put into the free-space database and reused only when no relevant reader can still observe them.
  • Write amplification. The first modification to an older page in a transaction copies that page, and parent/root changes can copy additional path pages.

COW vs WAL (and why this project uses WAL)

Shadow paging and write-ahead logging are two protocol families for publishing tree changes. Shadow paging retains old reachable pages and publishes new roots. WAL records page versions before the main database is updated; recovery can use committed log frames and checkpoint them later. Both require explicit commit ordering, corruption checks, and reclamation. This curriculum implements WAL in Phase 5 because that is the selected architecture, not because COW automatically supplies or lacks a particular isolation level.

✅ Concept check

1. In copy-on-write, why does changing one leaf also copy its ancestors?

Show answer
The copied leaf has a new page number, so its parent must point to the new page; that re-points cascades up to a new root.

2. How does COW give crash safety without a journal?

Show answer
Older reachable pages are preserved while new pages are written, and recovery chooses an acceptable committed metadata snapshot. That result depends on the full data/meta write and synchronization protocol, not COW alone.

3. Name two things COW does not provide automatically.

Show answer
Any two of: atomic+ordered root publication ordering, durability synchronization, reader tracking, free-page reclamation, or low write amplification.

4. Why might COW give cheap snapshots?

Show answer
A reader holding an old root sees a complete, immutable older version while writers build new pages elsewhere.
