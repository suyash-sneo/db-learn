3.9 — Source dive: LMDB / MDBX copy-on-write B+tree

Goal: See a completely different B+tree durability strategy — copy-on-write with no write-ahead log — and understand precisely what COW does and does not give you for free.
📂 Where to read

LMDB (Lightning Memory-Mapped Database, by Howard Chu / Symas) is a compact C B+tree engine; its core is mdb.c. libmdbx is a hardened fork with extended docs. Both are good reads because they are small and document their copy-on-write design. [src: LMDB documentation (lmdb.tech); libmdbx repository] ⚠️VERIFY precise LMDB internals against the source/design talk before asserting exact structures; this page teaches the COW idea, which is the transferable part.

The big idea: never overwrite a live page

LMDB's B+tree is copy-on-write (a.k.a. shadow paging). To modify a page, it does not write in place. Instead it:

  1. Copies the page being modified to a brand-new, currently-free page.
  2. Applies the change to the copy.
  3. Because the copy has a new page number, its parent must now point to the new page — so the parent is also copied and updated, and so on up to the root.
  4. The transaction commits by writing a new root and a new "meta page" that points at it.

The old pages are untouched and still form a complete, consistent older version of the tree until the new root is published.

What COW buys you

  • Crash safety without a WAL. A crash mid-write leaves the old root and all old pages intact; on restart you simply use the last fully-committed meta page. There is no journal to replay.
  • Cheap consistent snapshots / MVCC reads. A reader that pinned the old root keeps seeing a consistent older version while a writer builds a new one — readers never block writers and vice versa. LMDB uses exactly this for its single-writer, many-reader concurrency.
⚠️ COW does not give you these "for free"

This is the lesson's whole point, and a common over-claim. Copy-on-write provides the mechanism, but you still must implement the protocol around it:

  • Atomic root publication. Switching to the new root must itself be atomic and correctly ordered with syncs, or a crash mid-publish corrupts everything. LMDB uses two meta pages and writes them alternately, with sync ordering, so one is always valid. ⚠️VERIFY the exact two-meta-page protocol against LMDB docs.
  • Durability still needs fsync. Publishing a new root is only durable after the relevant sync; COW does not remove the need for the durability ordering from 2.6.
  • Space reclamation. Old pages from no-longer-referenced versions must be put back on a free list, or the file grows forever. LMDB maintains a free-page list keyed by the transaction that freed them.
  • Write amplification. Every change rewrites the whole root-to-leaf path, not one page — a real cost for deep trees or tiny updates.

COW vs WAL (and why this project uses WAL)

COW (shadow paging) and write-ahead logging are two answers to the same question — "how do I make an in-place B+tree crash-safe?" COW keeps old pages and atomically switches roots; WAL (Phase 5) logs changes before applying them and replays on recovery. Both are valid. This curriculum builds a WAL because it generalizes more directly to incremental durability and is the design SQLite uses, but COW is a legitimate (and in some ways simpler) alternative you should understand. A future v2 of your engine could adopt COW for snapshot isolation.

✅ Concept check

1. In copy-on-write, why does changing one leaf also copy its ancestors?

Show answer
The copied leaf has a new page number, so its parent must point to the new page; that re-points cascades up to a new root.

2. How does COW give crash safety without a journal?

Show answer
Old pages and the old root stay intact; a crash before publishing the new root just leaves the last committed version usable.

3. Name two things COW does not provide automatically.

Show answer
Any two of: atomic+ordered root publication, fsync durability, free-page reclamation, low write amplification.

4. Why might COW give cheap snapshots?

Show answer
A reader holding an old root sees a complete, immutable older version while writers build new pages elsewhere.
