1.2 — extern "C", #[no_mangle], #[repr(C)]

Goal: Learn the three Rust switches that turn a normal Rust function or struct into something C — and Dart — can call, and how to configure Cargo to emit a loadable library.

Why this matters to your DB

These are the literal building blocks of Layer 2. Every function the outside world calls on your engine will carry extern "C" and #[no_mangle]; every struct that crosses the boundary will carry #[repr(C)]. Misuse any of them and you get linker errors or, worse, silent memory corruption. This lesson makes them routine.

1. extern "C" — the calling convention

Putting extern "C" on a function tells Rust: "compile this with the C calling convention, not Rust's." That governs how arguments and return values are physically passed (recall Lesson 1.1). The function body is still normal Rust.

rust 
pub extern "C" fn add(a: i32, b: i32) -> i32 {
    a + b   // ordinary Rust inside
}

2. #[no_mangle] — keep the symbol name

By default Rust mangles symbol names (encodes module + type info), so the compiled symbol isn't simply add. A C/Dart caller looks the function up by name, so it needs the predictable name. #[no_mangle] says "export this symbol exactly as written."

rust 
#[no_mangle]
pub extern "C" fn add(a: i32, b: i32) -> i32 {
    a + b
}
// Now the binary contains a symbol literally named "add" that Dart can find.
⚠️ No overloading, unique names

Because #[no_mangle] uses the raw name, every exported function needs a globally unique name (no two adds, no overloads). Real C libraries solve this by prefixing: sqlite3_open, sqlite3_close. You'll do the same — mydb_open, mydb_insert — both to avoid clashes and to namespace your API.

3. #[repr(C)] — C-compatible data layout

Rust reserves the right to reorder struct fields and choose padding for efficiency. C does not — it lays fields out in declaration order with defined alignment. If you pass a struct by value across the boundary, both sides must agree, so you force C layout with #[repr(C)]:

rust 
#[repr(C)]
pub struct Point {
    x: f64,
    y: f64,
}   // guaranteed: x then y, C alignment — safe to share with C/Dart
You'll mostly avoid passing structs by value

Your design keeps the C surface tiny: documents cross as byte buffers (pointer + length), and the database itself is an opaque handle (Lesson 1.4) — neither requires the caller to know your struct layout. So #[repr(C)] matters most for the occasional small result struct (e.g. a buffer descriptor). When in doubt, prefer pointers + lengths over by-value structs.

Telling Cargo to build a C-callable library

A normal Rust library compiles to an .rlib (only Rust can link it). To get a dynamic library other languages can load (.so on Linux, .dylib on macOS, .dll on Windows), set the crate type:

Cargo.toml 
[package]
name = "mydb"
version = "0.1.0"
edition = "2021"

[lib]
# cdylib = a C-compatible dynamic library (what Dart loads at runtime)
# staticlib is for static linking; you'll want cdylib for Flutter desktop/dev
crate-type = ["cdylib", "staticlib", "rlib"]

Including "rlib" too lets your own Rust tests still link the crate normally. After cargo build, look in target/debug/ for libmydb.dylib / .so / mydb.dll ⚠️VERIFY exact file naming per platform.

Inspecting the exported symbol

Confirm your function is actually exported with the right name — a great sanity check before you ever touch Dart:

terminal (macOS/Linux) 
cargo build
# Linux:
nm -D target/debug/libmydb.so | grep add
# macOS:
nm -gU target/debug/libmydb.dylib | grep add
# expect to see a symbol named "add" (or "_add" with a leading underscore on macOS) ⚠️VERIFY
🦀 Rust just-in-time
  • The same three switches in concert is the canonical exported-function pattern: #[no_mangle] pub extern "C" fn ….
  • Newer Rust may nudge you toward #[unsafe(no_mangle)] spelling ⚠️VERIFY — check your compiler's warnings and the edition you chose.
  • Only C-compatible parameter/return types may appear in an extern "C" signature: integers, floats, raw pointers (*const T/*mut T), and #[repr(C)] types. No String, Vec, &str, or generics in the signature — you'll learn the buffer/pointer translations in 1.3 and 1.5.

✅ Concept check

1. What does each of the three do: extern "C", #[no_mangle], #[repr(C)]?

Show answer
extern "C" = C calling convention; #[no_mangle] = keep the symbol name unmangled so callers can find it; #[repr(C)] = lay out a struct/enum the C way so both sides agree on its bytes.

2. Why can't you have two #[no_mangle] functions named open?

Show answer
Unmangled symbols must be unique in the binary — there's no type info to disambiguate. Hence prefixing (mydb_open).

3. Which crate-type makes a library Dart can load at runtime?

Show answer
cdylib (a C-compatible dynamic library). You often list ["cdylib","staticlib","rlib"] so tests/static linking still work.

4. Why does your design rarely need #[repr(C)]?

Show answer
Because documents cross as byte buffers (pointer+length) and the DB is an opaque handle — neither exposes a struct layout to the caller. repr(C) only matters for the occasional by-value struct.
