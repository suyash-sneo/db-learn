1.3 — Passing primitives and pointers across the boundary

Goal: Know which types can legally cross the C ABI (the easy primitives), meet Rust's raw pointers (*const T / *mut T), and understand why touching them requires unsafe — and what unsafe really promises.

Why this matters to your DB

Everything richer than a number crosses the boundary as a pointer: a document buffer is a pointer to bytes, the database is a pointer to an opaque object, an error message is a pointer to text. Pointers are where FFI gets dangerous, so getting comfortable and disciplined with them now is what keeps the engine memory-safe later.

Primitives: the trivial case

Fixed-size scalar types map directly onto their C counterparts and pass by value with no fuss:

Prefer the explicitly-sized integer types (i32, u64, …) at the boundary — they have one fixed size everywhere, unlike C's int whose width is platform-dependent.

Rust's two reference worlds

Rust has safe references (&T, &mut T) — the borrow-checked ones from Phase 0 — and raw pointers (*const T, *mut T). Safe references can't cross the C ABI (they carry lifetime/borrow guarantees C knows nothing about). So at the boundary you use raw pointers, which are just addresses — no lifetime, no borrow rules, can be null, can dangle. They're exactly C pointers.

Out-parameters via pointers

C functions often return a value by writing through a pointer the caller supplies — an "out-parameter." You'll use this constantly (the result code is the real return; outputs go through pointers). To write through a raw pointer you need unsafe:

rust 
/// Writes a*b into *out. Returns 0 on success, 1 if out is null.
#[no_mangle]
pub extern "C" fn mul(a: i64, b: i64, out: *mut i64) -> i32 {
    if out.is_null() { return 1; }      // always null-check pointers from C
    unsafe { *out = a * b; }            // deref a raw pointer = unsafe
    0
}

What unsafe actually means

This is the most misunderstood word in Rust. unsafe does not turn off the borrow checker or mean "sloppy code." It unlocks exactly five extra abilities the compiler can't verify for you — most relevantly here, dereferencing a raw pointer and calling other unsafe/FFI functions. It is a contract: "I, the programmer, have personally verified the safety conditions the compiler couldn't."

⚠️ The obligations you take on inside unsafe

When you deref a *mut T from C you are promising it is: non-null, properly aligned, pointing at a valid T, and not aliased in a way that breaks Rust's rules. The compiler won't check — you must. That's why unsafe blocks should be small and every pointer gets a null check. FFI is inherently unsafe; your job is to wrap that unsafety in a thin, audited layer and keep the engine core 100% safe Rust.

Strings: a first look (full treatment in 1.5)

A C string is a *const c_char (a pointer to bytes ending in a \0). Rust strings are length-prefixed UTF-8 with no terminator, so they don't match — you convert with CStr/CString. But your design largely sidesteps C strings: documents travel as length-prefixed byte buffers (pointer + length), which is simpler and binary-safe. We cover buffers thoroughly in Lesson 1.5; just note here that "string across FFI" = "pointer to bytes," and the length question is the crux.

✅ Concept check

1. Why can't a Rust &mut i64 cross the C ABI, but a *mut i64 can?

Show answer
Safe references carry lifetime/borrow guarantees the C side knows nothing about and can't uphold. Raw pointers are just addresses — the same thing C already passes around.

2. Does unsafe disable the borrow checker?

Show answer
No. It only unlocks a few extra operations (like dereferencing raw pointers) and shifts responsibility for their safety conditions to you. The borrow checker still runs on everything else.

3. What's the first thing to do with any pointer C hands you?

Show answer
Null-check it (p.is_null()) and return an error code if it's null — before any dereference.

4. Why prefer i32/u64 over C's int at the boundary?

Show answer
The Rust types have a fixed, known size on every platform; C's int width can vary, which would make the ABI ambiguous.
