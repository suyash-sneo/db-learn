1.6 — Panics across FFI = undefined behavior; catch_unwind / panic=abort

Goal: Understand why a Rust panic must never cross the C ABI, and the two ways to guarantee it can't: catch every panic at the boundary with catch_unwind, or compile with panic = "abort". This is a correctness requirement, not a nicety.

Why this matters to your DB

Your engine code will panic sometimes — an unwrap you missed, an array index out of range, an arithmetic overflow in debug. Inside pure Rust that's a controlled crash. But if a panic unwinds out of an extern "C" function into Dart, the behavior is undefined — the host app can corrupt memory or crash unpredictably. A database that can crash its host app on a bad input is unacceptable, so every boundary function must contain panics.

What "panic" and "unwind" mean

A Rust panic! is an unrecoverable error. By default it unwinds the stack: it walks back up the call stack running destructors (like exceptions in C++). That unwinding machinery is Rust-specific. The C ABI has no notion of it — C functions don't unwind. So when Rust unwinding hits the extern "C" frame and tries to continue up into C/Dart frames, it's doing something the foreign code was never built to handle.

⚠️ Unwinding across an FFI boundary is undefined behavior

"Undefined behavior" (UB) is the worst category of bug: the compiler is allowed to do anything — crash, corrupt memory, appear to work then fail later. It is not a catchable exception on the Dart side. The only safe rule is: a panic must be stopped before it leaves your Rust function.

Solution A: catch_unwind at every boundary (recommended)

std::panic::catch_unwind runs a closure and, if it panics, catches the unwind and returns an Err instead of letting it propagate. You wrap the body of every extern "C" function in it and convert a caught panic into your error code:

rust 
use std::panic::catch_unwind;

#[no_mangle]
pub extern "C" fn db_do_thing(db: *mut Db) -> i32 {
    let result = catch_unwind(|| {
        // … real work; may panic …
        0i32   // success code
    });
    match result {
        Ok(code) => code,
        Err(_)   => -99,   // a panic was caught → return an error code, NOT UB
    }
}

Because every boundary function repeats this shape, real projects wrap it in a small helper macro or function so you write it once. The principle stands: no extern "C" function lets a panic escape.

A subtlety: UnwindSafe

catch_unwind wants the closure to be UnwindSafe — a marker that catching a panic mid-operation won't leave shared state in a broken, observable condition. For raw pointers and most boundary code this is satisfiable, sometimes with AssertUnwindSafe to tell the compiler you've reasoned about it. ⚠️VERIFY the exact mechanics in the std docs before leaning on AssertUnwindSafe.

Solution B: panic = "abort"

You can tell Cargo to make panics abort the process immediately instead of unwinding. Then there's no unwind to cross the boundary — but the whole app dies on any panic.

Cargo.toml 
[profile.release]
panic = "abort"

[profile.dev]
panic = "abort"

This is simpler and removes unwinding overhead, but it's a blunt instrument: a single panic kills the host Flutter app rather than returning a recoverable error. For a library others embed, you generally want Solution A — catch panics and return error codes — so the host can decide what to do. Some projects do both (abort as a backstop, catch_unwind for graceful errors).

🦀 Rust just-in-time — also: don't panic on bad input
  • catch_unwind is your safety net, not your error strategy. Normal failures (key not found, bad buffer) should be returned as error codes via Result (Lesson 1.7), not panics. Reserve panics (and the net) for genuine bugs.
  • Replace boundary-layer unwrap()/expect() with proper error handling; they're the most common accidental panic source.
  • Integer overflow panics in debug builds but wraps in release by default — another reason to validate inputs rather than assume.

✅ Concept check

1. What exactly is undefined behavior here — what crosses the boundary illegally?

Show answer
The stack unwind from a panic. C/Dart frames aren't built to be unwound by Rust's machinery, so letting the unwind propagate out of an extern "C" function is UB.

2. What does catch_unwind let you do at the boundary?

Show answer
Run the function body, catch any panic as an Err, and return a normal error code instead of letting the unwind escape.

3. Trade-off of panic = "abort" vs catch_unwind?

Show answer
abort is simple and removes unwinding, but kills the entire host app on any panic. catch_unwind lets you return a recoverable error code, which is better for an embedded library.

4. Should "key not found" be handled by catch_unwind?

Show answer
No — that's an expected failure; return it as an error code from a Result (Lesson 1.7). catch_unwind is only the net for unexpected panics/bugs.
