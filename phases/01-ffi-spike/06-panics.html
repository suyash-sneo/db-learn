1.6 — Panics at an FFI boundary: abort, unwind, or catch

Goal: Understand current Rust panic behavior at C ABI boundaries and choose deliberately between process abort and converting an unwinding panic into an error code.

Why this matters to your DB

A database bug must not become an unwinding mechanism that a Dart caller accidentally receives. Normal failures—bad input, missing keys, I/O errors—belong in Result and error codes. A panic means Rust code violated an assumption or chose to stop. The boundary must define what then happens to the host process and to the database handle's state.

Three separate ideas

  • Panic: Rust begins failure handling because code called panic!, indexed out of bounds, unwrapped an error, or hit another panic path.
  • Unwinding: stack frames are removed and local destructors run while the panic travels upward.
  • Aborting: the process terminates immediately; no panic is returned to Dart.

The compile-time panic strategy controls whether Rust panics unwind or abort. Separately, the ABI on each exported function controls whether unwinding is permitted across that boundary.

What current Rust does at an ABI boundary

BoundaryIf a Rust panic reaches itUse here
extern "C"Rust safely aborts the process instead of letting the unwind escapeYes; this is the normal C ABI
extern "C-unwind"Rust unwinding is permitted to cross the boundaryNo; Dart is not your Rust unwinder
foreign exception enters non-unwind Rust ABIundefined behaviorDo not permit it

So the old slogan “a Rust panic crossing extern "C" is undefined behavior” is no longer accurate. A non-unwinding ABI stops it by aborting. That is memory-safe, but still disastrous for an embedded library because it kills the host application. [src: Rustonomicon: FFI and unwinding]

Option A: catch an unwinding panic and return a code

std::panic::catch_unwind executes a closure and returns Err if that closure unwinds. Put the entire fallible boundary body inside it, then translate the result:

rust 
use std::panic::{catch_unwind, AssertUnwindSafe};

const OK: i32 = 0;
const ERR_OPERATION: i32 = 2;
const ERR_PANIC: i32 = 99;

#[unsafe(no_mangle)]
pub unsafe extern "C" fn db_operation(db: *mut Db) -> i32 {
    if db.is_null() { return 1; }

    match catch_unwind(AssertUnwindSafe(|| {
        // Validate pointers, borrow the handle, then call the safe core.
        // Convert the core's Result to OK or ERR_OPERATION here.
        OK
    })) {
        Ok(code) => code,
        Err(_) => ERR_PANIC,
    }
}

AssertUnwindSafe is not proof that recovery is correct. It says you accept the unwind-safety responsibility. If a panic can leave a handle half-mutated, returning it to Dart for continued use may violate database invariants. Design operations so state changes become visible only after a successful commit, or mark the handle poisoned/unusable after a caught panic. [src: catch_unwind]

Option B: compile panics to abort

Cargo.toml 
[profile.release]
panic = "abort"

This removes unwinding recovery: any panic terminates the process. Because there is no unwind, catch_unwind cannot turn such a panic into ERR_PANIC. Choose this only if terminating the whole Flutter process is the intended failure policy. Do not describe panic="abort" plus catch_unwind as graceful recovery; they are mutually exclusive for an aborting panic.

A panic net is not input validation

Return ordinary error codes for null pointers, invalid lengths, malformed documents, missing keys, and I/O failures. Remove avoidable panic sites from boundary code: do not index before checking, do not unwrap caller-controlled data, and use checked arithmetic for sizes. catch_unwind is a last-resort containment layer for bugs.

🦀 Rust just-in-time
  • catch_unwind catches only unwinding Rust panics, not aborts or every possible crash such as a segmentation fault.
  • A panic hook normally runs before the panic is caught, so logging may still occur.
  • Destructors run during unwinding, but that does not automatically restore logical database invariants.
  • Keep extern "C" for this API. C-unwind is for an intentionally unwind-aware foreign interface, not a shortcut for error handling.
✅ Concept check

1. What happens if an unwinding Rust panic reaches extern "C"?

Show answer
Rust aborts the process rather than allowing the unwind to cross that non-unwind ABI boundary.

2. Can catch_unwind recover a panic="abort" panic?

Show answer
No. An aborting panic does not unwind, so there is nothing for catch_unwind to catch.

3. Why can returning ERR_PANIC still be unsafe at the design level?

Show answer
The operation may have partially changed logical state before panicking. The handle needs transactional mutation or a poisoned state before callers may safely continue.

4. Should malformed input be handled by panicking?

Show answer
No. It is an expected failure and should become a normal Result and stable error code.
