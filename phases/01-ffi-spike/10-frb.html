1.10 — flutter_rust_bridge: what it automates (conceptual)

Goal: Understand what flutter_rust_bridge (frb) does, why it exists, and — importantly — why this curriculum has you do the FFI by hand first. You'll make an informed choice about whether to adopt it, instead of cargo-culting it.

Why this matters to your DB

frb is a feature-rich Rust↔Dart/Flutter binding generator and may be useful for the production binding. It automates exactly the machinery you just learned — pointers, ownership, free functions, the two-signature dance. If you adopt it without understanding what it generates, the day it breaks you'll be helpless. So: learn the manual path (1.1–1.9), then understand what frb is doing for you here.

The problem frb solves

Look back at Lesson 1.9's echoBytes: per function you hand-wrote two typedefs, allocated and freed native buffers, handled the result-buffer ownership, and translated error codes. That's a lot of error-prone boilerplate, repeated for every function, on both sides. Multiply by a real engine's API and it's a maintenance burden — and every hand-written pointer cast is a chance to introduce UB.

What frb generates for you

You write ordinary Rust functions (with real types — String, Vec<u8>, structs, even Result and async), annotate/expose them, and frb's codegen produces:

  • The Rust-side C-ABI glue (the extern "C" shims, with the marshalling).
  • The Dart-side bindings — idiomatic Dart functions you just call, no Pointer in sight.
  • Automatic conversion of rich types across the boundary (it serializes/marshals them for you).
  • Memory management for the buffers it creates (the free-function discipline, handled).
  • Sync and async calling modes, async Rust functions, streams, and Rust-to-Dart calls.

The current documentation labels this generation as v2 and describes ordinary Rust types, Result, async Rust, streams, and two-way calls. [src: flutter_rust_bridge v2 docs]

The trade-offs (decide deliberately)

Hand-rolled C ABI (Lessons 1.1–1.9)flutter_rust_bridge
Full control of the exact ABI surfaceABI shaped by the generator's conventions
Tiny, stable, language-agnostic C API — any language can bind itOptimized for Dart/Flutter specifically
You write/maintain all boilerplateBoilerplate generated; far less to maintain
You must get every pointer/ownership detail rightMarshalling & memory handled for you
You deeply understand every byte that crossesDetails hidden — a problem if you don't know them
A design tension worth naming

Your stated goal is a tiny, stable C ABI that any language can consume (Dart first, but not only). frb generates its own Rust/Dart bridge and codecs; it does not automatically turn your separately designed canonical C API into an idiomatic Dart wrapper. Therefore choose deliberately: make the hand-designed C ABI the product and bind it with ffigen, or make frb's generated bridge the Flutter-facing product. You can maintain both, but then both surfaces need compatibility, packaging, and test coverage.

Why the curriculum still teaches the manual path

  • Understanding before automation. When frb's generated code panics, mismatches a type, or leaks, you can only debug it if you understand pointers, ownership, and the free-function rule — which you now do.
  • The engine's core ABI is the product. The thing other languages call is the C ABI; that's a design artifact you want to own, not delegate.
  • It generalizes. The manual knowledge transfers to Python/Swift/Kotlin bindings; frb knowledge is Flutter-only.
🦀 Generator discipline
  • Pin the generator and runtime crate versions together; generated code and runtime support are a compatibility unit.
  • Commit the generator configuration and make regeneration reproducible in CI.
  • Test the generated bridge on every target you ship; successful desktop generation does not prove Android or iOS packaging.
  • Keep domain logic in ordinary Rust functions so it remains testable without Flutter.
✅ Concept check

1. In one sentence, what does frb generate?

Show answer
From your plain Rust functions, it generates both the Rust C-ABI glue and the idiomatic Dart bindings (plus the marshalling and memory management) — the boilerplate you wrote by hand in Lesson 1.9.

2. Why learn the manual FFI before adopting frb?

Show answer
So you can debug, audit, and reason about what frb hides (pointers, ownership, free functions); and because the underlying knowledge transfers to non-Dart bindings.

3. How does frb sit in tension with your "any language can use it" goal?

Show answer
frb optimizes the boundary for Dart/Flutter specifically, whereas your goal is a tiny, neutral C ABI consumable by any language. You can keep a hand-designed C ABI can remain canonical, but frb is a separate generated bridge rather than an automatic wrapper over that API.

4. What capability does frb add that's painful to hand-roll?

Show answer
Automatic marshalling of rich types (String, Vec, structs, Result) and async/off-isolate execution returning Dart Futures.
