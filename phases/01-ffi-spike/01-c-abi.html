1.1 — What a C ABI is; ABI vs API; why Rust can expose one

Goal: Understand precisely what "exposing a C ABI" means, why the C ABI is the universal language-interop layer, and why Rust is unusually well-suited to producing one.

Why this matters to your DB

Your engine's whole reason to exist is to be callable from any language — Dart today, Swift/Kotlin/Python tomorrow. None of those can call Rust directly. But all of them can call C. So the plan is: write the engine in Rust, then wrap it in a thin layer that looks like C from the outside. To design that layer well, you first need to know what "looks like C" actually means at the binary level.

API vs ABI — the core distinction

These sound alike and are constantly confused. Pin them down:

APIABI
Stands forApplication Programming InterfaceApplication Binary Interface
LevelSource codeCompiled machine code
Answers"What functions/types do I call, and with what arguments?""Once compiled, how are those calls physically made?"
Examplesfunction names, signatures, docsargument passing (registers/stack), name mangling, struct layout/padding, calling convention
Audiencethe programmerthe linker & CPU

Analogy: the API is the menu (what you can order, by name). The ABI is the kitchen's physical protocol — which window the order ticket goes through, how plates are stacked, who hands what to whom. Two restaurants can offer the same menu (API) while running totally different kitchens (ABI). For code in different languages to call each other, they must agree on the ABI, not just the API.

What's actually in an ABI

When language A calls a function compiled from language B, both sides must agree on the mechanics:

  • Calling convention — which CPU registers and stack slots hold the arguments and the return value; who cleans up the stack.
  • Name mangling — the actual symbol name in the compiled binary. C++ and Rust mangle names (encoding types into the symbol, so foo(int) and foo(float) differ). C does not — a C foo is just the symbol foo. Predictable names are what let another language find the function.
  • Data layout — the exact byte arrangement of a struct: field order, alignment, padding. If two sides disagree on layout, they read garbage.
  • Type sizes — how big an int, a pointer, etc. are.

Why "C" specifically is the lingua franca

There's no universal ABI across languages — but the C ABI of a given platform (e.g. the System V AMD64 ABI on Linux/macOS x86-64 ⚠️VERIFY name) is the closest thing. It's simple, stable, well-documented, and every serious language ships a way to call into it (Dart's dart:ffi, Python's ctypes/CFFI, Java's JNI/Panama, Swift's C interop, Go's cgo). So "expose a C ABI" = "present your library as if it were a plain C library," and instantly the whole ecosystem can call it.

Why Rust can expose a C ABI cleanly

Rust normally mangles names and lays out structs however the compiler likes (it doesn't promise a stable Rust ABI). But it gives you opt-in switches to produce C-compatible output for specific items:

  • extern "C" on a function → use the C calling convention.
  • #[no_mangle] → keep the symbol name exactly as written.
  • #[repr(C)] on a struct/enum → lay it out the way C would.

You'll meet all three in detail next lesson. The point here: Rust deliberately separates "fast, flexible internal code" from "stable C-shaped boundary code," and you choose which is which. Your engine core is the former; your shim is the latter.

The three-layer plan, restated in ABI terms

Layer 1 (Rust core): idiomatic Rust, mangled names, native layout — no ABI promises, maximum freedom. Layer 2 (C ABI shim): a thin set of extern "C" functions over opaque handles — the only thing the outside world sees. Layer 3 (bindings): Dart code that calls Layer 2 through dart:ffi. Keeping Layer 2 tiny is the whole strategy; the rest of Phase 1 is how to build it safely.

✅ Concept check

1. In one line each: API vs ABI?

Show answer
API = the source-level contract (names, signatures). ABI = the binary-level contract (calling convention, name mangling, data layout) that governs how compiled code actually calls compiled code.

2. Why does targeting C specifically let Dart, Python, Swift, and Kotlin all use your Rust engine?

Show answer
Because there's no universal cross-language ABI, but the platform's C ABI is the de-facto standard every language can call into. Presenting your library as "C-shaped" makes it callable from all of them.

3. What problem does name mangling cause for FFI, and how does C avoid it?

Show answer
Mangling encodes type info into symbol names (so they're unpredictable to other languages). C doesn't mangle — a function foo is literally the symbol foo — so callers can find it by name.

4. Does Rust have a stable ABI of its own? What does that imply for your design?

Show answer
No — Rust's own ABI is unstable by design. So cross-language interop must go through a C ABI boundary, which is exactly why Layer 2 (the shim) exists and stays thin.
