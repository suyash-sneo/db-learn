Build · AI off

Build: a byte buffer round-tripping Rust → C ABI → Dart

Goal: Retire the project's biggest risk. Prove a Uint8List can travel from a real Flutter app, into your Rust library through a C ABI, get transformed, and come back — with correct memory ownership and no leaks. When this works, the scariest unknown is gone and the engine has a safe place to land.
⚠️ The rule

AI off. You may use the Rust Book, the Rustonomicon, the dart:ffi/cbindgen docs, and the Stuck? ladder. You may not have AI write the FFI glue — that boundary code is exactly the skill this spike builds. Build pages give you stubs and tests; the implementation is yours.

Deliverable spec

A Cargo crate ffi_echo (crate-type cdylib) exposing a C ABI, a generated header, and a minimal Flutter app that calls it. The "echo" deliberately exercises every transfer rule: it takes input bytes and returns a new, Rust-allocated buffer that is the input with each byte incremented by 1 (so you can verify the bytes actually made the round trip and were processed, not just copied).

Required C ABI surface (names are fixed so the tests match):

  • echo_transform(in_ptr, in_len, out_ptr, out_cap, out_len) -> i32 — the caller passes input (borrowed) and an out-param trio; on success Rust allocates the result buffer, writes its pointer/len through the out-params, returns 0. (Use the buffer-descriptor or out-pointer style from Lesson 1.5 — your choice; keep it consistent with the tests below.)
  • echo_free(ptr, len, cap) — frees a buffer that echo_transform allocated.
  • echo_version() -> i32 — returns a constant version int (a trivial warm-up call).

Ordered milestones

  1. M1 — Library builds & exports. Crate compiles as cdylib; echo_version is exported. Verify with nm/cargo build. (Lessons 1.2)
  2. M2 — Pure-Rust core. A safe fn transform(input: &[u8]) -> Vec<u8> that increments each byte. Unit-test it directly. (Keep the logic in safe Rust; the boundary fn just calls it.)
  3. M3 — The C ABI shim. Implement echo_transform and echo_free: null-check pointers, borrow the input slice, call transform, hand out the Rust buffer, and free it correctly. Wrap the body in catch_unwind. (Lessons 1.3–1.7)
  4. M4 — Header. Generate ffi_echo.h with cbindgen. (Lesson 1.8)
  5. M5 — Flutter app. Load the library, bind the functions, send a Uint8List, display the returned bytes, free everything. (Lesson 1.9)

Acceptance criteria

  • cargo test green (core + boundary tests below).
  • The Flutter app, given input [1,2,3], displays [2,3,4].
  • No memory leaks: a loop calling echo_transform+echo_free thousands of times shows flat memory. (Manual check / a Rust loop test.)
  • Robustness: a null input pointer with non-zero length returns an error code, doesn't crash; a forced panic inside the boundary returns the panic code, doesn't unwind into the caller.
  • Empty input (len == 0) is handled (returns an empty result, no UB).

Failing test suite (Rust)

tests — drive M2 & M3 from Rust before touching Flutter 
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn core_increments() {
        assert_eq!(transform(&[1, 2, 3]), vec![2, 3, 4]);
        assert_eq!(transform(&[]), Vec::<u8>::new());
        assert_eq!(transform(&[255]), vec![0]);   // wrapping_add
    }

    #[test]
    fn boundary_round_trip() {
        let input = [10u8, 20, 30];
        let mut out_ptr: *mut u8 = std::ptr::null_mut();
        let mut out_len: usize = 0;
        let mut out_cap: usize = 0;

        let rc = echo_transform(
            input.as_ptr(), input.len(),
            &mut out_ptr, &mut out_len, &mut out_cap,
        );
        assert_eq!(rc, OK);
        assert_eq!(out_len, 3);

        let result = unsafe { std::slice::from_raw_parts(out_ptr, out_len) };
        assert_eq!(result, &[11u8, 21, 31]);

        echo_free(out_ptr, out_len, out_cap);   // must not crash / double-free
    }

    #[test]
    fn null_input_is_error_not_crash() {
        let mut out_ptr: *mut u8 = std::ptr::null_mut();
        let mut out_len: usize = 0;
        let mut out_cap: usize = 0;
        let rc = echo_transform(std::ptr::null(), 5,
            &mut out_ptr, &mut out_len, &mut out_cap);
        assert_eq!(rc, ERR_NULL);
    }

    #[test]
    fn no_leak_under_load() {        // watch memory stay flat while this runs
        let input = [0u8; 1024];
        for _ in 0..100_000 {
            let (mut p, mut l, mut c) = (std::ptr::null_mut(), 0usize, 0usize);
            echo_transform(input.as_ptr(), input.len(), &mut p, &mut l, &mut c);
            echo_free(p, l, c);
        }
    }
}

Reuse the structure from Lesson 1.9 (load lib, two typedefs per function, allocate input, read output, copy with Uint8List.fromList, then echo_free the Rust buffer and malloc.free your own). Acceptance: the screen shows [2, 3, 4].

Smoke-test checklist (Dart)
  • ☐ echo_version() returns 1 (proves the library loads & binds).
  • ☐ [1,2,3] → [2,3,4] displayed.
  • ☐ Empty input → empty result, no exception.
  • ☐ Pressing the button 1000× doesn't grow memory (you freed both sides).

Stuck? — graduated hints

M3: how do I "hand out" the Rust Vec?
Nudge: A Vec knows its pointer, length, and capacity. You need to give all three to the caller and stop Rust from freeing it.
Bigger: Grab v.as_mut_ptr(), v.len(), v.capacity(), write them through the out-params, then prevent the drop.
Technique: std::mem::forget(v) leaks it (Lesson 1.5). The matching reclaim in echo_free is Vec::from_raw_parts(ptr, len, cap) — which is why you must pass capacity back, not just length.
M3: where exactly does catch_unwind go?
Nudge: The whole body that could panic must be inside it; the out-param writes are the risky part if a pointer is bad.
Technique: Null-check first (return ERR_NULL), then put the borrow + transform + hand-out inside catch_unwind(|| { … OK }), and map Err(_) to ERR_PANIC (Lesson 1.6). Writing through a raw out-pointer inside the closure may need AssertUnwindSafe — read its docs.
M5: DynamicLibrary.open can't find my library
Nudge: The path is relative to where the app runs, not where the .dylib/.so is built.
Bigger: For desktop dev, copy the built library next to the executable, or pass an absolute path while iterating.
Technique: confirm the file name per platform (lib*.so/lib*.dylib/ *.dll) and that the symbol exists (nm). On iOS/Android you static-link and use DynamicLibrary.process() — that's a Phase 8.7 concern; do desktop first.
M5: the returned bytes are garbage or the app crashes on free
Nudge: Two classic causes — a wrong native typedef, or freeing with the wrong allocator.
Bigger: Re-check the native typedef matches the header exactly (Size vs Uint64, pointer-to-pointer for out-params). And confirm you copy with Uint8List.fromList before calling echo_free.
Technique: Dart frees only what malloc made; the Rust result buffer is freed only by echo_free. Never cross them (Lesson 1.5/1.9).

Definition of Done

  • ☐ All four Rust tests pass; the leak test keeps memory flat.
  • ☐ ffi_echo.h generated by cbindgen, declaring an opaque-free, length-prefixed API.
  • ☐ Flutter app shows [1,2,3] → [2,3,4].
  • ☐ Null input → error code; a forced panic → panic code; neither crashes the caller.
  • ☐ You can explain, in your own words, who allocates and who frees in each direction.
  • ☐ You wrote every line yourself.
📚 Why this is the right thing to have built first

You now have a proven, leak-free, panic-safe channel from Flutter to Rust. Every later phase builds the engine behind this channel; you never again have to wonder whether the boundary works. The riskiest unknown in the project is retired — exactly what a spike is for.
