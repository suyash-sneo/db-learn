Phase 1

FFI spike — de-risk the scariest unknown early

Before you build a single page or B-tree node, you'll prove the hardest, least familiar part of the whole project actually works: a byte buffer travelling from Rust, across a C ABI, into a running Flutter app — and back. This is a deliberate "spike": a small, focused experiment that retires the biggest risk so the rest of the engine has somewhere safe to land.

Why do this so early?

The C ABI / FFI boundary is where most ambitious Rust+Flutter projects quietly die — memory ownership across languages, panics turning into crashes, header generation, build/linking. If you're going to hit a wall, hit it now, on a trivial "echo" program, not after months of engine work. By the end of Phase 1 you'll have a real Flutter app calling real Rust.

What you'll be able to do after Phase 1

Expose a Rust function with a stable C ABI, hand out an opaque handle to a Rust object, pass byte buffers in and out with correct ownership and a free function, stop panics from crossing the boundary, report errors as integer codes, generate a C header with cbindgen, and call all of it from Dart with dart:ffi.
