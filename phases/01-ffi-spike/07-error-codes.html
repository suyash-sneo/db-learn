1.7 — Errors across a C ABI: integer codes + out-params

Goal: Translate Rust's rich Result<T, E> into something C understands: an integer return code plus output values written through pointers. Design a small, stable error-code scheme now so the whole engine speaks it.

Why this matters to your DB

C has no built-in equivalent of Rust's Result or Option. A common, portable C API convention is: the function returns an integer status code; output values are written to caller-provided pointers (out-params). This engine will follow that convention, so the caller can write if (rc != 0) handle_error(); in any language. Lesson 0.5 built your Rust error enum; this lesson maps it to the wire.

The convention: return code, output via pointer

SQLite is the model: nearly every function returns an int result code (0 = success, non-zero = a specific error), and produces its actual result through an out-parameter. You'll adopt the same shape:

rust 
// Real return = status. Real output (a found document's length) via *out_len.
#[unsafe(no_mangle)]
/// # Safety: db must be live and exclusive; out_len must be null or writable.
pub unsafe extern "C" fn db_find_len(db: *mut Db, id: i64, out_len: *mut usize) -> i32 {
    if db.is_null() || out_len.is_null() { return ERR_NULL; }
    // … look up; on success write the length and return OK …
    unsafe { *out_len = 42; }
    OK
}

Convention details worth fixing early: 0 means success; negatives or a numbered scheme for errors; out-params are only valid to read when the code is success.

A stable error-code scheme

Define your codes as named constants in one place. Stability matters: once shipped, a code's meaning shouldn't change, or you break every binding. Reserve 0 for OK.

rust 
pub const OK: i32 = 0;
pub const ERR_NULL: i32 = 1;        // a required pointer was null
pub const ERR_NOT_FOUND: i32 = 2;   // key/doc not found
pub const ERR_CORRUPT: i32 = 3;     // file/data invalid
pub const ERR_IO: i32 = 4;          // underlying I/O failure
pub const ERR_PANIC: i32 = 99;      // a panic was caught (Lesson 1.6)

The translation layer: Result → code

Keep your engine core returning idiomatic Result<T, DbError>. The shim is the only place that flattens it to a code. Write one mapping function and reuse it:

rust 
fn code_of(e: &DbError) -> i32 {
    match e {
        DbError::KeyNotFound(_) => ERR_NOT_FOUND,
        DbError::Corrupt(_)     => ERR_CORRUPT,
        DbError::Io(_)          => ERR_IO,
        DbError::BadMagic       => ERR_CORRUPT,
    }
}

// In a boundary fn: run engine logic, then translate.
match db.insert(&bytes) {            // engine returns Result<(), DbError>
    Ok(())  => OK,
    Err(e) => code_of(&e),
}
The full boundary pattern, assembled

Every public function now layers the four ideas you've learned: (1) null-check pointers → return ERR_NULL; (2) wrap the body in catch_unwind → return ERR_PANIC on a caught panic (1.6); (3) call the safe engine core, which returns a Result; (4) map that to a code, writing any output through out-params (1.5 for buffers). The shim is thin and mechanical; the intelligence stays in the safe core.

Richer error detail (optional, SQLite-style)

A bare code can't carry a message. SQLite's sqlite3_errmsg() returns UTF-8 text describing the most recent failed API call associated with a database connection; later calls may invalidate or replace that text. You can use the same general shape: store error text on the Db handle and expose db_last_error(db) -> *const c_char (whose buffer the handle owns). Adds richness without bloating every function's signature. Its lifetime and thread-safety contract must be explicit. This is a Phase 8 refinement; for the spike, codes suffice. [src: SQLite error APIs]

Out-params: the rules

  • Validate the out pointer (null-check) before writing — a null out-param is a caller bug; return ERR_NULL.
  • Only write outputs on success. If you return an error, leave out-params untouched (or zeroed) and document that the caller must not read them.
  • Two-call pattern for sizes: for variable-size output, one call reports the needed length, the caller allocates, a second call fills it — or use the buffer-descriptor approach from Lesson 1.5. Pick one and be consistent.
🦀 Rust just-in-time
  • Keep the mapping from DbError to ABI integers in one exhaustive match; adding an engine error then forces an explicit ABI decision.
  • Do not export a Rust enum directly as the long-term error contract unless its representation and numeric values are fixed deliberately. Named i32 constants are simpler here.
  • Initialize or leave out-parameters untouched according to one documented rule. This curriculum initializes buffer outputs to null/zero before work begins.
  • An error message pointer needs an ownership and lifetime rule just like any other buffer.
✅ Concept check

1. In the C convention, what does the function's integer return value represent, and where does the actual result go?

Show answer
The return value is a status/error code (0 = success). The real result is written through caller-provided out-pointers (out-params).

2. Why keep the engine core returning Result and only translate to codes in the shim?

Show answer
The core stays idiomatic, testable, and uses ? freely; the lossy flattening to integer codes happens once, at the boundary, where it's required. Separation of concerns.

3. When is it safe for the caller to read an out-param?

Show answer
Only when the function returned the success code. On error, out-params should be considered unset.

4. Why must error codes be stable once shipped?

Show answer
Bindings in every language compare against specific numbers; changing a code's meaning silently breaks all callers. Treat the code list as part of the ABI.
