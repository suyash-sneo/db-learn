1.4 — The opaque-handle pattern (Box::into_raw / from_raw)

Goal: Learn the single most important pattern in your whole C ABI: how to hand the caller a pointer to a Rust object they can hold and pass back, without exposing any of its internals — and how to give it back to Rust to be freed correctly.

Why this matters to your DB

The caller needs something that represents an open database — to pass to insert, find, close. That "something" is an opaque handle: a pointer to your real Rust Db struct that the caller treats as a meaningless token. They never see its fields, never know its size — they just keep the pointer and give it back. This is exactly how sqlite3 * works, and it's the backbone of Layer 2.

The problem it solves

You have a rich Rust type — Db { pager, btree, cache, … } — full of Vecs, HashMaps, and other non-C types. You can't pass that across the C ABI by value (C has no idea how to lay it out, and you don't want to expose it). But the caller needs a persistent reference to it across many calls. Solution: keep the object on the Rust heap, and give the caller only its address as an opaque pointer.

The lifecycle: into_raw on create, from_raw on destroy

Three moves make up the pattern:

  1. Box it — put the object on the heap with Box::new(...).
  2. Leak it out — Box::into_raw converts the Box into a raw pointer and tells Rust to stop managing it (so it won't be freed when the function returns). You hand this pointer to C.
  3. Reclaim it — when the caller is done, they pass the pointer back to your close function, which uses Box::from_raw to rebuild the Box. Now Rust manages it again, and dropping that Box frees the object correctly.

The code shape

This is FFI plumbing (not a database component), so it's fine to show the pattern in full:

rust 
pub struct Db { /* your real engine state */ }

// CREATE: heap-allocate, hand out the pointer, stop managing it.
#[unsafe(no_mangle)]
pub extern "C" fn db_open() -> *mut Db {
    let db = Box::new(Db { /* … */ });
    Box::into_raw(db)        // pointer escapes; Rust won't free it now
}

// USE: borrow the object from the pointer WITHOUT taking ownership.
#[unsafe(no_mangle)]
/// # Safety: `db` must be a live handle returned by db_open and exclusively used for this call.
pub unsafe extern "C" fn db_do_thing(db: *mut Db) -> i32 {
    if db.is_null() { return 1; }
    let db: &mut Db = unsafe { &mut *db };  // reborrow; does NOT free
    // … use db …
    0
}

// DESTROY: reclaim ownership so the Box drops and frees the Db.
#[unsafe(no_mangle)]
/// # Safety: `db` must be a live handle from db_open and must not have been closed before.
pub unsafe extern "C" fn db_close(db: *mut Db) {
    if db.is_null() { return; }
    unsafe { let _ = Box::from_raw(db); }   // rebuilt Box drops here → freed
}

Read the three carefully: into_raw is "ownership out," from_raw is "ownership back in," and the use-case in the middle only borrows (&mut *db) so it never frees anything. The functions are marked unsafe for Rust callers because a null check cannot prove that an arbitrary non-null address is a live handle. [src: Box::from_raw safety contract]

⚠️ The three rules that prevent disaster
  • Borrow in use, own in close. If db_do_thing accidentally did Box::from_raw, it'd free the Db while the caller still holds the pointer → use-after-free on the next call. Use &* / &mut * to borrow.
  • Free exactly once. Calling db_close twice → double-free. The caller owns the lifecycle; document it loudly.
  • Null after free (on the caller side). Dart should drop its reference / set it to null after close so it can't be reused.
  • No concurrent or reentrant use in this phase. Creating &mut Db promises exclusive access. The same handle must not be used by two calls at once, and a callback must not reenter it. A later API can put synchronized interior mutability behind the handle if concurrent access is required.

Why "opaque"?

From C/Dart's perspective the pointer points at an incomplete type — declared but never defined. The header says typedef struct Db Db; with no fields. The caller literally cannot read or construct a Db; they can only pass the pointer to your functions. That opacity is a feature: you can change Db's internals freely without breaking the ABI, and the caller can't corrupt your invariants. With an appropriate public API and configuration, cbindgen (Lesson 1.8) can emit the corresponding forward declaration.

🦀 Rust just-in-time
  • Box<T> is a smart pointer that owns a heap allocation; dropping the Box frees it. into_raw/from_raw are how you temporarily step outside that automatic management.
  • let _ = ...; binds-and-immediately-drops — here it makes the "rebuild the Box just to drop it" intent explicit.
  • &mut *ptr is a "reborrow": create a reference from a raw pointer. Only valid inside unsafe, and only if the pointer is truly valid (hence the null check).

✅ Concept check

1. What does Box::into_raw do to Rust's ownership of the object?

Show answer
It releases Rust's automatic management: the object stays alive on the heap and won't be freed until someone calls from_raw to take ownership back. You've "leaked" it deliberately.

2. Why must a "use" function borrow (&mut *ptr) rather than from_raw?

Show answer
Because from_raw takes ownership, and the resulting Box would be dropped at function end — freeing the object the caller still holds a pointer to (use-after-free). Borrowing leaves ownership where it is.

3. What goes wrong if the caller calls db_close twice?

Show answer
Double-free / undefined behavior. The lifecycle must be exactly one open ↔ one close; the caller is responsible, and you document it.

4. Why is it valuable that C sees Db as an incomplete (opaque) type?

Show answer
The caller can't read or build a Db, so they can't break your invariants, and you can change its internals without changing the ABI. The pointer is just a token.
