1.5 — Byte buffers across the boundary; ownership & the free function

Goal: Move arbitrary bytes between Dart and Rust with explicit lifetime, ownership, and deallocation rules. These bytes will later hold serialized documents.

Why this matters to your DB

The database accepts and returns binary documents. A C ABI has no Rust Vec<u8> or Dart Uint8List, so the shared representation is a pointer plus a byte length. The pointer says where; the length says how much; the API contract says who owns it and for how long.

The four facts every buffer contract must state

  1. Is the pointer readable, writable, or both?
  2. How many bytes are valid?
  3. How long may the receiver use them?
  4. Which function eventually frees the allocation?

Length-prefixed buffers are binary-safe: a document may contain 0x00, so it cannot be represented as a null-terminated C string.

Dart → Rust: borrow for one call, copy if retained

Dart allocates the input. Rust may read it only while the call is active and must not free it. If the engine needs the data after returning, it copies the slice into Rust-owned storage.

rust 
/// # Safety
/// `db` must be a live, exclusively used handle.
/// If `len > 0`, `ptr` must identify `len` readable bytes for this call.
#[unsafe(no_mangle)]
pub unsafe extern "C" fn db_insert(
    db: *mut Db,
    ptr: *const u8,
    len: usize,
) -> i32 {
    if db.is_null() || (ptr.is_null() && len != 0) {
        return 1;
    }

    let bytes: &[u8] = if len == 0 {
        &[]
    } else {
        unsafe { std::slice::from_raw_parts(ptr, len) }
    };

    let owned = bytes.to_vec(); // keep this; never keep `bytes` or `ptr`
    // pass `owned` to the safe engine core
    0
}
⚠️ Null + zero needs a separate branch

slice::from_raw_parts requires a non-null, aligned pointer even for a zero-length slice. C APIs often represent an empty buffer as (NULL, 0), so map that pair directly to &[]. Reject (NULL, nonzero). A non-null pointer still depends on the caller's validity, alignment, lifetime, and allocation-size contract. [src: slice::from_raw_parts]

Rust → Dart: transfer a boxed slice, then return it to Rust

Rust allocates the output. The caller may read it until it invokes the exported Rust free function. Do not call C's free() on it: the API that creates an allocation must define the matching deallocation operation. Even if two runtimes currently use the same system allocator, that is not a stable cross-language contract.

Use Box<[u8]> rather than exposing a Vec. A boxed slice has no spare capacity, so pointer + length are enough to reconstruct it.

rust 
#[repr(C)]
pub struct Buf {
    pub ptr: *mut u8,
    pub len: usize,
}

fn vec_into_buf(v: Vec<u8>) -> Buf {
    if v.is_empty() {
        return Buf { ptr: std::ptr::null_mut(), len: 0 };
    }
    let boxed: Box<[u8]> = v.into_boxed_slice();
    let len = boxed.len();
    let raw: *mut [u8] = Box::into_raw(boxed);
    Buf { ptr: raw as *mut u8, len }
}

/// # Safety
/// `ptr, len` must be one live pair returned by this library, unchanged,
/// and this function must be called exactly once for that pair.
#[unsafe(no_mangle)]
pub unsafe extern "C" fn db_free_buf(
    ptr: *mut u8,
    len: usize,
) {
    if ptr.is_null() { return; }
    let raw: *mut [u8] =
        std::ptr::slice_from_raw_parts_mut(ptr, len);
    unsafe { drop(Box::from_raw(raw)); }
}

Box::into_raw transfers ownership out without freeing. The free function rebuilds the original wide slice pointer from the data pointer and length, then Box::from_raw reclaims ownership so dropping the box frees it. [src: Box::into_raw; slice_from_raw_parts_mut]

Alternative: caller-provided output

An API can instead ask the caller for a writable pointer and capacity, then report either the bytes written or the required size. This avoids cross-boundary ownership transfer but needs retry logic for variable-size documents. The later API-design phase will compare both approaches.

🦀 Rust just-in-time
  • A raw pointer has no length or lifetime. Those facts exist only in the surrounding contract.
  • from_raw_parts creates a temporary view; it does not copy or take ownership.
  • Vec::into_boxed_slice removes spare capacity from the ABI design.
  • The caller copies Rust output into a Dart-owned Uint8List before freeing it.
✅ Concept check

1. Why use pointer + length instead of a C string?

Show answer
Serialized data may contain zero bytes; an explicit length is binary-safe.

2. Why copy borrowed Dart input before returning?

Show answer
Dart may free or reuse its allocation as soon as the call ends. The engine needs its own owned copy for longer storage.

3. Why is a null check not a complete pointer validation?

Show answer
A non-null address may still be dangling, misaligned, too short, read-only, or concurrently aliased. The caller contract must guarantee those properties.

4. Why does the output API not expose a Vec capacity?

Show answer
The vector is converted to a boxed slice. Its pointer and exact length are sufficient to reconstruct and free that allocation.
