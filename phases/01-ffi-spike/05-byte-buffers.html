1.5 — Byte buffers across the boundary; ownership & the free function

Goal: Master the single most important data transfer in your whole project: moving a buffer of bytes (a serialized document) between Rust and Dart, with rock-solid rules about who allocated it and who frees it. Get this wrong and you leak or crash; get it right and the engine's data path is trivial.

Why this matters to your DB

Recall the design: documents cross the FFI boundary as serialized byte buffers (BSON/MessagePack). So every meaningful operation — insert a document, return a found document — is a byte-buffer transfer. There are exactly two directions, and each has one correct ownership rule. This lesson is those two rules.

A buffer on the C ABI = pointer + length

There's no Vec in C. A byte buffer crosses as two values: a pointer to the first byte (*const u8 / *mut u8) and a length (usize). Length-prefixed, not null-terminated — because document bytes can legitimately contain 0x00, so a C-string terminator would corrupt them. Always carry the length explicitly.

Direction 1: Dart → Rust (caller owns)

Dart has the document bytes and wants Rust to read them. Dart allocates the buffer, passes pointer+len, Rust borrows it for the duration of the call, and Dart frees it afterward. Rust must not hold the pointer past the call or free it.

rust 
/// Rust reads `len` bytes that Dart owns. Returns 0 on success.
#[no_mangle]
pub extern "C" fn db_insert(db: *mut Db, ptr: *const u8, len: usize) -> i32 {
    if db.is_null() || (ptr.is_null() && len != 0) { return 1; }
    // Borrow Dart's bytes as a Rust slice for THIS CALL ONLY.
    let bytes: &[u8] = unsafe { std::slice::from_raw_parts(ptr, len) };
    // … deserialize & store a COPY; never keep `bytes` after returning …
    0
}
⚠️ The borrow is only valid during the call

slice::from_raw_parts creates a Rust slice view over memory Rust doesn't own. The moment db_insert returns, Dart may free that memory. So if you need to keep the bytes (you do — they go into a page), .to_vec() them into Rust-owned memory before returning. Stashing the raw slice/pointer is a dangling-pointer bug waiting to fire.

Direction 2: Rust → Dart (Rust owns, Dart must give it back)

This is the tricky direction. Rust produces bytes (a found document) and needs to hand them to Dart. Rust allocates the buffer. But here's the rule that trips everyone:

⚠️ Memory must be freed by the allocator that made it

Rust's allocator and the C/Dart allocator are different. If Rust mallocs (well, allocates a Vec) and Dart tries to free it, you get heap corruption. So when Rust hands out a buffer, you must also export a Rust free function and require the caller to return the buffer to it. "Whoever allocated it, frees it."

The clean pattern: return a small descriptor (ptr + len), keep Rust's ownership "leaked," and provide a matching free:

rust 
#[repr(C)]
pub struct Buf { ptr: *mut u8, len: usize, cap: usize }  // cap needed to rebuild the Vec

// Hand a Rust-allocated buffer to the caller.
fn vec_into_buf(mut v: Vec<u8>) -> Buf {
    let b = Buf { ptr: v.as_mut_ptr(), len: v.len(), cap: v.capacity() };
    std::mem::forget(v);   // leak: stop Rust from freeing it now
    b
}

// The REQUIRED matching free — caller must call this, not C free().
#[no_mangle]
pub extern "C" fn db_free_buf(b: Buf) {
    if b.ptr.is_null() { return; }
    unsafe { let _ = Vec::from_raw_parts(b.ptr, b.len, b.cap); } // rebuilt Vec drops → freed
}

Notice the symmetry with the opaque handle (1.4): mem::forget here plays the role of Box::into_raw (leak ownership out), and Vec::from_raw_parts in the free plays the role of Box::from_raw (reclaim and drop). Same idea, different container.

An alternative: caller-provides-buffer (out-param style)

SQLite-style APIs often avoid returning allocations by having the caller pass a buffer and a capacity; the function fills it and reports the needed size (call twice if too small). This dodges the cross-allocator problem entirely but is clumsier for variable-size documents. You'll weigh both in Phase 8; for the spike, the leak-out + free-back pattern above is simplest to demonstrate.

🦀 Rust just-in-time
  • slice::from_raw_parts(ptr, len) = "view these external bytes as &[u8]" (read); from_raw_parts_mut for &mut.
  • mem::forget(v) = "drop my responsibility to free v" (no destructor runs). It's the leak primitive; pair it with a matching reclaim.
  • Vec::from_raw_parts(ptr, len, cap) rebuilds a Vec — you need the original capacity, not just length, which is why Buf carries cap.
  • On the Dart side you'll copy the bytes into a Dart Uint8List and then call the free function (Lesson 1.9).

✅ Concept check

1. Why length-prefixed buffers instead of null-terminated C strings for documents?

Show answer
Serialized documents can contain 0x00 bytes; a null terminator would truncate them. A length is binary-safe.

2. Rust receives Dart's buffer in db_insert and needs to store it. Why must it .to_vec() before returning?

Show answer
The slice only borrows Dart's memory, which Dart may free as soon as the call returns. Copying into a Rust-owned Vec keeps the data valid for storage.

3. Why can't Dart just free() a buffer that Rust allocated?

Show answer
Rust and C/Dart use different allocators; freeing with the wrong one corrupts the heap. Memory must be freed by the same allocator that made it — hence an exported Rust free function.

4. What two std functions form the "leak out / reclaim" pair for a Vec<u8> crossing to C and back?

Show answer
mem::forget (leak it out, after grabbing ptr/len/cap) and Vec::from_raw_parts (rebuild and drop in the free function).
