1.8 — Generating the C header with cbindgen

Goal: Auto-generate a C header (.h) from your Rust extern "C" declarations, so C/Dart callers have an accurate, always-in-sync description of your ABI — without you hand-writing and hand-maintaining it.

Why this matters to your DB

A C library is two things: a compiled binary and a header file declaring its functions and types. Dart's FFI tooling reads that header to know what to bind. Writing the header by hand is tedious and drifts out of sync with the Rust source the instant you change a signature. cbindgen reads your Rust and emits the matching header automatically — your Rust stays the single source of truth.

What cbindgen does

cbindgen scans your crate for #[no_mangle] extern "C" functions, #[repr(C)] types, and constants, and produces a C (or C++) header declaring them. Opaque types (your boxed handles) become forward-declared incomplete structs; your error-code constants become #defines or an enum.

Two ways to run it

1. As a CLI (simplest for the spike)

terminal 
cargo install cbindgen           # once ⚠️VERIFY current install method
cbindgen --crate mydb --output mydb.h
# optionally with a config file:
cbindgen --config cbindgen.toml --crate mydb --output mydb.h

2. As a build script (auto-regenerate on every build)

A build.rs file runs before compilation; you can have it invoke cbindgen so the header is always fresh. This needs cbindgen as a build dependency:

Cargo.toml 
[build-dependencies]
cbindgen = "0.27"   # ⚠️VERIFY current version

The build.rs calls cbindgen's API to generate mydb.h into your output directory. ⚠️VERIFY the exact build.rs API against the cbindgen docs — it changes between versions.

Configuration (cbindgen.toml)

A small config controls language, naming, includes, and an autogenerated-warning banner:

cbindgen.toml 
language = "C"
include_guard = "MYDB_H"
autogen_warning = "/* Generated by cbindgen. Do not edit. */"
# style, prefixes, etc. ⚠️VERIFY exact keys against cbindgen docs

What the generated header looks like

For the opaque-handle + buffer + error-code API you've designed, cbindgen produces roughly:

mydb.h (generated — illustrative ⚠️VERIFY exact output) 
/* Generated by cbindgen. Do not edit. */
#ifndef MYDB_H
#define MYDB_H
#include <stdint.h>
#include <stddef.h>

typedef struct Db Db;            // opaque — no fields exposed

#define OK 0
#define ERR_NULL 1
#define ERR_NOT_FOUND 2

Db *db_open(void);
void db_close(Db *db);
int32_t db_insert(Db *db, const uint8_t *ptr, size_t len);

#endif  /* MYDB_H */

Note how the design pays off: the handle is opaque (typedef struct Db Db; with no definition), buffers are const uint8_t * + size_t, and results are int32_t codes — exactly the surface you built across Lessons 1.2–1.7. cbindgen just transcribes it.

🦀 Rust just-in-time
  • build.rs is a special file Cargo compiles and runs before building your crate — used for codegen, linking flags, finding native libs.
  • cbindgen only sees what's reachable and C-compatible: #[no_mangle] extern "C" functions, #[repr(C)] types, exported consts. Internal Rust types are correctly ignored.
  • Doc comments (///) on your exported items can be carried into the header as C comments — good for the people consuming your API. ⚠️VERIFY.
  • Read more: cbindgen docs/book — github.com/mozilla/cbindgen ⚠️VERIFY URL and the docs.md/book for config keys and the build.rs API.
✅ Concept check

1. What problem does cbindgen solve that hand-writing a header doesn't?

Show answer
It keeps the header automatically in sync with the Rust source (the single source of truth), eliminating drift and transcription errors when signatures change.

2. How does your opaque Db handle appear in the generated header?

Show answer
As a forward-declared incomplete type (typedef struct Db Db; with no fields) — callers can hold a Db * but can't inspect or construct it.

3. Which Rust items does cbindgen include in the header, and which does it ignore?

Show answer
It includes #[no_mangle] extern "C" functions, #[repr(C)] types, and exported constants; it ignores ordinary internal Rust types and functions.

4. What's the advantage of generating the header from a build.rs vs running the CLI by hand?

Show answer
The header regenerates automatically on every build, so it can never go stale relative to the code.
