1.8 — Generate the C header with cbindgen

Goal: Generate a C declaration of the Rust ABI, understand what cbindgen can and cannot infer, and keep the checked-in header synchronized with the Rust source.

Why this matters to your DB

The compiled library contains symbols and machine code, but consumers also need declarations: function names, parameter types, constants, and public C-layout types. Hand-written declarations can silently drift from Rust. cbindgen derives the header from the public Rust API. [src: cbindgen repository]

cbindgen and ffigen do opposite jobs

ToolInputOutput
cbindgenRust declarationsC/C++ header
package:ffigenC headerDart FFI bindings

dart:ffi is the runtime API used by those Dart bindings; it does not itself parse a header. Dart's official C interop guide identifies package:ffigen as the header-to-Dart generator. [src: Dart C interop]

Generate a C header

terminal 
cargo install cbindgen

# --lang c matters: the CLI's default output language is C++.
cbindgen --crate mydb --lang c --output include/mydb.h

# With a project config:
cbindgen --config cbindgen.toml --crate mydb --output include/mydb.h

The current cbindgen CLI documents C++ as its default language, so either pass --lang c or commit a config containing language = "C". [src: cbindgen CLI/config docs]

cbindgen.toml 
language = "C"
include_guard = "MYDB_H"
autogen_warning = "/* Generated by cbindgen. Do not edit. */"
documentation = true

What it sees

cbindgen follows the crate's public API and emits C declarations for compatible exported functions, constants, and types. Types with a guaranteed representation can be defined in the header; types without an exposed layout can appear as forward declarations and serve as opaque handles. Internal Rust implementation details remain absent. [src: cbindgen documentation]

Rust input 
pub struct Db { /* private engine fields */ }

pub const MYDB_OK: i32 = 0;

#[unsafe(no_mangle)]
pub extern "C" fn mydb_open() -> *mut Db {
    todo!()
}

#[unsafe(no_mangle)]
pub unsafe extern "C" fn mydb_close(db: *mut Db) {
    todo!()
}
Representative C output — inspect your generated file for exact formatting 
#ifndef MYDB_H
#define MYDB_H
#include <stdint.h>

typedef struct Db Db;

extern const int32_t MYDB_OK;

Db *mydb_open(void);
void mydb_close(Db *db);

#endif

Do not teach an illustrative rendering as byte-for-byte promised output. cbindgen formatting and whether a constant becomes a declaration, definition, enum, or macro depend on configuration and item shape. The generated header is the authority for the build that produced it.

Keep the generated file honest

  1. Commit cbindgen.toml and pin the cbindgen version used by development/CI.
  2. Generate include/mydb.h from the crate.
  3. Run generation in CI and fail if git diff --exit-code -- include/mydb.h is nonempty.
  4. Compile a tiny C smoke test against the header and library; generation alone does not prove that the final linker surface is correct.

A build.rs can invoke cbindgen automatically, but generating source-tree files during every Cargo build can surprise package consumers. For this curriculum, an explicit generation command plus a CI drift check is easier to reason about. If you choose the library API, pin its version and copy the exact call pattern from that version's documentation.

Header generation does not make a type safe

cbindgen transcribes declarations. It cannot prove pointer validity, ownership, thread safety, panic containment, or that Dart selected the correct dynamic library. Those contracts belong in Rust safety comments, C header documentation, tests, and the Dart wrapper.

🦀 Rust just-in-time
  • pub controls Rust visibility; extern "C" controls calling convention; #[unsafe(no_mangle)] controls the exported symbol name.
  • #[repr(C)] is needed for concrete structs shared by layout. Opaque handle fields remain private and are not shared by layout.
  • Put a version prefix on symbols and header guards early, for example MYDB_H and mydb_open.
✅ Concept check

1. Why pass --lang c?

Show answer
cbindgen's CLI defaults to C++ output; this project promises a C ABI and needs a C header.

2. Which tool converts the generated C header into Dart declarations?

Show answer
package:ffigen. dart:ffi is the runtime interop library.

3. Why still compile a C smoke test?

Show answer
A plausible header does not prove the library exports matching symbols and link-compatible signatures.

4. What should CI do after regenerating the header?

Show answer
Fail if regeneration changes the committed header, which detects source/header drift.
