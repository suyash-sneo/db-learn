1.9 — dart:ffi basics: DynamicLibrary, lookups, Pointer, malloc/free, Uint8List

Goal: From the Dart side, load your Rust library, find a function, call it with pointers and lengths, and move bytes in and out as Uint8List — closing the loop you opened in Lessons 1.1–1.8. You know Flutter; this is the FFI-specific part.

Why this matters to your DB

This is Layer 3 — the consumer. Everything you designed in Rust only matters if Dart can actually call it. dart:ffi is Dart's built-in mechanism to call C ABIs, so it calls your Rust exactly as if it were a C library. Getting a real Uint8List across is the literal goal of the Phase 1 Build.

Step 1: load the dynamic library

DynamicLibrary opens your compiled .so/.dylib/.dll at runtime. The path differs per platform; for desktop dev you point at the build output.

dart 
import 'dart:ffi';
import 'dart:io' show Platform;
import 'package:ffi/ffi.dart';   // for malloc/free, Utf8 helpers ⚠️VERIFY

DynamicLibrary openLib() {
  if (Platform.isMacOS) return DynamicLibrary.open('libmydb.dylib');
  if (Platform.isLinux) return DynamicLibrary.open('libmydb.so');
  if (Platform.isWindows) return DynamicLibrary.open('mydb.dll');
  // iOS/Android static-link differently — DynamicLibrary.process()/executable() ⚠️VERIFY
  throw UnsupportedError('platform');
}

Step 2: describe the function in two type systems

This is the part unique to FFI. Each foreign function needs two signatures: the native (C) signature using dart:ffi types, and the Dart signature using Dart types. lookupFunction ties them together.

dart 
// Rust:  int32_t echo(const uint8_t* ptr, size_t len, uint8_t** out, size_t* out_len)

// Native (C) signature — uses ffi types like Int32, Pointer, Size:
typedef EchoNative = Int32 Function(
    Pointer<Uint8> ptr, Size len, Pointer<Pointer<Uint8>> out, Pointer<Size> outLen);

// Dart signature — uses Dart types like int:
typedef EchoDart = int Function(
    Pointer<Uint8> ptr, int len, Pointer<Pointer<Uint8>> out, Pointer<Size> outLen);

final lib = openLib();
final echo = lib.lookupFunction<EchoNative, EchoDart>('echo');
🦀 The two-signature rule (the #1 dart:ffi confusion)

The native typedef must match the C ABI exactly (Int32, Size, Pointer<Uint8>). The Dart typedef is how you call it (int instead of Int32/Size). Pointers stay Pointer<…> in both. Mismatch the native side and you get crashes or garbage, not compile errors — so cross-check it against the cbindgen header.

Step 3: allocate, fill, call, read, free

Dart objects don't live in C-accessible memory, so you allocate a native buffer, copy your bytes in, pass the pointer, then free it. malloc/free come from package:ffi.

dart 
Uint8List echoBytes(Uint8List input) {
  // 1. allocate native input buffer and copy Dart bytes into it
  final inPtr = malloc<Uint8>(input.length);
  inPtr.asTypedList(input.length).setAll(0, input);

  // 2. out-params: a place for Rust to write the result ptr + len
  final outPtr = malloc<Pointer<Uint8>>();
  final outLen = malloc<Size>();

  try {
    final rc = echo(inPtr, input.length, outPtr, outLen);
    if (rc != 0) throw StateError('echo failed: rc=$rc');

    // 3. read Rust's buffer into a Dart-owned Uint8List (a COPY)
    final len = outLen.value;
    final resultPtr = outPtr.value;
    final result = Uint8List.fromList(resultPtr.asTypedList(len));

    // 4. give Rust's buffer BACK to Rust to free (Lesson 1.5!)
    freeBuf(resultPtr, len);   // a second bound Rust fn, e.g. db_free_buf

    return result;
  } finally {
    // 5. free the buffers DART allocated
    malloc.free(inPtr);
    malloc.free(outPtr);
    malloc.free(outLen);
  }
}

Key dart:ffi pieces, summarized

PieceRole
DynamicLibrary.open(path)load the compiled Rust library
lib.lookupFunction<Native, Dart>('name')bind an exported function (two-signature)
Pointer<T>a native pointer; .value deref, .asTypedList(n) view as a typed list
malloc<T>(n) / malloc.free(p)allocate/free native (C) memory (from package:ffi)
Uint8ListDart's byte buffer; fromList copies, asTypedList views
⚠️ Two ownership pitfalls to internalize
  • asTypedList gives a view over native memory — it becomes invalid once that memory is freed. To keep the data, Uint8List.fromList(...) to copy first (as above).
  • Free with the matching allocator: Dart frees what malloc made; Rust frees what Rust made (call the exported free fn). Mixing them corrupts the heap — the exact rule from Lesson 1.5, now from the Dart side.
🦀 Notes
  • package:ffi (separate from the built-in dart:ffi) provides malloc/calloc/free and string helpers. Add it to pubspec.yaml. ⚠️VERIFY current API (e.g. Size vs IntPtr, allocator names).
  • On iOS/Android you usually static-link the Rust and use DynamicLibrary.process()/executable() rather than a file path — covered in Phase 8.7 (packaging). ⚠️VERIFY
  • Long-running calls should run off the UI isolate (Phase 7.5); for the spike, a synchronous call is fine.
✅ Concept check

1. Why does each FFI function need two typedefs in Dart?

Show answer
One native typedef matching the C ABI exactly (Int32, Size, Pointer), and one Dart typedef with Dart types (int) for how you call it. lookupFunction bridges them.

2. After asTypedList(len) on a Rust pointer, why copy with Uint8List.fromList before freeing the Rust buffer?

Show answer
asTypedList is a view into the native memory; once Rust frees it, the view dangles. Copying gives you Dart-owned bytes that survive.

3. Who frees the Rust-allocated result buffer, and how?

Show answer
Rust does — Dart calls the exported Rust free function (e.g. db_free_buf) and passes the pointer back. Dart must not malloc.free a Rust allocation.

4. Which buffers in echoBytes does Dart free itself?

Show answer
The ones Dart allocated: inPtr, outPtr, outLen — freed in the finally block with malloc.free.
