2.4 — Reading/writing pages & the VFS abstraction

Goal: Put a thin abstraction — a "virtual file system" — between the pager and the actual OS file, so the engine talks to one small interface for block I/O and durability, and the backend (real file, memory, test double) can change underneath.

Why this matters to your DB

If the pager calls std::fs::File directly, you can't test it without touching the disk, can't run an in-memory database, and can't port to a platform with different I/O. A VFS is the seam that makes the engine testable and portable. SQLite's portability across operating systems comes almost entirely from this one idea.

What a VFS is

A VFS ("virtual file system") is the boundary between database logic and platform I/O. SQLite divides that boundary into two interfaces: sqlite3_vfs handles operations such as opening/deleting files, path names, dynamic loading, time, and randomness; each opened file then exposes sqlite3_io_methods for close, read, write, truncate, sync, size, locking, and related file controls. Unix and Windows VFS implementations sit behind the same interface. [src: SQLite VFS overview and method tables] This curriculum's BlockIo is deliberately narrower: it models one already-open byte store, not SQLite's complete VFS contract.

A minimal block-device trait in Rust

In Rust this maps naturally onto a trait. The pager is generic over "something I can read and write fixed-size blocks to, and ask to become durable."

rust — the seam (toy trait, not engine logic) 
pub trait BlockIo {
    /// Fill `buf` with `buf.len()` bytes starting at `offset`.
    fn read_at(&mut self, offset: u64, buf: &mut [u8]) -> std::io::Result<()>;
    /// Write all of `buf` starting at `offset`.
    fn write_at(&mut self, offset: u64, buf: &[u8]) -> std::io::Result<()>;
    /// Request that previously written bytes reach durable storage.
    fn sync(&mut self) -> std::io::Result<()>;
    fn len(&self) -> std::io::Result<u64>;
    fn set_len(&mut self, len: u64) -> std::io::Result<()>;
}

A file-backed implementation is just the 0.8 seek/read/write code behind these methods; an in-memory implementation is a Vec<u8> you index into. The pager never knows which one it has.

🦀 read_exact / write_all under the hood

read_at should loop or use read_exact so a short read is an error, not silent truncation; write_at should use write_all. On Unix, std::os::unix::fs::FileExt offers read_exact_at/write_all_at that take an explicit offset and do not change the file's cursor. [src: Rust standard library std::os::unix::fs::FileExt] That trait is Unix-specific. A portable single-threaded fallback is seek + read_exact/write_all; concurrent access then needs synchronization around the shared cursor.

Where durability lives

Crucially, the VFS is where sync lives. The layers above don't call fsync; they call vfs.sync() at the right moments (commit, checkpoint). This keeps all the platform-dependent durability behavior in one place — important because, as you saw in 0.8 and will deepen in 2.6, "the bytes were written" and "the bytes are durable" are different claims. File::sync_all() is the OS-level request that backs this method, and its actual guarantee depends on the OS, filesystem, device, and hardware. [src: File::sync_all]

Why this pays off

  • Tests run in memory. Unit-test the pager and B+tree against an in-memory BlockIo — fast, deterministic, no temp files.
  • Crash injection. A test VFS can simulate "the write succeeded but sync never happened" or "only the first half of this page reached disk," which is exactly what you need to test recovery in Phase 5.
  • Portability. New platform ⇒ new backend, not a rewrite of the engine.
✅ Concept check

1. What problem does putting a VFS trait under the pager solve?

Show answer
It decouples engine logic from the storage backend, enabling in-memory testing, crash injection, and portability without changing higher layers. The narrow trait also makes the exact semantics—full transfer or error, length changes, and synchronization—part of the project's own contract.

2. Why does sync belong in the VFS rather than scattered through the engine?

Show answer
Durability is platform-dependent and subtle; centralizing it keeps all OS-specific behavior in one auditable place that higher layers invoke at well-defined commit points.

3. How does a VFS help you test recovery?

Show answer
A test VFS can simulate partial writes or dropped syncs, reproducing crash states deterministically without a real power cut.

4. Why prefer read_exact/write_all inside the backend?

Show answer
Their method contract is complete transfer or an error, instead of silently accepting fewer bytes than a fixed-size page requires.
