2.1 — Why fixed-size pages

Goal: Understand why disk-oriented databases carve their one file into equal-sized blocks called pages, why power-of-two sizes are conventional, and what that choice buys and costs you.

Why this matters to your DB

The page is the atom of everything that follows. The B+tree stores one node per page; records live inside pages; the freelist hands out free pages; the WAL ships modified pages. If you pick the page abstraction well, every later layer is simpler. So before writing any code, you need to know exactly why the page exists and why it is a fixed size.

The problem: a file is just bytes

An on-disk file is a flat, one-dimensional array of bytes with no structure. A database needs to store many independent things — tree nodes, rows, free-space bookkeeping — and to update one of them without rewriting the whole file. To do that, it needs addressable units: chunks it can point at, read, modify, and write back independently. A page is that unit.

Why fixed size (not variable)?

You could imagine storing variable-length chunks back-to-back. Databases overwhelmingly choose fixed sizes instead, for several concrete reasons:

  • O(1) addressing. If every page is the same size S, then page n lives at a byte offset computed by simple arithmetic — no index of "where is each chunk" is needed just to find a page. (You'll derive the exact formula in 2.3.)
  • Simple allocation and reuse. When a page is freed, any other page can reuse that exact slot because all slots are interchangeable. Variable-length holes cause external fragmentation — a freed 700-byte hole can't cleanly hold an 800-byte chunk. Fixed pages turn the free-space problem into a simple list of free page numbers (Phase 4's freelist).
  • It cooperates with lower layers. Operating systems, filesystems, and storage devices also work in fixed-size units. A database page that is compatible with those granularities is easier to align and reason about, although buffered file I/O does not promise that one database page becomes one physical device operation.
  • Predictable caching. A buffer pool of uniform slots (2.5) is far simpler than managing a cache of arbitrary-sized objects.

Why a power of two?

Page sizes are conventionally powers of two (512, 1024, 2048, 4096, 8192, …). That convention has two practical benefits: multiplication/division by the page size is simple and exact in binary, and power-of-two boundaries compose cleanly with many lower-layer allocation units. It is a format choice, not a law of storage hardware.

  • Virtual-memory pages, filesystem blocks, and device sectors each have their own granularity. Those values vary by platform, filesystem, device, and configuration; they are not database constants.
  • A power-of-two database page makes alignment checks and offset calculations straightforward. If direct I/O is introduced later, that API's documented alignment rules must be checked separately.

Do not infer a one-to-one hardware transfer from this choice. The OS may merge, split, cache, or reorder buffered I/O. Page size gives your file format a stable unit; the storage stack decides how that request is carried out.

What SQLite actually does (your reference point)

SQLite stores the page size as a 2-byte big-endian value at offset 16 of its file header. It must be a power of two between 512 and 65536 inclusive; the value 0x0001 is the special encoding for 65536, because 65536 does not fit as an ordinary 16-bit value. [src: SQLite file format §1.3.2, "page size"] Every page in a SQLite database is the same size, and page numbers start at 1. [src: SQLite file format §1.2, "Pages"]

Design choice for your engine

You'll pick a single fixed page size for your database (a 4096-byte default is a reasonable, common choice) and store it in your own file header so the value travels with the file. This is a design decision you are copying the shape of from SQLite, not a guarantee you inherit from SQLite — your engine only behaves correctly if you implement the offset math and durability yourself.

The trade-offs of the size you pick

Smaller pages (e.g. 512–1024)Larger pages (e.g. 8192–65536)
Wasted space per pageLess internal fragmentation for tiny recordsMore slack if records are small
Tree heightFewer keys per node → taller B+tree → more I/O per lookupMore keys per node → shallower tree → fewer seeks
I/O granularityRead less to touch one recordRead more, but fewer operations for scans
CacheMore pages fit in a fixed-size cacheEach cached page covers more data

There's no universally correct answer; it depends on workload. The important property is that the size is fixed and recorded, so the file is self-describing.

✅ Concept check

1. Why does a fixed page size let you find page n without an index?

Show answer
Because every page is the same size, the offset of page n is pure arithmetic on n and the page size — no lookup table mapping page → location is required.

2. Why are power-of-two page sizes conventional?

Show answer
They make binary offset/alignment arithmetic simple and compose well with many lower-layer granularities. They do not guarantee that one page request becomes one physical device operation.

3. You choose a tiny 512-byte page. What happens to a B+tree's height and why does that matter?

Show answer
Fewer keys fit per node, so the tree is taller; each extra level is potentially another disk read per lookup, increasing I/O.

4. Why store the page size inside the file rather than hard-coding it in the engine?

Show answer
So the file is self-describing: any version of the engine can open it and compute offsets correctly without external configuration.
