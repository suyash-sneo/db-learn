2.3 — Page numbers ↔ byte offsets

Goal: Nail the arithmetic that converts a page number to a byte offset and back, decide whether your page numbers start at 0 or 1, and understand why this one formula is the pager's whole job.

Why this matters to your DB

Every read and write in the engine ultimately becomes "seek to offset X, transfer page-size bytes." Higher layers speak in page numbers; the pager is the only place that knows the formula translating a page number into a file offset. Get this off by one and the entire file is misread.

The core formula

If pages are numbered starting at 0:

offset (0-based numbering) 
offset = page_no * page_size

If pages are numbered starting at 1 (SQLite's convention), page 1 begins at byte 0:

offset (1-based numbering) 
offset = (page_no - 1) * page_size

SQLite numbers pages from 1, and computes the offset of page n as (n − 1) × page_size; page 1 starts at byte 0 and holds the 100-byte file header. [src: SQLite file format §1.2, "Pages" — pages numbered from 1; the database header is the first 100 bytes of page 1]

0-based or 1-based? Pick deliberately

Both work. 1-based mirrors SQLite and lets you reserve "page 0" as a sentinel meaning "no page" (handy for the freelist and tree pointers — a child pointer of 0 means "none"). 0-based makes the arithmetic marginally simpler. The important thing is to choose one, document it, and never mix conventions across the codebase. This curriculum uses SQLite-style 1-based numbering with page 0 reserved as the null page so your examples line up with the source dives.

In Rust

rust — the pager's central conversion (toy module) 
struct Layout { page_size: u64 }

impl Layout {
    // 1-based page numbers; page 0 is the reserved "null page".
    fn offset(&self, page_no: u64) -> u64 {
        debug_assert!(page_no >= 1, "page 0 is the null page");
        (page_no - 1) * self.page_size
    }

    fn page_count(&self, file_len: u64) -> u64 {
        file_len / self.page_size // assumes file_len is a whole multiple
    }
}
⚠️ Widen and check offset arithmetic

This curriculum stores page numbers as u32 on disk but computes byte offsets as u64. Convert before subtracting and multiplying, and use checked arithmetic. Do not use usize as the file-format type: it changes width across targets. A checked u64 offset also lets the pager reject page 0 and arithmetic overflow as ordinary errors instead of relying on a debug-only assertion.

Why this is literally the pager's job

Strip everything else away and the pager exposes two operations to the layers above:

  • read_page(n) -> bytes: compute offset(n), seek there, read page_size bytes (with read_exact, from 0.8).
  • write_page(n, bytes): compute offset(n), seek there, write page_size bytes (with write_all).

Everything else — caching, dirty tracking, durability — is an optimization or safety layer wrapped around these two. The next lesson (2.4) puts a small abstraction in front of the raw file so the rest never seeks directly.

✅ Concept check

1. With 1-based numbering and a 4096-byte page, what offset is page 7 at?

Show answer
(7 − 1) × 4096 = 24576.

2. Why reserve page number 0 instead of using it for data?

Show answer
So 0 can act as a null/"no page" sentinel in pointers (freelist links, tree child pointers) — there's no real page to confuse it with.

3. Why store page numbers as u32 but compute offsets as u64?

Show answer
The file format needs a fixed-width page identifier, while byte offsets need a wider type. Converting before checked multiplication avoids target-dependent usize behavior and catches overflow cleanly.

4. Which two operations are the irreducible core of the pager?

Show answer
read_page(n) and write_page(n), each built on the offset formula plus seek + exact read/write.
