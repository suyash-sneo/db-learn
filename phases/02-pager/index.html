Phase 2

The single file & the pager

Your whole database is going to live in one file. This phase builds the layer that turns that flat stream of bytes into a numbered array of fixed-size pages, caches them in memory, and writes them back. Every higher layer — the B+tree, records, the WAL — is built on top of the pager and never touches the file directly.

Why the pager is the foundation

A file is just a one-dimensional array of bytes. A database needs random access to structured chunks that it can read, modify, and write back through one controlled layer. The pager is the translation layer: "give me page 42" instead of "seek to byte 172032 and read 4096 bytes." Once it exists, the rest of the engine thinks in page numbers, not byte offsets. SQLite is the concrete reference used in this phase; other engines may divide responsibility among different modules.

What you'll be able to do after Phase 2

Lay out a file header, map page numbers to byte offsets, read and write fixed-size pages through a small abstraction (a "VFS"), cache hot pages in a buffer pool with dirty tracking and eviction, and reason precisely about when written bytes are actually durable. You'll finish by building a real pager in Rust with a failing test suite.
