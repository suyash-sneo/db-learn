2.2 — The file header

Goal: Design the first bytes of your database file — the header — so the file is self-describing: a magic number, a format version, the page size, and counters the engine needs at startup.

Why this matters to your DB

When your engine opens a file, it knows nothing yet. The header is how the file tells the engine what it is and how to interpret the rest. Get this right and you can evolve the format later without silently corrupting old databases. Get it wrong and you can't even safely detect "this isn't my file."

What every header needs

  • A magic number / signature. A fixed byte string at offset 0 that identifies the file as yours. On open you compare the first bytes; a mismatch means "not my file, refuse to touch it" rather than misinterpreting random bytes as pages.
  • A format version. So a newer engine can recognize an older file (and either upgrade it or refuse). Without a version field, format evolution is guesswork.
  • The page size. Stored in the file so offset math (2.3) is driven by data, not a compile-time constant. The file becomes portable across builds.
  • Bookkeeping counters. Things the engine needs before reading any tree: how many pages the file currently has, the head of the freelist (Phase 4), the root page of the top-level structure, and a change counter for cache validation.

SQLite's header as a worked example

SQLite reserves the first 100 bytes of page 1 as the database header. [src: SQLite file format §1.3, "The Database Header"] A few fields, with their exact offsets, to anchor your own design:

OffsetSizeField
016Magic string "SQLite format 3\0"
162Page size (big-endian; 0x0001 ⇒ 65536)
244File change counter
284Size of the database file in pages
324Page number of the first freelist trunk page (0 = none)
364Total number of freelist pages

[src: all rows above from SQLite file format §1.3, header field table; the 16-byte magic is the hex sequence 53 51 4c 69 74 65 20 66 6f 72 6d 61 74 20 33 00]

Notice the pattern: a signature, a self-describing page size, a count of pages, and a pointer into the free-space structure. Those four ideas are the backbone of almost any single-file format.

Endianness: pick one and write it down

Multi-byte integers must be stored in a defined byte order or a file written on one machine won't read correctly on another. SQLite uses big-endian for its header and B-tree integers. [src: SQLite file format §1.3 — "All multibyte fields … are stored with the most significant byte first (big-endian)"] For your engine, choosing a fixed endianness (big-endian to mirror SQLite, or little-endian to match most CPUs) is fine — the rule is that it is fixed and documented, and that you use explicit conversions, never raw memory casts.

rust — explicit, endianness-safe encoding (toy example) 
fn write_u32_be(buf: &mut [u8], at: usize, v: u32) {
    buf[at..at + 4].copy_from_slice(&v.to_be_bytes()); // big-endian, defined order
}

fn read_u32_be(buf: &[u8], at: usize) -> u32 {
    u32::from_be_bytes(buf[at..at + 4].try_into().unwrap())
}
⚠️ Never transmute a struct onto disk

It is tempting to #[repr(C)] a header struct and copy its raw bytes to the file. That couples the file format to your CPU's endianness, the compiler's padding, and field alignment, and breaks portability and forward compatibility. Serialize each field explicitly with to_be_bytes/from_be_bytes (or your serde format) so the on-disk layout is something you control, not something the compiler picks.

Validation on open

Opening a file is a sequence of checks, each of which can fail cleanly:

  1. Read the header bytes.
  2. Compare the magic. Mismatch ⇒ NotMyFile error.
  3. Read the format version. Newer-than-supported ⇒ refuse or upgrade.
  4. Read the page size; sanity-check it (power of two, within range).
  5. Read page count, freelist head, root page for later use.

A brand-new (empty) file is the special case: there's no header to read, so the engine initializes one — write the magic, version, page size, and zeroed counters, then sync — before treating the file as a database.

✅ Concept check

1. What does the magic number protect you from?

Show answer
Treating an unrelated or corrupt file as a database and misinterpreting arbitrary bytes as pages; it lets you fail fast with a clear error.

2. Why store the page size in the file instead of as a constant in the engine?

Show answer
So the file is self-describing and portable; offset math is driven by the file's own data and a file made with a different page size still opens correctly.

3. Why not just transmute a #[repr(C)] header to disk?

Show answer
It bakes in CPU endianness and compiler padding/alignment, breaking portability and format stability. Serialize fields explicitly.

4. What must opening a brand-new empty file do differently?

Show answer
There's no header yet, so it must initialize one (magic, version, page size, zeroed counters) and sync before use.
