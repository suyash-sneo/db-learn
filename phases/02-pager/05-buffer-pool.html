2.5 — The page cache / buffer pool

Goal: Keep recently used pages in memory, serve repeat reads without touching disk, track which cached pages were modified ("dirty"), and evict pages when memory is full — without ever losing a change.

Why this matters to your DB

Storage access is generally much more expensive than reading an already-resident memory buffer. A B+tree lookup repeatedly visits the root and upper levels, so caching those pages avoids repeated backend reads. The buffer pool is also the point that tracks which in-memory copies differ from the main file, making its writeback rules part of the later durability design.

What the buffer pool does

It sits between the engine and the VFS and maintains a fixed-capacity set of in-memory page frames plus a map from page number → frame. On get_page(n):

  1. If page n is already cached (a "hit"), return the in-memory copy.
  2. If not (a "miss"), find a free frame (evicting one if necessary), read page n from the VFS into it, record the mapping, and return it.

On modification, the engine marks the page dirty. A dirty frame cannot simply be discarded. What may be written, where it may be written, and when depends on the recovery protocol: Phase 2 temporarily writes directly to the main file, while Phase 5 introduces WAL ordering.

The pin/reference problem

While a higher layer is reading or editing a page, that frame must not be evicted out from under it. Buffer pools solve this with a pin count (a.k.a. reference count): a page in use is "pinned" and ineligible for eviction; when the caller is done it "unpins." Only unpinned pages can be evicted.

🦀 Rust makes this a borrow-checker question

In Rust, "this page must stay alive and not move while I hold a reference to it" is exactly what the borrow checker enforces. A simple design hands out a guard object (like a RefMut-style handle) whose Drop unpins the page, so you cannot forget to unpin. Designs using Rc<RefCell<Page>> or returning indices instead of references are possible; each trades ergonomics against runtime checking and ownership complexity. The Phase 2 build chooses a simpler contract: reads return owned bytes and edits happen inside a closure, so no borrowed page escapes the pager. A production guard design still needs a count, not a Boolean, because more than one caller may hold the same page.

Eviction policies

When the pool is full and you need a frame, you must choose a victim. Common policies:

PolicyIdeaTrade-off
FIFOEvict the oldest-loaded pageSimple; ignores how hot a page is
LRUEvict the least-recently-used pageGood general default; needs per-access bookkeeping
Clock (second chance)Approximate LRU with a reference bit and a rotating handCheaper than true LRU; widely used in real DBs

For your engine, LRU or Clock are both reasonable. Start simple; the policy is swappable as long as the invariant below holds.

⚠️ The one invariant you cannot break

A dirty page must not be forgotten when its frame is reused. In the pre-WAL Phase 2 exercise, eviction writes that page to the main file first. Once Phase 5 exists, an uncommitted dirty page may reach the main file only when the WAL protocol permits it; otherwise a crash could expose an uncommitted value. This is the database's STEAL/no-STEAL policy. A clean frame can be dropped because the authoritative storage copy is current. In either design, writeback is not the same as commit: commit requires the recovery protocol's ordering and synchronization steps.

Clean vs dirty vs durable

  • Clean cached page: in memory, identical to disk. Safe to drop.
  • Dirty cached page: in memory, newer than disk. Must be written before reuse; a crash now loses it.
  • Written but not synced: handed to the OS, but maybe still in the OS cache, not on the device. A crash may still lose it — the subject of 2.6.
✅ Concept check

1. Why does the buffer pool exist at all?

Show answer
To serve hot pages (e.g. tree upper levels) from RAM instead of re-reading them from much slower disk on every access.

2. What does pinning a page prevent?

Show answer
Eviction of a frame while a caller still holds a reference to it — which would leave a dangling/invalid view.

3. What must happen before a dirty frame is reused for another page?

Show answer
The change must be preserved according to the active recovery policy. Phase 2 writes it to the main file before reuse; after WAL is introduced, the log/order rules decide whether main-file writeback is legal.

4. Is writing a dirty page during eviction the same as committing it?

Show answer
No. The write reaches the OS but durability still requires a sync at a defined commit point; eviction order and commit order are separate.
