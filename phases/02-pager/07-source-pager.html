2.7 — Source dive: SQLite's pager

Goal: See how a production pager is organized in real code, so the concepts from 2.1–2.6 map onto names you can grep for — and learn how to read this file yourself rather than trusting a summary.
📂 Where to read

SQLite's pager lives in src/pager.c with its interface in src/pager.h. This lesson pins its source links to SQLite 3.53.2, commit ccc132c5be20ab5c755c97a08d06b1b592fef330, so later source changes do not silently invalidate the names below. The file opens with an extensive comment describing the rollback-mode state machine—start there. [src: SQLite 3.53.2 src/pager.c; Atomic Commit]

How to verify any claim below: open pager.c, search for the symbol in monospace, and read its surrounding comment. SQLite is unusually well-commented; the file's own header comments are the authority, not this page.

What the pager module is responsible for

SQLite's own description: the pager implements transactions, page caching, and the rollback journal / WAL on top of the OS file (via the VFS). It hands the B-tree layer above it pages by number and hides everything about how those pages are made durable. [src: SQLite architecture — "The Pager"] Notice this is a bigger scope than the Phase 2 pager: SQLite folds journaling into the same module you'll build incrementally across Phases 2 and 5.

The split between pager.c (transactions/journaling) and pcache.c (the buffer pool itself) mirrors the separation this curriculum draws between 2.5 (cache) and Phase 5 (durability). pcache.c is the pager-facing cache layer; pcache1.c is the default page-cache implementation. That is a useful separation to study, but it is not identical to this curriculum's smaller Rust API.

The state machine is the interesting part

In rollback-journal mode, the pinned source names seven pager states: PAGER_OPEN, PAGER_READER, PAGER_WRITER_LOCKED, PAGER_WRITER_CACHEMOD, PAGER_WRITER_DBMOD, PAGER_WRITER_FINISHED, and PAGER_ERROR. The opening comment states the lock/journal invariants for each. WAL mode follows different locking details, which is why the comment explicitly limits this state description to rollback mode. [src: pinned SQLite 3.53.2 src/pager.c opening state-machine comment] Read this section to see how caching, locking, and recovery are coordinated; do not copy all seven states into the Phase 2 exercise, which has no transaction protocol yet.

⚠️ Read for ideas, not to copy

This is a source dive, not a transplant. SQLite's pager is decades of hardening around edge cases (hot journals, super-journals, mmap, WAL) you don't need yet. Extract the shape — page handles, dirty tracking, a documented state machine — and build your own minimal version. The "no solution code" rule applies: you may quote short illustrative snippets from this public-domain file in your notes, but the engine you ship is one you wrote.

✅ Concept check

1. Why is SQLite's pager scope larger than the Phase 2 pager you'll build?

Show answer
It also implements the rollback journal / WAL and transactions, which this curriculum defers to Phase 5; Phase 2 is just paging + cache + sync awareness.

2. Which file holds the buffer-pool logic separate from the pager, and why split them?

Show answer
pcache.c (with pcache1.c) — separating the cache from transaction/journal logic keeps each concern independently understandable and replaceable. pcache.c presents the cache interface while pcache1.c supplies its default implementation.

3. What's the most valuable section of pager.c to read first?
