2.7 — Source dive: SQLite's pager

Goal: See how a production pager is organized in real code, so the concepts from 2.1–2.6 map onto names you can grep for — and learn how to read this file yourself rather than trusting a summary.
📂 Where to read

SQLite's pager lives in src/pager.c with its interface in src/pager.h, in the public-domain SQLite source tree. Read the amalgamation or the canonical repo. The file opens with an extensive comment block describing the pager's job and state machine — start there. [src: sqlite.org/src · src/pager.c; design overview at Atomic Commit]

How to verify any claim below: open pager.c, search for the symbol in monospace, and read its surrounding comment. SQLite is unusually well-commented; the file's own header comments are the authority, not this page.

What the pager module is responsible for

SQLite's own description: the pager implements transactions, page caching, and the rollback journal / WAL on top of the OS file (via the VFS). It hands the B-tree layer above it pages by number and hides everything about how those pages are made durable. [src: SQLite architecture — "The Pager"] Notice this is a bigger scope than the Phase 2 pager: SQLite folds journaling into the same module you'll build incrementally across Phases 2 and 5.

The state machine is the interesting part

The pager is essentially a state machine over the file's lock and journal state: no transaction → read transaction → write transaction → committing → done. The header comment in pager.c enumerates these states and the legal transitions; this is the single most valuable thing to read, because it shows how caching, locking (Phase 7), and durability (Phase 5) are coordinated by one component. Read it before you design your own — you will recognize the same states emerging in your build.

⚠️ Read for ideas, not to copy

This is a source dive, not a transplant. SQLite's pager is decades of hardening around edge cases (hot journals, super-journals, mmap, WAL) you don't need yet. Extract the shape — page handles, dirty tracking, a documented state machine — and build your own minimal version. The "no solution code" rule applies: you may quote short illustrative snippets from this public-domain file in your notes, but the engine you ship is one you wrote.
