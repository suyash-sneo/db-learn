Build: a single-file pager with a buffer pool

Goal: Turn one file into a numbered array of fixed-size pages with a header, a cache that tracks dirty pages and evicts safely, and an explicit sync. Every page above this in the engine will use it.
⚠️ AI off

Use the lessons, the SQLite file-format spec, primary docs, compiler errors, and the hint ladder. This page gives you contracts, stubs, and failing tests — not the implementation. The pager is a core database component, so per the project rules there is no worked solution here.

Deliverable

A Rust 2024 package pager exposing a Pager built over a small BlockIo trait, with a file-backed and an in-memory backend, page get/modify/sync, dirty tracking, capacity-bounded caching with safe eviction, and a self-describing header.

Fixed design decisions

  • Page size: a single value stored in the header; default 4096. Page numbers are 1-based; page 0 is the reserved null page.
  • Multi-byte header fields are big-endian, written with to_be_bytes (never transmute).
  • Header (kept simple, all big-endian): 16-byte magic "minidb format 1\0", 2-byte page size, 4-byte page count, 4-byte freelist head (0 = none; unused until Phase 4). The header lives at the start of page 1.
  • Durability is explicit: write_page does not sync; sync does.

Ordered milestones

  1. M1 — BlockIo. Define the trait; implement MemBlock (a Vec<u8>) and FileBlock (seek + read_exact/write_all; sync via sync_all).
  2. M2 — Header. Pager::create writes a fresh header; Pager::open reads and validates it (magic, page size). Reject a wrong magic with a typed error.
  3. M3 — Offset math + raw page I/O. Read and write a page by number through the backend using the 1-based formula.
  4. M4 — Cache + dirty tracking. A bounded cache mapping page no → frame; get_page serves hits from memory; modifications mark the page dirty.
  5. M5 — Safe eviction. When full, evict a victim; if it's dirty, write it out first; never evict a pinned page.
  6. M6 — sync + allocate. sync writes all dirty pages then syncs the backend; allocate_page grows the file by one page and updates the header count.

Acceptance criteria

  • All Rust tests below pass against both backends.
  • Opening a file written by create round-trips the page size and page count.
  • A wrong-magic file is rejected with the typed error (tested).
  • A written-then-evicted dirty page is not lost: re-reading returns the new bytes (tested).
  • Pinned pages are never chosen as eviction victims (tested).
  • Durability ordering is a documented manual check: write_page alone is not claimed durable; only post-sync state is. (No automated power-cut test in Phase 2; that arrives in Phase 5.)

⚠️ The borrow-checker design decision (read before M4)

Returning &[u8] from get_page borrows the whole Pager, so you can't get one page and then evict to make room for another in the same expression. This is the real design tension from 2.5. Options, in rough order of simplicity: (a) make modify_page take a closure so the borrow ends when the closure returns (shown above); (b) return a copy of the bytes for reads; (c) hand out a guard/handle that unpins on Drop; (d) interior mutability with RefCell. Pick one and justify it in your notes — this choice is part of the exercise, not an obstacle to route around with unsafe.

        let err = Pager::open(io, 8);
        assert!(matches!(err, Err(PagerError::BadMagic)));
    }

    #[test]
    fn dirty_page_survives_eviction() {
        let mut p = Pager::create(MemBlock::empty(), 4096, /* capacity */ 2).unwrap();
        let a = p.allocate_page().unwrap();
        p.modify_page(a, |b| b[0] = 42).unwrap();
        // touch enough other pages to force `a` out of a 2-frame cache
        for _ in 0..4 {
            let n = p.allocate_page().unwrap();
            let _ = p.get_page(n).unwrap();
        }
        assert_eq!(p.get_page(a).unwrap()[0], 42);
    }

    #[test]
    fn offset_is_one_based() {
        let p = Pager::create(MemBlock::empty(), 4096, 8).unwrap();
        assert_eq!(p.offset(1), 0);
        assert_eq!(p.offset(4), 3 * 4096);
    }
}

Stuck? — graduated hints

M2: how do I validate the page size?
Nudge: The valid set is powers of two in a fixed range.
Bigger: Check 512..=32768 and v.is_power_of_two().
Source: the same constraint SQLite enforces at header offset 16 [src: fileformat2 §1.3.2]; return BadPageSize otherwise.
M4: get_page won't compile — borrow conflicts
Nudge: A returned &[u8] borrows the whole pager for as long as you hold it.
Bigger: Separate "bring the page into a frame" (mutable, may evict) from "hand out a view" (immutable), so eviction never races a live borrow.
Technique: resolve all eviction before taking the final shared borrow, or use the closure-based modify_page so the borrow's scope is the closure.
M5: my eviction sometimes loses data
Nudge: What's true of a frame you're about to reuse?
Bigger: If dirty, you must write_at it before overwriting the frame.
Technique: never select a pinned frame; write-back-then-reuse for dirty victims; drop clean victims directly.
M6: when exactly do I sync?
Nudge: Per 2.6, write_at ≠ durable.
Technique: flush every dirty frame with write_at, clear their dirty bits, then call io.sync() once. Document that only this point is durable.
