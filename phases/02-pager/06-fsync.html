2.6 — Flushing, fsync, and the durability gap

Goal: Understand the full chain a byte travels from your program to durable storage, where it can be lost at each stage, and what fsync does and does not guarantee — because the entire durability story (Phase 5) is built on getting this exactly right.

Why this matters to your DB

"The database is durable" means: once a transaction is reported committed, its data survives a crash or power loss. Every layer above is pointless if this is wrong. And it is shockingly easy to get wrong, because at least four separate layers each buffer your write and each can lose it. You must know exactly which call pushes data past which layer.

The four layers a write passes through

  1. Your process buffer. If you wrote through a BufWriter or your own buffer, bytes sit in your program's memory until flushed. A crash loses them; even a clean process exit without flush can lose them. flush() (the Write trait) pushes this buffer to the OS. A raw File isn't buffered, so its flush is essentially a no-op.
  2. The OS page cache. After a successful write/write_all, the bytes are in the kernel's cache, not necessarily on the device. The write call returning success does not mean the data is durable. A power loss here loses the data even though every write returned Ok.
  3. The device cache. Drives have their own volatile write caches. Data can be "on the drive" in this cache yet still lost on power failure unless flushed to the medium.
  4. The storage system's stable-state contract. The OS, filesystem, device, and hardware together determine what a completed synchronization operation promises. The application cannot directly observe a universal "physical medium" boundary.

What fsync / sync_all actually does

Rust's File::sync_all() requests that the OS synchronize the file's content and metadata with the storage device; sync_data() may skip metadata that isn't needed to read the content back (a possible optimization). The standard library deliberately describes these as requests whose final guarantee depends on the OS, filesystem, device, and hardware. [src: File::sync_all / sync_data] Under the hood this is fsync/fdatasync on Unix-like systems and the platform equivalent elsewhere.

⚠️ The directory entry needs syncing too

Creating, renaming, or deleting a file changes the containing directory, not just the file. On platforms/filesystems whose documented crash protocol requires it, synchronizing the file is followed by synchronizing the containing directory so the name change is also persistent. SQLite's atomic-commit design explicitly reasons about file deletion and directory updates rather than assuming a file sync covers them. [src: SQLite: Atomic Commit In SQLite]

The cost, and why you can't just sync constantly

A synchronization call can be much more expensive than a buffered write because it may wait for lower storage layers. Its latency varies substantially by device, filesystem, workload, and durability settings, so this curriculum does not attach a universal number. Calling it after every page would also prevent useful batching. Recovery protocols therefore define a small number of ordered synchronization points—often around transaction commit—without weakening the stated crash contract.

The takeaway for the pager

  • write_page returning Ok means "handed to the OS," not "durable."
  • Durability requires an explicit sync at a defined point.
  • Ordering matters: in Phase 5 you'll learn you must sync the log before overwriting the real data, or recovery can't work.
  • Under a durability mode that promises crash persistence, the database does not report success until that mode's required synchronization step succeeds. A deliberately relaxed mode must state its weaker guarantee.

For Phase 2's build, your pager exposes a sync method and you reason about when it's called; the full crash-safe protocol is deferred to Phase 5, where it belongs.

✅ Concept check

1. Your write_all returned Ok(()) and the process is still running fine. Is the data safe across a power cut?

Show answer
No — it may be only in the OS page cache (and later another cache). It satisfies the database's durability promise only after the required synchronization protocol succeeds under the documented storage assumptions.

2. What's the difference between flush() and sync_all()?

Show answer
flush pushes your process's user-space buffer to the OS; sync_all asks the OS to push the file's content+metadata toward the device. They cross different layers.

3. Why might you also need to sync a directory?

Show answer
File creation/rename/delete updates the containing directory. On systems whose documented crash protocol requires it, syncing that directory is a separate step needed to persist the name change.

4. Why not just call fsync after every page write?

Show answer
A sync may wait on lower storage layers, and per-page synchronization prevents batching. A recovery protocol instead defines the minimum ordered sync points needed for its durability mode.
