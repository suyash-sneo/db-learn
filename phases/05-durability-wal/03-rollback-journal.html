5.3 — The rollback journal (undo logging)

Goal: Understand the first classic atomicity mechanism: before changing a page, save its original content to a side file, then modify the database in place. A crash mid-write is repaired by copying the originals back — rolling the database back to before the transaction.

Why this matters to your DB

The rollback journal is the simpler of the two mechanisms and the one SQLite used by default for years. Even though your build will target a WAL, understanding the rollback journal first makes the WAL obvious by contrast: same problem, opposite direction (undo vs redo).

The idea: keep an undo copy, modify in place

The main database file always holds the "real" pages. To change them atomically:

  1. For every page you're about to modify, first write its original bytes into a separate rollback journal file.
  2. Barrier: fsync the journal so the originals are durable.
  3. Now write the modified pages into the main database file in place.
  4. Barrier: fsync the database file.
  5. Delete the journal. This deletion is the commit point.

SQLite's atomic-commit write describes exactly this sequence, and names the journal deletion as "the instant where the transaction commits." [src: SQLite "Atomic Commit" §3.10 — "the rollback journal file is deleted. This is the instant where the transaction commits."]

Why this ordering is crash-safe

Consider where a crash can strike:

  • Before the journal is fully synced: the database hasn't been touched yet; the partial journal is discarded. No change — correct.
  • After journal sync, during the in-place db writes: the database may be torn, but the complete originals are durably in the journal. Recovery copies them back, undoing the partial write — back to the start state. Correct (atomic: none applied).
  • After the db is synced but before journal deletion: the new data is fully in the db, but the journal still exists, so recovery would roll back. To avoid losing a real commit, the deletion/zeroing of the journal header is the single act that flips the outcome — done after the db is durable. Correct.
  • After journal deletion: no journal, nothing to recover; the committed state stands. Correct (atomic: all applied).

The journal header and the torn-write defense

The journal starts with a header recording the original database size (so recovery can truncate back) and a count of how many page records follow. SQLite writes the page count into the header after the page records are synced; combined with the "sectors are linear" assumption, a journal whose header reads as zero/garbage is treated as not committed and ignored. [src: SQLite "Atomic Commit" §3.5–§3.7, §4.2 — journal header, original db size, page-count written last]

The cost: write amplification

Every changed page is written twice to stable storage — once as an original into the journal, once as the new value into the database — with two fsync barriers per transaction. That double write and the in-place modification of the main file (which forces an exclusive lock during the write) are the two drawbacks the WAL was designed to fix. [src: SQLite WAL docs — "writing the content once (versus twice for rollback-journal transactions)"]

⚠️ The commit point is a single durable act

Atomicity hinges on one operation flipping the system from "not committed" to "committed" — here, making the journal stop looking valid (delete it, truncate it, or zero its header) after the database is durable. If you blur that — e.g. start deleting the journal before the db sync completes — you open a window where a crash loses a committed transaction. One barrier-ordered flip, no ambiguity.

✅ Concept check

1. What does the rollback journal store, and in which direction does it recover?

Show answer
The original (pre-change) content of pages. Recovery rolls back by copying originals into the db — undo logging.

2. Where are the two fsync barriers, and what does each protect?

Show answer
One after writing the journal (so originals survive before the db is touched), one after writing the db (so changes are durable before the journal is removed).

3. What single act is the commit point?

Show answer
Deleting (or zeroing/truncating) the journal, performed after the database file is synced.

4. Why is the rollback journal said to write data twice?

Show answer
Each changed page goes to disk as an original in the journal and again as the new value in the db — double writes plus two syncs per transaction.
