5.3 — The rollback journal (undo logging)

Goal: Understand the first classic atomicity mechanism: before changing a page, save its original content to a side file, then modify the database in place. A crash mid-write is repaired by copying the originals back — rolling the database back to before the transaction.

Why this matters to your DB

The rollback journal is the simpler of the two mechanisms and the one SQLite used by default for years. Even though your build will target a WAL, understanding the rollback journal first makes the WAL obvious by contrast: same problem, opposite direction (undo vs redo).

The idea: keep an undo copy, modify in place

In this rollback-journal protocol, the main database file holds the current pages. To change them atomically:

  1. For every page you're about to modify, first write its original bytes into a separate rollback journal file.
  2. Finish and synchronize a valid journal before touching the database. SQLite's FULL mode may use more than one journal synchronization: page content is flushed before the page count is made valid, then the updated header is flushed. [src: SQLite Atomic Commit §3.7 and §6.1 — rollback journal synchronization]
  3. Now write the modified pages into the main database file in place.
  4. Barrier: fsync the database file.
  5. In SQLite's DELETE journal mode, delete the journal. That deletion is the commit point for the protocol described here.

SQLite's atomic-commit document describes this DELETE-mode sequence and names journal deletion as the commit instant. [src: SQLite "Atomic Commit" §3.10 — "the rollback journal file is deleted. This is the instant where the transaction commits."]

SQLite also supports rollback-journal modes such as TRUNCATE and PERSIST, where the journal is invalidated by truncating it or overwriting its header instead of unlinking the file. The durable invalidation mechanism changes, but it still occurs only after the database writes are safe. [src: SQLite PRAGMA journal_mode]

Creating or removing a side file also changes directory metadata. Whether that metadata needs a directory synchronization is platform/filesystem specific; a production design must include it in the durability contract rather than assuming that synchronizing the file also persists its directory entry. [src: Linux fsync(2) — file sync does not necessarily persist its directory entry]

Why this ordering is crash-safe

Consider where a crash can strike:

  • Before the journal is fully synced: the database hasn't been touched yet; the partial journal is discarded. No change — correct.
  • After journal sync, during the in-place db writes: the database may be torn, but the complete originals are durably in the journal. Recovery copies them back, undoing the partial write — back to the start state. Correct (atomic: none applied).
  • After the db is synced but before journal deletion: the journal is still hot, so recovery rolls back. The transaction has not reached its commit point yet, making that outcome correct. Deleting or otherwise invalidating the journal is the later act that flips the outcome.
  • After journal deletion: no journal, nothing to recover; the committed state stands. Correct (atomic: all applied).

The journal header and the torn-write defense

The journal starts with a header recording the original database size (so recovery can truncate back) and a count of how many page records follow. Under SQLite's FULL synchronous mode, SQLite first flushes journal page content while the count is zero, then writes the count and flushes the journal again. The separated header sector, checksums, and documented sector assumptions let recovery reject incomplete journal content rather than applying it as valid undo data. [src: SQLite "Atomic Commit" §3.5–§3.7, §4.2 — journal header, original db size, page-count written last]

The cost: write amplification

Every changed page contributes an original page image to the journal and a new page image to the database. SQLite's FULL rollback-journal path also synchronizes the journal as required by its header protocol and synchronizes the database before commit. WAL mode can make commit I/O more sequential and uses fewer synchronization operations in SQLite, though checkpointing still eventually writes pages into the main database. [src: SQLite Atomic Commit §3.7–3.10; SQLite WAL §1 — advantages and trade-offs]

⚠️ The commit point is a single durable act

Atomicity hinges on one operation flipping the system from "not committed" to "committed" — here, making the journal stop looking valid (delete it, truncate it, or zero its header) after the database is durable. If you blur that — e.g. start deleting the journal before the db sync completes — you open a window where a crash loses a committed transaction. One barrier-ordered flip, no ambiguity.

✅ Concept check

1. What does the rollback journal store, and in which direction does it recover?

Show answer
The original (pre-change) content of pages. Recovery rolls back by copying originals into the db — undo logging.

2. What ordering must the synchronization barriers establish?

Show answer
A complete, valid journal must be synchronized before the database is modified; the modified database must then be synchronized before the journal is invalidated. SQLite FULL mode may require two journal flushes to make its page-count header valid safely.

3. What single act is the commit point?

Show answer
Deleting (or zeroing/truncating) the journal, performed after the database file is synced.

4. Why is the rollback journal said to write data twice?

Show answer
Each changed page goes to disk as an original in the journal and again as the new value in the db. SQLite's exact number of synchronization operations depends on its synchronous mode and journal/header protocol.
