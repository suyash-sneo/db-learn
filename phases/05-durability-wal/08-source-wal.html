5.8 — Source dive: SQLite's WAL format & wal.c

Goal: Collect the exact on-disk WAL byte layout from the file-format spec (the authoritative contract) and the code routines in wal.c that implement append, checksum, recovery, and checkpoint — so your build can mirror a faithful (if simplified) version.
📂 Where to read

The file-format spec §4 defines the WAL bytes; src/wal.c implements the read/append/recover/checkpoint logic. Code links are pinned to SQLite 3.53.2, commit ccc132c5be20ab5c755c97a08d06b1b592fef330. [src: file format §4; pinned src/wal.c]

The checksum algorithm

SQLite's WAL checksum is a pair of 32-bit values computed over the bytes as an array of 32-bit integers, with a Fibonacci-style weighting; it is computed big-endian if the header magic is 0x377f0683 and little-endian if 0x377f0682. Each frame's checksum is cumulative — it includes the prior checksum — so a single corrupt or torn frame invalidates the rest of the chain, which is exactly what recovery uses to find the valid tail. [src: SQLite file format §4.2 — checksum algorithm, endianness from magic, cumulative chaining]

checksum shape (spec algorithm) 
// s0, s1 carry from the previous frame (or header for frame 1)
// for each pair of 32-bit words (x0, x1) in the buffer:
//   s0 += x0 + s1;
//   s1 += x1 + s0;
// (interpret words big- or little-endian per the magic's low bit)

⚠️ Source dive discipline

The byte offsets and constants above come from the specification. The function names were checked at the pinned revision, but their internal contracts are not part of SQLite's stable public API.

What to mirror in your build vs simplify

  • Mirror: a fixed-size WAL header, per-frame header with page number + commit marker, and per-frame checksums chained cumulatively. These are what make recovery correct.
  • Simplify: the shared-memory wal-index becomes an in-memory hash map rebuilt on open (single process); salt rotation can be minimal; you can checkpoint eagerly rather than at 1000 pages.
✅ Concept check

1. How big is the WAL header, and how big is each frame header?

Show answer
32-byte WAL header; 24-byte frame header, each frame header followed by one page of data.

2. Which frame-header field identifies a commit frame, and what value?

Show answer
The "database size in pages after commit" field at offset 4 — non-zero on a commit frame, zero otherwise.

3. Why does one torn frame invalidate the rest of the WAL chain?

Show answer
Checksums are cumulative (each includes the previous), so a break in the chain fails every subsequent frame — recovery stops at the last valid one.

4. For Phase 5 byte layouts, is the code or the spec the authority?

Show answer
The file-format spec §4; the code implements it. Cite §4.1/§4.2 for offsets and constants.
