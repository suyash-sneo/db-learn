Build: a write-ahead log with recovery & a crash test

Goal: Add atomic, durable transactions to your pager via a WAL: append dirty pages as checksummed frames, commit with one fsync, merge WAL over the db on reads, recover on open up to the last valid commit frame, and checkpoint the WAL back into the db — then prove it with a simulated crash test.
⚠️ AI off

The WAL, recovery, and checkpoint are the safety-critical core of the engine — no worked implementation here, only contracts, stubs, failing tests, and graduated hints. Cite the SQLite WAL file-format spec (§4) for the frame layout and checksum; write the code yourself.

Deliverable

A Rust 2024 module wal layered on the Phase 2 pager, plus a Txn handle. Writes go through a transaction; commit appends frames + a commit frame and fsyncs once; reads merge the WAL over the db; open runs recovery; a checkpoint folds the WAL into the db with the correct barrier ordering.

Fixed design decisions

  • WAL header (simplified): 4-byte magic, 4-byte page size, 4-byte salt; big-endian. Document any extra fields you add.
  • Frame = 24-byte header (page number u64, db-size-after-commit u32 [0 = non-commit], salt u32, checksum [u32,u32]) + one page of data. Non-zero db-size marks a commit frame, mirroring SQLite §4.2.
  • Checksums are cumulative (each frame's checksum chains from the previous), so a torn tail invalidates everything after the break.
  • wal-index = an in-memory HashMap<u64, u64> (page → byte offset of newest frame), rebuilt by scanning the WAL on open. Not persisted.
  • Durability default = FULL: commit fsyncs the WAL after the commit frame. A sync: bool knob may relax it (NORMAL-style) but defaults to true.
  • Recovery on open scans frames, validates checksums, stops at the last valid commit frame, builds the wal-index. Runs before any read/write.
  • Checkpoint ordering: sync WAL → copy newest frame of each page into db → sync db → reset WAL.

Ordered milestones

  1. M1 — Frame I/O + checksum. Serialize/parse a frame; compute and verify the cumulative checksum.
  2. M2 — Append & commit. Txn buffers dirty pages; commit appends frames, writes the commit frame, fsyncs once.
  3. M3 — Read merge. read_page returns the newest WAL frame for a page (via the wal-index) or falls back to the db.
  4. M4 — Recovery. open scans the WAL, validates checksums, recovers to the last commit frame, rebuilds the wal-index; ignores an uncommitted/torn tail.
  5. M5 — Checkpoint. Fold the WAL into the db with the correct barrier order; reset the WAL only after the db sync.

Acceptance criteria

  • All Rust tests below pass.
  • A committed transaction is readable after reopen (durability).
  • An uncommitted transaction (frames appended, no commit frame) is invisible after reopen (atomicity / rollback of the tail).
  • A truncated/torn WAL tail is ignored; everything up to the last valid commit frame survives.
  • After a checkpoint, the db file alone (empty WAL) reflects all committed data.
  • A crash during checkpoint (simulated: stop after copying some pages but before resetting the WAL) still recovers all committed data on reopen.

Stuck? — graduated hints

M1: my checksum doesn't match across frames
Nudge: The checksum is cumulative, not per-frame-independent.
Technique: seed frame N's checksum with frame N−1's result (frame 1 seeds from the header). Process the buffer as 32-bit word pairs; pick endianness from the magic's low bit (spec §4.2).
M2: how do I know which frame is the commit?
Nudge: The commit marker lives in the frame header, not a separate record.
Technique: write db_size_after = 0 for every dirty page except the last, which gets the post-commit page count (> 0). Recovery treats a non-zero value as "transaction ends here, committed."
M4: recovery includes a half-written transaction
Nudge: You must stop at the last valid commit frame, not the last frame.
Technique: while scanning, remember the index state as of the most recent commit frame; if the scan later hits a checksum failure or EOF mid-transaction, roll the index back to that remembered commit point.
M5: a crash during checkpoint loses data
Nudge: Ordering. The WAL is your safety net until the db is durable.
Technique: sync WAL → copy frames → sync db → only then reset the WAL. If you reset before the db sync, a crash leaves the db torn and the WAL gone. Re-running recovery after a mid-checkpoint crash must reproduce the same committed state.
