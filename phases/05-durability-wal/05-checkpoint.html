5.5 — Checkpointing

Goal: Fold the WAL back into the database file safely, then recycle the WAL — with the exact barrier ordering that prevents corruption, and an understanding of when checkpoints run and why the WAL can be reset.

Why this matters to your DB

The WAL (5.4) is a staging area that grows forever unless something drains it. Checkpointing is that drain: it copies committed frames into the main file so the WAL can be truncated/reset and reads stay fast. Get the barrier ordering wrong here and you can corrupt the database even though every individual transaction was committed correctly.

What a checkpoint does

"Moving the WAL file transactions back into the database is called a 'checkpoint'." It takes the newest version of each page from the WAL and writes it into the corresponding slot of the database file. [src: SQLite WAL §3 — definition of checkpoint] After a checkpoint has transferred everything and the WAL is no longer needed by any reader, the WAL can be rewound and reused from the beginning.

The barrier ordering (do not reorder)

SQLite states the required syncs precisely: "The WAL must be synced to persistent storage prior to moving content from the WAL into the database and the database file must be synced prior to resetting the WAL." [src: SQLite WAL §3 — checkpoint sync requirements] So the sequence is:

  1. Barrier: ensure the WAL is fully synced (it must be durable before you trust its frames as the source of truth).
  2. Copy each committed page's newest frame from the WAL into the database file.
  3. Barrier: fsync the database file (now it durably holds everything the WAL did).
  4. Only now reset/truncate the WAL (rewind to the start), or mark it empty.

Why this ordering is crash-safe

A checkpoint is just another in-place write of the db file — exactly the thing that can be torn (5.2). But it's safe because the WAL still holds every committed frame until step 4. If a crash interrupts the copy, recovery (5.6) simply replays the WAL again on next open; the partially-copied db pages are overwritten by the authoritative WAL frames. The WAL is only discarded after the db is proven durable. This is why checkpointing never loses a committed transaction no matter when it's interrupted.

When checkpoints happen, and WAL reset

SQLite checkpoints automatically when the WAL reaches a threshold — by default 1000 pages (about 4 MB at a 4 KiB page size), after which the WAL is recycled. [src: SQLite WAL §4 — "automatically when the WAL file reaches a threshold size of 1000 pages"] Resetting the WAL (rewinding to the beginning to overwrite old frames) is only safe when the entire WAL has been transferred and synced into the db and no reader still needs those frames; "the writer checks how much progress the checkpointer has made… and if no readers are making use of the WAL, then the writer will rewind the WAL back to the beginning." [src: SQLite WAL §4 — WAL rewind conditions]

Checkpoint vs readers (concurrency note)

A checkpoint can run while readers are active, but it must stop at any page beyond the end mark of the oldest live reader — it can't overwrite db pages that an in-flight reader still expects to read from the WAL snapshot. It remembers its progress and resumes next time. [src: SQLite WAL §3 — checkpoint stops at the oldest reader's end mark, resumes later] For a single-threaded embedded build you can checkpoint when no read is in progress and skip this complication initially; it returns in Phase 7.

⚠️ Never reset the WAL before the db sync completes

The single most dangerous bug in a WAL implementation is truncating/rewinding the WAL before the checkpoint's db-file fsync has returned. If you do, a crash leaves the db half-updated and the WAL gone — unrecoverable corruption of committed data. The WAL is your safety net during checkpoint; cut it only after the db is provably durable (step 3 before step 4, always).

✅ Concept check

1. In one sentence, what does a checkpoint do?

Show answer
Copies the newest committed version of each page from the WAL into the main database file, so the WAL can later be recycled.

2. State the two checkpoint barriers and their order.

Show answer
Sync the WAL before copying frames into the db; sync the db before resetting/truncating the WAL.

3. Why can't a crash during checkpoint lose a committed transaction?

Show answer
The WAL still holds every committed frame until the db is synced; recovery just replays the WAL, overwriting any partially-copied db pages.

4. What is SQLite's default auto-checkpoint threshold?

Show answer
1000 WAL pages (~4 MB at a 4 KiB page size), after which the WAL is checkpointed and recycled.
