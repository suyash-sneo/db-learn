5.7 — Transactions at the engine level

Goal: Wrap the WAL mechanism in a clean begin / commit / rollback API so the rest of the engine (B+tree, record store) gets atomicity "for free" — every page mutation routes through the transaction, and the durability barrier lives in exactly one place.

Why this matters to your DB

The mechanisms in 5.3–5.6 are plumbing. The rest of your engine shouldn't know about WAL frames — it should just say "begin, do a bunch of page edits, commit." This lesson defines that seam so atomicity is structural, not something every caller must remember to do.

The transaction as the unit of atomicity

A transaction collects a set of page mutations and makes them visible together. The contract:

  • begin() — start buffering changes; reads see the committed state (db + already-committed WAL frames).
  • page edits — instead of writing the db directly, dirty pages are recorded in the transaction (and staged as WAL frames or held until commit).
  • commit() — append all dirty pages as WAL frames, write the commit frame, one fsync barrier, then return success. Now durable.
  • rollback() — discard the buffered/staged frames without writing a commit frame; nothing becomes visible. (On a crash this is automatic: no commit frame ⇒ recovery ignores the tail.)

Routing page writes through the transaction

The key refactor: the pager's modify_page from Phase 2 no longer writes the db file directly during a transaction. Instead it records the page as dirty in the active transaction. At commit, those dirty pages are flushed to the WAL. Reads consult the WAL-index first (newest frame for the page), then the db file — the same merge logic as 5.4.

rust — the seam (shape only; you implement it in the build) 
// pub fn begin(&mut self) -> Txn<'_, B>
// impl Txn {
//   fn modify_page(&mut self, pno: u64, f: impl FnOnce(&mut [u8]));  // marks dirty
//   fn read_page(&mut self, pno: u64) -> &[u8];  // wal-index newest, else db
//   fn commit(self) -> Result<()>;  // append frames + commit frame + ONE fsync
//   fn rollback(self);             // drop staged frames; nothing committed
// }

One barrier, one commit point

This design concentrates the entire durability story into commit(): the fsync of the WAL after the commit frame is the moment the transaction becomes durable, mirroring the rollback journal's "delete the journal" commit point but cheaper (one sequential append + one sync). Everywhere else in the engine is free of durability concerns.

Grouping and the durability/throughput trade-off

fsync is expensive, so the number of commits dominates write throughput. Two standard levers:

  • Batch logical work into fewer transactions — one commit's fsync covers many page edits.
  • Relax the sync — like SQLite's PRAGMA synchronous=NORMAL in WAL mode, you may skip the per-commit WAL fsync, trading a window of durability (a crash can lose the last few commits) for speed; the database remains consistent, just not maximally durable. [src: SQLite WAL — FULL syncs WAL each commit; NORMAL "omit this sync", sacrificing durability on power loss but not integrity]
⚠️ Consistency ≠ durability when you relax sync

Skipping the per-commit fsync (NORMAL-style) does not risk corruption — recovery still only replays whole, checksum-valid, commit-bounded frames — but it does mean a power loss can lose recently "committed" transactions. Be explicit in your API about which mode a caller chose; silently dropping durability is a footgun. Default to the safe (FULL) mode and make relaxation opt-in.

✅ Concept check

1. What four operations make up the engine-level transaction API?

Show answer
begin, page edits (dirty tracking), commit (append frames + commit frame + one fsync), rollback (discard staged frames).

2. How does modify_page change once transactions exist?

Show answer
It no longer writes the db file directly; it records the page as dirty in the active transaction, flushed to the WAL at commit.

3. Where is the single durability barrier, and why concentrate it there?

Show answer
In commit(), the fsync after the commit frame. Concentrating it keeps the rest of the engine free of durability logic and gives one unambiguous commit point.

4. What does relaxing the per-commit sync risk, and what does it not?

Show answer
It risks losing recently committed transactions on power loss (durability), but not corruption/consistency — recovery still replays only whole, valid, commit-bounded frames.
