5.1 — ACID at the storage layer

Goal: Pin down what the four ACID properties actually demand of a storage engine, which of them this phase delivers (atomicity + durability), and which depend on Phase 7 (isolation), so you know exactly what "a transaction committed" must guarantee.

Why this matters to your DB

"ACID" is thrown around loosely. To build correctly you need the engineering version: each letter is a concrete obligation on bytes and fsyncs, not a slogan. This phase is responsible for two of the four letters; getting clear on which two keeps you from over- or under-building.

The four properties, as obligations

PropertyWhat it demands of the engineDelivered by
AtomicityA transaction's changes are all applied or none are — no partial transaction is ever visible, even after a crash mid-write.This phase (journal / WAL)
ConsistencyApplication/schema and engine invariants are true before commit and remain true afterward. Atomicity prevents a partial invariant update, but does not invent or validate every business rule.Transaction logic, validation, and atomicity
IsolationDefines which effects concurrent transactions may observe. Serializability is one strong level, not the definition of every isolation level.Phase 7 policy (locking / MVCC)
DurabilityOnce a durable-mode commit returns success, the change survives the crashes covered by the documented storage contract.This phase's ordered WAL synchronization protocol

The classic ACID definition originates with Haerder & Reuter's 1983 survey, which formalized the transaction guarantees databases are expected to provide. [src: Haerder & Reuter, "Principles of Transaction-Oriented Database Recovery", ACM Computing Surveys 15(4), 1983]

Atomicity and durability are a pair

These two are what a crash-recovery mechanism provides, and they only mean something together:

  • Durability without atomicity would let a half-finished transaction be the thing that durably survives — useless.
  • Atomicity without durability would correctly apply all-or-nothing in memory but lose the commit on power failure — also useless.

So the commit protocol must make the whole change recoverable before it reports success. That ordering — log the change durably first, declare victory second — is the heart of every mechanism in this phase.

⚠️ "Write returned" is not "committed"

From Phase 2 you already know write() only hands bytes to the OS page cache, and even sync_all is a request to the OS/hardware to flush. Durability rests on the barrier actually being honored end-to-end. This phase builds the protocol assuming fsync works; the caveat that some hardware lies about fsync is real and is why SQLite documents fsync reliability as an assumption. [src: SQLite, "Atomic Commit In SQLite" §3.7 — fsync as a critical step / assumption]

What "a transaction" means here

For this engine a transaction is a batch of page mutations that must become visible together. Even a single logical insert can be multi-page (a B+tree split writes two leaves plus a parent; an overflow record writes a cell plus chain pages plus a freelist update). Atomicity is therefore not a luxury for "explicit transactions" — it's required for any operation that touches more than one page, which is almost all of them.

✅ Concept check

1. Which two ACID properties does this phase deliver, and which is deferred to Phase 7?

Show answer
Atomicity and durability here; isolation is Phase 7 (concurrency). Consistency still depends on correct application/engine rules and validation; atomicity only prevents a partial update from publishing.

2. Why are atomicity and durability useless without each other?

Show answer
Durable-but-not-atomic could persist a half transaction; atomic-but-not-durable could lose a committed one on power loss. Commit must make the whole change recoverable before returning success.

3. Why does even a single insert need atomicity in this engine?

Show answer
One logical insert can touch several pages (split, overflow chain, freelist). All must land together or none — that's a multi-page atomic change.

4. Why is "write() returned" not enough for durability?

Show answer
write() only reaches the OS cache; durability needs the data flushed to stable storage (fsync) — and even that is a request the hardware must honor.
