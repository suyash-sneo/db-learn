5.2 — The crash model

Goal: State precisely what a crash is allowed to do to your file — reordered writes, lost cache contents, torn (partial) sector writes — so that the journal and WAL you build defend against the actual failure modes, not imagined ones.

Why this matters to your DB

You can't prove a recovery scheme correct without an enemy model: exactly what the crash may have done. Build against too weak a model and recovery has real holes; too strong a model and you pay for protection you don't need. The model below is the standard one storage engines assume.

What a crash can do

  1. Lose everything in volatile caches. Any bytes still in the OS page cache (or a disk's volatile write cache) that were never flushed are gone. Only data that reached stable storage via a successful fsync is guaranteed present.
  2. Reorder writes. Without an explicit barrier, the OS and device may commit writes to stable storage in a different order than you issued them. So "I wrote A then B" does not guarantee that if B survived, A did too — unless an fsync separated them.
  3. Tear a multi-sector write. A single write() of a 4 KiB page is not atomic at the hardware level: a crash can leave some sectors of the page updated and others holding old bytes — a torn write.

The sector assumption

The standard simplifying assumption is about a single hardware sector. SQLite states it directly: it does not assume a sector write is atomic, but it does assume a sector write is linear — the hardware writes a sector byte-by-byte from one end to the other, so an interrupted sector write leaves a known-direction partial result rather than random garbage. [src: SQLite "Atomic Commit" §2.2 — "a sector write is not atomic … SQLite does always assume that a sector write is linear"] This is why recovery can detect an incomplete journal: the page-count field is written last, so a torn journal header reads as "0 pages" and is treated as not-yet-committed.

What a crash is assumed not to do

  • It does not corrupt bytes that were never written in this session and were already durably on disk (we assume the media is otherwise reliable; bit-rot detection is a separate concern, handled by checksums — see 5.4/5.8).
  • It does not silently mis-direct a write to the wrong location (again, checksums catch the cases that matter).
  • fsync, when it returns, is assumed to mean the data is durable. This is an assumption, not a theorem — some hardware has historically lied — but every mechanism here is built on it. ⚠️VERIFY the fsync behavior of your target platform/filesystem if durability is safety-critical.
⚠️ The barrier is the whole game

Because writes can be reordered and caches can vanish, the only tool that gives ordering is the durability barrier (sync_all / fsync). Every correct commit protocol is fundamentally a sequence of "write these bytes, barrier, then write those bytes, barrier." Remove a barrier and you reintroduce a crash window. Memorize where the barriers go — that's what 5.3 and 5.4 are really teaching.

✅ Concept check

1. What three things can a crash do to your file in this model?

Show answer
Lose unflushed cache contents, reorder writes that weren't separated by a barrier, and tear a multi-sector write (partial page).

2. What does "a sector write is linear but not atomic" let recovery do?

Show answer
Detect an incomplete write: if a written-last field (e.g. journal page count) is torn, it reads as not-yet-committed, so the partial record is ignored.

3. What is the only mechanism that gives you ordering guarantees across a crash?

Show answer
The durability barrier (fsync / sync_all). Writes between barriers may be reordered or lost.

4. Why is "fsync means durable" called an assumption?

Show answer
Because some hardware/firmware has historically acknowledged fsync before data was truly on stable media; correctness depends on the stack honoring the barrier.
