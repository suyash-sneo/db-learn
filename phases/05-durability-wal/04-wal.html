5.4 — Write-ahead logging (redo logging)

Goal: Understand the WAL: leave the database file untouched and append new page versions to a log. A commit is a record in the log; readers merge log + db; a crash is repaired by replaying committed log frames forward (redo). This is the mechanism your build targets.

Why this matters to your DB

The WAL flips the rollback journal around. In SQLite WAL mode, writer traffic appends page versions and can use fewer synchronization operations; readers use committed WAL frames plus the database. These are properties of SQLite's protocol and workload, not a universal promise that every WAL is faster. This project selects the same broad architecture.

The idea: append new versions, never overwrite (yet)

SQLite's WAL keeps the original pages in the database while appending changed page images to the separate WAL; a transaction commits when its commit-marked frame is appended. [src: SQLite WAL §1–2] So a write transaction:

  1. Appends a frame (a page number + that page's new content) to the WAL for each changed page.
  2. Marks the last frame of the transaction as a commit frame.
  3. Barrier: fsync the WAL (under PRAGMA synchronous=FULL) — after this, the transaction is durable. [src: SQLite WAL — "Writers sync the WAL on every transaction commit if PRAGMA synchronous is set to FULL"]

The writer does not apply its transaction pages to the main database during commit; a separate checkpoint may write the database at another time.

How readers see a consistent view

A reader can't just read the db file — newer data may be in the WAL. When a read transaction starts, it records the position of the last valid commit record: its end mark. For each page it needs, it uses the most recent WAL frame for that page at or before the end mark; if there is none, it reads the db file. Because the end mark is fixed for the transaction's duration, the reader sees a single point-in-time snapshot. [src: SQLite WAL §2.2 — end mark & reader snapshot]

Scanning the whole WAL per page would be slow, so SQLite keeps a wal-index in shared memory mapping pages to frames; the WAL documentation says the wal-index is transient, normally small, and need not be synchronized because it can be reconstructed from the WAL. [src: SQLite WAL §2.3 — wal-index purpose, shared memory, size, and reconstruction] For a single-process embedded engine you can keep this index as an ordinary in-memory hash map rebuilt on open.

The WAL frame, concretely

SQLite's WAL file is a 32-byte header followed by zero or more frames; each frame is a 24-byte frame header followed by one page of data. The frame header records the page number and, for commit frames, the database size in pages after the commit (a non-zero value marks the frame as a commit frame); the rest is salt and checksums. [src: SQLite file format §4 (WAL): 32-byte WAL header, 24-byte frame header, "db size in pages after commit" non-zero ⇒ commit frame] You'll mirror a simplified version of this in the build (5.8 dives into the exact bytes).

Why SQLite WAL can reduce commit work

  • No undo copy on the commit path: the transaction appends each new page image to the WAL rather than first saving the old page and then overwriting the database. A later checkpoint still writes the selected page image into the main database.
  • More sequential commit I/O: WAL appends are sequential, which can be favorable compared with scattered in-place writes; the benefit remains device- and workload-dependent. [src: SQLite WAL §1 — I/O tends to be more sequential and uses fewer fsync() operations]
  • Readers don't block the writer and vice versa: the writer appends to the WAL while readers read the db + earlier WAL frames. There is still only one writer at a time (one WAL). [src: SQLite WAL — "readers do not block writers and a writer does not block readers"; "only one writer at a time"]
⚠️ The WAL grows until you checkpoint

Because transaction pages accumulate in the WAL until checkpointing, an unchecked WAL keeps growing and can increase read work. The WAL is not a permanent home for data — it's a fast staging area that must periodically be folded back into the db (5.5). Under FULL synchronization, a committed WAL transaction can already satisfy durability; checkpointing is maintenance, not an additional condition for that commit to count.

✅ Concept check

1. During a WAL write transaction, is the main database file modified? What is?

Show answer
The writer appends new page versions to the WAL rather than applying them to the db during commit. A separate checkpoint may modify the db. Under FULL mode, commit includes the WAL synchronization step.

2. How does a reader get a consistent snapshot with data split across db and WAL?

Show answer
It fixes an end mark (last valid commit record) at start; per page it uses the newest WAL frame ≤ end mark, else the db page — a single point-in-time view.

3. What marks a frame as a commit frame in SQLite's format?

Show answer
A non-zero "database size in pages after commit" field in the 24-byte frame header (zero on non-commit frames).

4. Why can SQLite WAL mode make commits faster than rollback journaling?

Show answer
The commit path avoids writing undo copies and modifying scattered database pages, uses append-oriented I/O, and generally uses fewer synchronization operations. Checkpoint work still exists, and the result depends on workload and storage.
