5.6 — Recovery on open

Goal: Turn the mechanisms into a concrete routine that runs every time the database is opened: validate recovery state, decide what's committed, and expose a consistent state — roll back a hot journal or reconstruct the committed WAL view — before any normal access is allowed.

Why this matters to your DB

Durability is only real if recovery actually runs and is correct. The recovery routine is the payoff of everything in this phase: it's the code that makes "committed" survive a crash. Crucially, it must run before the first read, and must be safe to run repeatedly (a crash during recovery must leave recovery still correct on the next attempt).

Recovery for a rollback journal

On open, check for a hot journal — a journal that needs replaying because a writer crashed mid-commit. SQLite's exact conditions: the journal exists, is non-empty, has no reserved lock on the db, has a well-formed (non-zeroed) header, and either names no super-journal or its super-journal exists. [src: SQLite "Atomic Commit" §4.2 — "A rollback journal is a 'hot' journal if all of the following are true…"] If hot, recovery:

  1. Takes an exclusive lock on the db.
  2. Copies each original page from the journal back to its location in the db (undo).
  3. Truncates the db to the original size recorded in the journal header.
  4. Barrier: fsync the db.
  5. Deletes the journal.

Result: the interrupted transaction is undone; the db is back to its pre-transaction state.

Recovery for a WAL (your build)

WAL contains new page images, so it is a redo log. On open, scan the WAL from the start, validating each frame's checksums, and find the position of the last valid commit frame — frames after it (an interrupted, uncommitted transaction) are ignored. Everything up to and including the last commit frame is the committed content.

  1. If the WAL is absent or empty, there is no WAL work. If a nonempty header has the wrong magic/version/page size or a bad header checksum, report corruption/incompatibility rather than silently treating it as empty.
  2. Walk frames, verifying each frame's checksum against the running checksum; stop trusting frames at the first checksum failure. The checksum proves that the frame is invalid, not whether the cause was a torn write, bit flip, or some other fault.
  3. Record the latest frame for each page, up to the last valid commit frame. (Build the in-memory wal-index here.)
  4. Either leave the WAL in place (reads will merge it) or immediately checkpoint it into the db (5.5) and reset it.

The checksum-validated, commit-frame-bounded scan identifies the valid committed prefix and rejects an invalid or uncommitted tail. Keeping that committed prefix indexed already restores the logical database view; copying it into the main file is a checkpoint, not a prerequisite for opening. [src: SQLite file format §4 (WAL) — per-frame checksums; commit frame = non-zero db-size field]

Recovery must be idempotent

A crash can happen during recovery. Both schemes survive this because recovery is re-runnable: redoing the same WAL frames produces the same db pages; re-applying the same journal originals produces the same rollback. As long as you don't remove the log (delete journal / reset WAL) until after the db's recovering writes are synced, a crash mid-recovery just means recovery runs again — same result.

⚠️ Recovery runs before the first byte of normal access

If you let a read or write touch the db before recovery completes, you can read half-applied state or overwrite the log you still needed. Open must be: detect → recover (with its barrier) → only then admit normal operations. In your build, fold this into open() so it's impossible to skip.

✅ Concept check

1. What does each recovery scheme reconstruct?

Show answer
A rollback journal restores original pages (undo). WAL recovery validates and indexes committed new page images (redo); checkpointing may later copy those images into the main database.

2. In WAL recovery, what is the cutoff between recovered and discarded data?

Show answer
The last checksum-valid commit frame. Frames after it (uncommitted or torn) are ignored.

3. Why must recovery be idempotent?

Show answer
A crash can interrupt recovery itself; re-running must yield the same consistent result, which holds as long as the log isn't removed before the recovering db writes are synced.

4. Where in the engine should recovery live, and why?

Show answer
Inside open(), before any read or write, so half-applied state can never be observed and the log isn't clobbered prematurely.
