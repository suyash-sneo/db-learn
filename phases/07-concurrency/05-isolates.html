7.5 — Flutter isolates across FFI

Goal: Understand Dart's concurrency model — isolates with separate memory that communicate by message passing — and what it means to call your Rust engine across the FFI from more than one isolate, so the consumer-side concurrency story matches the engine-side one.
📂 Where to read

Dart's concurrency model is documented under "Isolates" and "Concurrency"; the FFI boundary under "C interop with dart:ffi." This lesson is pinned to the Dart 3.12.2 documentation shown by the API site on June 21, 2026. [src: dart.dev — Concurrency (Dart 3.12.2); dart.dev — C interop]

Why this matters to your DB

Your engine's first consumer is Dart/Flutter. A database call can be slow (disk I/O, a big query), and on a phone you must not block the UI. The Dart answer is to move heavy work to a background isolate. But isolates have a memory model very different from threads, and that changes how — and whether — two isolates can share one engine handle. Getting this right is the difference between a responsive app and either a frozen UI or a corrupted database.

An isolate is not a thread

Dart concurrency is built on isolates. Each isolate has its own state and event loop, and ordinary Dart objects are not jointly mutable across isolates. Ports carry asynchronous messages; immutable values may be shared internally, other sendable objects are copied, and TransferableTypedData supports transfer-oriented byte messaging. [src: Dart concurrency — isolates and message types; SendPort.send (Dart 3.12.2)]

Isolate.run(fn) is useful for one computation: it creates an isolate, runs and awaits the computation there, sends the result back using Isolate.exit, then the worker terminates. That lifecycle is a poor fit for a persistent database handle. Use a long-lived Isolate.spawn worker with request/reply ports when one isolate must own an open engine. [src: Isolate.run; Dart concurrency — background workers]

The FFI boundary, precisely

dart:ffi lets Dart Native programs call C-compatible APIs and access native memory. Library loading is platform/package-layout specific, as Phase 1 explains; after loading, symbols are looked up and converted to Dart-callable functions. [src: dart.dev — C interop] Two consequences matter for concurrency:

  • An ordinary FFI function call is synchronous from Dart's point of view. Until it returns, the isolate is still executing that event and cannot process the next queued UI event. Dart's concurrency guide warns that lengthy synchronous operations make a client unresponsive and explicitly says a worker isolate may block without affecting other isolates. [src: Dart concurrency — event handling and background workers]
  • Native memory is outside Dart's garbage collector. Pointers you get back from C are raw addresses; you must free native allocations yourself (e.g. via the allocator from package:ffi), and the allocate/free pair must match the engine's ownership contract from Phase 1 / Phase 8. [src: Dart C interop]
⚠️ Do not send the FFI handle to another isolate

In Dart 3.12.2, DynamicLibrary and Pointer are among the object types that cannot be sent through isolate messages. Passing pointer.address as an integer and rebuilding it with Pointer.fromAddress bypasses that guard, but it does not establish lifetime, ownership, synchronization, or use-after-free safety. This curriculum does not use that escape hatch. Create and destroy the native handle in the same long-lived worker isolate. [src: Dart concurrency — message type exceptions (Dart 3.12.2); Pointer API]

Two architectures — with different engine requirements

PatternHowTrade-off
One DB-owning workerA long-lived isolate loads the library, opens one handle, and services request/reply messages.One clear owner and serialized ABI entry. Message data must be sendable and copied/transferred. Recommended here.
Independent handlesEach isolate loads/binds the library itself and opens the same database path separately.The engine must coordinate distinct handles to one file, including cross-process cases. A per-handle Arc<RwLock> is insufficient because the handles do not necessarily share that Arc.

The owning-isolate design serializes calls entering that one handle, but it does not protect against a second process or another independently opened handle. If the product permits those, the engine still needs file-level coordination or an explicit refusal policy.

Crossing back: callbacks and async

Dart 3.12.2 offers several callback shapes. NativeCallable.isolateLocal must be invoked on the creating isolate's mutator thread and aborts the process if native code calls it from another thread. NativeCallable.listener may be invoked from any thread; it sends arguments to the creating isolate asynchronously, supports only void return, and requires pointed-to resources to stay valid until the later callback. Both must be closed; invoking after close is undefined behavior. [src: NativeCallable.isolateLocal; NativeCallable.listener (Dart 3.12.2)]

This curriculum's Phase 8 API remains synchronous request/response and uses a worker isolate, so it does not need native-to-Dart callbacks. Also avoid using mutually blocking FFI calls as a synchronous channel between isolates: Dart's current guide warns that this can deadlock when runnable isolates exceed the VM's parallel-isolate limit. [src: Dart concurrency — synchronous blocking communication between isolates]

✅ Concept check

1. How do two Dart isolates share state, and how do they not?

Show answer
They communicate with asynchronous port messages. Depending on the sendable value, the VM may share immutable data, copy mutable data, or transfer supported typed data; the isolates do not jointly mutate an ordinary Dart object.

2. Why does a slow database call belong on a worker isolate?

Show answer
The call occupies the current synchronous event until native code returns, delaying queued UI work. A worker isolate may block without stopping the main isolate's event loop.

3. Can you send a Pointer or DynamicLibrary to the worker?

Show answer
No in Dart 3.12.2. The worker should load the library and open the handle itself. Sending an integer address and reconstructing a pointer bypasses the restriction without solving ownership or synchronization.

4. Why is the DB-owning-isolate pattern a safe default?

Show answer
It gives the library, bindings, and handle one owner and serializes entry to that handle. The engine still needs a policy for a second handle or process opening the same file.

5. How do isolateLocal and listener callbacks differ?

Show answer
isolateLocal is synchronous and thread-affine. listener can be called from any native thread but delivers later through a port, returns void, and requires argument resources to remain valid.

🦀 Rust just-in-time

An opaque Rust handle confined to one worker isolate may remain single-caller. If the C ABI later permits concurrent calls or independent handles to one file, encode that contract in the Rust types and synchronization design; Dart isolation does not supply Rust Send/Sync or OS file locking.
