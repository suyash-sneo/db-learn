7.5 — Flutter isolates across FFI

Goal: Understand Dart's concurrency model — isolates with separate memory that communicate by message passing — and what it means to call your Rust engine across the FFI from more than one isolate, so the consumer-side concurrency story matches the engine-side one.
📂 Where to read

Dart's concurrency model is documented under "Isolates" and "Concurrency"; the FFI boundary under "C interop with dart:ffi." [src: dart.dev — Isolates; dart.dev — C interop]

Why this matters to your DB

Your engine's first consumer is Dart/Flutter. A database call can be slow (disk I/O, a big query), and on a phone you must not block the UI. The Dart answer is to move heavy work to a background isolate. But isolates have a memory model very different from threads, and that changes how — and whether — two isolates can share one engine handle. Getting this right is the difference between a responsive app and either a frozen UI or a corrupted database.

An isolate is not a thread

Dart concurrency is built on isolates, each with its own memory and its own event loop. The defining property: "Isolates can't 'see' inside each others' memory, which is where the name 'isolate' comes from." They do not share mutable objects. [src: dart.dev — Isolates] Communication is by message passing over ports (SendPort/ReceivePort), and the data is transferred, not aliased — for the result of Isolate.run, "The worker isolate transfers the memory holding the result to the main isolate. It does not copy the data." [src: dart.dev — Isolates]

Isolate.run(fn) is the high-level helper: it spawns an isolate, runs fn on it, returns the result (as a Future), terminates the isolate, and re-throws any exception to the caller. [src: dart.dev — Isolates, Isolate.run()]

The FFI boundary, precisely

dart:ffi lets Dart on the native platform "call native C APIs, and to read, write, allocate, and deallocate native memory." A library is loaded with DynamicLibrary.open(path) and a function is bound via lookup<NativeFunction<…>>(name).asFunction(). [src: dart.dev — C interop] Two consequences matter for concurrency:

  • A synchronous FFI call runs on the calling isolate's thread and does not yield to the event loop until it returns. A long blocking C call therefore freezes that isolate — which is why a slow database call belongs on a worker isolate, not the main (UI) one. ⚠️VERIFY the exact threading/blocking wording against the current dart:ffi and API reference.
  • Native memory is outside Dart's garbage collector. Pointers you get back from C are raw addresses; you must free native allocations yourself (e.g. via the allocator from package:ffi), and the allocate/free pair must match the engine's ownership contract from Phase 1 / Phase 8.
⚠️ Two isolates, two views of one native handle

Dart's "no shared memory" rule is about Dart objects. A native pointer is just an integer address; nothing stops two isolates from each holding a binding that points at the same Rust engine. If they call into it concurrently, you are back to 7.1 — the engine itself must be safe under concurrent entry. So the safe patterns are: (1) confine the engine to one isolate and send requests to it by message (the engine sees a single caller), or (2) make the Rust side internally synchronized (7.4) so concurrent FFI entry is well-defined. Choose one explicitly; do not let two isolates aliase an unsynchronized handle by accident.

Two workable architectures

PatternHowTrade-off
DB-owning isolateOne isolate opens the engine; others send query/CRUD messages and await replies.Simple and matches Dart's model; the owning isolate serializes all access (a clean single-writer funnel). Adds message-passing latency.
Shared synchronized handleMultiple isolates bind the same library; the Rust side uses internal locking (Arc<RwLock>, 7.4).More parallel reads, but you must implement and prove the engine-side synchronization, and native memory lifetimes get trickier.

For a first embedded database, the DB-owning-isolate pattern is the lower-risk default: it makes the "single writer" rule fall out of the architecture instead of requiring engine-side locking to enforce it.

Crossing back: callbacks and async

Calling C from Dart is the easy direction. Having C call back into Dart, or surfacing native events asynchronously, is more constrained and version-sensitive (it has evolved across Dart releases). This curriculum's API (Phase 8) is deliberately synchronous, request/response, byte-buffer-based — which sidesteps callback complexity. If you later need async native notifications, treat it as a separate design with its own verification. ⚠️VERIFY current callback/async-FFI capabilities against the dart:ffi docs for the Dart version you target.

✅ Concept check

1. How do two Dart isolates share state, and how do they not?

Show answer
They do not share mutable Dart memory; they communicate only by passing messages over ports (data is transferred/copied, not aliased).

2. Why does a slow database call belong on a worker isolate?

Show answer
A synchronous FFI call runs on the calling isolate's thread without yielding to its event loop, so a long call on the main isolate freezes the UI. A worker isolate keeps the UI responsive.

3. Does Dart's "no shared memory" rule prevent two isolates from racing on the engine?

Show answer
No. A native pointer is just an address; two isolates can hold bindings to the same Rust handle. The rule is about Dart objects, so the engine must be confined to one isolate or be internally synchronized.

4. Why is the DB-owning-isolate pattern a safe default?

Show answer
All access funnels through one isolate that serializes calls, so the single-writer rule emerges from the architecture rather than needing engine-side locking to enforce it.
