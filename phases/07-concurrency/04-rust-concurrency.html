7.4 — Rust concurrency for this

Goal: Pick the right Rust primitive for sharing your engine across threads — Arc, Mutex, RwLock — and understand precisely what the Send/Sync marker traits do and do not guarantee, so the locking layer you build rests on the compiler's checks rather than hope.

Why this matters to your DB

Phase 1 wrapped the engine in an opaque handle and noted that boundary code creating &mut Db is not concurrently usable. To support more than one actor you need a deliberate synchronization design. Rust is unusually well-suited here: its type system rejects data races at compile time, so "did I forget a lock?" becomes a build error rather than a once-in-a-million corruption. But the compiler protects memory safety, not your database invariants — you must still choose the locking granularity and the isolation model. This lesson is about using the language's guarantees without overclaiming them.

What Rust guarantees: data-race freedom

Rust's ownership and borrowing rules extend to threads. The core guarantee, in the Rust Book's framing, is that safe Rust prevents data races — concurrent unsynchronized access to the same memory where at least one access is a write — at compile time. [src: The Rust Programming Language, ch. 16 "Fearless Concurrency"] Two marker traits encode the rules:

  • Send: a type is safe to transfer ownership to another thread.
  • Sync: a type is safe to share by reference (&T) across threads — equivalently, &T is Send.

These are auto-derived for types built entirely from Send/Sync parts, and the compiler refuses to move a non-Send value across a thread boundary or share a non-Sync value. Rc is neither (non-atomic refcount); Arc is both (atomic refcount). [src: Rust Book ch. 16.4 — Send and Sync]

⚠️ Data-race freedom is not database consistency

The compiler guarantees you won't have two threads scribbling the same bytes unsynchronized. It does not guarantee your transactions are isolated, that you took the lock at the right granularity, or that two well-synchronized writers can't still produce a lost update at the logical level. Memory safety and logical correctness are different layers; Rust gives you the first for free in safe code and leaves the second to your design.

The combination Arc<Mutex<Engine>> is the standard "share one mutable thing across threads" pattern: Arc for shared ownership, Mutex for safe mutation. RwLock swaps the mutex for the "many readers or one writer" policy — which maps directly onto SQLite's SHARED-vs-EXCLUSIVE distinction (7.2). [src: Rust Book ch. 16.3 — Shared-State Concurrency; std::sync::RwLock]

// The canonical shared-handle shape (illustrative — not your engine's logic).
use std::sync::{Arc, RwLock};

let engine = Arc::new(RwLock::new(open_engine()?));

// reader thread:
let r = Arc::clone(&engine);
let guard = r.read().unwrap();   // many readers may hold this at once

// writer thread:
let w = Arc::clone(&engine);
let mut guard = w.write().unwrap(); // exclusive; waits for readers to drain

The guard pattern and poisoning

Locking returns an RAII guard; the lock is released when the guard is dropped, so you cannot forget to unlock. Two precise caveats for a database:

  • Poisoning. If a thread panics while holding a Mutex/RwLock, the lock becomes poisoned; subsequent lock()/write() calls return an Err. This is std's way of warning that the protected data may be in a half-updated state — which is exactly the situation AGENTS calls out: do not keep using a handle whose invariants a panic may have broken. Decide deliberately whether to propagate the error or treat the engine as failed. [src: std::sync::Mutex — Poisoning]
  • Deadlock is still possible. The borrow checker prevents data races, not deadlocks: acquiring two locks in inconsistent orders, or re-locking a non-reentrant Mutex on the same thread, can hang. Lock ordering is your responsibility.

How this maps onto the build

The build implements a single-writer, multi-reader policy in-process. A RwLock (or an explicit lock-state field guarded by a Mutex) models the SHARED vs. EXCLUSIVE distinction; Arc lets multiple threads hold the handle. Because the tests run in one process with in-memory state, they exercise the policy logic — who is allowed in when — not real cross-process OS file locking or true hardware parallelism. The build page states this limit explicitly.

🦀 Rust just-in-time
  • Arc::clone(&x) bumps the refcount and is cheap; it does not deep-copy the data. Prefer it over x.clone() to make the intent ("another owner") obvious.
  • lock()/read()/write() return a Result because of poisoning — handle it, don't blanket-unwrap() in engine code.
  • A guard borrows the lock; keep its scope as small as possible so you hold the lock briefly (mirrors SQLite holding EXCLUSIVE only for the write).

✅ Concept check

1. What exactly do Send and Sync mean?

Show answer
Send: ownership can be transferred to another thread. Sync: &T can be shared across threads (i.e. &T is Send). The compiler enforces both at the type level.

2. What does Rust's data-race freedom not protect?

Show answer
Logical database consistency, correct lock granularity, isolation, and deadlock freedom. It prevents unsynchronized memory access, not bad concurrency design.

3. Why might RwLock fit this engine better than Mutex?

Show answer
It allows many concurrent readers while still giving the writer exclusivity — directly modeling SQLite's SHARED-vs-EXCLUSIVE policy, which a plain Mutex (one accessor of any kind) does not.

4. What is lock poisoning and why does it matter for a database handle?

Show answer
If a thread panics while holding the lock, later lock attempts return Err to signal the protected data may be half-updated. For a DB it's a cue not to keep using a possibly-inconsistent handle — handle the error deliberately.
