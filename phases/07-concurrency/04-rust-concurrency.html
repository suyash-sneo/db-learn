7.4 — Rust concurrency for this

Goal: Pick the right Rust primitive for sharing your engine across threads — Arc, Mutex, RwLock — and understand precisely what the Send/Sync marker traits do and do not guarantee, so the locking layer you build rests on the compiler's checks rather than hope.

Why this matters to your DB

Phase 1 wrapped the engine in an opaque handle and noted that boundary code creating &mut Db is not concurrently usable. To support more than one actor you need a deliberate synchronization design. Safe Rust's type system prevents data races in the Rust memory it can reason about. Unsafe code, raw pointers, FFI callers, files shared with another process, and logically incorrect lock scopes remain your responsibility. The compiler protects memory safety within its contracts, not your database's transaction invariants.

What Rust guarantees: data-race freedom

Rust's ownership and borrowing rules extend to threads. The core guarantee, in the Rust Book's framing, is that safe Rust prevents data races: concurrent accesses to one memory location where at least one is a write and the accesses are not synchronized. [src: The Rust Programming Language, ch. 16 "Fearless Concurrency"] Two marker traits encode the rules:

  • Send: a type is safe to transfer ownership to another thread.
  • Sync: a type is safe to share by reference (&T) across threads — equivalently, &T is Send.

These unsafe auto traits are inferred for many composite types from their fields, and the compiler refuses to move a non-Send value across a thread boundary or share a non-Sync value through APIs that require those bounds. Rc<T> is neither because its reference count is non-atomic. Arc<T> uses an atomic reference count, but it is Send/Sync only when T has the required traits; wrapping RefCell<T> in Arc does not make the inner mutation thread-safe. [src: Rust Book ch. 16.4; std::sync::Arc — thread safety]

⚠️ Data-race freedom is not database consistency

Two methods may each lock correctly and still form a lost update if the application reads under one guard, drops it, computes, then writes under another. Rust also cannot make an advisory file lock mandatory for a noncooperating process. Memory safety, file-level exclusion, and transaction isolation are different claims.

Arc<Mutex<Engine>> is a common coarse-grained starting point: Arc supplies ownership and Mutex supplies mutual exclusion. An RwLock permits multiple in-process read guards, but it does not reproduce SQLite's RESERVED/PENDING ladder, snapshots, or cross-process file locks. The standard library also leaves RwLock priority policy to the operating system, so do not promise writer fairness. [src: Rust Book ch. 16.3 — Shared-State Concurrency; std::sync::RwLock]

// Toy shared value: demonstrates the wrapper shape, not database isolation.
use std::sync::{Arc, RwLock};

let counter = Arc::new(RwLock::new(0_u64));

{
    let r = Arc::clone(&counter);
    let guard = r.read().unwrap();
    assert_eq!(*guard, 0); // read guard drops at this brace
}
{
    let w = Arc::clone(&counter);
    let mut guard = w.write().unwrap();
    *guard += 1; // exclusive within this one RwLock
}

The guard pattern and poisoning

Locking returns an RAII guard; dropping the guard releases the lock. Three caveats matter:

  • Poisoning is advisory, not proof. A standard Mutex is normally poisoned when a panic unwinds through its guard; RwLock poisoning is triggered by a panic while exclusively locked, not by a reader panic. The standard docs also list circumstances in which poisoning detection is not guaranteed. Treat a poison error as a warning that invariants may be broken, never as the mechanism that makes unsafe recovery sound. [src: Mutex poisoning; RwLock poisoning]
  • Deadlock is still possible. The borrow checker prevents data races, not deadlocks: acquiring two locks in inconsistent orders can create a cycle. Calling lock again from the same thread while it already holds a standard Mutex is explicitly unspecified: it may panic or deadlock. [src: Mutex::lock]
  • Atomics cover only the stated atomic value. An atomic ID increment does not make a multi-step record insertion atomic, and relaxed ordering does not publish unrelated state. Use atomics only with a written memory-ordering argument. [src: std::sync::atomic]

How this maps onto the build

The build separates a pure lock-state table from an Arc<Mutex<LockTable>> wrapper. The table makes compatibility rules deterministic and testable; the mutex prevents data races in that table; RAII guards release owner records. This still provides only in-process coordination. It neither locks the database file against another process nor creates WAL reader snapshots.

🦀 Rust just-in-time
  • Arc::clone(&x) increments the strong count and does not deep-copy T. The allocation is released after the final strong owner is dropped. [src: Arc]
  • lock()/read()/write() return a Result because of poisoning — handle it, don't blanket-unwrap() in engine code.
  • A guard borrows the lock; keep its scope as small as possible so you hold the lock briefly (mirrors SQLite holding EXCLUSIVE only for the write).

✅ Concept check

1. What exactly do Send and Sync mean?

Show answer
Send: ownership can be transferred to another thread. Sync: &T can be shared across threads (i.e. &T is Send). The compiler enforces both at the type level.

2. What does Rust's data-race freedom not protect?

Show answer
Logical database consistency, correct lock granularity, isolation, and deadlock freedom. It prevents unsynchronized memory access, not bad concurrency design.

3. Why might RwLock fit this engine better than Mutex?

Show answer
It can allow several in-process read guards while giving one write guard exclusivity. It does not model RESERVED/PENDING, snapshots, or cross-process locking, and its fairness policy is unspecified.

4. What is lock poisoning and why does it matter for a database handle?

Show answer
A standard Mutex normally reports poison after a panic unwinds through its guard; an RwLock does so for a writer panic, not a reader panic. Poisoning is advisory and not guaranteed to detect every invariant-breaking panic, so database recovery still needs a rollback-or-fail-closed policy.
