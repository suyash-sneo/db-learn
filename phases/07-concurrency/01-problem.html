7.1 — The concurrency problem

Goal: See exactly how two operations running at the same time can corrupt data or produce results that no serial order could, so that the rest of the phase is about preventing specific named failures rather than chasing a vague feeling of "thread safety."

Why this matters to your DB

Your engine so far assumes one actor: one thread opens the file, reads and writes pages, commits. The moment a second actor touches the same file — another thread, another isolate, another process — the interleaving of their individual steps becomes something neither of them controls. The hardware and OS are free to run their instructions in any order consistent with each thread's own sequencing. Most interleavings are fine; a few are catastrophic. Concurrency control is the discipline of forbidding exactly the bad ones while allowing as many good ones as possible.

Anatomy of a race: lost update

A "transaction" you think of as atomic is, at the machine level, many steps: read a page, modify bytes in memory, write the page back. Interleave two such sequences and the result depends on timing. Take a counter stored in a document, two transactions each doing "increment by 1":

Nothing here is a "bug" in either transaction. Each one read, added one, and wrote — correct in isolation. The corruption emerges only from the interleaving. This is the lost update anomaly, and it is the canonical reason a read-modify-write must be protected as a unit.

The catalogue of failures

It helps to name the failures precisely, because isolation levels (7.3) are defined as "which of these may occur." The SQL standard frames concurrency phenomena as things that must not happen:

PhenomenonWhat happens
Dirty read"A transaction reads data written by a concurrent uncommitted transaction" — data that may be rolled back.
Nonrepeatable read"A transaction re-reads data it has previously read and finds that data has been modified by another transaction (that committed since the initial read)."
Phantom read"A transaction re-executes a query returning a set of rows that satisfy a search condition and finds that the set of rows … has changed due to another recently-committed transaction."
Serialization anomaly"The result of successfully committing a group of transactions is inconsistent with all possible orderings of running those transactions one at a time."

[src: PostgreSQL — Transaction Isolation §13.2, which quotes the SQL-standard phenomena]

Lost update is closely related — it is the write-write version of the read anomalies above, and some systems treat it as a distinct phenomenon. The point is not to memorize a taxonomy but to see that "concurrency correctness" decomposes into concrete, testable failures.

Corruption vs. anomaly: two different stakes

There are really two levels of damage, and they need different defenses:

  • Structural corruption. Two writers updating the same B+tree page or the same freelist simultaneously can leave the on-disk structures themselves invalid — a torn split, a double-freed page, a dangling pointer. After this, even single-threaded reads return garbage or crash. This must be prevented absolutely.
  • Logical anomalies. The structures stay valid, but a transaction observes a state that no serial execution would have produced (lost update, dirty read, …). The database is not broken, but an application invariant may be.

A single-writer rule (only one transaction may write at a time) is enough to prevent structural corruption from concurrent writers, which is why it is the simplest correct design and the one your build targets. Preventing the full set of logical anomalies is a stronger, separate requirement that needs an isolation policy (7.3).

⚠️ "It worked when I tested it" is not evidence of correctness

Concurrency bugs are timing-dependent: a racy program can produce the right answer thousands of times and the wrong answer once, on a different machine, under load. You cannot prove concurrency correctness by running it and observing success. You establish it by argument — showing which interleavings are possible and that each preserves your invariants — and you defend it with primitives whose guarantees are specified. This is also why the in-memory tests in this phase's build validate logic, not real parallel timing.

What "doing nothing" actually costs

It is tempting to skip concurrency entirely: "my app is single-threaded." But the embedded target for this curriculum is consumed from Dart/Flutter (Phase 1, Phase 8), where work is commonly offloaded to background isolates (7.5), and a mobile app can be backgrounded, killed, and relaunched while a write is in flight (Phase 5's crash model). Even a "single-threaded" application can end up with two live handles to one file. Deciding your concurrency model explicitly — even if the decision is "single writer, fail fast on conflict" — is part of building a correct engine.

✅ Concept check

1. Why is a lost update possible even though each transaction's logic is correct?

Show answer
Each transaction is internally correct, but a read-modify-write is several steps; if both read the old value before either writes, the second write overwrites the first. The fault is in the unguarded interleaving, not the logic.

2. Distinguish structural corruption from a logical anomaly.

Show answer
Structural corruption leaves the on-disk data structures themselves invalid (torn split, double free) so even serial reads break. A logical anomaly keeps structures valid but lets a transaction observe a state no serial order would produce.

3. Name three of the standard concurrency phenomena.

Show answer
Any three of: dirty read, nonrepeatable read, phantom read, serialization anomaly (and the related lost update).

4. Why can't you establish concurrency correctness by testing alone?

Show answer
Races are timing-dependent; a wrong interleaving may occur rarely and only under specific load/hardware. Correctness comes from reasoning about possible interleavings plus primitives with specified guarantees, not from observed passes.
