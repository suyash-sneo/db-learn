7.1 — The concurrency problem

Goal: See exactly how two operations running at the same time can corrupt data or produce results that no serial order could, so that the rest of the phase is about preventing specific named failures rather than chasing a vague feeling of "thread safety."

Why this matters to your DB

Your engine so far assumes one actor: one thread opens the file, reads and writes pages, commits. The moment a second actor touches the same file — another thread, another isolate, another process — their relative execution order is unspecified unless the actors coordinate. Compilers, CPUs, runtimes, and the operating system all participate in that execution; source-code order in one actor does not by itself create a synchronization edge with another actor. Most interleavings are harmless; some violate a database invariant. Concurrency control identifies the conflicting operations and orders or rejects them. [src: std::sync — synchronization primitives; Rust atomic memory orderings]

Anatomy of a race: lost update

A "transaction" you think of as atomic is, at the machine level, many steps: read a page, modify bytes in memory, write the page back. Interleave two such sequences and the result depends on timing. Take a counter stored in a document, two transactions each doing "increment by 1":

Each transaction's arithmetic is correct by itself. The failure appears because the database did not protect the whole read-modify-write as one conflict unit. This is a lost update. Serializing only the final writes is insufficient: both transactions may already have computed from the stale value. The lock, validation, or atomic operation must cover the dependency from read through write.

The catalogue of failures

Isolation levels (7.3) are described by the phenomena they must prevent. The table below paraphrases the SQL phenomena listed by PostgreSQL's standards-based isolation documentation:

PhenomenonWhat happens
Dirty readA transaction observes another transaction's uncommitted write, which may later be rolled back.
Nonrepeatable readThe same transaction reads one item twice and sees a committed change made between those reads.
Phantom readRepeating a predicate query produces a different matching set because another transaction committed inserts, deletes, or qualifying updates.
Serialization anomalyThe committed outcome cannot be explained by any one-at-a-time ordering of the transactions.

[src: PostgreSQL 18 — Transaction Isolation, Table 13.1 and the phenomenon definitions]

Lost update is another useful named failure, even though engines and literature classify it differently. The practical lesson is to write down the histories your engine permits rather than claim vaguely that it is “thread-safe.”

A schedule is the object you reason about

A schedule is the combined order of steps from concurrent transactions. A serial schedule runs all of T1, then all of T2 (or the reverse). A concurrent schedule interleaves them. An interleaved schedule is serializable when its externally visible effect matches some serial schedule. This definition is stronger than “the file did not corrupt”: the B+tree can remain structurally valid while the counter example still loses an update. [src: PostgreSQL 18 — Serializable Isolation Level]

Corruption vs. anomaly: two different stakes

There are really two levels of damage, and they need different defenses:

  • Structural corruption. Two writers updating the same B+tree page or the same freelist simultaneously can leave the on-disk structures themselves invalid — a torn split, a double-freed page, a dangling pointer. After this, even single-threaded reads return garbage or crash. This must be prevented absolutely.
  • Logical anomalies. The structures stay valid, but a transaction observes a state that no serial execution would have produced (lost update, dirty read, …). The database is not broken, but an application invariant may be.

A single-writer rule prevents two cooperating writers from mutating storage structures simultaneously, provided every write path obeys the same lock and the lock covers the complete storage transaction. It does not by itself prevent readers from seeing partially published state, nor does it prevent a lost update when an application reads outside the protected write transaction and later writes a stale result. Structural exclusion, publication rules, and transaction isolation are separate obligations.

⚠️ "It worked when I tested it" is not evidence of correctness

Concurrency bugs are timing-dependent: a racy program may pass repeatedly and still have an allowed failing execution. Tests are useful for exercising transitions and known regressions, but observed passes do not prove every schedule safe. Establish correctness by stating invariants, identifying every state transition and synchronization edge, and then using tests to challenge that argument. The build includes deterministic policy tests and a small threaded stress check; neither substitutes for the argument.

What "doing nothing" actually costs

It is tempting to skip concurrency entirely: "my app is single-threaded." But the embedded target for this curriculum is consumed from Dart/Flutter (Phase 1, Phase 8), where work is commonly offloaded to background isolates (7.5), and a mobile app can be backgrounded, killed, and relaunched while a write is in flight (Phase 5's crash model). Even a "single-threaded" application can end up with two live handles to one file. Deciding your concurrency model explicitly — even if the decision is "single writer, fail fast on conflict" — is part of building a correct engine.

✅ Concept check

1. Why is a lost update possible even though each transaction's logic is correct?

Show answer
Each transaction is internally correct, but a read-modify-write is several steps; if both read the old value before either writes, the second write overwrites the first. The fault is in the unguarded interleaving, not the logic.

2. Distinguish structural corruption from a logical anomaly.

Show answer
Structural corruption leaves storage structures invalid, so later ordinary reads can fail. A logical anomaly can leave every page valid while producing an application-visible history that violates the promised isolation rule.

3. Name three of the standard concurrency phenomena.

Show answer
Any three of: dirty read, nonrepeatable read, phantom read, serialization anomaly (and the related lost update).

4. Why can't you establish concurrency correctness by testing alone?

Show answer
Races are timing-dependent; a wrong interleaving may occur rarely and only under specific load/hardware. Correctness comes from reasoning about possible schedules and synchronization guarantees; tests only sample executions.
