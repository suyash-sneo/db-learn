Phase 7

Concurrency

Phase 5 gave you the A and D of ACID. This phase is about the I: what two or more transactions are allowed to observe about each other. You'll see why uncontrolled concurrent access corrupts data, study the two dominant families of solution — locking and multi-version concurrency control (MVCC) — using SQLite as the concrete reference, then map them onto the tools you actually have: Rust's ownership-based concurrency and Dart's isolate model across the FFI boundary. The build is deliberately modest: a correct single-writer locking layer for one embedded file, with an honest account of what it does and does not guarantee.

Why concurrency comes after durability

Isolation is only meaningful once atomic, durable commits exist (Phase 5): you cannot reason about "what a concurrent transaction sees" until "a transaction" is a well-defined, all-or-nothing unit. Concurrency control then decides which interleavings of those units are permitted. Putting it here means every rule in this phase is layered on a commit protocol you already built and understand.

What you'll be able to do after Phase 7

Name the concrete anomalies uncontrolled concurrency causes; explain SQLite's five rollback-journal lock states and how WAL mode changes the reader/writer story; describe the SQL isolation levels and where MVCC and snapshot isolation sit among them; choose the right Rust synchronization primitive for a shared handle and justify it with Send/Sync; reason about calling your engine from multiple Dart isolates across the FFI; and implement a single-writer/multi-reader locking layer with tests that are honest about their limits.
