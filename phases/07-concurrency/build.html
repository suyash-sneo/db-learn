Build: a single-writer lock policy

Goal: Implement a deterministic owner-aware lock table, wrap it in thread-safe in-process coordination, and route one document-store handle through RAII read/write guards. The policy mirrors SQLite rollback-mode SHARED → RESERVED → PENDING → EXCLUSIVE transitions without pretending to implement SQLite's VFS or cross-process locks.
⚠️ AI off

This page contains contracts, todo!() stubs, failing tests, and graduated hints. Implement the state transitions yourself. Do not ask an AI to fill them in.

⚠️ Exact scope

This build has two layers: a pure LockTable that decides legal transitions, and a LockManager that shares that table through Arc<Mutex<…>>. It coordinates threads using this manager instance. It does not coordinate another process or an independently constructed manager, does not create reader snapshots, and returns WouldBlock instead of waiting. Real file locking belongs in a later platform-specific VFS layer.

The model you must implement

Every holder has an OwnerId. A writer first holds SHARED, then uniquely holds RESERVED. An EXCLUSIVE attempt enters PENDING; while PENDING, new SHARED requests fail. EXCLUSIVE becomes grantable when the only remaining SHARED owner is the writer itself. This counts the writer's own read lock explicitly rather than hiding it in a global boolean.

RequestGrant conditionEffect
SHARED(owner)owner is new; no PENDING or EXCLUSIVE writeradd owner to readers
RESERVED(owner)owner already has SHARED; no writer existsrecord the one RESERVED writer
EXCLUSIVE(owner)owner holds RESERVED; no other SHARED owner remainspass through PENDING, then replace the writer's SHARED+RESERVED state with EXCLUSIVE
failed EXCLUSIVEanother SHARED owner remainskeep this writer PENDING so new readers cannot jump ahead
releasethe same owner currently holds that roleremove exactly that owner's state; dropping a writer clears its own SHARED plus writer state

[src: SQLite lockingv3 §3 and §5. The owner-aware Rust API and fail-fast behavior are this curriculum's design choices.]

Ordered milestones

  1. M1 — pure table. Implement owner allocation, SHARED acquisition/release, a visible state summary, and invariant checks without threads or synchronization.
  2. M2 — writer path. Require SHARED before RESERVED; enforce one writer; implement RESERVED → PENDING → EXCLUSIVE and retry from PENDING after readers drain.
  3. M3 — failure atomicity. A failed compound manager operation must undo only the state it added. A failed second writer must not leak its temporary SHARED owner.
  4. M4 — synchronized wrapper. Put the table behind Arc<Mutex<LockTable>>. Return Poisoned rather than silently ignoring mutex poison. Allocate owner IDs without wrapping.
  5. M5 — RAII guards. Read and write guards release exactly their own owner state on drop. A write guard retains PENDING after WouldBlock and can retry EXCLUSIVE.
  6. M6 — document-store integration. On one Phase 6 handle, hold a read guard for the complete lookup/query and a write guard from before mutation begins through commit or rollback.

Acceptance criteria

  • All ten supplied unit tests pass unchanged.
  • The table never contains two writer owners, PENDING without a writer, or EXCLUSIVE alongside a reader owner.
  • RESERVED requires the same owner to hold SHARED first. PENDING rejects new readers while allowing existing readers to release.
  • A failed second-writer acquisition leaves no leaked reader owner.
  • Cloned managers share one table; the type is Send + Sync; the threaded race grants exactly one reservation.
  • Owner allocation fails closed at exhaustion and never wraps to a reused ID.
  • Mutex poisoning is surfaced as LockError::Poisoned. Do not use PoisonError::into_inner as an automatic “recovery.”
  • Manual integration check: instrument one Phase 6 handle and demonstrate that every public read/write path acquires the appropriate guard for the whole operation. Record that this does not protect another process or separately created manager.

append as inline unit tests in src/lockmgr.rs 
#[cfg(test)]
mod tests {
    use super::*;
    use std::sync::Barrier;

    const A: OwnerId = OwnerId(1);
    const B: OwnerId = OwnerId(2);
    const C: OwnerId = OwnerId(3);

    #[test]
    fn many_shared_owners_coexist() {
        let mut t = LockTable::new();
        t.acquire_shared(A).unwrap();
        t.acquire_shared(B).unwrap();
        assert_eq!(t.view(), LockView::Shared { readers: 2 });
    }

    #[test]
    fn reserved_requires_shared_and_is_unique() {
        let mut t = LockTable::new();
        assert_eq!(t.acquire_reserved(A), Err(LockError::InvalidTransition));
        t.acquire_shared(A).unwrap();
        t.acquire_reserved(A).unwrap();
        t.acquire_shared(B).unwrap();
        assert_eq!(t.acquire_reserved(B), Err(LockError::AlreadyWriter));
    }

    #[test]
    fn failed_exclusive_stays_pending_and_blocks_new_readers() {
        let mut t = LockTable::new();
        t.acquire_shared(A).unwrap(); // existing reader
        t.acquire_shared(B).unwrap(); // future writer's SHARED
        t.acquire_reserved(B).unwrap();
        assert_eq!(t.try_exclusive(B), Err(LockError::WouldBlock));
        assert_eq!(t.view(), LockView::Pending { readers: 2 });
        assert_eq!(t.acquire_shared(C), Err(LockError::WouldBlock));
    }

    #[test]
    fn pending_writer_retries_after_existing_reader_drains() {
        let mut t = LockTable::new();
        t.acquire_shared(A).unwrap();
        t.acquire_shared(B).unwrap();
        t.acquire_reserved(B).unwrap();
        assert_eq!(t.try_exclusive(B), Err(LockError::WouldBlock));
        t.release_shared(A).unwrap();
        t.try_exclusive(B).unwrap();
        assert_eq!(t.view(), LockView::Exclusive);
    }

    #[test]
    fn release_is_owner_checked() {
        let mut t = LockTable::new();
        t.acquire_shared(A).unwrap();
        assert_eq!(t.release_shared(B), Err(LockError::NotOwner));
        assert_eq!(t.view(), LockView::Shared { readers: 1 });
    }

    #[test]
    fn guard_drop_restores_unlocked() {
        let m = LockManager::new();
        { let _r = m.acquire_read().unwrap(); }
        assert_eq!(m.view(), Ok(LockView::Unlocked));
        { let mut w = m.acquire_write().unwrap(); w.try_exclusive().unwrap(); }
        assert_eq!(m.view(), Ok(LockView::Unlocked));
    }

    #[test]
    fn cloned_managers_share_one_table() {
        let a = LockManager::new();
        let b = a.clone();
        let _r = a.acquire_read().unwrap();
        assert_eq!(b.view(), Ok(LockView::Shared { readers: 1 }));
    }

    #[test]
    fn owner_ids_never_wrap() {
        let mut t = LockTable::with_next_owner(u64::MAX);
        assert_eq!(t.allocate_owner(), Err(LockError::OwnerExhausted));
    }

    #[test]
    fn mutex_poison_is_reported_not_ignored() {
        let manager = LockManager::new();
        let poisoner = manager.clone();
        let _ = std::thread::spawn(move || {
            let _guard = poisoner.inner.lock().unwrap();
            panic!("poison the lock-table mutex");
        }).join();
        assert_eq!(manager.view(), Err(LockError::Poisoned));
        assert_eq!(manager.acquire_read().unwrap_err(), LockError::Poisoned);
    }

    #[test]
    fn two_threads_racing_for_writer_grant_exactly_one() {
        fn assert_send_sync<T: Send + Sync>() {}
        assert_send_sync::<LockManager>();

        let manager = LockManager::new();
        let start = Arc::new(Barrier::new(2));
        let finish = Arc::new(Barrier::new(2));
        let mut joins = Vec::new();

        for _ in 0..2 {
            let m = manager.clone();
            let s = start.clone();
            let f = finish.clone();
            joins.push(std::thread::spawn(move || {
                s.wait();
                let held = m.acquire_write().ok();
                let won = held.is_some();
                f.wait(); // winner retains its guard until both attempted
                won
            }));
        }

        let wins = joins.into_iter()
            .map(|j| j.join().unwrap())
            .filter(|won| *won)
            .count();
        assert_eq!(wins, 1);
        assert_eq!(manager.view(), Ok(LockView::Unlocked));
    }
}

⚠️ A lock guard is not a rollback protocol

If a panic or error occurs after the engine has partially changed in-memory transaction state, releasing the lock does not repair that state. The Phase 5 transaction must roll back safely, or the handle must fail closed. Standard mutex poisoning cannot prove the database remains valid because the lock-table mutex is held only while changing lock metadata, not for the whole database operation.

Stuck? — graduated hints

The state combinations are getting tangled
Nudge: Write assert_valid before writing transitions.
Bigger nudge: The writer is either absent or exactly one (OwnerId, WriterPhase). RESERVED/PENDING requires that owner in readers; EXCLUSIVE requires an empty reader set.
Technique: mutate one transition, assert invariants, and return. Do not derive policy independently in the wrapper.
A failed second writer leaks a reader
Nudge: acquire_write is a compound operation.
Bigger nudge: Record which sub-step this call added.
Technique: if RESERVED fails after this call added SHARED, remove that same owner's SHARED before returning the original error.
The pending writer can never retry
Nudge: WouldBlock is not a rollback to RESERVED.
Bigger nudge: PENDING must remain visible so new readers are refused while old readers drain.
Technique: accept try_exclusive from either RESERVED or PENDING; only the sole-reader condition changes it to EXCLUSIVE.
Drop and mutex poison conflict
Nudge: a destructor cannot return Result.
Bigger nudge: do not panic from Drop during unwinding.
Technique: define and document a fail-closed release path: ordinary methods surface Poisoned; the destructor makes a non-panicking best effort and the owning engine is considered unusable if lock metadata cannot be safely reached.
