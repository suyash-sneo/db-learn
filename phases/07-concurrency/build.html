Build: a single-writer lock manager

Goal: Implement the policy half of SQLite's rollback-journal locking (7.2) as an in-process state machine: a lock manager that grants or refuses SHARED / RESERVED / PENDING / EXCLUSIVE according to the compatibility rules, enforcing "many readers OR one writer." Then drive your engine's reads and writes through it.
⚠️ AI off

This page gives the state model, contracts, failing tests, and todo!() stubs only — no worked lock manager. Concurrency control is correctness-critical; implement the transition logic yourself. Cite lockingv3 for the compatibility rules you encode.

⚠️ What this build is and is not

You are modeling the decision logic of locking — given the locks currently held, is a requested lock granted or refused? — as a deterministic in-process state machine. You are not building OS cross-process file locking, and the tests do not use real threads or real fcntl locks, so they validate the compatibility matrix and escalation order, not true parallel timing or cross-process exclusion. A request that "would block" in a real system is returned here as Err(WouldBlock) rather than parking a thread; a production version would wait on a condition variable instead. The acceptance criteria include a manual note about what real blocking and OS locking would require.

The state model

Five logical states, escalating; the manager tracks the locks currently held by all participants:

Held / requestedRule encoded
SHAREDGranted unless a PENDING or EXCLUSIVE lock is held. Any number may be held at once.
RESERVEDGranted only if no RESERVED, PENDING, or EXCLUSIVE is held. Coexists with existing SHARED locks. At most one.
PENDINGThe escalation step from RESERVED toward EXCLUSIVE. While held, new SHARED requests are refused; existing SHARED locks continue.
EXCLUSIVEGranted only when no other lock of any kind is held (SHARED count is 0 and no other writer). At most one; coexists with nothing.

[src: sqlite.org — lockingv3 — the compatibility rules above are this page's stated semantics]

Deliverable

A Rust 2024 module lockmgr. A LockManager holds the current lock census and hands out RAII guards; dropping a guard releases that lock. A reader takes a SHARED guard; a writer takes RESERVED, then escalates to EXCLUSIVE (via PENDING) only when no SHARED locks remain. The write path is expressed as explicit transitions so the escalation order is testable. Wire your Phase 6 document store so that a read holds SHARED for its duration and a write holds the write path around the commit.

Ordered milestones

  1. M1 — census + SHARED. Track shared_count, reserved, pending, exclusive. Implement acquire_shared and its release; refuse SHARED while PENDING/EXCLUSIVE is held.
  2. M2 — RESERVED. acquire_reserved: at most one, coexists with SHARED, refused if another writer (reserved/pending/exclusive) is present.
  3. M3 — escalation to EXCLUSIVE. From RESERVED, move to PENDING (blocking new SHARED), then to EXCLUSIVE only when shared_count == 0. If readers remain, the request is WouldBlock — not granted.
  4. M4 — guards + invariants. RAII guards that release on drop; an internal assert_census_valid that the held set is always one of the legal combinations.
  5. M5 — wire the engine. Route Phase 6 reads through a SHARED guard and writes through the write path; show a read and a write cannot both hold incompatible locks.

Acceptance criteria

  • All supplied tests pass unchanged.
  • Any number of SHARED locks coexist; a new SHARED is refused while PENDING or EXCLUSIVE is held.
  • At most one RESERVED; it coexists with existing SHARED locks; a second writer is refused.
  • EXCLUSIVE is granted only when nothing else is held; the write path refuses (returns WouldBlock) while readers remain.
  • The census is never in an illegal combination (e.g. EXCLUSIVE with a nonzero SHARED count).
  • Manual note (write it in a comment or README): describe what a real implementation would add — condition-variable waiting instead of WouldBlock, OS file locks for cross-process exclusion, and the platform caveats (advisory locks, network filesystems) from 7.2 — and state plainly that these tests do not cover real parallel timing.

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn many_shared_coexist() {
        let mut m = LockManager::new();
        let a = m.acquire_shared().unwrap();
        let b = m.acquire_shared().unwrap();
        drop((a, b)); // both held at once, then released
    }

    #[test]
    fn reserved_coexists_with_shared_but_is_unique() {
        let mut m = LockManager::new();
        let _r = m.acquire_reserved().unwrap();   // writer intent
        assert_eq!(m.acquire_reserved().unwrap_err(), LockError::AlreadyWriter);
    }

    #[test]
    fn pending_blocks_new_shared() {
        let mut m = LockManager::new();
        let mut w = m.acquire_reserved().unwrap();
        // no readers, so escalation should reach exclusive
        w.escalate_to_exclusive().unwrap();
        assert_eq!(m.acquire_shared().unwrap_err(), LockError::WouldBlock);
    }

    #[test]
    fn exclusive_waits_for_readers() {
        let mut m = LockManager::new();
        let reader = m.acquire_shared().unwrap();
        let mut w = m.acquire_reserved().unwrap(); // allowed alongside the reader
        assert_eq!(w.escalate_to_exclusive().unwrap_err(), LockError::WouldBlock);
        drop(reader);                       // reader leaves
        w.escalate_to_exclusive().unwrap(); // now exclusive is grantable
    }

    #[test]
    fn exclusive_is_alone() {
        let mut m = LockManager::new();
        let mut w = m.acquire_reserved().unwrap();
        w.escalate_to_exclusive().unwrap();
        assert_eq!(m.acquire_shared().unwrap_err(), LockError::WouldBlock);
        assert_eq!(m.acquire_reserved().unwrap_err(), LockError::AlreadyWriter);
    }

    #[test]
    fn release_restores_unlocked() {
        let mut m = LockManager::new();
        { let _a = m.acquire_shared().unwrap(); }
        { let mut w = m.acquire_reserved().unwrap(); w.escalate_to_exclusive().unwrap(); }
        // after all guards drop, a fresh writer can take the whole path again
        let mut w = m.acquire_reserved().unwrap();
        w.escalate_to_exclusive().unwrap();
    }
}

Stuck? — graduated hints

can_grant feels like a pile of if-statements
Nudge: There are only five request kinds; write one match arm per kind.
Bigger: SHARED depends only on !pending && !exclusive; the write path depends on "no other writer."
Technique: make can_grant the single source of truth and have every acquire_* call it, so the table lives in exactly one place.
Exclusive sometimes granted with readers still present
Nudge: PENDING and EXCLUSIVE are two different steps.
Technique: set pending = true first (this is what blocks new readers), then only flip to EXCLUSIVE when shared_count == 0; otherwise return WouldBlock while keeping PENDING set.
Census drifts into an illegal state
Nudge: Check the invariant after every mutation, not just in tests.
Technique: call assert_census_valid at the end of each acquire/release; legal combinations are limited (e.g. exclusive ⇒ shared_count==0 && !reserved).
How do guards release exactly once?
Nudge: Let Drop do it; don't add a manual release() the caller must remember.
Technique: each guard holds &mut LockManager and calls the matching private release in its Drop — mirrors how a real lock releases when its guard leaves scope.
