7.2 — File locking & SQLite lock states

Goal: Learn the classic locking solution by studying SQLite's rollback-journal mode precisely: its five lock states, the order a reader and a writer move through them, and the rule that makes "many readers or one writer" work without starving or corrupting anyone.
📂 Where to read

SQLite documents its rollback-journal locking model in "File Locking And Concurrency In SQLite Version 3." The lock-state names and the quoted definitions below are from that page. [src: sqlite.org — lockingv3]

Why this matters to your DB

Locking is the oldest and most direct answer to 7.1: before you touch shared data, acquire a lock that excludes the actors who would conflict with you; release it when done. The whole art is in choosing lock modes so that operations which don't conflict (two reads) can proceed together, while operations that do conflict (a write vs. anything) are serialized. SQLite's rollback-journal scheme is a compact, battle-tested example you can study end to end, and its single-writer rule is the exact model your build implements.

The five lock states

SQLite's rollback-journal mode defines five states a connection can hold on the database file. They are an escalation ladder: a writer climbs from a read lock toward exclusivity in stages, so that it can do as much preparatory work as possible before it has to shut readers out.

StateDefinition (quoted)
UNLOCKED"No locks are held on the database. The database may be neither read nor written."
SHARED"The database may be read but not written. Any number of processes can hold SHARED locks at the same time, hence there can be many simultaneous readers."
RESERVED"A RESERVED lock means that the process is planning on writing … at some point in the future but … is currently just reading. Only a single RESERVED lock may be active at one time, though multiple SHARED locks can coexist with a single RESERVED lock."
PENDING"A PENDING lock means that the process holding the lock wants to write … as soon as possible and is just waiting on all current SHARED locks to clear … No new SHARED locks are permitted … if a PENDING lock is active, though existing SHARED locks are allowed to continue."
EXCLUSIVE"An EXCLUSIVE lock is needed in order to write … Only one EXCLUSIVE lock is allowed … and no other locks of any kind are allowed to coexist with an EXCLUSIVE lock."

[src: sqlite.org — lockingv3 §"Database File Locking"]

The compatibility rules, distilled

  • Many readers: any number of SHARED locks coexist.
  • One intending writer alongside readers: a single RESERVED lock can coexist with the SHARED locks. The writer can prepare (build its rollback journal) while readers keep reading.
  • PENDING is the gate: once a writer escalates to PENDING, no new readers may start, but existing readers drain naturally. This is what stops a steady stream of readers from starving a waiting writer forever.
  • EXCLUSIVE is alone: the actual write to the database file happens only when no other lock of any kind is held.

A read transaction's lock sequence

  1. From UNLOCKED, acquire a SHARED lock.
  2. If a "hot journal" is present (a previous crash left one), recover first: escalate to PENDING then EXCLUSIVE, roll back, drop back to SHARED. (This is Phase 5's recovery, gated by locks.)
  3. Read pages while holding SHARED.
  4. Drop SHARED back to UNLOCKED when done.

A write transaction's lock sequence

  1. Acquire SHARED (handling a hot journal if needed).
  2. Acquire RESERVED to announce the intent to write while still only reading.
  3. Create the rollback journal and record original page contents (Phase 5).
  4. When ready to write the database file itself, escalate to PENDING then EXCLUSIVE — waiting for existing SHARED locks to clear.
  5. Write the modified pages.
  6. Commit (flush + delete the journal — the commit point), then drop EXCLUSIVE and PENDING.

Note how locking and durability interlock: the journal is built under RESERVED (cheap, concurrent with readers), and the database file is mutated only under EXCLUSIVE (expensive, exclusive). The lock ladder exists precisely to minimize how long anyone holds the exclusive state.

⚠️ This is OS file locking, with real-world caveats

SQLite implements these states using the host operating system's file-locking facilities, and the page is explicit that the guarantees are only as good as the platform's locking. Advisory locks can be ignored by code that doesn't ask for them; some network filesystems implement locking incorrectly or not at all, which can break the scheme across machines. Treat "file locking gives mutual exclusion" as a claim that holds on a correctly behaving local filesystem, not unconditionally. [src: sqlite.org — lockingv3 (locking relies on OS primitives; network filesystems are called out)]

What this buys, and its ceiling

Rollback-journal locking gives "many readers OR one writer." That is correct and simple, but a writer ultimately needs EXCLUSIVE, which means writers block readers and readers block writers during the write. For workloads with frequent writes, that serialization is the bottleneck — and it is exactly the limitation WAL mode (next lesson, and Phase 5's WAL) was designed to relax.

✅ Concept check

1. How many SHARED locks can coexist, and how many RESERVED?

Show answer
Any number of SHARED locks (many simultaneous readers); only one RESERVED lock at a time, and it can coexist with the SHARED locks.

2. What problem does the PENDING state solve?

Show answer
It blocks new readers from starting while letting existing readers finish, so a continuous stream of readers can't starve a waiting writer indefinitely.

3. Why does a writer build its rollback journal under RESERVED rather than EXCLUSIVE?

Show answer
Journal preparation only reads the database, so it can run while readers continue (RESERVED coexists with SHARED). EXCLUSIVE is held only for the brief actual write, minimizing the time readers are shut out.

4. Why is "file locking gives mutual exclusion" a conditional claim?

Show answer
It depends on the OS/filesystem implementing locking correctly; advisory locks can be bypassed and some network filesystems implement locking incorrectly, so the guarantee holds on a correctly behaving local filesystem, not universally.
