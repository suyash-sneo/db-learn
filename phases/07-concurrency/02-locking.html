7.2 — File locking & SQLite lock states

Goal: Learn the classic locking solution by studying SQLite's rollback-journal mode precisely: its five VFS lock states, the reader and writer paths, and the PENDING step that reduces writer starvation by stopping new readers while existing readers finish.
📂 Where to read

SQLite documents this rollback-journal model separately from WAL locking. The descriptions below are original paraphrases of that authoritative page. [src: sqlite.org — lockingv3]

Why this matters to your DB

Locking is the oldest and most direct answer to 7.1: before you touch shared data, acquire a lock that excludes the actors who would conflict with you; release it when done. The whole art is in choosing lock modes so that operations which don't conflict (two reads) can proceed together, while operations that do conflict (a write vs. anything) are serialized. SQLite's rollback-journal scheme is a compact example you can study end to end. Your build implements an educational, in-process version of its compatibility policy, not SQLite's OS-locking mechanism.

The five lock states

From the viewpoint of one SQLite process/file handle, the database file can be in five VFS lock states. The VFS tracks all five; SQLite's pager tracks four because PENDING is a temporary state while trying to reach EXCLUSIVE. [src: SQLite lockingv3 §3]

StateMeaning in rollback-journal mode
UNLOCKEDThis handle holds no database-file lock; cached content must not be trusted without revalidation.
SHAREDThe holder may read. Multiple SHARED holders can coexist, but the database file cannot be written while any remain.
RESERVEDOne holder announces an intended write. Existing and new SHARED readers may coexist with that one reservation.
PENDINGThe writer is waiting for existing SHARED holders to drain. Existing readers continue; new SHARED acquisitions are refused.
EXCLUSIVEThe holder may write the database file. No other file lock may coexist.

[src: sqlite.org — lockingv3 §"Database File Locking"]

The compatibility rules, distilled

  • Many readers: any number of SHARED locks coexist.
  • One intending writer alongside readers: a single RESERVED lock can coexist with the SHARED locks. The writer can prepare (build its rollback journal) while readers keep reading.
  • PENDING is the gate: once a writer reaches PENDING, no new readers may start, while existing readers may finish. This addresses the continuous-arrival form of writer starvation, assuming existing readers eventually release their locks. [src: SQLite lockingv3 §5.1]
  • EXCLUSIVE is alone: the actual write to the database file happens only when no other lock of any kind is held.

A read transaction's lock sequence

  1. From UNLOCKED, acquire a SHARED lock.
  2. Check for a hot rollback journal. If recovery is needed, request EXCLUSIVE without first taking RESERVED; the VFS's path may pass through PENDING. Skipping RESERVED matters because RESERVED would make another observer conclude the journal is not hot.
  3. Read pages while holding SHARED.
  4. Drop SHARED back to UNLOCKED when done.

[src: SQLite lockingv3 §4.1 — hot-journal read sequence]

A write transaction's lock sequence

  1. Acquire SHARED (handling a hot journal if needed).
  2. Acquire RESERVED to announce the intent to write while still only reading.
  3. Create the rollback journal and record original page contents (Phase 5).
  4. When page-cache spill or commit requires database-file writes, request EXCLUSIVE. The VFS obtains PENDING on the route when necessary and waits/retries according to the configured busy policy while existing SHARED locks remain.
  5. Write the modified pages.
  6. Complete the rollback-journal commit protocol, finalize the journal according to its configured journal mode, then lower the file lock. In DELETE mode, deleting the journal is the documented commit point; TRUNCATE and PERSIST use different journal-finalization operations.

[src: SQLite lockingv3 §5; SQLite Atomic Commit §3.8–3.12]

Note how locking and durability interlock: the journal is built under RESERVED (cheap, concurrent with readers), and the database file is mutated only under EXCLUSIVE (expensive, exclusive). The lock ladder exists precisely to minimize how long anyone holds the exclusive state.

⚠️ This is OS file locking, with real-world caveats

The mechanism is VFS- and platform-specific. SQLite's documented Unix path uses POSIX advisory locks; its Windows path uses Windows file-lock APIs. Advisory locking coordinates only cooperating programs, and SQLite warns that broken or absent network-filesystem locking can lead to corruption. Therefore “the lock excludes other writers” is conditional on every accessor following the protocol and the OS/filesystem implementing its locking primitive correctly. [src: SQLite lockingv3 §6; SQLite VFS documentation]

What this buys, and its ceiling

During the period in which a rollback-journal writer holds EXCLUSIVE, readers cannot hold SHARED, so readers and the database-file write exclude one another. RESERVED deliberately allows much of the transaction's preparation to overlap with readers. WAL mode changes this reader/writer relationship by appending new page images while readers retain an earlier end mark, but it still serializes writers. [src: SQLite WAL §1–2]

✅ Concept check

1. How many SHARED locks can coexist, and how many RESERVED?

Show answer
Any number of SHARED locks (many simultaneous readers); only one RESERVED lock at a time, and it can coexist with the SHARED locks.

2. What problem does the PENDING state solve?

Show answer
It blocks new readers from starting while letting existing readers finish, so a continuous stream of readers can't starve a waiting writer indefinitely, provided existing readers eventually finish. It is not a general fairness guarantee.

3. Why does a writer build its rollback journal under RESERVED rather than EXCLUSIVE?

Show answer
RESERVED lets one writer announce intent and prepare journaled changes while readers continue. EXCLUSIVE is deferred until database-file pages must be written.

4. Why is "file locking gives mutual exclusion" a conditional claim?

Show answer
It depends on the OS/filesystem implementing locking correctly; advisory locks can be bypassed and some network filesystems implement locking incorrectly, so the guarantee holds on a correctly behaving local filesystem, not universally.

🦀 Rust just-in-time

std::sync::RwLock implements an in-process many-readers/one-writer primitive, but it does not reproduce SQLite's RESERVED or PENDING states and it cannot coordinate another process. Treat the policy and the enforcement mechanism as separate design layers.
