7.6 — Source dive: SQLite locking

Goal: Connect the locking model of 7.2 to the actual SQLite code that implements it — where the lock states are defined, where they map onto OS calls, and the clever byte-range trick that encodes five logical states on a file — reading the source at a pinned version rather than trusting memory.
📂 Where to read

SQLite ships as one amalgamation but is developed as separate files. The locking logic of interest: the VFS-level OS locking in os_unix.c (and os_win.c), the lock-state constants in sqliteInt.h, and the page-cache layer that requests locks in pager.c. The documentation page lockingv3 is the prose companion to this code. Read at a specific release/check-in, not "trunk." [src: sqlite.org — lockingv3; source browsable at sqlite.org/src]

⚠️ Version & platform discipline

SQLite's locking is per-VFS and per-platform: the Unix path (os_unix.c) and the Windows path (os_win.c) are different implementations of the same logical states, and Unix has multiple locking styles (POSIX advisory, dot-file, flock, etc.) selectable at build/runtime. Any claim about a specific function name, byte offset, or syscall below is tagged ⚠️VERIFY and must be confirmed against the exact file at the exact check-in you read — do not blend versions or platforms into one description.

What the docs establish (solid)

From 7.2, treat as confirmed by lockingv3: the five states (UNLOCKED, SHARED, RESERVED, PENDING, EXCLUSIVE); many SHARED coexist; one RESERVED; PENDING blocks new readers; EXCLUSIVE is alone. The source dive is about how these are realized.

The byte-range locking trick (read this carefully in the source)

SQLite encodes its logical lock states using locks on specific byte ranges of the database file, rather than one OS lock per state. There is a reserved region of the file (historically near a fixed offset) where distinct bytes/ranges represent the PENDING byte, the RESERVED byte, and a SHARED range. Holding a read lock means a shared lock somewhere in the SHARED range; an EXCLUSIVE lock means an exclusive lock over the whole region. This lets several logical states be expressed through the OS's single advisory-locking mechanism.

  • Constants such as the names for the lock states and the special byte offsets are defined in a header. ⚠️VERIFY the exact macro names (e.g. the lock-level constants and the PENDING_BYTE/SHARED_FIRST/SHARED_SIZE region) and their values in sqliteInt.h at your check-in.
  • The function that escalates/de-escalates a connection's lock translates a requested logical state into the corresponding byte-range lock/unlock syscalls. ⚠️VERIFY the function name (e.g. the unixLock/unixUnlock family) and which syscall it uses (fcntl with POSIX advisory locks on the default Unix VFS) in os_unix.c.

Why a trick at all? OS advisory locking gives you shared/exclusive on byte ranges; SQLite needs five graded states. Mapping states onto distinct ranges is how it gets RESERVED and PENDING — "I intend to write" and "I'm waiting to write" — out of primitives that only offer shared/exclusive.

Where the pager asks for locks

The lock policy (when to escalate) lives above the OS layer, in the pager/transaction code: beginning a read takes SHARED; beginning a write takes RESERVED; spilling/committing escalates to EXCLUSIVE. ⚠️VERIFY the specific functions in pager.c that request each transition (the "begin read", "begin write", and "commit phase" paths) against your check-in. This is the same ordering you read as prose in 7.2 — now you can see the policy and the mechanism are cleanly separated, exactly as your build should separate them.

WAL mode is a different file

Everything above is rollback-journal locking. WAL mode uses the shared-memory wal-index (the -shm file) and a different concurrency scheme (7.3); its locking lives in wal.c. Don't describe WAL concurrency using rollback-journal lock states — they are separate mechanisms. ⚠️VERIFY any specific wal.c claim against your check-in. [src: sqlite.org — WAL for the model; wal.c for the code]

What to take back to your build

  • Separate policy from mechanism. SQLite keeps "when to lock" (pager) apart from "how to lock" (VFS). Your build should keep its lock-state policy separate from whatever primitive (an in-process RwLock, an OS file lock) enforces it.
  • Express graded states explicitly. The five-state ladder exists to maximize concurrency; even your simplified single-writer layer benefits from naming its states rather than using one boolean.
  • Respect the platform boundary. Real file locking is platform-specific and fallible; your in-process build sidesteps that, and you should say so rather than imply OS-grade cross-process locking you didn't build.
✅ Concept check

1. How does SQLite get five logical lock states out of shared/exclusive OS locks?

Show answer
By locking distinct byte ranges of the file (a PENDING byte, a RESERVED byte, and a SHARED range), so different logical states correspond to different range locks rather than one OS lock per state.

2. Which layer decides when to escalate locks, and which decides how?

Show answer
The pager/transaction layer decides when (begin read → SHARED, begin write → RESERVED, commit → EXCLUSIVE); the VFS (e.g. os_unix.c) decides how, translating states into syscalls.

3. Why must WAL-mode locking be described separately?

Show answer
WAL mode uses the shared-memory wal-index and a different concurrency scheme (wal.c), not the rollback-journal five-state ladder. Mixing them produces false claims.

4. Why is every code-level claim here tagged ⚠️VERIFY?

Show answer
Function names, macro names, byte offsets, and syscalls are platform- and version-specific; they must be confirmed against the exact file at the exact check-in rather than asserted from memory.
