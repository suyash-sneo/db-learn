7.6 — Source dive: SQLite locking

Goal: Connect the locking model of 7.2 to the actual SQLite code that implements it — where the lock states are defined, where they map onto OS calls, and the clever byte-range trick that encodes five logical states on a file — reading the source at a pinned version rather than trusting memory.
📂 Where to read

This lesson is pinned to SQLite 3.53.0, released April 9, 2026, source ID 4525003a53a7fc63ca75c59b22c79608659ca12f0131f52c18637f829977f20b. Use the official source archive or the Fossil file links at that check-in, not moving trunk. [src: SQLite 3.53.0 release log; official 3.53.0 source archive]

⚠️ Version & platform discipline

SQLite's locking is per-VFS and per-platform: the Unix path (os_unix.c) and the Windows path (os_win.c) are different implementations of the same logical states, and Unix has multiple locking styles. The function and constant claims below describe the POSIX unix path in the pinned source; they are not universal descriptions of every SQLite VFS.

Start in src/os.h, not sqliteInt.h

The five integer lock levels are NO_LOCK=0, SHARED_LOCK=1, RESERVED_LOCK=2, PENDING_LOCK=3, and EXCLUSIVE_LOCK=4. The byte-range constants are also in src/os.h: RESERVED_BYTE=PENDING_BYTE+1, SHARED_FIRST=PENDING_BYTE+2, and SHARED_SIZE=510. In a normal build PENDING_BYTE refers to the global sqlite3PendingByte, whose default value is 0x40000000 in src/global.c. It is a configurable testing hook, so describe that number as the pinned default, not an immutable file-format law. [src: SQLite 3.53.0 src/os.h and src/global.c]

How unixLock maps states to byte ranges

For the pinned POSIX path, unixLock raises a file's lock level and unixUnlock lowers it. The helper unixFileLock routes ordinary POSIX locking through osSetPosixAdvisoryLock; without the optional set-lock-timeout build feature, that macro invokes fcntl(..., F_SETLK, ...). SQLite also keeps per-inode in-process bookkeeping because POSIX process-associated locks have semantics that cannot be represented by a file descriptor alone. [src: SQLite 3.53.0 src/os_unix.c: osSetPosixAdvisoryLock, unixFileLock, unixLock, unixUnlock]

  • SHARED: briefly read-lock the PENDING byte, read-lock the SHARED range, then release the PENDING-byte lock. Holding PENDING exclusively therefore prevents new SHARED acquisition.
  • RESERVED: write-lock the one RESERVED byte while retaining SHARED.
  • PENDING: while a RESERVED holder requests EXCLUSIVE, write-lock the PENDING byte. If the SHARED range cannot yet be taken exclusively, the connection remains PENDING.
  • EXCLUSIVE: write-lock the complete SHARED range, which conflicts with other readers' read locks.

One subtle source fact: SQLite does not call sqlite3OsLock(..., PENDING_LOCK) directly. unixLock inserts PENDING as an intermediate state during an EXCLUSIVE request. This matches the documentation's statement that the pager does not track PENDING as a stable pager state. [src: SQLite 3.53.0 unixLock assertions and transition table; lockingv3 §3]

Where the pager asks for locks

pagerLockDb is the pager wrapper around sqlite3OsLock, and pager_wait_on_lock adds busy-handler retry behavior where the caller permits it. sqlite3PagerSharedLock obtains SHARED and handles hot-journal recovery. sqlite3PagerBegin requests RESERVED for a normal rollback-mode write transaction (or EXCLUSIVE immediately when requested). sqlite3PagerExclusiveLock requests EXCLUSIVE before rollback-mode database-file writes when needed. sqlite3PagerCommitPhaseOne performs the journal/database synchronization and write work while the transaction has the required lock. [src: SQLite 3.53.0 src/pager.c: pagerLockDb, pager_wait_on_lock, sqlite3PagerSharedLock, sqlite3PagerBegin, sqlite3PagerExclusiveLock, sqlite3PagerCommitPhaseOne]

WAL mode has a separate lock namespace

In the pinned wal.c, lock-slot macros name WAL_WRITE_LOCK, WAL_CKPT_LOCK, WAL_RECOVER_LOCK, and indexed WAL_READ_LOCK(i) slots. Helpers such as walLockShared and walLockExclusive operate on wal-index shared-memory locks. These are not the rollback-journal database-file ladder above. Keep WAL writer, checkpoint, recovery, and reader-mark coordination conceptually separate. [src: SQLite 3.53.0 src/wal.c: WAL lock macros and walLockShared/walLockExclusive; SQLite WAL]

What to take back to your build

  • Separate policy from mechanism. SQLite keeps "when to lock" (pager) apart from "how to lock" (VFS). Your build should keep its lock-state policy separate from whatever primitive (an in-process RwLock, an OS file lock) enforces it.
  • Express graded states explicitly. The five-state ladder exists to maximize concurrency; even your simplified single-writer layer benefits from naming its states rather than using one boolean.
  • Respect the platform boundary. Real file locking is platform-specific and fallible; your in-process build sidesteps that, and you should say so rather than imply OS-grade cross-process locking you didn't build.
✅ Concept check

1. How does SQLite get five logical lock states out of shared/exclusive OS locks?

Show answer
By locking distinct byte ranges of the file (a PENDING byte, a RESERVED byte, and a SHARED range), so different logical states correspond to different range locks rather than one OS lock per state.

2. Which layer decides when to escalate locks, and which decides how?

Show answer
The pager/transaction layer decides when (begin read → SHARED, begin write → RESERVED, commit → EXCLUSIVE); the VFS (e.g. os_unix.c) decides how, translating states into syscalls.

3. Why must WAL-mode locking be described separately?

Show answer
WAL mode uses the shared-memory wal-index and a different concurrency scheme (wal.c), not the rollback-journal five-state ladder. Mixing them produces false claims.

4. Where are the five lock constants in SQLite 3.53.0?

Show answer
In src/os.h, not sqliteInt.h. The default runtime value behind PENDING_BYTE is initialized in src/global.c.

5. Does the pager explicitly request PENDING_LOCK?

Show answer
No. The pager requests EXCLUSIVE; the Unix VFS can enter PENDING as an intermediate state while existing SHARED locks drain.
