7.3 — Isolation & MVCC overview

Goal: Understand isolation as a spectrum of levels (not a single on/off property), see how multi-version concurrency control (MVCC) and snapshot isolation let readers and writers run concurrently without locking each other, and place SQLite's WAL mode on that map.

Why this matters to your DB

Locking (7.2) is correct but pessimistic: a writer's EXCLUSIVE lock stalls readers. The alternative is to keep multiple versions of data so a reader can be handed a stable old version while a writer produces a new one — nobody waits. This is MVCC, and it is how WAL mode (the durability mechanism you already built in Phase 5) delivers concurrent reads and writes. Knowing where each technique sits on the isolation spectrum lets you state honestly what your engine guarantees instead of hand-waving "it's isolated."

Isolation is a level, not a boolean

The SQL standard defines isolation levels by which of the 7.1 phenomena they permit. Stronger levels forbid more anomalies and cost more concurrency.

⚠️ "Isolated" does not mean "serializable"

A common error is to treat isolation as binary and equate it with serializability. Serializable is only the strongest level: "any concurrent execution of a set of Serializable transactions is guaranteed to produce the same effect as running them one at a time in some order." [src: PostgreSQL §13.2] Most databases default to a weaker level (often Read Committed). When you describe your engine, name the exact level it provides and the anomalies it still allows — do not claim serializability you did not implement.

MVCC: keep old versions instead of overwriting in place

Under MVCC, a write does not destroy the previous value; it creates a new version and leaves the old one readable. Each transaction reads as of a snapshot — a consistent point in time — so it sees a stable view even while writers move on. The mechanics:

  • Each version is tagged with the transaction/commit that produced it.
  • A transaction's snapshot determines which versions are visible to it (those committed "before" its snapshot point).
  • Readers never block writers and writers never block readers, because they touch different versions. Write-write conflicts still need resolution (a lock on the row being written, or abort-on-conflict).
  • Obsolete versions no longer visible to any live snapshot are eventually reclaimed (garbage collection / vacuuming).

PostgreSQL is a canonical MVCC engine: it implements the isolation levels "using multiversion concurrency control (MVCC)," where a query sees "a snapshot … as of the start" of the statement or transaction depending on level. [src: PostgreSQL §13.2; §13.1 MVCC Introduction]

Snapshot isolation, concretely: SQLite WAL mode

You already built a WAL in Phase 5. Its concurrency payoff is a form of snapshot isolation. When a read begins in WAL mode, SQLite "remembers the location of the last valid commit record in the WAL … the 'end mark' … unchanged for the duration of the transaction, thus ensuring that a single read transaction only sees the database content as it existed at a single point in time." [src: sqlite.org — WAL §"Concurrency"]

Consequently, in WAL mode "readers do not block writers and a writer does not block readers," but "there can only be one writer at a time" because there is a single WAL file. [src: sqlite.org — WAL] That is the precise shape your build adopts: multiple concurrent readers on snapshots, a single writer.

✅ Concept check

1. Why is it wrong to call a database simply "isolated"?

Show answer
Isolation is a spectrum of levels defined by which anomalies they allow. "Isolated" is meaningless without naming the level; serializable is just the strongest one, and many engines default to weaker levels.

2. How does MVCC let a reader and a writer run without blocking each other?

Show answer
A write creates a new version rather than overwriting; the reader keeps reading its snapshot's (older) version while the writer produces the new one, so neither waits.

3. What exactly is a WAL-mode reader's snapshot in SQLite?

Show answer
The location of the last valid commit record in the WAL at read start (the "end mark"), held fixed for the transaction so it sees the database as of one point in time.

4. In WAL mode, how many concurrent writers are allowed, and why?

Show answer
One, because there is a single WAL file that all writes append to; readers, however, can run concurrently with the writer.
