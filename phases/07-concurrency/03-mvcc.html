7.3 — Isolation, snapshots & MVCC

Goal: Understand isolation as a spectrum of levels (not a single on/off property), distinguish an isolation guarantee from a version-storage mechanism, and place PostgreSQL-style MVCC and SQLite WAL snapshots accurately on that map.

Why this matters to your DB

Rollback-journal EXCLUSIVE locks temporarily exclude readers. Versioned designs can instead let a reader keep using an older committed view while a writer prepares a newer one. That improves one kind of concurrency, but it does not eliminate locks, waits, write conflicts, or isolation anomalies. Isolation level describes permitted histories; MVCC and a WAL snapshot are mechanisms that may help implement those histories.

Isolation is a level, not a boolean

The SQL standard defines minimum phenomena each level must prevent. An implementation may provide stronger behavior than the minimum, so the level name alone is not a complete implementation spec.

⚠️ "Isolated" does not mean "serializable"

A serializable implementation makes each committed concurrent history equivalent to some serial ordering. A stable snapshot alone does not establish that property. When you describe your engine, specify when the snapshot is chosen, how write-write and predicate conflicts are handled, and which anomalies remain. [src: PostgreSQL 18 Transaction Isolation — Serializable]

MVCC: visibility over multiple versions

In a typical tuple-version MVCC design, updates create a new row version while older versions remain available to transactions whose snapshots can still see them. The implementation needs:

  • Version metadata that identifies which transaction created or invalidated a version.
  • Visibility rules that choose one visible version for the reader's snapshot.
  • Conflict handling for concurrent writers; multiple versions do not make two writers to the same logical item automatically compatible.
  • Reclamation that removes versions only after no relevant live snapshot can need them.

PostgreSQL documents MVCC as its method for giving each statement a database snapshot while minimizing read/write lock contention. At Read Committed it chooses a new snapshot for each command; at Repeatable Read it uses the transaction's initial query snapshot. [src: PostgreSQL 18 §13.2; §13.1 MVCC Introduction]

Snapshot isolation can still allow write skew

Suppose a rule requires at least one of two doctors, A or B, to remain on call. T1 and T2 both read the same snapshot showing A and B on call. T1 turns A off; T2 turns B off. They update different rows, so a policy that checks only same-row write conflicts may commit both. The final state violates the rule even though each transaction read a stable snapshot. This is write skew: a serialization anomaly that snapshot isolation alone may permit. Serializable Snapshot Isolation adds conflict detection that can abort a dangerous transaction. [src: PostgreSQL 18 — Serializable Isolation Level]

SQLite WAL: page versions plus a fixed end mark

When a SQLite WAL read transaction begins, it fixes an end mark at the last commit visible then. For each page, the reader uses the newest applicable frame at or before that end mark, otherwise the database-file page. Later commits append beyond the reader's end mark, so that reader continues to see its earlier snapshot. [src: SQLite WAL §2.2]

SQLite therefore documents WAL-mode isolation as snapshot isolation: readers may continue while one writer appends, but writers are serialized. A read transaction cannot necessarily upgrade to a writer after another connection has changed its snapshot; SQLite can return SQLITE_BUSY_SNAPSHOT. [src: SQLite Isolation — WAL mode and SQLITE_BUSY_SNAPSHOT]

⚠️ SQLite WAL is not PostgreSQL-style tuple MVCC

Both preserve older committed views, but the representation and lifetime differ. PostgreSQL keeps row versions with visibility metadata and vacuums obsolete tuples. SQLite WAL keeps newer page images in an append-only log until checkpoint/reset and uses the reader's end mark plus the wal-index. Call SQLite WAL a snapshot mechanism or page-version design; do not infer a general tuple-MVCC subsystem that SQLite does not claim.

✅ Concept check

1. Why is it wrong to call a database simply "isolated"?

Show answer
Isolation is a spectrum of levels defined by which anomalies they allow. "Isolated" is meaningless without naming the level; serializable is just the strongest one, and many engines default to weaker levels.

2. How does MVCC let a reader and a writer run without blocking each other?

Show answer
It retains older committed versions and applies visibility rules so a reader can select its snapshot's version while a writer creates a newer one. Writers still need conflict handling, and readers may still wait for unrelated resources.

3. What exactly is a WAL-mode reader's snapshot in SQLite?

Show answer
The location of the last valid commit record in the WAL at read start (the "end mark"), held fixed for the transaction so it sees the database as of one point in time.

4. In WAL mode, how many concurrent writers are allowed, and why?

Show answer
One writer at a time. SQLite serializes writes to the WAL; readers can normally retain earlier snapshots concurrently.

5. Why does a stable snapshot not imply serializability?

Show answer
Transactions can read the same snapshot and make disjoint writes whose combination violates an invariant, as in the two-doctor write-skew example. Serializable execution requires additional conflict handling.
