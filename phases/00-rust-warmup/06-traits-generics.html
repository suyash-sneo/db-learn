0.6 — Traits & generics (just enough)

Goal: Understand trait (shared behavior, like an interface) and generics (code that works over many types), at the depth the engine needs — no more. These two features power Read/Write, serde, and the standard collections you're about to use.

Why this matters to your DB

You'll write functions that work with "anything you can read bytes from" — a file, an in-memory buffer, a network stream. That "anything readable" is the Read trait. You'll derive Serialize for documents — that's a trait. Knowing how traits and generics work means these stop being magic and become tools you can wield.

Traits — shared behavior

A trait is a set of method signatures a type can promise to implement. If Dart/Java is your background: a trait is close to an interface (with optional default methods).

rust 
trait Summary {
    fn summarize(&self) -> String;          // required
    fn preview(&self) -> String {            // default impl
        format!("({}…)", self.summarize())
    }
}

struct Doc { title: String }

impl Summary for Doc {
    fn summarize(&self) -> String { self.title.clone() }
    // preview() comes free from the default
}

Note the shape: impl TraitName for TypeName. You can implement a trait for a type even if you didn't define the type (with some rules), which is how the standard library can make your types printable, comparable, hashable.

Derivable traits — the ones you'll use daily

Many common traits can be auto-implemented with #[derive(...)], saving you the boilerplate:

TraitGives you
Debug{:?} formatting for printing/inspecting
Clonea .clone() method (deep copy)
Copyimplicit bitwise copy (only for simple types)
PartialEq, Eq== comparison
PartialOrd, Ord< > sort() — vital for B+tree keys
Hashuse as a HashMap key
Serialize, Deserializeserde conversion (Lesson 0.9)
rust 
#[derive(Debug, Clone, PartialEq, Eq, PartialOrd, Ord)]
struct Key(i64);   // now Key is printable, cloneable, comparable, sortable

Generics — write once, work for many types

A generic function/struct has a type parameter (usually T) filled in at each call site. The compiler generates a specialized version per concrete type ("monomorphization") — so generics are zero runtime cost.

rust 
// works for any T, returns the first element if present
fn first<T: Clone>(items: &[T]) -> Option<T> {
    items.first().cloned()
}

let a = first(&[1, 2, 3]);          // T = i32  → Some(1)
let b = first(&["x", "y"]);          // T = &str → Some("x")

Trait bounds: "T, but only if it can do X"

The <T: Clone> above is a trait bound: it constrains T to types that implement Clone, which is what lets the body call .cloned(). Bounds are how generics stay safe — you can only use capabilities you've required. To sort keys, you'd require T: Ord:

rust 
fn sorted<T: Ord + Clone>(items: &[T]) -> Vec<T> {
    let mut v = items.to_vec();
    v.sort();   // allowed because T: Ord
    v
}

Traits as parameters & the two ways to be generic

You'll see two syntaxes for "accept anything implementing a trait":

rust 
// 1. static dispatch (impl Trait / generic) — fast, resolved at compile time
fn print_summary(item: &impl Summary) {
    println!("{}", item.summarize());
}

// 2. dynamic dispatch (dyn Trait) — one function, type chosen at runtime via a vtable
fn print_dyn(item: &dyn Summary) {
    println!("{}", item.summarize());
}

For now: prefer the first (generics / impl Trait) — it's the common case and has no runtime cost. Reach for dyn Trait only when you need a collection of mixed concrete types behind one interface (e.g. a Vec<Box<dyn Trait>>).

🦀 Rust just-in-time
  • where clauses move long bounds out of the signature for readability: fn f<T>(x: T) -> T where T: Ord + Clone { … }.
  • You'll constantly use trait methods from the standard library: .iter(), .map(), .collect() come from the Iterator trait.
  • For the B+tree: your key type needs Ord so the tree can compare and order keys. Keep that in mind when you design it.

✅ Concept check

1. In one sentence, what is a trait?

Show answer
A named set of behaviors (method signatures) that types can implement — Rust's version of an interface, with optional default methods.

2. Which derived trait must your B+tree key type have, and why?

Show answer
Ord (and its prerequisites Eq/PartialOrd/PartialEq) — a search tree must compare and order keys to decide left/right.

3. What does the bound in fn f<T: Clone>(…) let the body do?

Show answer
Call Clone methods (like .clone()) on values of type T. Without the bound, the compiler can't assume T is cloneable.

4. Static dispatch (generics) vs dynamic dispatch (dyn): which has zero runtime cost, and which should you default to?

Show answer
Generics/impl Trait are monomorphized at compile time — zero runtime cost — and are the default. dyn uses a runtime vtable; use it only when you need mixed types behind one pointer.
