0.1 — Why I'm building this & how to use this curriculum

Goal: Leave this page knowing exactly what you're building, why "from scratch in Rust" is the right call, and the one discipline that makes this whole thing work — AI as tutor, never as author.

The thing you're building

A schemaless, MongoDB-like, single-file, embedded database, written in Rust, exposing a C ABI so any language can call it — with Flutter/Dart as the first consumer. "Embedded" means it runs inside your app, like SQLite, with no server process. "Single-file" means the entire database — documents, indexes, everything — lives in one file on disk. "Schemaless" means you store JSON-ish documents without declaring tables and columns up front.

That is genuinely ambitious, and you are deliberately not taking the shortcut of wrapping SQLite. You're building the storage engine yourself: the part that decides how bytes are laid out in the file, how they're indexed, and how they survive a crash. SQLite is your teacher, not your foundation.

📌 Decisions already made (the lessons teach toward these)
  • From-scratch storage engine in Rust — not a layer over SQLite.
  • B+tree as the core index — fits the single-file goal. (LSM trees are a valid alternative you'll learn about and could revisit in a v2.)
  • Three layers: (1) a pure, idiomatic Rust core engine; (2) a thin C ABI shim; (3) language bindings, Dart first.
  • Documents cross the language boundary as serialized byte buffers (BSON or MessagePack), so the C surface stays tiny.

Why Rust, why from scratch

A storage engine spends its life doing two dangerous things: manipulating raw bytes, and managing memory and concurrency. In C, both are landmines (buffer overruns, use-after-free, data races). In a garbage-collected language, you can't control memory layout tightly enough and you pay latency you don't want in an embedded engine. Rust is the sweet spot: it gives you C-level control over bytes and layout, and a compiler that statically prevents the classic memory bugs. It also produces a clean C ABI, which is exactly what you need for the FFI layer.

From scratch because the goal isn't "ship a database fast" — it's understand databases deeply enough to have built one. Wrapping SQLite would teach you its API, not its internals. Writing the pager, the B+tree, the freelist, and the WAL yourself is the only way the knowledge becomes yours.

The one rule: AI as tutor, never author

Be honest about the trap you're climbing out of: over-relying on AI to write code wrecked your focus and hollowed out your understanding. This project is the cure, and it only works if you enforce a single rule.

📐 The rule

You write every line of the database yourself. AI explains concepts, reviews your code, and answers "why does this compile error happen" — it never hands you a working pager, B+tree, freelist, WAL, query evaluator, or C-ABI function.

This curriculum is built to enforce that. Lesson pages teach concepts in full. But every Build page gives you only: a spec, ordered milestones, acceptance criteria, a failing test suite, and todo!() stubs to implement against. There is never a reference implementation of a database component anywhere in here. If you could paste a page's contents and get a working engine, the page would have failed its job.

When you're stuck on a Build, you have two legitimate moves: (1) use the graduated Stuck? hint ladder on that page, which nudges → nudges harder → names the technique, but never writes the solution; (2) ask AI to explain the concept or debug your code — not to produce the component. The difference between "explain why my split logic loses a key" and "write the split function" is the entire difference between this project working and not.

How the curriculum is structured

  • Phases are an ordering, not a schedule. No day/week targets. You move when you understand, not when a calendar says so.
  • Lessons are self-contained, ~10–20 minutes. Each one teaches its concept from the page alone — you shouldn't need the book or a video open to follow it. The Go deeper / verify box at the end tells you where the same material lives if you want more depth.
  • Builds are AI-off. When you hit a build.html, that's the deliverable for the phase. Close the assistant and implement.
  • Two reference tracks run throughout. The lower / storage track uses SQLite (its file format, WAL, locking, and public-domain source). The upper / document track uses Isar, a real Rust embedded NoSQL DB for Flutter, for the document model and the Rust↔Dart boundary.

How to read the trust markers

A database is built out of exact facts: byte offsets, header sizes, default page sizes, algorithm steps. AI can state any of these wrongly with full confidence. So this curriculum makes its claims checkable:

  • A precise claim carries an inline pointer like [src: SQLite fileformat2.html §1.3] — go there to confirm it.
  • Anything reconstructed from memory wears a ⚠️VERIFY tag. Treat it as unconfirmed until you've checked the primary source. The Verification page collects every one of these into a single checklist for your review pass.
✅ Concept check

1. Why not just build this on top of SQLite and save months?

Show answer
Because the goal is to understand database internals deeply enough to have built one. Wrapping SQLite teaches its API, not how a storage engine actually works. The pager, B+tree, freelist, and WAL are the learning.

2. You're stuck on the B+tree split during a Build. Which of these is allowed: (a) ask AI to write the split function, (b) ask AI why your split drops a key, (c) read the Stuck? hints?

Show answer
(b) and (c). Asking AI to debug your code or explain a concept is tutoring. Asking it to write the component is authoring — the one thing the rule forbids.

3. A lesson says "the page size defaults to 4096 bytes" with a ⚠️VERIFY tag. What should you do before hard-coding 4096?

Show answer
Confirm it against the primary source (the SQLite file-format doc) — and note that your engine can pick its own default anyway. The tag means "don't trust this number on my say-so."

4. Why does the C ABI keep documents as opaque byte buffers instead of exposing a rich struct API?

Show answer
To keep the C surface tiny and stable. A schemaless document has no fixed shape, so serializing it to bytes (BSON/MessagePack) lets one buffer-in/buffer-out function carry any document across the boundary.
