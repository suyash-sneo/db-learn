0.9 — Serialization with serde (JSON to start)

Goal: Turn Rust values into bytes and back with serde. Start with JSON because it's human-readable while you learn; later (Phase 6) you'll swap the format to a compact binary one (BSON/MessagePack). Many data types can stay the same, but format differences still require compatibility tests and sometimes custom serialization.

Why this matters to your DB

Two reasons. (1) In the warm-up build you'll persist key-value pairs to a file — serde is the easy way. (2) Conceptually, this is what your engine does: a document is a Rust value, and storing it means serializing it to bytes that go into a page; reading it means deserializing bytes back. serde is also exactly how documents will cross the FFI boundary as byte buffers. Learning the pattern now means Phase 6 is "swap the format," not "learn serialization."

What serde is

serde ("serialize/deserialize") is Rust's standard serialization framework. It splits the job in two: a data model (your types implement Serialize / Deserialize) and a format (a separate crate like serde_json, rmp-serde for MessagePack, or a BSON crate). The same annotated type can be written to many supported formats. The abstraction removes most format-specific code from your types; it does not guarantee that every format represents every Rust value identically. [src: Serde framework]

Setup

Cargo.toml 
[dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"

The "derive" feature is what enables the #[derive(Serialize, Deserialize)] macros — without it, you'd implement the traits by hand.

Deriving on your types

rust 
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug, Clone)]
struct User {
    id: u64,
    name: String,
    email: String,
    active: bool,
}

That's all it takes — the derive generates the conversion code. It works on nested types too: a struct containing a Vec<User> or an Option<String> just works, because those standard types already implement the traits.

Serialize → bytes, deserialize → value

rust 
let u = User { id: 1, name: "Ada".into(), email: "ada@x.io".into(), active: true };

// to a JSON String
let json: String = serde_json::to_string(&u)?;
// → {"id":1,"name":"Ada","email":"ada@x.io","active":true}

// to JSON bytes (Vec<u8>) — this is the shape the engine cares about
let bytes: Vec<u8> = serde_json::to_vec(&u)?;

// back from bytes to a User
let back: User = serde_json::from_slice(&bytes)?;
assert_eq!(back.name, "Ada");

Notice the round-trip: to_vec gives you a Vec<u8> you could write into a page; from_slice takes an &[u8] you sliced out of a page. That's the entire document storage idea in miniature.

Schemaless data: serde_json::Value

Your database is schemaless — documents don't have a fixed Rust struct. serde models "arbitrary JSON" with the Value enum (an object, array, string, number, bool, or null). This is the closest taste of the document model you'll build:

rust 
use serde_json::{json, Value};

let doc: Value = json!({
    "_id": 1,
    "name": "Ada",
    "tags": ["math", "engines"],
    "meta": { "active": true }
});

// dynamic field access, like a schemaless DB query would do:
if let Some(name) = doc["name"].as_str() {
    println!("name = {}", name);
}
let blob = serde_json::to_vec(&doc)?;   // store these bytes as a document

In Phase 6 you'll decide whether to keep documents as JSON Value or a binary model, but the access pattern — "given bytes, get field X" — is what an index and a query evaluator are built on.

🦀 Rust just-in-time
  • Field control with attributes: #[serde(rename = "_id")] changes the encoded key, #[serde(default)] fills missing fields, #[serde(skip)] omits one.
  • "Ada".into() and "Ada".to_string() both make an owned String from a &str; .into() uses the From trait you met in Lesson 0.5.
  • Same serde traits, different formats later: MessagePack and BSON both have serde-compatible Rust crates. Phase 6 will choose and pin the concrete crate after comparing their data models.
  • Read more: the Serde guide; Rust Book Ch.10 (the traits behind the derive).
✅ Concept check

1. serde splits serialization into two parts. What are they, and why does that matter to your engine?

Show answer
A data model (your types' Serialize/ Deserialize impls) and a format (serde_json, MessagePack, BSON). Decoupling means Phase 6 can change the on-disk/wire format without rewriting your data types.

2. Which serde_json functions produce/consume Vec<u8> / &[u8] rather than String?

Show answer
to_vec (value → bytes) and from_slice (bytes → value). These match the page read/write shape from Lesson 0.8.

3. Your DB is schemaless. What serde type lets you hold an arbitrary document with no fixed struct?

Show answer
serde_json::Value — an enum covering object/array/string/number/bool/null.

4. Why start with JSON if the engine will eventually use a binary format?

Show answer
JSON is human-readable, so you can inspect what you stored while learning. Serde keeps most model code independent of the format, but the later switch still needs round-trip, range, and compatibility tests.
