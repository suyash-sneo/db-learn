0.3 — Ownership, borrowing, moves & references

Goal: Understand the single idea that makes Rust different from every language you know — ownership — well enough that the borrow checker becomes a helper, not an enemy. This is the lesson to actually slow down on.

Why this matters to your DB

A storage engine constantly passes around big buffers of bytes (pages) and shares them between the cache, the B+tree, and the writer. In C that sharing is where use-after-free and double-free bugs live. Rust's ownership rules prevent those at compile time — but only once you think in ownership. Every "cannot borrow ... as mutable" error you'll hit while building the pager comes back to this page.

The problem ownership solves

Every value in memory must eventually be freed — exactly once. Free too early → use-after-free. Free twice → double-free. Never free → leak. Other languages solve this with a garbage collector (runtime cost, unpredictable pauses) or by trusting the programmer (C — and the bugs that follow).

Rust solves it at compile time with three rules:

🦀 The ownership rules
  1. Each value has exactly one owner (a variable).
  2. When the owner goes out of scope, the value is dropped (freed) — automatically.
  3. There can be only one owner at a time. Assigning or passing a value moves ownership.

Move: the surprising default

For types that own heap memory (like String or Vec), assignment moves ownership — the original variable becomes unusable:

rust 
let a = String::from("page data");
let b = a;            // ownership MOVES from a to b
// println!("{}", a);  // ❌ compile error: value borrowed after move
println!("{}", b);     // ✅ b owns it now

This isn't Rust being difficult — it's the rule "one owner" being enforced. If both a and b were usable, both would try to free the same memory at scope end (double-free). By making the move explicit-ish, Rust guarantees exactly one free.

The same thing happens when you pass a value to a function — the function takes ownership:

rust 
fn consume(s: String) {           // s is owned here
    println!("got {}", s);
}                                  // s dropped (freed) here

let doc = String::from("hi");
consume(doc);                       // doc moved IN
// doc is gone now — you gave it away

Copy types are the exception

Primitive numeric types, bool, and char implement Copy. Assignment duplicates their value instead of invalidating the source. Copy is a trait decision, not a promise that a value happens to live on the stack; types with destructors cannot implement it. [src: Copy trait]

rust 
let x = 5;
let y = x;          // x is COPIED, not moved
println!("{} {}", x, y);  // ✅ both fine

Borrowing: using without owning

Moving ownership every time you want to use a value would be miserable. So Rust lets you borrow — take a reference (&) without taking ownership. The owner keeps the value; the borrow is temporary access.

rust 
fn len_of(s: &String) -> usize {   // borrows, doesn't own
    s.len()
}                                       // borrow ends; nothing freed

let doc = String::from("hello");
let n = len_of(&doc);              // lend a reference
println!("{} is {} bytes", doc, n); // ✅ doc still owned & usable

Shared vs mutable borrows — the core rule

There are two kinds of references, and one rule governs them:

  • &T — a shared (immutable) reference. You can have many at once.
  • &mut T — a mutable reference. You can have exactly one, and no shared ones at the same time.
🦀 The borrowing rule (memorize this)

At any given time you may have either any number of &T (readers) or exactly one &mut T (writer) — never both.

This is "no shared mutation." It's exactly the discipline that prevents data races, and it's why Rust can be safely concurrent. It also catches single-threaded bugs like mutating a collection while iterating it. [src: Rust Book: references and borrowing]

Lifetimes (a gentle first look)

A reference must never outlive the data it points to — otherwise it'd be a dangling pointer. The compiler tracks how long each reference is valid; that span is a lifetime. Most of the time the compiler infers lifetimes and you write nothing. Occasionally (e.g. a struct that holds a reference) you'll annotate one as &'a T. You don't need to master lifetimes now — just know why they exist: to guarantee no reference ever points at freed memory.

rust 
// This won't compile: r would point at x's memory after x is dropped.
let r;
{
    let x = 5;
    r = &x;            // borrow x
}                      // x dropped here
// println!("{}", r); // ❌ `x` does not live long enough
✅ Concept check

1. Why does let b = a; make a unusable for a String but not for an i32?

Show answer
String owns heap memory, so it moves (to keep one owner / one free). i32 is Copy — copying its few stack bytes is harmless, so both variables are valid.

2. Can you have a &mut v and a &v alive at the same time?

Show answer
No. The rule is: many shared borrows OR one mutable borrow, never both. A reader could see data change underneath it otherwise.

3. You pass page_buf (a Vec<u8>) to fn write(buf: Vec<u8>) and then try to use page_buf afterward. What happens, and how do you fix it?

Show answer
Compile error — you moved page_buf into the function. Fix by borrowing instead: make the parameter buf: &[u8] (or &Vec<u8>) and call write(&page_buf).

4. In plain words, what is a lifetime?

Show answer
The span during which a reference is guaranteed to point at still-valid data. The compiler uses it to reject references that would outlive what they point to.
🧭 When the borrow checker yells at you

Read the error fully — Rust's are unusually good and often suggest the fix. The three reflexes that resolve most cases: (1) borrow (&) instead of moving; (2) if you need to keep a copy, call .clone() (fine while learning — optimize later); (3) shrink the scope of a &mut so it ends before the next borrow starts.
