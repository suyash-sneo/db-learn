0.8 — Files & I/O: std::fs, Read/Write/Seek

Goal: Read and write a file at arbitrary byte offsets — open it, seek to a position, read or write N bytes, and flush. This is the literal mechanism the pager uses to turn a file into a numbered array of pages.

Why this matters to your DB

The entire database is one file. "Read page 5" means: seek to offset 5 × page_size, read page_size bytes. "Write page 5" means: seek to that offset, write the bytes, and (for durability) flush. Master Seek + read_exact + write_all + flush and you have the pager's I/O core.

Opening files

File::open opens read-only; File::create creates/truncates for writing. For a database you want read and write on an existing-or-new file, which means OpenOptions:

rust 
use std::fs::OpenOptions;

let mut file = OpenOptions::new()
    .read(true)
    .write(true)
    .create(true)     // create if missing, keep if present
    .open("data.db")?; // Result<File, io::Error> — note the ?

read, write, create are the options you'll use for the DB file. (create_new(true) fails if the file already exists; truncate(true) zeroes it — you do not want truncate on a real database.)

The three I/O traits

File I/O in Rust is built on traits, so the same code works on files, in-memory buffers, sockets, etc. The three that matter here:

TraitKey methodsWhat it does
Readread, read_exactpull bytes from the source into a buffer
Writewrite, write_all, flushpush bytes to the sink; flush user-space buffers
Seekseekmove the read/write cursor to a byte offset

You must use std::io::{Read, Write, Seek, SeekFrom}; to call these methods — they're only in scope when the trait is imported (a common beginner stumble).

Seeking to a page and reading it

rust 
use std::io::{Read, Write, Seek, SeekFrom};

const PAGE_SIZE: u64 = 4096;

fn read_page(file: &mut std::fs::File, page_no: u64) -> std::io::Result<Vec<u8>> {
    let offset = page_no * PAGE_SIZE;
    file.seek(SeekFrom::Start(offset))?;   // move cursor to the page
    let mut buf = vec![0u8; PAGE_SIZE as usize];
    file.read_exact(&mut buf)?;             // fill the whole buffer or error
    Ok(buf)
}
🦀 read vs read_exact (this distinction bites everyone)

read(&mut buf) may read fewer bytes than the buffer holds and returns how many — you'd have to loop. read_exact(&mut buf) reads exactly buf.len() bytes or returns an error. For fixed-size pages, read_exact is almost always what you want. Same idea for writing: prefer write_all over write.

Writing a page

rust 
fn write_page(file: &mut std::fs::File, page_no: u64, data: &[u8]) -> std::io::Result<()> {
    file.seek(SeekFrom::Start(page_no * PAGE_SIZE))?;
    file.write_all(data)?;   // write every byte
    Ok(())               // note: NOT durable yet — see below
}

SeekFrom has three modes: Start(n) (from the beginning), End(n) (from the end, e.g. to append), Current(n) (relative). For paged access you'll almost always use Start.

flush vs sync — the durability preview

This is your first real glimpse of the hardest problem in the engine, so read carefully:

  • flush() (from Write) pushes Rust's user-space buffer to the operating system. (A raw File isn't buffered, so this is often a no-op; a BufWriter is.)
  • file.sync_all() / sync_data() asks the operating system to flush its caches all the way to the physical disk. This is the fsync system call.
⚠️ "Written" ≠ "durable"

After write_all returns, your data may still live only in the OS's page cache in RAM. A power loss there loses it, even though the call succeeded. Only after sync_all() returns is the data guaranteed on stable storage. This gap — fast writes vs. crash-safe writes — is the entire reason WAL exists (Phase 5). For now, just know the two calls are different and which one means "really saved."

rust 
file.write_all(data)?;
file.sync_all()?;   // fsync — now it's actually on disk (slow, but durable)

Convenience helpers (for the warm-up build, not the engine)

rust 
use std::fs;
fs::write("notes.txt", b"hello")?;          // whole file at once
let bytes: Vec<u8> = fs::read("notes.txt")?;  // read whole file
let text:  String  = fs::read_to_string("notes.txt")?;

(b"hello" is a byte string literal — a &[u8] instead of a &str.) These whole-file helpers are great for the KV warm-up; the real engine uses the seek/read/write path above because it touches pages, not whole files.

✅ Concept check

1. What byte offset is page number 5 at, with a 4096-byte page size?

Show answer
5 × 4096 = 20480. In code: SeekFrom::Start(5 * PAGE_SIZE).

2. Why read_exact instead of read for a page?

Show answer
read may return fewer bytes than requested; you'd have to loop. read_exact fills the entire fixed-size buffer or errors — exactly the semantics for a known-size page.

3. write_all returned Ok(()). Is the data safe across a power cut? What makes it safe?

Show answer
Not necessarily — it may sit in the OS cache. file.sync_all() (fsync) forces it to physical storage. This gap is why WAL exists.

4. You call file.seek(...) and the compiler says no method named seek. Likely cause?

Show answer
You didn't bring the trait into scope. Add use std::io::Seek; (and friends). Trait methods are only callable when the trait is imported.
