Build: a file-backed key-value store CLI

Goal: Prove you can write real Rust unassisted, and reset the AI-dependence habit. You'll build a persistent string→string key-value store with a tiny CLI. This is your Rust litmus test before the engine begins.
⚠️ The rule for every Build page

Close the AI assistant now. You may use the Rust Book, the std docs, and the Stuck? ladder on this page. You may not ask an AI to write any of this. Debugging your own compiler errors by reading them is the skill you're building.

Deliverable spec

A crate kvstore exposing a Store type that persists a map of String → String to a single file, plus a CLI binary that drives it. Concretely:

  • Store::open(path) — load an existing store, or start with an empty map when the path does not exist. The file may be created on the first successful mutation.
  • store.set(key, value) — insert or overwrite, and persist.
  • store.get(key) — return Option<String>.
  • store.delete(key) — remove a key (return whether it existed), and persist.
  • store.keys() — list all keys.
  • A CLI: kvstore set name Ada, kvstore get name, kvstore del name, kvstore list — each opens the file, does one op, exits.
  • Data survives process restarts (it's on disk, re-read on open).
Allowed tools

Use std::collections::HashMap for the in-memory map, serde + serde_json for persistence (serialize the whole map to the file on each change), and std::fs for the file. This is deliberately the “load and rewrite the whole file” approach. It proves ordinary persistence across clean process restarts, but it is not atomic or crash-safe: interruption during a rewrite may leave partial JSON. The real engine replaces it with paged storage and a commit protocol.

Ordered milestones

  1. M1 — In-memory only. Store wraps a HashMap; set/get/delete/keys work, no file yet. Make the in-memory tests pass.
  2. M2 — Persist on change. open reads the file (or starts empty if missing); every mutation serializes the map back to the file. Make the persistence tests pass.
  3. M3 — CLI. Parse std::env::args(), dispatch to the right method, print results. Manual check.
  4. M4 — Robustness. Missing file is fine (empty store); a get on a missing key returns None, not an error; corrupt file returns a clean Err, not a panic.

Acceptance criteria

  • cargo test is fully green.
  • Running kvstore set a 1 then, in a new process, kvstore get a prints 1 — proving on-disk persistence.
  • No .unwrap() / .expect() in src/lib.rs outside of tests; fallible operations return Result.
  • Opening a non-existent path creates an empty, usable store without error.

        let path = tmp("overwrite");
        let mut s = Store::open(&path).unwrap();

    }

        let path = tmp("delete");

        s.set("k", "v").unwrap();
        assert_eq!(s.delete("k").unwrap(), true);
        assert_eq!(s.delete("k").unwrap(), false);
        assert_eq!(s.get("k"), None);
    }

    #[test]
    fn persists_across_reopen() {     // the important one
        let path = tmp("persist");
        {
            let mut s = Store::open(&path).unwrap();
            s.set("a", "1").unwrap();
            s.set("b", "2").unwrap();
        } // s dropped — simulate process exit

        assert_eq!(s2.get("a"), Some("1".to_string()));
        assert_eq!(s2.get("b"), Some("2".to_string()));

        assert!(Store::open(&path).is_err());

Stuck? — graduated hints

Use these in order. Each reveals a little more; none writes it for you.

Stuck on M1 (get returns the wrong type)
Nudge: The map owns its String values. You can't return a reference that outlives the borrow, and the test wants an owned Option<String>.
Bigger: HashMap::get returns Option<&String>. You need to turn the inner &String into a String.
Technique: map over the option and clone — i.e. reach for .cloned() (or .map(|v| v.clone())). Look it up in the Option docs.
Stuck on M2 (reading the file on open)
Nudge: "File might not exist yet" is not an error for you — it just means empty.
Bigger: Check existence (e.g. path.exists()) before reading, or match on the read result and treat a not-found error as "empty map."
Technique: read bytes with std::fs::read, then serde_json::from_slice into a HashMap<String,String>. The supplied From<serde_json::Error> conversion lets ? return StoreError::Json.
Stuck on M2 (persisting)
Nudge: Simplest correct approach: after every mutation, write the entire map to the file.
Bigger: Write a private persist(&self) -> Result<()> and call it at the end of set and (when something changed) delete.
Technique: serde_json::to_vec(&self.map)? then std::fs::write(&self.path, bytes)?. (Yes, rewriting the whole file every time is wasteful — feel that pain; it's the motivation for paging.)
Stuck on the CLI (M3)
Nudge: std::env::args() yields the program name first, then the arguments.
Bigger: Collect into a Vec<String>, then match on args[1].as_str() for "set"|"get"|"del"|"list".
Technique: decide a fixed db path (e.g. "kv.json"), open the store, dispatch, and println! the result. Handle wrong arg counts with a usage message.
Stuck: borrow checker fights you in set
Nudge: You probably want to insert into self.map and then call self.persist(). Both touch self.
Technique: Do the mutation first (the &mut borrow ends at the statement), then call the persist helper — they don't need to overlap. If you tried to hold a reference into the map across the persist call, drop it first.

Definition of Done

  • ☐ All six tests pass with cargo test.
  • ☐ kvstore set a 1 in one run, kvstore get a in another, prints 1.
  • ☐ No unwrap/expect in non-test library code.
  • ☐ Missing file → empty store; missing key → None; corrupt file → clean Err.
  • ☐ You can explain, out loud, why rewriting the whole file on every set is a problem a real engine must avoid. (That explanation is your on-ramp to Phase 2.)
  • ☐ You wrote every line yourself.
📚 Reflect before moving on

This store is correct but does everything the engine won't: it loads the whole dataset into RAM, has no concept of pages, and rewrites the entire file on each change. Hold onto that discomfort — Phase 2 replaces "rewrite the whole file" with "read/write one fixed-size page," and every phase after builds on that.
