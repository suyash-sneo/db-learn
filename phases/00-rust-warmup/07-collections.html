0.7 — Collections: Vec, HashMap, slices & byte buffers

Goal: Get fluent with Vec<T>, HashMap, slices, and especially byte buffers (&[u8] / Vec<u8>) — because a storage engine is, at bottom, a machine for moving bytes in and out of fixed-size buffers.

Why this matters to your DB

A page is a Vec<u8> (or fixed array) of, say, 4096 bytes. The buffer pool is a HashMap<u32, Page> mapping page number → cached page. Reading a record means slicing bytes out of a page. Writing one means copying bytes into the right offset. If you're comfortable with these four things, the pager is mostly mechanical.

Vec<T> — a growable array

rust 
let mut v: Vec<i64> = Vec::new();
v.push(10);
v.push(20);
let first = v[0];              // index (panics if out of range)
let maybe = v.get(99);          // Option<&i64> — safe, returns None
let n = v.len();

// the vec! macro builds and fills in one step:
let zeros = vec![0u8; 4096];   // 4096 bytes, all 0 — a blank page

v[i] panics on a bad index; v.get(i) returns an Option so you can handle absence. In engine code that parses untrusted bytes, prefer .get() and range checks — a malformed file shouldn't crash you.

Slices — a borrowed window into a sequence

A slice &[T] is a reference to a contiguous run of elements — a (pointer, length) pair, no ownership, no copy. Slicing a page to read a record is the single most common operation you'll do.

rust 
let page: Vec<u8> = vec![0u8; 4096];
let header: &[u8] = &page[0..16];     // first 16 bytes (no copy)
let body:   &[u8] = &page[16..];      // 16 to end
println!("{} {}", header.len(), body.len());
⚠️ Slice ranges panic if out of bounds

&page[0..5000] on a 4096-byte page panics. When offsets come from data you're parsing, validate first or use page.get(start..end) which returns an Option<&[u8].

Byte buffers: Vec<u8> and &[u8]

These are your bread and butter. u8 is an 8-bit unsigned byte. A Vec<u8> is an owned, growable buffer; &[u8] is a borrowed view. Conventions you'll lean on:

  • Take &[u8] for "read these bytes" and return / take Vec<u8> for "here are owned bytes." This is the same convention as &str vs String.
  • Copy a slice into an owned buffer with .to_vec().
  • Write bytes into an existing buffer at an offset with buf[off..off+n].copy_from_slice(src) (lengths must match).
rust 
// read a u32 page number stored little-endian at offset 0
fn read_u32_le(buf: &[u8]) -> u32 {
    u32::from_le_bytes([buf[0], buf[1], buf[2], buf[3]])
}

// write a u32 into a buffer at offset `off`
fn write_u32_le(buf: &mut [u8], off: usize, val: u32) {
    buf[off..off + 4].copy_from_slice(&val.to_le_bytes());
}

to_le_bytes/from_le_bytes convert between integers and their byte representation (little-endian here). Choosing and fixing an endianness for your file format is a real decision you'll make in Phase 2 — these are the methods you'll use.

HashMap<K, V> — key→value lookup

rust 
use std::collections::HashMap;

let mut cache: HashMap<u32, Vec<u8>> = HashMap::new();
cache.insert(1, vec![0u8; 4096]);    // page 1 → its bytes

if let Some(page) = cache.get(&1) {  // Option<&Vec<u8>>
    println!("cached, {} bytes", page.len());
}
let entry = cache.entry(2).or_insert_with(|| vec![0u8; 4096]); // get-or-create

Keys must implement Eq + Hash (recall Lesson 0.6 — derive them). The entry API ("get the slot for this key, inserting a default if absent") is exactly the pattern a buffer pool uses to load-on-miss.

Iterating

rust 
for b in &buf { /* b: &u8, read each byte */ }
for (i, b) in buf.iter().enumerate() { /* i: index, b: &u8 */ }
let total: u64 = nums.iter().map(|&x| x as u64).sum();

|x| … is a closure (an anonymous function). Iterator chains (.iter().map().filter().collect()) are idiomatic and compile down to tight loops.

🦀 Rust just-in-time
  • Fixed-size array vs Vec: [u8; 4096] is a stack array with a compile-time length; Vec<u8> is heap-allocated and growable. Pages can be either; a fixed array makes "this is exactly 4096 bytes" a type-level guarantee.
  • &str (borrowed string slice) is to String as &[u8] is to Vec<u8>. A String is just a Vec<u8> known to be valid UTF-8.
  • Convert text↔bytes: s.as_bytes() (&str → &[u8]) and String::from_utf8(vec) (bytes → Result<String>).
  • Read more: Book Ch.8 (Vec, String, HashMap), Ch.4.3 (slices).
✅ Concept check

1. Difference between Vec<u8> and &[u8]?

Show answer
Vec<u8> owns a growable heap buffer. &[u8] is a borrowed, fixed view into bytes someone else owns — no allocation, no copy.

2. Why prefer page.get(start..end) over &page[start..end] when offsets come from a file you're parsing?

Show answer
Indexing/slicing out of bounds panics. .get() returns None instead, so a corrupt or malicious file produces a handled error rather than a crash.

3. What two traits must a type implement to be a HashMap key?

Show answer
Eq and Hash (both derivable).

4. You have a u32 page number and a mutable page buffer. How do you write it at offset 8?

Show answer
Pick an endianness and buf[8..12].copy_from_slice(&n.to_le_bytes()) (or to_be_bytes). Read it back with u32::from_le_bytes.
