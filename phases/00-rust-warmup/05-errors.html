0.5 — Error handling: Result, Option, ?, custom errors

Goal: Handle failure the Rust way. Rust has no exceptions — errors are values you return. Master Result, Option, the ? operator, and your own error enum, because the engine's API is "give me a Result for everything that can fail."

Why this matters to your DB

Opening a file, reading a page, parsing a header, finding a key — every one can fail. Your engine must surface those failures cleanly (no panics in normal operation), because later the C ABI turns each Err into an integer error code crossing the FFI boundary. Get the error model right now and the whole engine is easier.

Two failure shapes: absence vs error

Rust splits failure into two enums:

  • Option<T> = Some(T) or None — for "the value might not be there" (a key not found, an empty list's first element). Absence is not an error.
  • Result<T, E> = Ok(T) or Err(E) — for "the operation might fail, and there's a reason." The reason is the E.
rust 
fn first_byte(buf: &[u8]) -> Option<u8> {
    if buf.is_empty() { None } else { Some(buf[0]) }
}

use std::fs::File;
fn open_db(path: &str) -> Result<File, std::io::Error> {
    File::open(path)   // File::open already returns Result<File, io::Error>
}

Handling a Result without panicking

The honest way is match:

rust 
match open_db("data.db") {
    Ok(file)  => println!("opened {:?}", file),
    Err(e)    => eprintln!("failed: {}", e),
}
⚠️ unwrap / expect — fine for tests, not for the engine

.unwrap() returns the inner value or panics on None/Err. .expect("msg") is the same with a custom panic message. Convenient in tests and quick experiments—but an uncaught panic reaching a normal extern "C" boundary aborts the host process (Phase 1.6). In the engine, propagate expected errors instead of unwrapping.

The ? operator — the workhorse

Manually matching every Result to early-return its error is tedious. ? does it for you: on Ok(v) it unwraps to v; on Err(e) it returns Err(e) from the current function immediately.

rust 
use std::io::Read;

fn read_all(path: &str) -> Result<String, std::io::Error> {
    let mut f = File::open(path)?;     // if Err, return it now
    let mut s = String::new();
    f.read_to_string(&mut s)?;        // same
    Ok(s)                            // explicit Ok on success
}

Read ? as "or bail." It works inside a function whose return type supports propagating that residual value—most commonly a compatible Result or Option. In these examples it performs an early return; it is not exception handling. [src: ? operator]

Your own error type

Real engines define one error enum that covers every way they can fail. Each variant describes a category; some carry data.

rust 
#[derive(Debug)]
enum DbError {
    Io(std::io::Error),       // wraps a lower-level error
    BadMagic,                 // file isn't ours
    KeyNotFound(i64),
    Corrupt(String),           // with a message
}

Making ? convert errors for you

The magic of ?: if the error it hits isn't your function's error type, it calls From to convert. Implement From<io::Error> for DbError and now ? on a file operation automatically becomes a DbError::Io:

rust 
impl From<std::io::Error> for DbError {
    fn from(e: std::io::Error) -> DbError { DbError::Io(e) }
}

fn open_checked(path: &str) -> Result<File, DbError> {
    let f = File::open(path)?;   // io::Error auto-converts to DbError::Io
    Ok(f)
}

This is why a clean error enum pays off: every layer can use ? and errors bubble up with the right type, ready to become an FFI error code at the top.

🦀 Rust just-in-time
  • Handy combinators (use sparingly while learning): .ok_or(err) turns an Option into a Result; .map(...) transforms the Ok/Some value; .unwrap_or(default) supplies a fallback.
  • type Result<T> = std::result::Result<T, DbError>; — a type alias so you can write Result<Page> everywhere. Common in real crates.
  • The std::error::Error trait plus the thiserror crate can reduce error boilerplate. Pin dependencies and follow the version's own documentation.

✅ Concept check

1. "Key not found" — Option or Result? Why?

Show answer
Either is defensible, but it's a judgment call: if absence is normal/expected, Option<V> (return None); if a caller treats a missing key as a failure with a reason, Result<V, DbError> with KeyNotFound. The key point: absence isn't automatically an error.

2. What exactly does ? do on an Err value?

Show answer
It returns that error from the current function immediately (after converting it via From if needed), so the rest of the function doesn't run.

3. Why is .unwrap() dangerous in the engine but okay in a test?

Show answer
It panics on failure. A panic in a test just fails the test; an uncaught panic reaching a normal extern "C" boundary aborts the host process. Engine code should return expected errors rather than panic.

4. What lets ? turn an io::Error into your DbError automatically?

Show answer
An impl From<io::Error> for DbError. ? calls From::from on the error before returning it.
