0.2 — Rust + Cargo setup, project layout, running tests

Goal: Install Rust, create a project, understand the file layout Cargo gives you, and run your first test — so that when a Build page says "make this failing test pass," you know exactly what to type.

Why this matters to your DB

Your whole workflow for the engine is test-driven: every Build page hands you failing tests and stubs, and you implement until cargo test is green. So before any database ideas, you need the loop — write code, run tests, read failures — to be second nature.

Installing Rust

Rust is installed via rustup, which manages the compiler (rustc), the build tool/package manager (cargo), and toolchain versions. On macOS/Linux:

terminal 
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
# then restart your shell, or:  source "$HOME/.cargo/env"

rustc --version    # e.g. rustc 1.x.y
cargo --version

The Rust project publishes this command as the supported rustup installer for macOS, Linux, and other Unix-like systems. [src: Install Rust]

What Cargo is

Coming from Flutter, the mental model is easy: Cargo is to Rust what pub/flutter + pubspec.yaml are to Dart. It creates projects, downloads dependencies, builds, tests, and runs. A package is the directory described by one Cargo.toml. A crate is a compilation unit rooted at a file such as src/lib.rs or src/main.rs; one package can contain at most one library crate and multiple binary crates. [src: Rust Book: packages and crates]

Creating your first project

terminal 
cargo new kvstore        # a binary (has main.rs)
# or for a library:
cargo new mylib --lib    # a library (has lib.rs)

cd kvstore
cargo run                # builds + runs; prints "Hello, world!"

The project layout Cargo gives you

Cargo.toml

Cargo.toml 
# package metadata
[package]
name = "kvstore"
version = "0.1.0"
edition = "2024"   # current edition used by `cargo new`

# dependencies (crates from crates.io)
[dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"

The test loop — the most important thing in this lesson

Rust has testing built in. You don't need a framework. A test is a function annotated with #[test] that panics on failure (usually via an assertion macro). You can put tests right next to the code in a #[cfg(test)] mod tests block.

src/lib.rs 
pub fn add(a: i64, b: i64) -> i64 {
    a + b
}

// Only compiled when running tests, thanks to #[cfg(test)].
#[cfg(test)]
mod tests {
    use super::*;          // bring add() into scope

    #[test]
    fn adds_two_numbers() {
        assert_eq!(add(2, 3), 5);
    }

    #[test]
    fn handles_negatives() {
        assert_eq!(add(-4, 1), -3);
    }
}
terminal 
cargo test                    # run every test
cargo test adds_two           # run tests whose name contains "adds_two"
cargo test -- --nocapture     # show println! output even on passing tests

A failing test looks like this — learn to read it, because you'll see it constantly:

---- tests::adds_two_numbers stdout ----
thread 'tests::adds_two_numbers' panicked at src/lib.rs:14:9:
assertion `left == right` failed
  left: 6
 right: 5

It tells you the test, the file:line, and (for assert_eq!) the actual (left) vs expected (right) values. This is the entire Build workflow: run, read the panic, fix, repeat.

🦀 Rust just-in-time
  • fn name(args) -> ReturnType { … } declares a function. No return keyword needed: the last expression (no semicolon) is the return value.
  • pub makes an item public (visible outside its module) — like not prefixing with _ in Dart, but explicit.
  • #[...] is an attribute — metadata the compiler reads. #[test], #[cfg(test)], and later #[derive(...)] are all attributes.
  • Names ending in ! (assert_eq!, println!) are macros, not functions. For now: treat them as built-in helpers.
  • Read more: Book Ch.1 (setup) and Book Ch.11 (testing).
✅ Concept check

1. What does #[cfg(test)] on a module mean?

Show answer
That module is only compiled when you run tests, so your test code (and any test-only helpers) doesn't bloat the real build.

2. In assert_eq!(add(2,3), 5), when it fails the output shows left and right. Which is your code's output?

Show answer
left is the first argument — your actual result. right is the expected value.

3. You want to see println! debug output from a test that's passing. What flag?

Show answer
cargo test -- --nocapture. By default Rust captures (hides) stdout from passing tests.

4. Why keep engine logic in lib.rs rather than main.rs?

Show answer
Library code can be tested by the tests/ directory and reused by multiple binaries — and crucially, later compiled into a C-ABI cdylib. A binary's main.rs can't be linked into by other code.
